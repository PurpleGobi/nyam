# 기록물 전체 로직 — 스키마 · 프라이버시 · 노출 · UI

---

## 1. 핵심 데이터 구조

### 1-1. 테이블 관계

```
restaurants (1) ──┐
                  ├── (N) lists (1) ── (N) records
wines (1) ────────┘        │
                           │  UNIQUE(user_id, target_id, target_type)
                           │
                    users (1) ── (N) lists
```

- **restaurants / wines**: 대상 엔티티. 모든 사용자가 공유하는 "장소/와인 카드".
- **lists**: 사용자 × 대상의 **관계** 테이블. 한 유저가 같은 대상에 대해 하나의 관계만 가짐.
  - `status`: visited(방문) / wishlist(찜) / cellar(보유 와인) / tasted(시음)
  - 찜(wishlist/cellar)은 기록 없이도 존재 가능 → "기록 없는 식당/와인도 lists에는 존재할 수 있다"
- **records**: 개별 방문/시음 기록. 하나의 list에 여러 기록 가능 (재방문, 재시음).

### 1-2. 기록 생성 흐름

```
1. 사용자가 식당/와인을 특정 (검색, 카메라, 직접 등록)
2. restaurants/wines 테이블에 해당 대상이 없으면 → 새 행 INSERT
3. lists에 (user_id, target_id, target_type) 조합이 없으면 → 새 행 INSERT
4. records에 새 기록 행 INSERT (list_id 참조)
5. 사진이 있으면 record_photos에 INSERT
6. 버블 공유 설정에 따라 bubble_shares에 INSERT
```

### 1-3. 대상(식당/와인)의 생존 조건

- 스키마상 restaurants/wines는 독립 엔티티이지만, **사용자에게 노출되려면 반드시 기록(records)이 필요**.
- discover 페이지가 삭제된 현재, 기록 없는 식당/와인을 독립적으로 탐색할 UI 경로가 없음.
- 식당/와인 검색(`search()`)은 기록 플로우 안에서만 호출됨 → 검색 → 기록 생성이 한 흐름.
- **결론: 레코드 매칭 없는 식당/와인은 사용자 관점에서 존재하지 않는다.**
  - 예외: wishlist(찜)로만 추가한 경우 lists에는 있지만 records는 0개. 이 경우 홈 wishlist 탭에서 노출.

## 2. 프라이버시 — 기록물 노출 규칙

### 2-1. 기본 원칙

```
기본: 나의 기록 = 나만 볼 수 있음
공개: is_public = true인 사용자의 기록 → 모든 인증 사용자에게 공개
오버라이드: 비공개 사용자라도 아래 경로로 특정 관계에 공개됨
```

### 2-2. 노출 계층 (RLS 5개 정책)

| 우선순위 | 정책 | 조건 | 설명 |
|---------|------|------|------|
| 1 | `records_own` | `user_id = auth.uid()` | 내 기록은 항상 보임 |
| 2 | `records_public` | 작성자 `is_public = true` | 공개 유저의 기록 |
| 3 | `records_followers` | `follows.status='accepted'` + 작성자 `follow_policy != 'blocked'` | 승인된 팔로워에게 공개 |
| 4 | `records_bubble_shared` | `bubble_shares` 경유 | 버블에 공유된 기록 |
| 5 | `records_bubble_member_read` | 같은 버블 active 멤버 | 같은 버블 멤버의 기록 |

### 2-3. 오버라이드 관계

```
비공개 사용자 (is_public = false)
├── follow_policy에 따라 팔로워에게 기록 공개
│   ├── auto_approve: 팔로우 즉시 → 기록 열람 가능
│   ├── manual_approve: 승인 후 → 기록 열람 가능
│   ├── conditional: 조건 충족 시 → 기록 열람 가능
│   └── blocked: 팔로우 자체 불가 → 기록 비공개 유지
└── 버블 공유: is_public과 무관하게, 버블에 공유한 기록은 버블 멤버에게 공개
```

### 2-4. 가시성 토글 (항목 단위 제어)

기록이 "보이는" 것과 "어디까지 보이는" 것은 별개:

```
우선순위: 버블별 커스텀 > 버블 기본 토글 > 전체 공개 토글
```

| 계층 | DB 필드 | 설명 |
|------|---------|------|
| 전체 공개 | `users.visibility_public` | is_public=true일 때 모든 사용자에게 |
| 버블 기본 | `users.visibility_bubble` | 모든 버블 멤버에게 기본값 |
| 버블별 커스텀 | `bubble_members.visibility_override` | 특정 버블에서만 |

토글 대상 7개: score, comment, photos, level, quadrant, bubbles, price

## 3. 미평가 기록

### 3-0. 개념

기록(record) 행은 존재하지만 아직 평가(axis_x, axis_y)가 없는 상태.
찜(wishlist)과는 다르다 — 찜은 lists에만 있고 records가 없지만, 미평가는 records가 있되 점수만 NULL.

| 상태 | lists 존재 | records 존재 | 점수(axis_x/y) |
|------|-----------|-------------|----------------|
| 찜 | O (wishlist/cellar) | X | - |
| **미평가** | O (visited/tasted) | **O** | **NULL** |
| 완료 | O (visited/tasted) | O | 있음 |

### 3-1. 사용 시나리오

- 식당에 들어갔는데 아직 먹기 전
- 와인을 구매했는데 셀러에 보관 중
- 먹고/마시고 다 했는데 세부 평가 작성 전

### 3-2. UI — "아직이에요" 버튼

기록 폼에서 평가 섹션("어떤 곳이었나요?" / "테이스팅") 옆에 버튼 배치.
클릭 시 사분면/세부평가 전부 건너뛰고 즉시 저장.

```
┌────────────────────────────────────┐
│ 어떤 곳이었나요?      [ 아직이에요 ] │  ← 클릭 시 즉시 저장
│ ┌──────────────┐                   │
│ │  사분면 입력    │                   │  ← 평가하면 정상 플로우
│ └──────────────┘                   │
└────────────────────────────────────┘
```

- 저장 데이터: 대상(식당/와인) + visit_date(오늘) + 사진(있으면) — axis_x/y/satisfaction = NULL
- 사분면에는 표시되지 않음

### 3-3. 홈 필터 — "미평가" 칩

보기 필터(multi-select)에 `미평가` 옵션 추가. 식당/와인 동일.

- 식당: 방문 / 찜 / **미평가** / 팔로잉 / 버블 / 공개
- 와인: 시음 / 찜 / 셀러 / **미평가** / 팔로잉 / 버블 / 공개

쿼리 조건: `records WHERE axis_x IS NULL AND user_id = me`

### 3-4. 미평가 → 완료 전환

미평가 카드 클릭 → 상세 페이지 → "평가하기" → 기존 기록 폼 진입 (기입력 데이터 유지, 평가 섹션부터 시작)

## 4. 선택 뷰 — 사용자에게 보이는 기록물

### 4-1. 선택 뷰가 등장하는 곳

| 위치 | 탭 | 사용 가능 뷰 모드 |
|------|------|----------------|
| 홈페이지 | 식당 / 와인 | 카드, 리스트, 달력, 지도(식당만) |
| 버블 페이지 | 식당 / 와인 | 리스트, 지도(식당만) |

### 4-2. 뷰 모드 4종

| 모드 | 설명 |
|------|------|
| **카드 뷰** | 사진 + 점수 + 기본 정보. 시각적 탐색 |
| **리스트 뷰** | 컴팩트한 1줄 카드. 빠른 스캔 |
| **달력 뷰** | 월별 그리드. **나의 기록만** 표시 |
| **지도 뷰** | 지도 위 마커. 식당에 한해서만 |

### 4-3. 선택 뷰에서 보여주는 항목

#### 공통 (카드/리스트/지도)

| 항목 | 설명 |
|------|------|
| **대표 점수 + 사분면** | source 우선순위(나 → 팔로잉 → 버블 → 공개)에 따라 가장 상위 소스의 평균 점수 표시 |
| **기본 정보 (식당)** | 위치, 최신 기록일, 기록물 수 |
| **기본 정보 (와인)** | 빈티지, 산지, 품종, 기록물 수 |

#### 달력 뷰 (예외)

- **나의 기록만** 표시
- 각 날짜에 당일 생성한 **기록물 갯수**만 표시
- 날짜 클릭 시 → 당일 기록한 나의 기록물 리스트 표시

### 4-4. 점수 소스 우선순위

```
1. 나 (my)       — 내 기록이 있으면 최우선
2. 팔로잉        — 내가 팔로우하는 사람들의 기록 평균
3. 버블          — 내 버블에 공유된 기록 평균
4. 공개 (nyam)   — 전체 공개 기록 평균
```

가장 상위에 존재하는 소스의 점수만 표시. 복수 소스를 동시에 보여주지 않음.

### 4-5. 클릭 동선

모든 뷰 모드에서 항목 클릭 → **상세 페이지** (식당 상세 / 와인 상세)로 이동.

## 5. 데이터 흐름 다이어그램

```
[기록 생성]
  사용자 → 대상 특정 (검색/카메라)
       → restaurants/wines 없으면 INSERT
       → lists 없으면 INSERT (status: visited/tasted)
       → records INSERT
       → record_photos INSERT (사진 있으면)
       → bubble_shares INSERT (공유 설정에 따라)

[홈 선택 뷰 조회]
  lists JOIN records
    WHERE user_id = 나                    → 내 기록
    + 팔로잉/버블/공개 기록 (RLS 통과분)   → 점수 소스 폴백용
    → GroupedTarget 생성 (대상 + 소스별 점수 집계)
    → 선택한 뷰 모드로 렌더링

[상세 페이지 조회]
  restaurants/wines WHERE id = ?          → 기본 정보
  records WHERE target_id = ? AND ...     → 소스별 기록 (나/팔로잉/버블/공개)
  → 소스별 점수 카드 렌더링
  → 사분면 차트에 점 렌더링
```

## 6. 핵심 요약

| 개념 | 설명 |
|------|------|
| 3-테이블 구조 | restaurants/wines → lists → records. lists가 "관계"를 정의하고, records가 "기록"을 쌓는다 |
| 프라이버시 기본값 | 비공개. follow/bubble이 오버라이드 |
| 노출은 RLS + application layer | RLS가 행 접근 제어, application이 항목(컬럼) 제어 |
| 선택 뷰 4종 | 카드/리스트/달력/지도. 같은 데이터를 다른 형식으로 |
| 점수 우선순위 | 나 → 팔로잉 → 버블 → 공개. 폴백 순서 |
| 달력은 예외 | 내 기록만, 갯수만, 클릭 시 당일 리스트 |
| 미평가 기록 | records 행 있지만 점수 NULL. 찜과 다름. "아직이에요" 버튼으로 즉시 저장 |
| 보기 필터 통일 | 식당/와인 모두 `view`(multi-select). 미평가/팔로잉/버블/공개 공통 |
