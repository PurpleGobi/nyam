# 생활권(area_zones) 반경 설정 기준

> 마이그레이션: `029_area_zones.sql` (초기), `036_area_zones_radius_fix.sql` (보정)

---

## 설정 원칙

1. **중심점** = 해당 생활권의 대표 랜드마크/역 출구
2. **반경** = 식당·카페 밀집 구역의 끝단까지 + 200~300m 여유
3. 인접 생활권과 겹침은 허용 (가장 가까운 곳으로 판정하므로 문제 없음)
4. 겹침이 있더라도 각 생활권의 "끝단 식당"을 놓치지 않는 게 우선
5. **대형 생활권(강남 등)은 다중 원** 사용 — 단일 대형 원은 인접 구까지 침범하므로, 작은 원 여러 개로 실제 영역에 맞게 커버. 같은 `name`으로 여러 행 INSERT

---

## 다중 원 방식

### 왜 다중 원인가

강남구 전체를 단일 원으로 커버하려면 반경 3500m+ 필요 → 서초구·송파구까지 침범.
polygon은 관리 복잡도가 높으므로, **같은 이름의 원 여러 개**로 영역을 채우는 방식 채택.

```
area_zones 테이블에 name UNIQUE 제약 없음
→ 같은 name으로 N개 행 INSERT 가능
→ 매칭 로직: 어느 원이든 하나만 통과하면 해당 생활권으로 판정
→ 세부 생활권(역삼/삼성 등)이 더 가까우면 세부 쪽에 우선 배정
```

## 생활권 매칭 로직

### 전체 흐름

```
식당 등록 (POST /api/restaurants)
  │  lat/lng 저장
  │
  ├─ 클라이언트가 GET /api/location/resolve?lat=...&lng=... 호출
  │    │
  │    ├─ 1. 카카오 역지오코딩 → district (구/군) 추출
  │    │     coord2regioncode API → region_2depth_name → "강남구"
  │    │
  │    └─ 2. area_zones 전체 조회 → haversine 거리 계산 → 가장 가까운 zone 선택
  │          → area = "역삼" (또는 null)
  │
  ├─ 결과: { country, city, district, area }
  │
  └─ restaurants 테이블에 area 저장
```

### haversine 매칭 상세

```
GET /api/location/resolve
```

**입력**: `lat`, `lng` (식당 좌표)

**Step 1 — district 결정** (카카오 역지오코딩)

```
카카오 API: coord2regioncode.json?x={lng}&y={lat}
→ region_1depth_name: "서울특별시" → city: "서울"
→ region_2depth_name: "강남구"     → district: "강남구"
```

- `KAKAO_REST_API_KEY` 없으면 스킵 (district = null)
- API 실패해도 에러 무시, district만 null

**Step 2 — area 결정** (area_zones 매칭)

```typescript
// area_zones 전체 행 SELECT
for (각 zone) {
  dist = haversine(식당.lat, 식당.lng, zone.lat, zone.lng)
  if (dist <= zone.radius_m) {
    // 반경 내 → 후보에 추가
    if (dist < 현재최소거리) {
      closest = zone.name   // 더 가까운 zone으로 교체
    }
  }
}
area = closest?.name ?? null
```

- 반경 내 zone이 여러 개 → **가장 가까운 것 1개만 선택**
- 다중 원(강남 3원, 여의도 2원) → 같은 name이라 어느 원에 걸려도 동일 결과
- 반경 내 zone 없음 → area = null
- haversine 공식: 지구 반지름 R=6,371km 기준 대원거리 (미터 단위)

### 다중 원에서의 매칭 예시

```
식당 좌표: 대치동 (37.494, 127.058)

강남 ① 강남역·논현 (37.498, 127.028) 1500m → dist = 2,620m → ✗ 반경 초과
강남 ② 대치·도곡  (37.494, 127.060) 1500m → dist = 174m   → ✓ 반경 내
강남 ③ 개포·수서  (37.478, 127.072) 1500m → dist = 2,100m → ✗ 반경 초과
삼성          (37.509, 127.063) 1400m → dist = 1,720m → ✗ 반경 초과

→ 가장 가까운 반경 내 zone = 강남 ② (174m)
→ area = "강남"
```

### 세부 생활권 우선 배정 원리

```
식당 좌표: 역삼역 근처 (37.501, 127.037)

역삼    (37.501, 127.037) 1300m → dist = 50m    → ✓
강남 ① (37.498, 127.028) 1500m → dist = 830m   → ✓
삼성    (37.509, 127.063) 1400m → dist = 2,400m → ✗

→ 역삼(50m) < 강남 ①(830m)
→ area = "역삼" (세부 생활권 우선)
```

세부 생활권의 중심이 더 가까우니 자연스럽게 우선 배정. 별도 로직 불필요.

### 홈 필터에서의 area 매칭

홈 화면 필터는 **클라이언트 사이드 문자열 비교**:

```typescript
// filter-matcher.ts
if (attribute === 'area') {
  const arr = record['area'] as string[] | null
  return arr?.includes(String(value)) ?? false
}
```

- DB에 저장된 area 문자열과 필터 값을 `includes`로 비교
- 좌표 재계산 없음 (등록 시점에 결정된 값 사용)
- area가 null인 레코드는 어떤 생활권 필터에도 안 걸림

### 현재 한계

| 항목 | 상태 | 비고 |
|------|------|------|
| 식당 등록 시 area 자동 저장 | ⚠️ 미구현 | POST에서 area를 INSERT하지 않음 — 별도 구현 필요 |
| 기존 식당 area 백필 | ⚠️ 미실행 | lat/lng 있는 기존 식당에 area 소급 적용 필요 |
| area = null인 식당 | 필터 누락 | 어떤 생활권 필터에도 안 잡힘 |

## 반경 사이즈 가이드 (신규 생활권 추가 시)

| 상권 규모 | 반경 | 예시 |
|----------|------|------|
| 골목형 (1~2블록) | 800~1000m | 가로수길, 녹사평, 한양대 |
| 소형 (역 1개 중심) | 1000~1200m | 서촌, 북촌, 명동, 상수 |
| 중형 (역 1~2개) | 1200~1500m | 역삼, 청담, 연남, 합정, 건대 |
| 대형 (역 2~3개) | 1500~1800m | 홍대, 이태원, 잠실 |
| 구 단위 / 섬 | **다중 원** (1200~1500m × N) | 강남(3원), 여의도(2원) |

### 다중 원 추가 시 규칙

1. 같은 `name`으로 area_zones에 N개 행 INSERT
2. 각 원의 반경은 1200~1500m 범위 (인접 구 침범 방지)
3. 원끼리 200~500m 겹침은 OK (빈 틈 방지)
4. 세부 생활권이 이미 있는 영역은 다중 원에서 제외해도 됨
