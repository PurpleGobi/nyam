## 1. 배경: 왜 리팩토링이 필요한가

### 1-1. Nyam의 재정립된 정체성

> **Pain Point**: 제대로 된, 신뢰할 만한 맛집 정보를 찾기 어렵다.
>
> **Solution**: 자기 자신을 위해 시작한 기록이, 버블이라는 신뢰 단위를 거쳐, 경험치라는 자격 증명을 통해 더 넓은 맛집 정보 생태계로 확장되는 구조.

핵심 전제: **이타적 기록은 지속되지 않는다.** 동기 자체를 이기적으로 설계해야 한다. 기록의 시작점은 나 자신을 위한 것이고, 공유는 그 부산물이다.

### 1-2. 기존 버블 시스템의 문제점

| # | 문제 | 상세 |
|---|------|------|
| 1 | **콜드스타트 데이터 공백** | 경험치 0인 신규 유저는 어떤 버블 데이터에도 접근 불가. 홈 화면이 텅 빈 상태 → "이 앱 왜 쓰지?" → 이탈 |
| 2 | **XP가 "양"만 측정** | 이름만 등록 100개 = 높은 XP. 실제 방문 경험과 무관하게 레벨업 가능 → 신뢰 지표로서 무의미 |
| 3 | **버블 진입 허들의 불투명성** | XP 레벨로 가입 제한 → 들어가는 쪽은 뭐가 있는지 모르고, 받는 쪽은 누가 들어오는지 모름 |
| 4 | **개인 팔로우 vs 버블 차이 불명확** | 1인 버블 ≈ 개인 팔로우. 버블의 관리 부담 대비 차별적 가치 불분명 |
| 5 | **데이터 복제/무임승차** | 팔로우로 전체 리스트 접근 → 복제 가능. 지인 한 명이 여러 버블 가입 후 데이터 공유하면 허들 무력화 |
| 6 | **공유형 유저의 홈 경험** | 버블 공유가 주된 동기인 유저에게도 홈이 "내 기록"만 보여줌. 버블 데이터는 별도 페이지에 격리 |

### 1-3. 버블 사용의 본질 재확인

버블은 **뉴스 피드가 아니다.** 실시간 타임라인을 소비하는 소셜 미디어가 아니라, 내가 어디 갈지 고민할 때 꺼내보는 **신뢰할 수 있는 맛집 DB(공유 데이터베이스)**에 가깝다.

같은 맥락 안의 사람들이 모아놓은 리스트를 보면서 선택하고, 실제 경험 후 개인 기록을 업데이트하는 흐름.

## 2. 핵심 변경: 데이터 접근 계층 모델

### 2-1. 설계 원칙

> **기여도에 비례한 접근 권한** — 더 많이 기여할수록 더 깊은 정보에 접근 가능

### 2-2. 버블에서의 4단계 접근

| 단계 | 역할 | 조건 | 보이는 것 | 할 수 있는 것 |
|------|------|------|----------|-------------|
| **탐색** | 비팔로워 | public 버블이면 누구나 | 버블 이름, 설명, 멤버 수, 통계 요약 | 검색, 열람 |
| **팔로우** | follower | XP 불필요, 누구나 가능 | + 식당/와인 **이름, 평균 점수, 지역** | + 좋아요, 찜 |
| **멤버** | member | 가입 승인 필요 | + **개별 리뷰, 사진, 팁, 누가 몇 점인지** | + 기록 공유, 댓글, 리액션 |
| **운영** | admin/owner | 오너 지정 | 전체 | + 멤버 관리, 설정 변경 |

**팔로워 = 맛보기.** 식당 이름과 평균 점수는 알 수 있지만, "왜 좋은지, 뭘 먹어야 하는지"는 멤버만 볼 수 있다.

**복제 방어:** 이름+점수만으로는 가치가 제한적. Spotify 플레이리스트 곡 제목만 복사하는 것과 같음 — 진짜 가치는 지속적인 큐레이션과 상세 맥락.

### 2-3. 개인 팔로우에서의 3단계 접근

| 단계 | 관계 | 조건 | 보이는 것 |
|------|------|------|----------|
| **비팔로우** | 없음 | — | 공개 프로필만 (레벨, 기록 수, 버블 수) |
| **일방 팔로우** | 내가 팔로우 | 상대가 public 프로필 | + 식당/와인 **이름, 점수, 지역** (맛보기) |
| **맞팔로우** | 상호 팔로우 | 양쪽 모두 팔로우 | + **풀 액세스** (리뷰, 사진, 팁, 상세 기록) |

**핵심: 맞팔로우 = 기브앤테이크의 구현**

- "이 사람 기록 제대로 보고 싶다" → 팔로우
- 상대가 "이 사람도 볼만하네" → 맞팔
- 맞팔 조건 = 내 기록의 질이 좋아야 상대가 팔로우할 동기가 생김
- → 기록 품질 향상 동기 자연스럽게 발생

### 2-4. 버블 멤버 → 개인 팔로우 연결

```
버블 가입
  → 멤버 목록 열람 가능
  → 특정 멤버의 기록이 좋다 → 개인 팔로우
  → 상대도 나를 팔로우 → 맞팔 = 버블 밖에서도 풀 액세스
  → 버블을 넘어선 개인 네트워크 형성
```

이를 통해 **개인 팔로우의 차별점**이 명확해진다:
- **버블** = 그룹 단위의 맥락적 큐레이션 (여러 사람의 기록이 하나의 주제로 모임)
- **개인 팔로우** = 특정 개인의 취향을 깊게 추적 (버블 가입 후 발견한 취향 맞는 사람)
- **버블이 개인 팔로우의 발견 채널** 역할

## 3. XP 시스템 — 품질 가중 + 활성 XP + 어뷰징 방지

### 3-1. XP 이원화: 총 XP + 활성 XP

```
총 XP    = 누적 (절대 안 줄어듦). 레벨 표시에 사용. "이 사람의 역사"
활성 XP  = 최근 6개월 기록 기반 (자연 변동). 버블 진입 기준에 사용. "이 사람의 현재"
```

**왜 분리하나:**
- 총 XP만 있으면 인플레이션 → 24개월차에 Lv.10+ 21.7% (시뮬레이션 검증)
- 활성 XP는 6개월 윈도우로 자연 감쇠 → 비활동 유저의 활성 XP는 0으로 수렴
- **레벨(총 XP 기반)은 절대 안 깎임** → 손실 회피 심리 회피

```
예시:
  파워유저 A: 총 XP 2,700 (Lv.54) | 활성 XP 450 (최근 6개월 풀 기록 30개)
  → 버블 auto_approve 기준: 활성 검증 기록 10개 → ✅ 통과

  은퇴유저 B: 총 XP 2,700 (Lv.54) | 활성 XP 0 (6개월간 기록 0)
  → 같은 레벨이지만 활성 XP 0 → ❌ auto_approve 실패
  → 다시 기록 5개만 하면 활성 XP 회복
```

**버블 가입 기준 설정 (오너가 선택):**

| 기준 조합 | 의미 | 사용 예 |
|-----------|------|---------|
| 총 XP만 | 역사적 공헌 중시 | "오래 쓴 사람이면 OK" |
| 활성 XP만 | 현재 활동 중시 | "지금 활발한 사람만" |
| 총 XP AND 활성 XP | 둘 다 충족 | "오래 쓰고 + 지금도 활발" |
| 총 XP OR 활성 XP | 하나만 충족 | "느슨한 기준" |

### 3-2. 기록 품질 가중 XP

사진의 GPS 좌표 + 타임스탬프는 **실제 방문의 물리적 증거**. 대규모 위조가 극도로 어렵다.

| 기록 수준 | XP | 비고 |
|---|---|---|
| 이름만 등록 | **+1** | 누구나 가능, 거의 무의미 |
| + 사분면 점수 | **+3** | 최소한의 평가 |
| + 사진 (EXIF GPS+시간 포함) | **+10** | 실제 방문 증거 |
| + 상세 (리뷰 + 상황 + 메뉴) | **+15** | 풀 기록 |

### 3-3. 점수 제한: 같은 식당은 6개월에 1회만

```
같은 식당(target_id)에 대해:
  - 리뷰/사진/메모: 얼마든지 추가 가능 (방문할 때마다)
  - 만족도 점수: 6개월에 1회만 부여/갱신 가능
  - XP는 점수 부여 시에만 획득

→ 광고주가 20개 계정으로 자사 식당 점수 조작 시도해도:
  계정당 반기 1회 = 반기에 20점뿐 → 영향 미미
→ 정상 유저도 같은 식당 재방문 시:
  리뷰는 남기되, 점수는 6개월 후 갱신 가능
  → 자연스러운 UX (재방문 기록은 자유, 점수는 신중하게)
```

### 3-4. EXIF 검증 로직

```
사진 업로드 시:
  1. EXIF 메타데이터 추출 (GPS 좌표, 촬영 시간)
  2. GPS 좌표 ↔ 식당 위치 비교
  3. 반경 200m 이내 → has_exif_gps: true, exif_verified: true
  4. GPS 없거나 불일치 → 사진 XP는 부여, "검증됨" 배지 없음
  5. 검증됨 배지 수 = 프로필의 핵심 신뢰 지표
```

### 3-5. 평균 점수 산출: Trimmed Mean (양끝 5%)

```
식당/와인의 표시 점수 = trimmed_mean(모든 점수, trim=0.05)
  → 양쪽 끝 5%를 제거한 후 평균
  → 트롤(극단 저점수)과 팬(극단 고점수) 모두 자동 제거
  → 개인 점수는 그대로 유지 (본인에게는 자기 점수가 보임)
  → 시스템이 "이상한 점수"를 판단하지 않음 — 통계적으로 자연 제거

예시:
  스시코우지 점수 10개: [30, 85, 88, 90, 91, 92, 93, 95, 96, 99]
  일반 평균: 85.9
  Trimmed mean (5%): [85, 88, 90, 91, 92, 93, 95, 96] → 91.3
  → 30점(트롤)과 99점(극단)이 자동 제거됨
  → 패널티 없이 데이터 품질 유지
```

### 3-6. 소셜 XP (기존 유지 + 추가)

| 활동 | XP | 일일 상한 | 비고 |
|------|-----|----------|------|
| 버블에 공유 | +1 | 10 | 기존 유지 |
| 좋아요/찜 받음 | +1 | 10 | 기존 유지 |
| 팔로워 획득 (버블/개인) | +1 | 10 | — |
| 맞팔 성사 | +2 | 10 | 양쪽 모두에게 |

**설계 원칙:** 소셜 XP는 전체의 ~5-8% 수준 유지. **기록이 압도적 주 소스.**

### 3-7. 어뷰징 방지 — 다층 방어

**계정 레벨:**
- **소셜 로그인만 허용** (카카오/구글/애플)
- **1 소셜 계정 = 1 Nyam 계정** (중복 가입 차단)
- → 봇 팜 1,000개 = 소셜 계정 1,000개 필요 → 비용 급증 → 가성비 없음

**기록 레벨:**

| 위협 | 방어 |
|------|------|
| 이름만 대량 등록 | +1 XP뿐. 프로필에 "검증 0" 노출 → 가입 승인 거절 |
| 같은 식당 반복 점수 | **6개월 1회 제한** → 다중 계정 없이는 의미 없음 |
| EXIF 위조 | 기술적 가능하나 대규모 위조는 비현실적 |
| 하루 기록 수 상한 | 20개 (정상 파워유저도 3~5개) |

**소셜 레벨:**

| 위협 | 방어 |
|------|------|
| 자작 팔로우/좋아요 팜 | 같은 IP/디바이스 중복 무효 + 일일 상한 |
| 기록 0 계정의 팔로우 | XP 미부여 |
| 비정상 팔로우 패턴 | 1분 내 10+ 팔로우 → 제한 |
| 고레벨 지인이 데이터 중계 | 팔로워는 이름+점수만 보임. 상세는 멤버/맞팔만 |

## 4. 버블 공개 설정 + 가입 정책 분리

### 4-1. 기존 모델의 한계

```
기존: visibility = private | members | public
  → "볼 수 있는가"와 "가입할 수 있는가"가 하나로 묶여 있음
  → "공개하되 가입은 안 받는" 설정 불가
  → 콜드스타트 유저에게 열어줄 방법 없음
```

### 4-2. 새 모델: visibility + join_policy 분리

```
[visibility — 누가 볼 수 있는가?]
  private  → 초대받은 사람만 (검색 불가)
  public   → 누구나 검색/팔로우 가능

[join_policy — 누가 멤버가 될 수 있는가?] (public 버블에만 해당)
  closed         → 가입 안 받음 (팔로우만 가능)
  manual_approve → 오너가 프로필 보고 승인/거절
  auto_approve   → 검증 기록 N개 이상이면 자동 승인
  open           → 누구나 가입 가능
```

### 4-3. 가입 시 양방향 프리뷰

기존 문제: "XP 레벨 5" 같은 숫자만으로 판단 → 양쪽 모두 불투명.

**가입 희망자가 보는 것 (가입 전 미리보기):**

| 정보 | 예시 |
|------|------|
| 멤버 수 | 12명 |
| 총 기록 | 234개 |
| 주요 지역 | 을지로, 광화문 |
| 평균 점수 | 87 |
| 나와 겹치는 곳 | 5개 |
| 취향 유사도 | 78% |

→ "이 버블에 들어갈 가치가 있겠다"

**버블 오너가 보는 것 (가입 신청 시):**

| 정보 | 예시 |
|------|------|
| 검증 기록 (EXIF) | 28개 |
| 단순 등록 | 5개 |
| 주요 지역 | 을지로, 성수 |
| 취향 유사도 | 78% |
| 최근 활동 | 3일 전 |

→ "이 사람을 받아도 되겠다" or "거절"

**핵심:** XP "레벨" 단일 숫자 대신, **기록의 질적 프로필**이 판단 기준.

### 4-4. 기존 → 새 모델 마이그레이션 매핑

| 기존 visibility | 변경 |
|---|---|
| `private` | visibility=`private`, join_policy=`invite_only` |
| `members` (경험치 기반) | visibility=`public`, join_policy=`auto_approve` |
| `public` (누구나 가입) | visibility=`public`, join_policy=`open` |

## 5. 역할 체계 (개정)

| 역할 | 데이터 열람 | 좋아요/찜 | 댓글 | 기록 공유 | 멤버 관리 | 버블 삭제 |
|------|-----------|----------|------|----------|----------|----------|

> *댓글은 `allow_comments` 설정에 따라
> 기존 `subscriber` → `follower`로 명칭 변경

## 6. 버블 유형별 시나리오

### A: 개인 큐레이션 채널

```
"을지로 맛집 가이드" — 1인 운영
  visibility: public
  join_policy: closed

  → 팔로워들이 이름+점수 리스트로 참고
  → 좋아요/찜 → 오너 XP 상승
  → 오너는 성실하게 기록 → 팔로워 증가 → 선순환
  → 관리 부담: 최소 (가입 안 받으니 관리할 멤버 없음)
```

### B: 지인 모임 (관리 최소)

```
"직장 점심" — 5명
  visibility: private
  join_policy: invite_only

  → 초대받은 멤버만 접근
  → 외부에 노출 안 됨
  → 내부에서 자유롭게 기록 공유 + 댓글/리액션
  → 관리 부담: 없음 (소규모 + 아는 사람)
```

### C: 오픈 커뮤니티 (카페형, 소셜 강화)

```
"강남 와인바 모임" — 30명
  visibility: public
  join_policy: manual_approve

  → 누구나 팔로우 (이름+점수 열람)
  → 가입 신청 → 오너가 프로필(검증 기록 수, 취향 유사도) 보고 승인
  → 멤버: 풀 액세스 + 기록 공유 + 댓글/리액션
  → 멤버 목록에서 개인 팔로우 → 맞팔로 확장
  → 관리 부담: 있음 (가입 승인, 오염원 관리). 대신 리텐션 강화
```

### D: 콜드스타트 유저 여정

```
가입 직후 (기록 0, XP 0)

  1. 온보딩에서 버블 1개 생성 (내 모임)
  2. "버블 탐색"으로 공개 버블 둘러보기
  3. 취향 유사도 높은 버블 3~5개 팔로우 (무료, XP 불필요)
  4. 홈에서 팔로잉 버블의 이름+점수 리스트 즉시 활용
  5. "여기 괜찮겠다" → 방문 → 사진+리뷰 풀 기록 (+15 XP)
  6. 검증 기록 쌓임 → 프로필 신뢰도 상승
  7. 멤버 가입 신청 → 오너 승인 → 풀 액세스 (상세 리뷰, 사진, 팁)
  8. 멤버 목록에서 취향 맞는 개인 팔로우
  9. 맞팔 성사 → 버블 밖에서도 풀 액세스
```

## 7. 버블 공개/팔로우 동기 구조

```
왜 버블을 공개하나?
├── 팔로워 획득 → XP (+1/팔로워)
├── 좋아요/찜 수신 → XP (+1/건)
├── 자기 만족 (큐레이션 인정)
└── XP 축적 → 더 나은 버블 가입 자격 확보

왜 팔로우하나? (버블)
├── 신뢰할 수 있는 맛집 리스트 확보
├── "어디 갈지" 고민할 때 참고
├── 취향 유사도 높은 버블 발견
└── 무료 (XP 불필요)

왜 맞팔하나? (개인)
├── 특정 개인의 상세 기록/팁에 접근
├── 버블 밖에서도 지속적 관계
├── 상대의 기록 품질을 확인한 후 결정
└── 양쪽 모두 XP +3

선순환 (전체 플라이휠):
  나를 위한 기록 (이기적 동기)
    → XP 축적 (노력이 자산)
    → 버블 생성/공개 → 팔로워 유입
    → 좋은 멤버 합류 → 버블 가치 상승
    → 개인 팔로우/맞팔로 네트워크 확장
    → 더 많은 신뢰 데이터 접근
    → 더 나은 기록 동기
    → (반복)

콜드스타트 진입점:
  팔로우 (무료) → 맛보기 → 가치 체감 → 기록 시작 → 생태계 합류
```

## 8. 홈 화면 영향 (별도 상세 설계 필요)

팔로잉한 버블/개인의 데이터가 홈에서 접근 가능해야 콜드스타트가 해결된다.

### 방향: 서브탭 추가

```
기존:  방문(N) | 찜(N) | 추천(N)
변경:  방문(N) | 찜(N) | 추천(N) | 팔로잉(N)
```

- **팔로잉 서브탭**: 팔로우한 버블 + 맞팔 개인의 식당/와인 통합 리스트
- 보이는 것: 이름 + 평균 점수 + 지역 (팔로우 맛보기 수준)
- 필터: 버블별, 지역별, 점수순 등
- 콜드스타트에도 팔로잉만 있으면 콘텐츠 있음
- 맞팔 개인의 데이터는 풀 상세로 보임

> 홈 UI 상세 설계는 이 버블 시스템 확정 후 별도 진행

```sql
-- 매일 또는 주간 실행: 활성 XP 재계산
UPDATE users SET
  active_xp = (
    SELECT COALESCE(SUM(record_quality_xp), 0)
    FROM records
    WHERE user_id = users.id
      AND created_at > NOW() - INTERVAL '6 months'
  ),
  active_verified = (
    SELECT COUNT(*)
    FROM records
    WHERE user_id = users.id
      AND exif_verified = true
      AND created_at > NOW() - INTERVAL '6 months'
  );
```
