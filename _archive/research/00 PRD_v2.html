1. 제품 정체성

Nyam은
나의 식도락 여정을 기록하고, 분석하고, 신뢰할 수 있는 사람들과 공유하는 플랫폼
선택만으로 기록이 시작되고, 쌓일수록 나를 알아가며,
버블 안에서 오염 없는 진짜 추천을 주고받는다.

정체성 우선순위

순위가치설명
1나를 위한 기록나의 식도락 경험을 DB화. 언제, 어디서, 누구랑, 뭘 먹었고, 어땠는지. 콜드스타트 없이 첫날부터 풍성한 리스트.
2나를 아는 분석기록이 쌓이면 상황별 취향이 드러난다. 혼밥의 나와 접대의 나는 다르다. 와인도 마찬가지.
3오염 없는 공유버블(폐쇄형 그룹) 안에서만 공유. 광고 리뷰, 가짜 평가가 구조적으로 불가능.
4신뢰 기반 발견아는 사람들의 기록으로 만드는 추천. 낯선 동네에서 현지인 가이드가 된다.
5식도락 정체성기록이 곧 나의 프로필. 취향이 명함이 되고, 경험치가 신뢰의 근거.

Nyam이 아닌 것

  • 리뷰 플랫폼이 아니다 — 불특정 다수에게 공개되는 리뷰를 쓰는 곳이 아님
  • 예약 서비스가 아니다 — 거래/결제를 직접 처리하지 않음
  • 맛집 검색 엔진이 아니다 — 인터넷 정보 기반 추천은 최소한으로
  • 다이어트/영양 앱이 아니다 — 식도락을 즐기는 사람을 위한 도구

도메인 범위

포함제외 (v2.0)
식당 — 모든 외식 경험 (레스토랑, 카페, 바, 프랜차이즈, 포장마차 등)
와인 — 기호식품/취미로서의 와인 경험 		요리(쿠킹) — 향후 확장 가능성은 열어두되, v2에서는 제외
배달 — 외식 경험에 집중. 배달은 추후 검토

2. 문제 정의

2-1. 핵심 페인포인트

"나의 식도락 여정을 기록하고 싶다."
언제 누구랑 어디서 뭘 먹었고, 어땠는지. 와인도 마찬가지.

"그런데 기록은 너무 귀찮다."
사진 찍는 것도 까먹고, 앱 열어서 입력하는 건 더 귀찮다.

2-2. 확장 페인포인트

"낯선 동네에서 뭐 먹지?"

출장, 여행, 처음 가는 동네. 네이버 검색하면 광고와 협찬 리뷰뿐. 진짜 믿을 만한 추천이 없다.

"맛집 리뷰를 믿을 수 없다"

광고 리뷰, 블로그 체험단, 별점 조작. 정보가 많을수록 오히려 신뢰가 떨어진다.

"와인 뭐 마실지 모르겠다"

와인은 취향이 뚜렷한 기호식품인데, 내 취향을 체계적으로 파악할 방법이 없다.

"내가 예전에 갔던 그 집이 뭐였지?"

기록이 흩어져 있다. 카톡, 인스타, 메모앱, 머릿속. 검색도 정리도 안 됨.

2-3. 기존 서비스의 한계

서비스하는 것못하는 것
네이버/카카오맵검색, 공개 리뷰개인 기록, 취향 분석, 오염 없는 추천
인스타그램사진 공유, 팔로우구조화된 기록, 검색/필터, 분석
Vivino와인 스캔/기록식당 통합, 상황별 분석, 소셜 깊이
캐치테이블예약기록 도구 아님, 대상 식당 제한
망고플레이트리뷰/평점오픈 리뷰의 신뢰 문제, 개인화 약함

3. 해결 전략

3-1. 핵심 전환: "기록해라" → "골라라"

레터박스 전략의 식도락 적용

레터박스: 인기 영화 리스트에서 "봤던 거 체크" → 한 번에 수십 개 기록 생성
Nyam: 지역 기반 식당 리스트에서 "가봤던 곳 체크" → 한 번에 수십 개 기록 생성

사용자가 "쓰는" 게 아니라 "고르는" 것. 마찰이 근본적으로 다르다.

3-2. 전략 3단계

단계전략설명참고 모델
1 나의 식도락 DB 구축 선택 기반 기록으로 콜드스타트 해결. 개인 식도락 아카이브를 만든다. 재방문마다 디테일이 쌓인다. 레터박스 Watched/Log
2 버블로 오염 없이 공유 폐쇄형 그룹 안에서만 공유. 필터링된 내 기록을 신뢰할 수 있는 사람들에게. 리뷰의 리뷰로 토론. 당근 동네 인증 + 레터박스 소셜
3 경험치로 신뢰 구축 지역별/와인 종류별 활동량 기반 경험치. 기록이 많은 사용자의 추천이 더 무게를 갖는 구조. Nyam 고유

3-3. 와인 — 왜 식당과 함께인가

와인은 식도락보다 더 강한 기록 동기를 가진다:
  • 기호식품/취미 — 취향이 더 뚜렷하고 탐구 욕구가 있음
  • 객관적 축이 존재 — 품종, 산지, 빈티지, 스타일이 구조화되어 있음
  • 기록 동기 검증됨 — Vivino 사용자가 이미 와인 기록을 함 (단, 소셜이 약함)
  • 식당 경험과 자연스럽게 연결 — "이 식당에서 이 와인을 마셨다"
식당과 와인을 통합 관리함으로써 Vivino에도 네이버에도 없는 "식도락 전체 경험의 아카이브"가 된다.

4. 사용자 저니 — Phase 1: 나의 기록

4-1. 온보딩

목표: 5분 안에 "나의 취향"이 대략 파악되고, "나의 맛집 리스트"가 10개 이상 채워진 상태로 시작.

Step 1: 취향 카드 선택

설문지 형태가 아닌 2지선다 이미지 카드로 빠르게 취향 파악.

  • "새로운 도전 vs 익숙한 맛" (이미지: 이국적 요리 vs 한식)
  • "분위기 중시 vs 맛 중시" (이미지: 파인다이닝 vs 허름한 맛집)
  • "가성비 vs 프리미엄" (이미지: 백반집 vs 오마카세)
  • "혼밥 OK vs 무조건 같이" (이미지: 카운터석 vs 단체테이블)

와인: "레드 vs 화이트", "가볍게 vs 묵직하게", "가격대 선택" 등 별도 카드 세트

Step 2: 주요 활동 지역 선택

지도에서 주요 활동 지역 1~3곳 선택 (직장 동네, 집 동네, 자주 가는 동네)

당근식 동네 인증 방식. 인증 없이도 사용 가능하지만, 인증 시 해당 동네 경험치에 가산.

Step 3: 가본 곳 체크 (핵심 — 콜드스타트 해결)

선택한 지역의 대표 식당 리스트를 보여주고 가본 곳 체크.

  • 미슐랭/블루리본 선정 식당
  • 유명 프랜차이즈
  • 지역 대표 맛집 (카카오/네이버 상위 랭킹)

체크한 식당에 바로 Nyam 평가 툴(후술)로 간단 평가.

목표: 온보딩 완료 시 최소 10~30개 식당이 "나의 리스트"에 있는 상태.

Step 4: 와인 취향 (선택)

와인 관심 있는 사용자에게만. 지역/국가별 인기 와인 선택 + 점수 부여.

품종/산지/스타일 태그가 자동 할당됨.

Step 5: 직접 검색 추가 (선택)

리스트에 없지만 기억나는 식당/와인을 직접 검색해서 추가 가능.

4-2. 홈 화면 — 나의 식도락 대시보드

홈 = 나의 식도락 기록 관리 & 열람 페이지

상단: 추천 영역

탭내용
내 경험에서내가 높게 평가한 곳 중 재방문 추천. 상황별 필터 가능 ("오늘 데이트인데...")
새로운 도전찜(Wishlist) 목록, 버블에서 추천받은 곳, 권위 리뷰(미슐랭/블루리본) 기반
와인내 취향 기반 와인 추천 + 와인 버블 추천

인터넷 정보 기반 추천은 최소한. 특히 콜드스타트 때는 권위 있는 리뷰 업체 정보(미슐랭, 블루리본) + 네이버/다음/구글맵 평점 정도만 제공.

메인: 나의 기록 타임라인

  • 최근 식당 & 와인 기록이 시간순으로 표시
  • 뷰 전환: 캘린더 뷰 / 리스트 뷰 / 썸네일(그리드) 뷰 / 지도 뷰
  • 필터: 장르, 지역, 상황, 평점 범위, 기간
  • 정렬: 최신순, 평점순, 방문 횟수순

4-3. 새로운 기록 등록

두 가지 진입점:

방법 1 검색으로 시작

식당/와인 이름 검색 → 선택 → Nyam 평가 툴로 평가

가장 기본적이고 확실한 방법.

방법 2 사진으로 시작

사진 선택 → EXIF(GPS/날짜) + Vision AI로 식당/메뉴 자동 추정 → 확인 → 평가

와인은 라벨 사진 → Vision AI로 와인 정보 자동 인식.

지역 기반 푸시 알림 (새로운 지역 접근 시)

새로운 지역에 갔을 때 푸시: "○○동에 오셨네요! 이 동네 맛집 둘러볼까요?"
  • 해당 지역의 대표 식당 리스트 제시
  • 가본 곳 체크 유도 (컬렉션 확장 동기)
  • 못 가본 곳은 찜(Wishlist) 유도
  • 추천 역할 겸용 — 찜한 곳은 나중에 추천 영역에 노출

4-4. 식당/와인 상세 페이지

레터박스의 영화 상세 페이지와 같은 구조. 정보 + 나의 기록 + 커뮤니티 반응.

식당 상세 페이지 구성

영역내용
기본 정보가게명, 주소, 장르, 영업시간, 가격대, 대표 사진 — 카카오/네이버/구글 교차 취합
외부 평가미슐랭/블루리본 선정 여부, 네이버/다음/구글맵 평점 (각각 표시)
나의 기록방문 횟수, 날짜별 기록, 평가 점수, 한줄평, 메뉴별 별표, 사용팁 등
재방문 기록방문마다 새 기록 추가 가능. 상황/경험/메뉴가 매번 다를 수 있음.
버블 리뷰(Phase 2) 내가 속한 버블에서 이 식당에 대한 다른 멤버들의 리뷰

와인 상세 페이지 구성

영역내용
기본 정보와인명, 품종, 산지, 빈티지, 타입(레드/화이트/스파클링 등), 알코올 도수
나의 기록마신 날짜, 장소(식당과 연결 가능), 평가, 페어링 메모, 한줄평
재음용 기록같은 와인도 빈티지/상황에 따라 경험이 다름. 매번 새 기록.

4-5. Nyam 평가 툴 (독창적 개발)

설계 원칙:
  • 모바일 UI 사용성 최고 — 한 손으로, 3초 안에 평가 가능
  • 재미 — 평가 자체가 즐거운 경험 (게임적 요소)
  • 풍부한 정보 — 단순 별점이 아니라, 맛/분위기/가성비 등 다축 정보 수집
  • 유연한 깊이 — 최소 1탭(총점)부터, 원하면 상세 평가까지

구체적 UI 디자인은 DESIGN_SPEC에서 정의. 여기서는 요구사항만 명시.

4-6. 재방문 — 기록의 깊이가 쌓이는 구조

같은 식당이라도 매 방문은 새로운 경험.
  • 방문마다 독립된 기록 생성 (날짜, 상황, 평가가 매번 다를 수 있음)
  • 총점은 최신 방문 기준 + 과거 방문 이력 한눈에 열람
  • "3번 갔는데, 혼밥 95점, 회식 70점, 데이트 85점" — 상황별 경험 차이가 데이터로 쌓임
  • 이 데이터가 상황별 취향 분석(Taste DNA × Scene)의 원천

5. 사용자 저니 — Phase 2: 버블 & 소셜

5-1. 버블 개념

버블 = 초대제 폐쇄형 맛집 공유 그룹
오염(광고 리뷰, 가짜 평가) 문제를 구조적으로 해결하는 핵심 장치.

버블 유형

유형설명예시
양방향 버블모든 멤버가 기록 공유 + 리뷰 + 리액션직장 동료 맛집 버블, 동네 맘 버블, 와인 동호회
열람 전용 버블운영자의 기록만 공개. 구독자는 보기/찜만 가능맛잘알의 큐레이션 채널, 소믈리에 추천

5-2. 공유 플로우

Step 1: 공유할 기록 필터링

내 기록 중 공유하고 싶은 식당/와인을 필터로 추린다.

예: "광화문" + "90점 이상" → 내가 광화문에서 높게 평가한 곳만 추출

Step 2: 버블에 공유

선택된 기록을 특정 버블에 공유. 버블 멤버들의 피드에 노출.

Step 3: 반응 & 토론

  • 찜 — 가보고 싶으면 내 Wishlist에 추가
  • 좋아요 / 싫어요 — 가벼운 리액션 (익명 선택 가능)
  • 리뷰 — 같은 식당에 대한 본인의 리뷰 작성 (독립된 리뷰 ID)
  • 댓글 — 공유자의 리뷰에 대한 댓글 (리뷰의 리뷰)

5-3. 리뷰 구조 (중요)

식당 1개 = 고유 ID 1개
리뷰는 개별 ID — 한 식당에 여러 사용자의 리뷰가 독립적으로 달림.
댓글은 리뷰에 대한 댓글 — 식당 자체가 아니라, 특정 리뷰에 대한 반응/토론.

구조: 식당(1) → 리뷰(N) → 댓글(N per 리뷰)

5-4. 프로필 & 가시성

항목설명
개인 프로필닉네임, 경험치/레벨, 주요 활동 지역, 취향 요약. 숨김 가능.
소속 버블 표시리뷰어 클릭 시 소속 버블 확인 가능. 숨김 가능.
익명 옵션리뷰/반응 시 익명 요청 가능 (버블 내에서만).

5-5. 버블 페이지

  • 버블 내 공유된 전체 식당/와인 목록 열람
  • 소팅: 최신순, 평점순, 리뷰수순, 찜 수순
  • 필터: 지역, 장르, 상황, 평점 범위
  • 재미 요소 / 통계:
    • 누가 가장 많이 공유했는지 (공유 랭킹)
    • 어떤 식당이 리뷰가 가장 많은지
    • 누구의 리뷰가 좋아요/댓글이 가장 많은지
    • 버블 전체 방문 트렌드 (이번 달 핫한 동네/장르)

5-6. 팔로우 (버블 외)

  • 버블과 별도로 개인 팔로우 가능
  • 팔로우한 사람의 공개 기록이 피드에 노출
  • 취향이 비슷한 사용자 자동 추천 (Phase 3)

7. 데이터 아키텍처

7-1. Nyam이 소유하는 데이터 vs 외부 위임

Nyam 소유 (핵심 자산)

  • 사용자의 기록 (방문/음용 히스토리)
  • 평가 데이터 (총점, 상황, 세부축)
  • 상황별 취향 프로필 (Taste DNA × Scene)
  • 소셜 그래프 (팔로우, 버블 관계)
  • 리뷰/댓글 UGC
  • 리스트/큐레이션 콘텐츠
  • 경험치/활동 데이터

외부 위임 (API 활용)

  • 식당 기본 정보 — 카카오/네이버/구글 교차 취합
  • 와인 기본 정보 — 와인 DB API
  • 외부 평점 — 네이버/다음/구글 맵 평점
  • 권위 리뷰 — 미슐랭/블루리본 선정 정보

7-2. 외부 데이터 캐싱 전략

원칙: 실시간 API 호출 최소화. 사용자가 추가한 식당/와인만 캐싱.
  • 사용자가 식당/와인을 처음 추가할 때 → 외부 API 호출 → 정보 캐싱
  • 2주 간격으로 캐싱 데이터 업데이트
  • 업데이트는 일괄 처리하지 않음 — 데이터 첫 입력 시점 기준으로 2주+ 시간차를 두고 분산
  • 대기열 방식으로 API 호출 분산 (과부하 방지)
  • 모든 식당을 크롤링하지 않음 — 사용자가 추가한 것만 관리

7-3. 식당 ID 통합

과제접근
동일 식당의 카카오/네이버/구글 ID 매칭가게명 + 주소 기반 fuzzy matching → 수동 검증 보완
폐업/이전 감지2주 업데이트 시 상태 확인. 폐업 시 사용자에게 알림 + 기록은 유지(아카이브)
동명 식당 구분주소(지역) 기반 구분. 사용자 선택 시 지도 위치로 확인 유도

8. 경험치 & 게이미피케이션

8-1. 경험치 체계

지역별 + 와인 종류별 독립 경험치
"을지로 Lv.7" / "내추럴 와인 Lv.4" — 활동 영역마다 별도 레벨.

경험치 획득 기준

활동경험치조건
식당/와인 기록 추가기본해당 지역/종류에 경험치 부여
상세 리뷰 작성추가한줄평 이상, 세부 평가 포함 시 가산
사진 포함추가사진이 있는 기록
버블 공유추가기록을 버블에 공유
리뷰가 좋아요/댓글 받음추가커뮤니티 가치 기여
재방문 기록추가같은 곳 재방문 기록 시 약간의 가산
동네 인증보너스해당 지역 인증 시 일회성 보너스

8-2. 경험치의 의미

  • 신뢰 지표 — "을지로 Lv.7"인 사람의 을지로 맛집 추천은 무게가 다름
  • 프로필 정체성 — 나의 식도락 전문 영역이 시각적으로 드러남
  • 버블 내 차등 — 경험치 높은 멤버의 리뷰가 상위 노출 (옵션)
  • 기록 동기 — 레벨업이 기록을 계속하게 만드는 동기

9. 프라이버시 & 위치 정보 정책

핵심 원칙: 위치 정보는 수집하되, 절대 외부 활용하지 않는다.
항목정책
위치 수집 목적지역 식당 리스트 제공, 새 지역 푸시 알림, 동네 인증 — 이것만.
수동 선택 디폴트위치 권한 없어도 사용 가능. 지역을 수동으로 선택하는 것이 기본.
백그라운드 위치사용자가 명시적으로 켜야만 동작. 기본 OFF.
배터리/데이터 최소화geofencing 방식으로 지속 GPS 추적 없이 영역 진입만 감지. 배터리/데이터 영향 최소화.
데이터 비공유위치 데이터는 제3자에게 절대 공유/판매하지 않음. 개인 식별 불가능한 통계만 내부 서비스 개선에 사용.
삭제 권한사용자가 언제든 위치 기록 삭제 가능.

11. 리스크 & 미해결 과제

#리스크심각도대응 / 상태
R1 기록 동기 부족 높음 대응: 선택 기반 기록(마찰↓) + 경험치/레벨(게이미피케이션) + 지역 푸시(컬렉션 동기) + 재방문 구조(깊이 동기)
미해결: 첫 온보딩 이후 지속적 기록 습관 형성 검증 필요
R2 콜드스타트 완화됨 온보딩 리스트 체크로 첫날부터 10~30개 기록 확보. Import 기능 추가 예정.
단, 버블/소셜의 콜드스타트는 별도 — 사용자 확보 전략 필요
R3 외부 DB 의존 중간 카카오/네이버/구글 교차로 단일 의존 회피. 캐싱 + 분산 업데이트.
API 정책 변경 리스크 존재. 대체 소스 확보 계획 필요.
R4 네이버/카카오 직접 진출 높음 대기업이 못하는 것에 집중: 폐쇄형 신뢰 네트워크(버블), 상황별 추천, 커뮤니티 문화.
"기능"이 아니라 "문화"가 해자(moat).
R5 Nyam 평가 툴 UX 중간 미해결: "재미있고 빠르면서 풍부한 정보를 수집하는" 평가 UI 구체 설계 필요. DESIGN_SPEC에서 프로토타이핑.
R6 와인 DB 확보 중간 식당은 카카오/네이버로 해결 가능하나, 와인 DB는 별도 소스 필요.
후보: Wine-Searcher API, 자체 크롤링, 사용자 입력 기반 축적
R7 네이티브 앱 필요성 중간 위치 푸시, 백그라운드 geofencing, 사진 EXIF → 네이티브 or PWA 결정 필요.
MVP는 웹앱으로 시작, 검증 후 네이티브 전환 검토.
