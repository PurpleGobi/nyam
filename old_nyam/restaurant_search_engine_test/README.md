# Restaurant Search Engine Test

식당 검색 엔진 파이프라인 비교 테스트.
"LLM 추천 → 검증" 방식의 환각 문제를 해결하기 위해
"실제 데이터 수집 → LLM 평가" 방식을 테스트했다.

**결과: Pipeline F가 채택되어 프로덕션에 적용됨** (`src/app/api/discover/route.ts`)

## 파이프라인 비교

| 파이프라인 | 방식 | 장점 | 단점 | 상태 |
|-----------|------|------|------|------|
| A (구) | LLM 추천 → 카카오 검증 | 다양한 추천 | 환각 80%+, 느림 | 폐기 |
| B | 카카오 검색 → LLM 평가 | 실존 보장 | 카카오 검색 범위 제한 | 실험 |
| C | 멀티소스 수집 → LLM 평가 | 풍성한 후보 | API 비용 | 실험 |
| D | 멀티소스 + 외부 시그널 → LLM 평가 | 가장 정확 | 가장 복잡 | 실험 |
| E | LLM 키워드 생성 + 체인 필터 + 강화 프롬프트 | 스마트 검색어 | LLM 2회 호출 | 실험 |
| **F (현행)** | **템플릿 키워드 + 그리드 검색 + 체인 필터 + 사전 랭킹 + LLM 평가** | **비용 최적, 환각 0%** | - | **프로덕션** |

## 현행 파이프라인 (Pipeline F)

```
1. 템플릿 키워드 생성 (LLM 불필요, 비용 0)
   └── 씬/장르별 사전 정의된 검색 키워드

2. 키워드 검색 + 그리드 카테고리 검색
   ├── 카카오맵 키워드 검색 (씬 맞춤)
   ├── 카카오맵 그리드 카테고리 검색 (교차 검증)
   └── 네이버 지역 검색 (교차 출처)

3. 중복 제거 + 체인/패스트푸드 필터링
   └── 블랙리스트 기반 자동 제거

4. 장르 분류 + 사전 랭킹 (Top 20 선별)
   └── 키워드 히트, 복수 출처, 리뷰/평점, 거리, 씬 매칭

5. LLM 평가 (Gemini 2.5 Flash, 1회만)
   ├── 씬별 가중치 적용
   ├── 감점 규칙 (체인, 씬 부적합)
   └── 최종 점수 + 추천 이유 생성
```

## 실행

```bash
cd restaurant_search_engine_test
pip install -r requirements.txt
cp .env.example .env  # API 키 설정

# 개별 파이프라인 테스트
python test_pipeline_b.py  # 카카오 단독 → LLM 평가
python test_pipeline_e.py  # LLM 키워드 + 체인 필터 + 강화 프롬프트
python test_pipeline_f.py  # 현행: 템플릿 키워드 + 그리드 + 체인 필터

# 전체 비교 (B, E, F)
python compare_all.py [test_index]
# test_index: 0=강남혼밥, 1=성수데이트, 2=종로라멘, 3=홍대친구, 4=여의도비즈니스
```
