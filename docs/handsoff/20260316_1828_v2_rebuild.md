- **기록 작성 플로우**: 사진 촬영 → AI 인식 → 기록 저장 전체 미구현
- **Taste DNA 재계산 로직**: DB 트리거가 아닌 application 레벨 (기록 CRUD 시 비동기 재계산)
- **Experience Atlas XP 적립**: 기록 생성 시 3축 XP 갱신 로직
- **실시간 데이터 연결**: 컨테이너 ↔ repository hook 연결
- **.env 파일**: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 필요
- **이미지 업로드**: Supabase Storage 버킷 미생성

## 3-1. 기록 작성 플로우 상세 설계

PRD 섹션 8-2 Flow A 기반. 냠의 핵심 루프이자 Taste DNA + Experience Atlas의 데이터 원천.

### 사용자 플로우 (7단계)

```
Step 1: 진입
  중앙 FAB(+) 탭 → /record 페이지

Step 2: 사진
  카메라 촬영 / 갤러리 선택 (다중 가능)
  → Supabase Storage 업로드 → record_photos 저장
  → (병렬) AI 음식 인식 호출

Step 3: AI 자동 분류
  사진 + GPS에서 record_type 자동 판별:
  ├── 음식 사진 + GPS(외출) → 🍽️ 식당
  ├── 와인 라벨/병 감지    → 🍷 와인
  └── 음식 사진 + GPS(집)  → 🍳 직접만든 요리
  사용자가 수동 전환도 가능 (탭 1번)

Step 4: 식당 매칭 (식당 타입만)
  위치 기반 근처 식당 후보 제시 (Kakao Local API)
  + AI 인식 메뉴명으로 후보 축소
  → 사용자 1탭 확인 또는 수동 검색/선택
  → restaurants 테이블에 없으면 자동 생성

Step 5: 평가 (선택)
  record_type에 따라 평가 축 분기:
  ├── 식당: 맛·가성비·서비스·분위기·청결·양 (6축, 1-5)
  ├── 와인: 향·바디감·산미·여운·밸런스·가성비 (6축, 1-5)
  └── 홈쿡: 맛·난이도·소요시간·재현성 (4축, 1-5)
  → rating_overall 자동 산출 (DB 트리거 이미 구현됨)

Step 6: 태그 + 코멘트 (선택)
  상황 태그 (tags[]): 데이트, 혼밥, 점심... (SITUATIONS 상수)
  맛 태그 (flavor_tags[]): AI 자동 제안 + 사용자 선택 (FLAVOR_TAGS 상수)
  질감 태그 (texture_tags[]): (TEXTURE_TAGS 상수)
  분위기 태그 (atmosphere_tags[]): 식당만 (ATMOSPHERE_TAGS 상수)
  한 줄 코멘트 (comment)
  공개 범위 (visibility): private | group | public

Step 7: 저장 + 즉시 피드백
  records INSERT → 트리거가 completeness_score, rating_overall 자동 산출
  → 후처리 (application 레벨):
    ├── Taste DNA 재계산 (P1에서 구현)
    ├── Experience Atlas 3축 XP 적립 (P1에서 구현)
    └── user_stats 갱신
  → 즉시 피드백 UI:
    "기록 완료! +5 포인트"
    "📍 을지로 +12 XP (Lv.8)"
    "🍣 일식 +12 XP (Lv.7)"
```

### 구현 파일 계획

```

domain/entities/record.ts          — FoodRecord, RecordType, Ratings 타입

# Application (새로 만들어야 함)

application/hooks/use-image-upload.ts    — Supabase Storage 업로드

# Presentation (새로 만들어야 함)

presentation/components/record/
  ├── photo-picker.tsx         — 카메라/갤러리 선택 UI
  ├── record-type-selector.tsx — 식당/와인/홈쿡 탭 전환
  ├── restaurant-matcher.tsx   — 식당 검색/매칭 UI
  ├── rating-input.tsx         — 슬라이더 또는 탭 평가 입력
  ├── tag-selector.tsx         — 맛/질감/분위기/상황 태그 선택
  ├── visibility-selector.tsx  — 공개 범위 선택

```

### AI 음식 인식 전략 (PRD 섹션 5-2)

```
비용: Gemini 무료 티어 (15 RPM, 1500 RPD)
정확도 전략: 이미지만 73% → 메뉴 컨텍스트 추가 시 90%+

파이프라인:
  사진 → Gemini Vision (음식 분류 + 메뉴명 추정)
  → 위치 기반 매장 특정 → 메뉴 DB 조회 → 후보 축소
  → 상위 2-3개 후보 제시 → 사용자 1탭 확인
  → (DB 없으면) 사용자 수동 입력 폴백
  → 사용자 수정 데이터로 정확도 개선 (human-in-the-loop)
```

### 선행 조건
- **인증 필수**: 기록 저장에 user_id 필요 → P0-1 인증 먼저
- **Supabase Storage 버킷**: `record-photos` 버킷 생성 필요
- **Gemini API 키**: .env에 `GEMINI_API_KEY` 추가 필요
- **Kakao REST API 키**: .env에 `KAKAO_REST_API_KEY` 추가 필요

### MVP 단순화 (첫 구현)
첫 구현에서는 AI 인식 없이 **수동 입력 플로우만** 먼저 완성하고, AI는 이후 추가:
1. 사진 선택 → 업로드
2. record_type 수동 선택
3. 식당 수동 검색 (Kakao API)
4. 평가 + 태그 + 코멘트
5. 저장 + 피드백

AI 인식은 이 위에 얹는 형태로 증분 추가.
