# Nyam - Product Requirements Document (PRD)

> 버전: 1.0.0 | 작성일: 2026-03-16
> 상태: Draft

---

## 1. 제품 개요

### 1-1. 한 줄 정의

**Nyam은 식당, 와인, 직접만든 요리 — 세 가지 식음료 경험을 기록하고, 즉시 활용하며, 신뢰할 수 있는 그룹과 공유하는 "맛의 Second Brain"이다.**

### 1-2. 문제 정의

기존 맛집 추천/리뷰 앱은 두 가지 근본 문제를 해결하지 못하고 있다:

#### 문제 1: 리뷰를 믿을 수 없다

| 문제 | 심각도 | 현 상황 |
|------|--------|--------|
| 리뷰 신뢰도 붕괴 | 매우 높음 | Google이 2023년 한 해 1.7억 개 가짜 리뷰 삭제 |
| 광고-추천 이해충돌 | 매우 높음 | 광고비 지불 식당이 상위 노출. "상위 노출 = 돈 많은 곳" |
| AI 가짜 리뷰 급증 | 높음 | 2026년 말 AI 리뷰가 인간 작성 리뷰를 초과할 전망 (Whitespark 2025) |
| 별점 시스템 무의미화 | 높음 | 대부분 식당이 4.0-4.5점대 집중. 변별력 소멸 |
| 리뷰 피로 | 높음 | Yelp 활성 리뷰어 수 2019년 대비 20-30% 감소 추정 |

#### 문제 2: 나에게 맞는 추천이 없다

| 문제 | 심각도 | 현 상황 |
|------|--------|--------|
| 개인화 부족 | 높음 | 모든 사용자에게 동일한 위치 기반 인기순 결과 |
| 취향 데이터 없음 | 높음 | 플랫폼이 사용자의 미각 선호를 모른다 |
| 상황별 추천 부재 | 중간 | 데이트, 혼밥, 비즈니스 — 같은 지역이라도 답이 다르다 |
| 과거 경험 활용 불가 | 중간 | "지난번 을지로에서 갔던 그 집" 을 찾을 방법이 없다 |

### 1-3. 솔루션

**개인 기록 → 즉시 활용 → 신뢰 네트워크**

핵심 인사이트: 믿을만한 리뷰와 개인화 추천을 위해선 **사용자의 식음료 경험 데이터**가 필요하다. 그러나 사람들은 "기록"을 싫어한다 (식단 앱 97%가 1주 내 이탈). 해법은 **기록하는 순간 즉시 가치를 돌려주는 것**이다.

```
기록 → 즉시 활용 (추천, 통계, 인사이트)
  ↓
축적 → Taste Graph (개인 취향 데이터)
  ↓
공유 → 버블 네트워킹 (신뢰 그룹 내 추천)
  ↓
발견 → 기록 검색 → 소유자 → 그룹 (bottom-up)
```

### 1-4. 핵심 가치 제안

> **"기록하면 똑똑해지는, 나만의 맛 비서."**
>
> 당신이 먹고 마신 모든 경험이 쌓일수록,
> 더 정확한 추천, 더 멋진 통계, 더 깊은 발견.

### 1-5. 정체성 우선순위

| 순위 | 정체성 | 설명 |
|------|--------|------|
| **1** | 개인 기록 (Second Brain) | 식당·와인·직접만든 요리를 기록하고, 통계화하고, 즉시 활용 |
| **2** | 정보 공유 & 네트워킹 | 버블형 그룹으로 경험 공유. 회사/써클/가족/친구 |
| **3** | 정보 검색 & 발견 | 개별 기록 단위로 탐색 → 소유자 → 그룹 발견 |

### 1-6. 왜 이 접근인가

| 장점 | 설명 |
|------|------|
| **기록 = 활용** | 기록하는 순간 즉시 추천·인사이트 반환. "기록 피로" 해소 |
| **데이터 소유** | 나의 취향 데이터를 나만의 자산으로 축적 |
| **신뢰 기반 추천** | 익명 별점이 아닌, 내가 아는 사람들의 실제 경험 |
| **조작 내성** | 버블 네트워크 내 리뷰 → 가짜 리뷰 동기 자체가 없음 |
| **AI 비용 제로** | 사용자의 개인 LLM 활용 + 무료 티어 AI API |
| **축적 해자** | 기록이 쌓일수록 전환 비용 증가. 네이버/카카오가 복제 불가 |

### 1-7. 참고 모델

| 모델 | 영감 포인트 | 냠에 적용 |
|------|-----------|----------|
| **Obsidian** | 기록이 연결되어 가치가 커지는 PKM | Taste Graph — 기록 간 연결과 패턴 발견 |
| **Letterboxd** | 영화 로그 → 17M 소셜 디스커버리 | 식음료 로그 → 취향 기반 발견 |
| **Untappd** | 맥주 체크인 즉시 프로필 업데이트 + 추천 | 기록 즉시 활용 UI |
| **Strava** | 운동 기록 → 클럽 100만+ | 활동 기록 → 버블 그룹 |
| **Telegram** | 소규모 그룹 → 채널 → $14억 매출 | 버블 네트워킹 → 유료 그룹 확장 |
| **Spotify Wrapped** | 행동 데이터 → 공유 가능한 인사이트 | Year in Review, 맛 통계 시각화 |

## 2. 타겟 사용자

### 2-1. 주요 페르소나

#### 페르소나 A: "기록형 미식가" 민지 (28세, 직장인)

- **행동**: 인스타에 음식 사진을 자주 올리지만, 나중에 "그때 어디서 먹었더라?" 기억 못함
- **불만**: "인스타 피드를 뒤져봐도 식당 이름이 없고, 맛이 어땠는지 기록이 없다"
- **Nyam 가치**: 사진 찍으면 AI가 자동 인식 → 식당/메뉴 매칭 → 평가 한 번이면 기록 완료. "이 근처에서 내가 좋아한 곳" 즉시 추천

#### 페르소나 B: "맛집 탐험가" 준호 (33세, 프리랜서)

- **행동**: 새로운 동네의 숨은 맛집 찾기가 취미. 와인에도 관심. 친구들에게 맛집 추천을 자주 함
- **불만**: "내가 아는 맛집 정보를 공유할 데가 없다. 인스타는 너무 피상적이고, 블로그는 귀찮다"
- **Nyam 가치**: 맛집 탐험 그룹 운영, 지역 마스터 타이틀, 와인 기록 + 취향 프로필. 기록이 곧 콘텐츠

#### 페르소나 C: "실용적 소비자" 소영 (26세, 대학원생)

- **행동**: 친구들과 식사 장소 정할 때 항상 30분 이상 고민. 네이버 리뷰 불신
- **불만**: "너무 많은 선택지, 다 비슷해 보여서 못 고르겠다. 믿을 만한 추천이 없다"
- **Nyam 가치**: 친구 그룹의 실제 경험 기반 추천. "우리 그룹에서 4명이 좋다고 한 곳"

#### 페르소나 D: "그룹 운영자" 현우 (38세, 회사 팀장)

- **행동**: 회식 장소, 거래처 접대 식당을 자주 찾아야 함. 팀원들에게 맛집 정보 공유
- **불만**: "매번 검색하기 귀찮고, 팀원들의 피드백을 모아둘 곳이 없다"
- **Nyam 가치**: "마케팅팀 맛집" 그룹으로 팀 내 맛집 DB 축적. 상황별 필터 (접대/회식/간단 점심)

### 2-2. 타겟 시장

- **1차**: 25-39세, 도시 거주 직장인/프리랜서. 주 2회+ 외식, 음식 사진 습관
- **2차**: 20-24세, Z세대. 음식 기록에 SNS적 감성을 원하는 층
- **3차**: 와인 애호가, 요리 커뮤니티
- **지역**: 서울/수도권 → 광역시 → 전국 순차 확대

### 2-3. 시장 포지셔닝

냠은 **"식단 기록 앱"이 아니라 "식음료 경험 지식 도구"**다.

| 비교 기준 | 식단 기록 앱 (MFP) | 맛집 리뷰 앱 (네이버) | **Nyam** |
|----------|-------------------|---------------------|----------|
| 핵심 목적 | 칼로리 추적 | 식당 검색 | 경험 기록 + 활용 + 공유 |
| 기록 동기 | 건강 목표 | 없음 (소비만) | 즉시 활용 (추천·통계) |
| 개인화 | 영양소 기반 | 위치 기반 | Taste Graph 기반 |
| 신뢰 모델 | N/A | 익명 별점 | 신뢰 그룹 경험 |
| 네트워크 | 없음 | 오픈 리뷰 | 버블 네트워킹 |

## 3. 핵심 컨셉

### 3-1. 기록 범위와 AI 자동 분기

냠은 **세 가지 카테고리**의 식음료 경험을 기록한다:

| 카테고리 | 설명 | AI 인식 신호 | 평가 축 |
|---------|------|------------|--------|
| **식당** | 외식 경험 (카페/디저트 포함) | 음식 사진 + GPS 있음 | 맛·가성비·서비스·분위기·청결·양 (6각) |
| **와인** | 와인 경험 (식당 와인 포함) | 와인 라벨/병 감지 | 향·바디감·산미·여운·밸런스·가성비 (6각) |
| **직접만든 요리** | 요리, 레시피 | 음식 사진 + GPS 없음 (집) | 맛·난이도·소요시간·재현성 (4각) |

**기록 플로우에서 사용자는 카테고리를 선택하지 않는다.** AI가 사진과 위치에서 자동 분류:

```
FAB 탭 → 사진 촬영
           ↓
      AI 자동 분류
      ├── 음식 사진 + GPS(외출) → 🍽️ 식당 기록
      ├── 와인 라벨/병 감지    → 🍷 와인 기록
      └── 음식 사진 + GPS(집)  → 🍳 직접만든 요리
           ↓
      (사용자가 수동 전환도 가능: 탭 1번)
```

### 3-2. 이중 데이터 시스템 — Taste DNA + Experience Atlas

냠의 데이터 시스템은 **두 개의 독립된 축**으로 사용자를 이해한다:

| 구분 | Taste DNA (취향 DNA) | Experience Atlas (경험 아틀라스) |
|------|---------------------|-------------------------------|
| **성격** | 능동적 — 사용자가 평가한 취향 데이터 | 수동적 — 기록하면 자동 누적되는 경험 메타데이터 |
| **질문** | "이 사람은 무엇을 좋아하는가?" | "이 사람은 어떤 영역을 잘 아는가?" |
| **데이터 원천** | 평가 점수, 맛 태그, 선호 패턴 | 기록의 메타데이터 (지역, 카테고리, 상황, 시간) |
| **활용** | **취향 유사도 매칭** (비슷한 사람 찾기) | **보완성 매칭** (내가 모르는 영역의 전문가 찾기) |
| **시각화** | 맛 레이더 차트, 선호 프로필 카드 | 영역 히트맵, 전문성 레벨 배지 |

```
기록 1건이 생성되면:

  ┌─ Taste DNA 업데이트 (능동적 평가 데이터)
  │   "매운 파스타를 4.5점, 분위기 좋음" → 매운맛 선호 ↑, 양식 만족도 ↑, 분위기 민감도 업데이트
  │
  └─ Experience Atlas 업데이트 (자동 누적 메타데이터)
      "을지로 이탈리안에서 데이트 저녁" → 📍을지로 +XP, 🍝양식 +XP, 💕데이트 +XP
```

#### Taste DNA — "무엇을 좋아하는가"

사용자의 **주관적 취향**을 벡터화한 개인 미각 지문. 기록 시 사용자가 능동적으로 입력한 **평가·태그·패턴**에서 산출한다.

##### 카테고리별 평가 축

| 카테고리 | 평가 축 (기록 시 입력) | 파생 취향 벡터 |
|---------|----------------------|--------------|
| **식당** | 맛·가성비·서비스·분위기·청결·양 (6축) | 맛 프로필, 가격 민감도, 분위기 선호, 서비스 기대치 |
| **와인** | 향·바디감·산미·여운·밸런스·가성비 (6축) | 바디 선호, 산미 선호, 향 계열 선호, 가격대 |
| **직접만든 요리** | 맛·난이도·소요시간·재현성 (4축) | 요리 스타일, 난이도 선호, 시간 투자 성향 |

##### 기록 시 수집하는 취향 태그 (선택형, AI 자동 제안)

기록의 평가 점수만으로는 세밀한 취향 벡터를 구성할 수 없다. 기록 시 **맛 태그**와 **분위기 태그**를 선택형으로 수집하여 Taste DNA의 정밀도를 높인다:

```
맛 태그 (flavor_tags):
  매운 · 달콤한 · 짭짤한 · 시큼한 · 감칠맛 · 담백한 · 기름진 · 고소한 · 향긋한 · 깔끔한

질감 태그 (texture_tags):
  바삭한 · 부드러운 · 쫄깃한 · 크리미한 · 아삭한 · 촉촉한

분위기 태그 (atmosphere_tags):   -- 식당 기록에만
  조용한 · 활기찬 · 캐주얼 · 포멀 · 아늑한 · 개방적 · 감성적 · 모던한
```

사진과 메뉴명에서 AI가 태그를 자동 제안하고, 사용자가 1탭으로 확인/수정한다. 이 태그들이 누적되어 Taste DNA 벡터를 형성한다.

##### 공통 맛 프로필 (Taste DNA 핵심 벡터)

```
맛 선호 (0.0 ~ 1.0, 기록 누적에서 자동 산출):
  매운맛 ─────── 0.0 (기피) ← → 1.0 (즐김)
  단맛   ─────── 0.0 ← → 1.0
  짠맛   ─────── 0.0 ← → 1.0
  신맛   ─────── 0.0 ← → 1.0
  감칠맛 ─────── 0.0 ← → 1.0
  기름진/담백 ── 0.0 (담백) ← → 1.0 (리치)

질감 선호 (0.0 ~ 1.0):
  바삭한 ─────── 0.0 ← → 1.0
  부드러운 ───── 0.0 ← → 1.0
  쫄깃한 ─────── 0.0 ← → 1.0

분위기 선호 (양극 스케일 -1.0 ~ 1.0):
  조용한 ← → 활기찬
  캐주얼 ← → 포멀
  아늑한 ← → 개방적

가격-가치 성향:
  가성비 민감도   0.0 (가격 무관) ← → 1.0 (가성비 중시)
  선호 가격 범위  [min, max] 원
```

##### 와인 특화 취향 벡터

```
구조감 (양극 스케일 0.0 ~ 1.0):
  바디감     ─── 라이트 ← → 풀바디
  산미       ─── 낮음 ← → 높음
  타닌       ─── 부드러운 ← → 강한
  당도       ─── 드라이 ← → 스위트

향 계열 선호 (0.0 ~ 1.0, 각각 독립):
  과일향 · 꽃향 · 스파이시 · 오크/바닐라 · 미네랄 · 허브/풀

선호 품종 상위 N개  (기록 빈도 + 높은 평가에서 자동 산출)
선호 산지 상위 N개
선호 가격대 [min, max]
```

##### 직접만든 요리 특화

```
선호 조리법 (빈도 기반 순위):
  구이 · 볶음 · 찜 · 튀김 · 생식 · 베이킹 · 발효

난이도 선호:     간편(0.0) ← → 도전적(1.0)
시간 투자 성향:  빠른(0.0) ← → 여유(1.0)
선호 요리 장르 상위 N개
```

##### Taste DNA 산출 방식

```
트리거: 기록 생성/수정/삭제 시
방식:   최근 100건 기록 기반 가중 이동 평균 (최신 기록 가중치 ↑)

맛 프로필:
  각 맛 태그가 포함된 기록 비율 + 해당 기록의 평가 점수 가중
  예) 매운맛 = Σ(매운 태그 기록의 맛 점수) / Σ(전체 기록의 맛 점수)

질감/분위기:
  태그 빈도 기반 정규화

가격 민감도:
  가성비 평가 점수의 분산 분석 — 가격 대비 평가가 크게 변동하면 민감도 ↑

맛 성격 유형 (MBTI식 4자 코드):
  축1: 모험형(A) vs 안정형(S)  — 신규 식당 비율
  축2: 미식형(E) vs 실속형(P)  — 평균 가격대 + 평가 까다로움
  축3: 소셜형(S) vs 솔로형(I)  — 그룹 공유 비율 + 동행자 수
  축4: 자주형(F) vs 가끔형(R)  — 주간 기록 빈도
```

##### Taste DNA 활용

- "이 근처에서 당신이 좋아할 곳" — DNA 기반 미방문 식당 추천
- "당신과 맛 취향이 87% 비슷한 사람" — DNA 벡터 코사인 유사도
- "지난번 마셨던 그 와인과 비슷한 것" — 와인 DNA 매칭
- "이번 달 외식 리포트" — 소비 패턴 시각화
- 맛 성격 유형 공유 카드 — "Bold Flavor Seeker (AESI)"

#### Experience Atlas — "어떤 영역을 잘 아는가"

사용자의 **객관적 경험 범위와 깊이**를 구조화한 전문성 지도. 기록의 메타데이터에서 **자동 산출**되며, 사용자가 별도로 입력하는 것이 없다.

##### 3축 영역 시스템

| 축 | 기준 | 측정 지표 | 예시 |
|---|------|----------|------|
| **📍 지역** (WHERE) | 어디서 먹었는가 | 기록 수, 고유 식당 수, 하위 동네 커버리지, 최근성 | 을지로 Lv.8, 강남 Lv.5 |
| **🍴 장르** (WHAT) | 무엇을 먹었는가 | 기록 수, 하위 카테고리 다양성, 평가 일관성 | 일식 Lv.7, 와인 Lv.6 |
| **🎭 씬** (WHY) | 어떤 상황이었는가 | 기록 수, 상황 내 다양성, 동행자 다양성 | 데이트 Lv.5, 혼밥 Lv.4 |

##### 영역 레벨 산출 공식

```
영역 Lv. = f(Volume, Diversity, Recency, Consistency)

Volume (40%):     해당 영역 기록 수 (기본 경험량)
Diversity (25%):  하위 카테고리 커버리지 (폭넓은 경험)
  - 📍 지역: 해당 동네의 서로 다른 식당 수 / 동네 내 알려진 식당 수
  - 🍴 장르: 하위 카테고리 수 (일식 → 라멘/스시/이자카야/오마카세/돈까스/소바...)
  - 🎭 씬: 해당 상황의 서로 다른 식당 유형 (데이트 → 캐주얼/파인/분위기/바...)
Recency (20%):    최근 활동도 가중 (최근 3개월 기록 가중치 2배)
Consistency (15%): 기록 품질 일관성 (평가 완성도, 코멘트 유무, 사진 품질)

레벨 구간:
  Lv.1-2  입문     (1-4건)     — 시작 단계
  Lv.3-4  경험자   (5-14건)    — 기본 지식 확보
  Lv.5-6  전문가   (15-29건)   — 프로필 노출 시작
  Lv.7-8  마스터   (30-59건)   — 추천 신뢰도 높음
  Lv.9-10 전설     (60건+)     — 해당 영역 최고 권위
```

XP는 기록 품질에 따라 가변 (사진+평가+코멘트 완성도 높을수록 XP 보너스).

##### Experience Atlas 활용

- **프로필 표시**: "📍 을지로 Lv.8 · 🍣 일식 Lv.7 · 💕 데이트 Lv.5" (Lv.5+ 영역만 최대 3개)
- **그룹 참여 조건**: "📍 을지로 Lv.3+" 또는 "🍷 와인 Lv.5+"
- **보완성 매칭**: "당신은 일식에 강하고, 이 사람은 중식에 강합니다" → 정보 다양성을 위한 추천
- **그룹 구성 분석**: "이 그룹은 강남 전문가는 많지만 홍대 전문가가 부족합니다"
- **검색 신뢰도 가중**: 영역 레벨이 높은 사용자의 기록이 검색에서 상위 노출

#### 이중 매칭 시스템

두 시스템을 결합하여 **두 가지 서로 다른 매칭**을 수행한다:

```
1. 취향 유사도 (Taste DNA → similarity)
   두 사용자의 taste_dna 벡터 간 코사인 유사도 × 100
   → "맛 취향 궁합: 87%"
   → 비슷한 취향의 사람 추천, 그룹 내 맛 동료 찾기
   용도: "나와 비슷한 사람"

2. 경험 보완성 (Experience Atlas → complementarity)
   사용자 A에게 약하고(Lv.3 이하) B에게 강한(Lv.5 이상) 영역의 레벨 차이 합산
   + B에게 약하고 A에게 강한 영역의 레벨 차이 합산
   → 정규화하여 0-100
   → "정보 보완 궁합: 72%"
   용도: "나에게 없는 정보를 가진 사람"

3. 종합 궁합 (상황에 따라 가중치 가변)
   종합 = similarity × α + complementarity × β
   - "맛 동료 찾기" 시: α=0.7, β=0.3
   - "그룹 영입 추천" 시: α=0.3, β=0.7
   - "그룹 다양성 최적화" 시: α=0.2, β=0.8
```

**설계 원칙**: Taste DNA는 "공감"을 만들고, Experience Atlas는 "발견"을 만든다. 두 축이 독립적으로 작동하면서도 종합 궁합에서 결합될 때 최대 가치를 발휘한다.

### 3-3. 버블 네트워킹 — 신뢰 기반 그룹

텔레그램식 소규모 신뢰 그룹. 초기 미슐랭 가이드처럼 내부 서클에서 시작하여 확장.

#### 그룹 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **비공개** | 초대만 가능, 외부 노출 없음 | 가족 맛집, 친한 친구 |
| **공개** | 누구나 열람 + 참여 가능 | 을지로 맛집 탐험대 |
| **보기 전용** | 누구나 열람, 참여는 승인제 | 와인 전문가 모임 |
| **유료** | 열람에 결제 필요 | 미슐랭급 큐레이션 그룹 |

#### 그룹 참여 문턱

개인의 **활동 데이터를 분석하여 identity와 레벨**을 산출하고, 이를 기반으로 그룹 운영자가 참여 조건을 설정할 수 있다:

- 기록 수 (최소 N개 이상)
- 기록 빈도 (주 N회 이상 활동)
- 특정 카테고리 기록 수 (와인 N개 이상)
- 냠 레벨 (Lv.15 이상만)

#### 발견 플로우 (Bottom-up)

```
개별 기록 발견 (검색/추천/탐색)
  → 기록이 마음에 듦
  → 기록 소유자 프로필 확인
  → 소유자가 속한 그룹 목록 확인
  → 그룹 열람/참여 시도
```

그룹을 직접 검색하는 것이 아니라, **좋은 기록 → 좋은 사람 → 좋은 그룹** 순으로 자연스럽게 발견한다.

### 3-4. 레벨 시스템

> 상세 설계: `docs/research/09 scoring-expertise-system.html` 참조

#### 종합 레벨: 냠 Lv.

사용자가 보는 것은 **"냠 Lv." 숫자 하나**. 뒷단에서 5대 요소(Activity, Quality, Consistency, Community, Calibration)의 복합 점수로 산출.

| Lv. | 해금 기능 |
|-----|----------|
| 1+ | 기록, 검색, 그룹 1개 참여 |
| 5+ | Taste DNA 상세 보기, 그룹 3개, 기본 통계 |
| 15+ | 그룹 무제한, 나의 Top N, 상세 통계 |
| 30+ | 그룹 생성 (무료), Wrapped, 공유 카드 |
| 50+ | 유료 그룹 생성 |

#### 영역 레벨 (Experience Atlas 연동)

기록 1건은 **Experience Atlas의 3개 축에 동시에 XP를 적립**한다. 영역 레벨 산출 공식은 섹션 3-2의 Experience Atlas 참조.

```
기록: "을지로 일식집에서 데이트 저녁"
  → Taste DNA 업데이트 (맛 태그, 평가 반영)
  → 📍 을지로  +12 XP  (Experience Atlas - 지역)
  → 🍣 일식    +12 XP  (Experience Atlas - 장르)
  → 💕 데이트  +12 XP  (Experience Atlas - 씬)
  → 종합 냠 Lv. +5 포인트
```

**프로필 표시**: Lv.5+ 달성한 영역만 대표 프로필에 최대 3개 노출 + 맛 성격 유형 코드.

```
👤 민지
냠 Lv.23 · AESI (모험적 미식 소셜 식도락가)
📍 을지로 Lv.8 · 🍣 일식 Lv.7 · 💕 데이트 Lv.5
```

#### 기록 시 즉각 피드백

```
기록 완료!  +5 포인트
  🧬 Taste DNA: 매운맛 선호 ↑  양식 만족도 ↑
  📍 을지로  +12 XP  (Lv.8)
  🍣 일식    +12 XP  (Lv.7)
  💕 데이트  +12 XP  → Lv.5 달성! 🎉
```

#### 버블 문턱에서의 활용

그룹 운영자가 **종합 레벨 + Experience Atlas 영역 레벨**을 조합하여 참여 조건 설정:

- "냠 Lv.15+"
- "📍 을지로 Lv.3+ (경험자 이상)"
- "🍷 와인 Lv.5+ (전문가 이상)"

#### 그룹 랭킹

그룹의 활동성과 품질을 데이터로 산출:

- 멤버 활동 빈도
- 기록 품질 평균
- 성장률 (신규 멤버, 기록 증가)
- 외부 인용 (그룹 기록이 다른 사용자에게 저장된 횟수)
- **멤버 Experience Atlas 다양성** — 그룹 내 영역 커버리지 분석

## 5. AI 통합 전략

### 5-1. 원칙

**비용 제로**: 앱이 LLM API 비용을 부담하지 않는다.

AI는 세 가지 역할로 통합한다:

| 역할 | 설명 | 비용 구조 |
|------|------|----------|
| **인식 보조** | 음식 사진 → 메뉴/식당 매칭 | 무료 AI API (Gemini) 또는 경량 모델 |
| **기록 보조** | 코멘트 작성 도움, 기록 구조화 | 사용자 개인 LLM 활용 |
| **추천 비서** | Taste Graph 기반 맞춤 추천 | Phase 2: BYOK 또는 프롬프트 브릿지 |

### 5-2. 음식 인식 — 멀티모달 교차 검증

이미지만 사용하면 혼합 요리 정확도 73%에 불과하다 (ISIA Food-200). 메뉴 컨텍스트를 추가하면 90%+ 달성 가능 (Menu-Match 연구: 25.5% → 64.8%).

```
사진 촬영
  → Vision AI (Gemini 무료 티어) 음식 분류
  → 위치 기반 매장 특정 → 메뉴 DB 조회 → 후보 축소
  → 상위 2-3개 후보 제시 → 사용자 1탭 확인
  → (DB 없으면) 메뉴판 OCR 폴백 → 후보 축소
  → 사용자 수정 데이터로 모델 개선 (human-in-the-loop)
```

### 5-3. AI 기록 보조 — "쓰기 쉽게, 대신 쓰지 않게"

AI가 리뷰를 **대신 쓰는 것이 아니라 쓰기 쉽게 돕는 것**. (AI 전체 생성 시 진정성 인식 38%로 급락 — Liu et al., 2024)

| 보조 방식 | 예시 | AI 개입 수준 |
|----------|------|------------|
| 질문 프롬프트 | "가장 기억에 남는 메뉴는?" | 낮음 (안전) |
| 태그 서제스트 | 사진에서 "#파스타 #을지로 #데이트" 자동 제안 | 낮음 (안전) |
| 문장 다듬기 | 사용자 키워드 → 자연스러운 문장으로 개선 | 중간 (허용) |
| 기록 요약 | 그룹 내 여러 기록을 요약 (원문 항상 노출) | 중간 (허용) |

### 5-4. AI 추천 — 비용 제로 전략

| 우선순위 | 방식 | 비용 | Seamless | 설명 |
|---------|------|------|---------|------|
| 1 | **Gemini 무료 티어** | 제로 | 높음 | 간단 추천/분석에 Gemini API 무료 한도 활용 |
| 2 | **프롬프트 브릿지** | 제로 | 중간 | Taste Graph + 상황 → 프롬프트 자동 생성 → 공유 시트로 AI 앱 전달 |
| 3 | **BYOK** | 사용자 | 높음 | 파워유저가 자기 API 키 입력 → 앱 내 직접 호출 |

### 5-5. 앱이 하지 않는 것

- LLM API 비용을 앱이 부담하지 않음
- AI가 리뷰를 전체 생성하지 않음 (보조만)
- AI 응답의 정확성을 보증하지 않음
- 사용자의 AI 계정 정보를 수집하지 않음
- 식당으로부터 광고비를 수취하지 않음

## 6. 심리적 참여 설계

### 6-1. "기록 = 활용" — 기록 피로 해소

기록 앱의 97% 이탈은 **보상 지연**에서 온다. 냠은 기록하는 순간 즉시 가치를 돌려준다:

| 기록 행위 | 즉시 활용 가치 | 참고 모델 |
|----------|-------------|----------|
| 식당 방문 기록 | "이 근처에서 당신이 좋아한 곳" 즉시 표시 | Google Timeline + Obsidian |
| 와인/음료 평가 | 비슷한 와인 추천 + 취향 프로필 업데이트 | Vivino 스캔 → 추천 |
| 메뉴 사진 촬영 | AI가 메뉴 인식 → 자동 기록 + 비슷한 메뉴 추천 | Google Lens |
| 5건 이상 축적 | "당신의 맛 지도" 시각화 | Spotify Wrapped |
| 그룹에 공유 | 그룹 멤버 반응 + 배지 획득 | Untappd + Strava 쿠도스 |

### 6-2. 적용 심리 원리

| 원리 | 적용 방식 | 기대 효과 |
|------|----------|----------|
| **IKEA 효과** | 직접 기록한 경험 → "내가 쌓은 데이터" 소유감 | 앱 충성도 ↑, 전환 비용 ↑ |
| **사회적 증거** | "이 식당을 내 그룹 4명이 추천" (버블 내 신뢰) | 익명 별점 대비 신뢰도 ↑ |
| **가변 보상** | 기록 시 새로운 인사이트 발견, 배지 획득 | 반복 기록 행동 유도 |
| **목표 그래디언트** | 레벨업 진행 바 "Lv.16까지 3건!" | 기록 완주율 ↑ |
| **자기결정성** | 능동적 기록 (자율성), 취향 분석 (유능감), 그룹 (관계성) | 내재적 동기 |
| **축적 효과** | 기록 50개+ → 전환 비용 중간, 100개+ → 높음 | 이탈 방지 |
| **Dunbar's Number** | 5-15인 "맛 동료" 그룹 = 심리적 최적점 | 참여율 30-50% (오픈 플랫폼 1%) |

### 6-3. 사용자 참여 퍼널

```
Stage 1: Lv.1-5 (기록 습관 형성)
  행동: 첫 기록, 기록 습관 잡기
  목표: "기록 = 즉시 활용" 첫 경험
  넛지: "이 근처에서 좋아한 곳이 있어요"

    ↓

Stage 2: Lv.5-15 (네트워킹 시작)
  행동: 꾸준한 기록 + 그룹 참여/공유
  목표: 신뢰 네트워크 형성
  넛지: "친구를 초대하면 맛 궁합을 알 수 있어요"

    ↓

Stage 3: Lv.15-30 (영역 전문화)
  행동: 깊이 있는 평가 + 영역 레벨 성장 + 그룹 운영
  목표: 특정 영역 전문가 정체성
  넛지: "📍 을지로 Lv.5 달성! 프로필에 표시됩니다"

    ↓

Stage 4: Lv.30+ (커뮤니티 리더)
  행동: 그룹 생성/운영, 유료 그룹
  목표: 사회적 자본 + 수익
  넛지: "🍷 와인 Lv.8 — 와인 그룹을 만들어보세요"
```

### 6-4. 리텐션 루프

```
트리거 → "점심 뭐 먹지?" (시간 기반 / 위치 기반)
행동   → 근처 추천 확인 또는 기록 시작
보상   → 즉시 활용 (추천, 통계 업데이트), 그룹 반응
투자   → 기록 이력, Taste Graph, 그룹 관계 (이탈 비용)
```

| 주기 | 넛지 |
|------|------|
| 매일 | "이 근처에서 좋아한 곳" 위치 기반 서피싱 |
| 매주 | "이번 주 맛 리포트" (기록 요약, 새로운 발견) |
| 매월 | "이달의 맛 지도" + 그룹 활동 리포트 |
| 매년 | Year in Review (Wrapped) — 공유 가능한 연간 리포트 |

### 8-2. 핵심 사용자 플로우

#### Flow A: 기록하기 (핵심 플로우)

```

2. 사진 촬영/선택
3. AI가 음식 인식 → 메뉴 후보 2-3개 제시

7. 공유 대상 선택 (비공개 / 특정 그룹 / 공개)
8. "기록 완료" → Taste Graph 업데이트 애니메이션
9. "이 근처에서 당신이 좋아한 곳" 즉시 표시
```

#### Flow B: 근처 추천 받기

```
1. 홈에서 "이 근처 추천" 카드 확인
2. 과거 기록 + Taste Graph 기반 추천 목록
3. 식당 탭 → 내 기록 + 그룹 멤버 기록 확인
4. (선택) AI 추천 비서에게 상세 질문
```

#### Flow C: 기록 발견 → 그룹 참여

```
1. 발견 탭에서 공개 기록 탐색
2. 마음에 드는 기록 발견 → 저장
3. 작성자 프로필 탭 → 레벨, 기록 수, 소속 그룹 확인
4. 관심 그룹 탭 → 그룹 정보 확인
5. 참여 조건 충족 시 → 참여 요청
```

## 10. 데이터 아키텍처

### 10-1. 데이터 레이어 개요

냠의 데이터는 3개 레이어로 구분된다:

| 레이어 | 성격 | 예시 | 갱신 빈도 |
|--------|------|------|----------|
| **원본 데이터** | 사용자가 직접 생성한 1차 데이터 | 기록, 사진, 평가, 코멘트, 일기 | 실시간 (사용자 행동 시) |
| **파생 데이터** | 원본을 집계·분석하여 산출 | Taste Graph, 레벨, 지역 특화, 그룹 품질 | 배치 (기록 생성 시 or 주기적) |
| **외부 데이터** | 외부 API에서 가져온 데이터 | 식당 정보, 메뉴, 위치 | 캐싱 (24h ~ 7d) |

### 10-4. 파생 데이터 산출 로직

#### 레벨 산출

```
XP 획득 규칙:
  기록 생성           +10 XP
  사진 포함           +5 XP
  평가 입력           +3 XP
  코멘트 입력         +3 XP
  일기 작성           +5 XP
  그룹에 공유         +5 XP
  좋아요 받음 (건당)  +1 XP
  연속 7일 기록 보너스 +15 XP

레벨 기준:
  냠 Lv. = f(Activity, Quality, Consistency, Community, Calibration)

  등급 매핑:
  Lv. 1-5    기록 3-10건 정도. 기본 기능
  Lv. 5-15   기록 ~30건. Taste DNA, 그룹 3개
  Lv. 15-30  기록 ~80건. 그룹 무제한, 상세 통계
  Lv. 30-50  기록 ~200건. 그룹 생성, Wrapped
  Lv. 50+    전체 사용자의 1-3%만 도달. 유료 그룹 생성
```

#### Taste DNA 재계산

```
트리거: 기록 생성/수정/삭제 시 (비동기)
방식: 최근 100건 기록 기반 가중 이동 평균 (최신 기록 가중치 ↑)

맛 선호 벡터:
  각 맛 태그(flavor_tags)가 포함된 기록에서 해당 맛 점수를 가중 평균
  예) flavor_spicy = Σ(매운 태그 기록 × rating_taste × recency_weight) / Σ(전체 기록 × recency_weight)

질감 벡터:
  질감 태그(texture_tags) 빈도 기반 정규화 (0.0 ~ 1.0)

분위기 벡터:
  분위기 태그(atmosphere_tags) 빈도에서 양극 스케일 산출
  예) 조용한(3회) vs 활기찬(7회) → atmosphere_noise = +0.4

카테고리 선호도:
  각 카테고리 기록 수 / 전체 기록 수 (가중 평균, 최근 기록 우선)

가격 민감도:
  가성비 평가(rating_value)의 분산 분석
  가격대별 평가 점수 회귀 분석 → 가격 변동에 대한 평가 민감도

모험 지수:
  최근 30일 내 처음 방문한 식당 비율

맛 성격 유형 (4축 → 4자 코드):
  축1: 모험형(A) vs 안정형(S)  — 신규 식당 비율 (50% 컷)
  축2: 미식형(E) vs 실속형(P)  — 평균 가격대 상위 50% + 평가 분산 높음
  축3: 소셜형(S) vs 솔로형(I)  — 그룹 공유 비율 (30% 컷) + 동행자 태그 빈도
  축4: 자주형(F) vs 가끔형(R)  — 주간 기록 빈도 (2회 컷)

와인 특화 (taste_dna_wine):
  와인 기록의 평가 축에서 직접 산출 (향/바디감/산미/여운 → 구조감 벡터)
  향 태그에서 향 계열 선호 산출
  높은 평가(4.0+) 와인의 품종/산지 빈도 → 선호 목록

직접만든 요리 특화 (taste_dna_homecook):
  난이도/소요시간 평가에서 성향 산출
  메뉴명 + 카테고리에서 조리법 분류 → 빈도 기반 선호도
```

#### Experience Atlas 재계산

```
트리거: 기록 생성/삭제 시 (해당 영역만 incremental update)

XP 획득:
  기록 생성           +10 XP (기본)
  사진 포함           +3 XP
  평가 완성 (전축)    +3 XP
  코멘트 포함         +2 XP
  새로운 식당 (첫 방문) +5 XP (보너스)

영역 Lv. 산출:
  raw_score = volume_score × 0.40
            + diversity_score × 0.25
            + recency_score × 0.20
            + consistency_score × 0.15

  volume_score:      기록 수 기반 (1건=5, 5건=25, 15건=50, 30건=75, 60건+=100)
  diversity_score:   하위 카테고리/식당/동네 커버리지 비율 × 100
  recency_score:     최근 3개월 기록 비율 × 2 (가중), 상한 100
  consistency_score: 평균 기록 완성도 (사진+평가+코멘트 유무)

  level = floor(raw_score / 10)  -- 0-100 → Lv.1-10
```

#### 그룹 품질 점수 산출

```
overall_score = (
  activity_score × 0.30      -- 주간 기록수 / 멤버수
  + quality_score × 0.25     -- 멤버 평균 기록 완성도
  + diversity_score × 0.15   -- 멤버 Experience Atlas 다양성
  + external_citation × 0.20 -- 외부 인용 (정규화)
  + growth_rate × 0.10       -- 4주 성장률
)
갱신: 1시간마다 배치
```

#### 이중 매칭 궁합 산출

```
1. 취향 유사도 (Taste DNA 기반):
   벡터 A = [flavor_spicy, flavor_sweet, ..., texture_crispy, ..., atmosphere_noise, ...]
   벡터 B = 동일 구조
   similarity = cosine_similarity(A, B) × 100
   → "맛 취향 궁합: 87%"

2. 경험 보완성 (Experience Atlas 기반):
   A의 모든 영역 Lv.과 B의 모든 영역 Lv.을 비교
   complementarity = Σ(max(0, B_level - A_level)) + Σ(max(0, A_level - B_level))
                     / (2 × max_possible_diff)  × 100
   → "정보 보완 궁합: 72%"
   (A에게 약하고 B에게 강한 영역이 많을수록 보완성 ↑)

3. 종합 궁합:
   combined = similarity × α + complementarity × β
   - 맛 동료 찾기:     α=0.7, β=0.3
   - 그룹 영입 추천:   α=0.3, β=0.7
   - 그룹 다양성 분석: α=0.2, β=0.8
```

## 11. 수익 모델

**핵심 원칙: 수익 모델이 추천 결과를 왜곡하지 않는다. 수익화는 커뮤니티 성숙 후.**

Discord (2년), Telegram (8년), Strava (11년) 모두 커뮤니티를 먼저 키우고 수익화했다.

### Phase 1: 무료 (0-18개월)

- 모든 기본 기능 무료 제공
- 제품-시장 적합성 검증에 전념
- PLG (Product-Led Growth) — 친구 초대, 그룹 공유로 성장 → CAC ≈ $0

### Phase 2: 프리미엄 도입 (18-36개월)

| 모델 | 설명 | 사용자 이익 충돌 |
|------|------|----------------|
| 프리미엄 통계 | 상세 Taste Graph 분석, 고급 인사이트 | 없음 |
| 무제한 그룹 | 기본 N개 이상 그룹 생성/참여 | 없음 |
| Year in Review | 프리미엄 Wrapped 리포트 | 없음 |
| 유료 그룹 수익 분배 | 유료 그룹 열람 수익의 일부를 플랫폼이 수취 | 없음 (운영자 자발적) |

### Phase 3: 확장 (36개월+)

| 모델 | 설명 | 사용자 이익 충돌 |

## 13. 리스크 및 대응

### 13-1. 치명적 리스크

| 리스크 | 대응 | 해소 수준 |
|--------|------|----------|
| 97%가 1주 내 이탈 (식단 기록 앱 패턴) | "기록 = 즉시 활용" UI. 식단 기록이 아닌 경험 기록 + 즉시 추천/인사이트 반환 | 높음 |
| 시장 규모 $1.5억으로 극소 | 시장 재정의: "식단 기록"이 아닌 "개인 지식 도구(PKM)" 시장 ($13억→$27억) | 높음 |
| 네이버/카카오 복제 위협 | 3중 해자: Taste Graph + 특화 UX 깊이 + 버블 커뮤니티 관계 | 높음 |
| 수익 모델 미검증 | 수익화 유예 + PLG. Discord/Telegram/Strava 모델 | 높음 |
| "모든 것을 하는 앱" 실패 역사 | 단일 도메인(식음료) 내 깊이. Letterboxd 모델 | 높음 |
| 네트워크 효과 부재 | 버블형 네트워킹 (Telegram/Strava 클럽 모델). 다크소셜 활용 | 높음 |

### 13-2. 심각 리스크

| 리스크 | 대응 | 해소 수준 |
|--------|------|----------|
| AI 음식 인식 73% | 멀티모달 교차 검증 (이미지+메뉴DB+위치) → 90%+ | 높음 |
| GPS 도심 20-50m 오차 | 반자동 복합 위치 확인 (GPS+WiFi+간판 사진+사용자 선택) | 중-높 |
| AI 리뷰 불신 (55%) | AI는 "생성"이 아닌 "보조"만. 버블 내 실제 경험 기반 신뢰 | 높음 |
| 위치 추적 프라이버시 | "사용 시에만" 순간 접근. 백그라운드 추적 없음 | 높음 |

### 13-3. 운영 리스크

| 리스크 | 심각도 | 대응 |
|--------|--------|------|
| 초기 콘텐츠 부족 (Cold Start) | 높음 | 시드 그룹 10-20개 직접 운영, 초대 기반 성장 |
| 유료 그룹 품질 관리 | 중간 | 활동 데이터 기반 자동 품질 지표, 신고 시스템 |
| 사진 스토리지 비용 | 중간 | 이미지 최적화, CDN 캐싱, 무료 티어 내 용량 제한 |
| 배터리 소모 우려 | 낮음 | 순간 구동 + 서버사이드 AI. 백그라운드 프로세스 없음 |

## 15. 경쟁 우위 요약

| 기존 맛집 앱 | Nyam |
|-------------|------|
| 앱이 추천한다 | 내 기록이 추천한다 |
| 별점을 믿어라 | 내 그룹을 믿어라 |
| 광고비 = 노출 | 기록 품질 = 신뢰 |
| 리뷰 써라 (피로) | 사진 + 1탭 (간편) |
| 수동적 소비자 | 능동적 기록자 |
| AI 비용 = 앱 부담 | AI 비용 = 제로 |
| 블랙박스 알고리즘 | 투명한 Taste Graph |
| 오픈 리뷰 (조작) | 버블 네트워크 (신뢰) |
| 기록해도 활용 없음 | 기록 = 즉시 활용 |
| 전환 비용 없음 | Taste Graph + 그룹 = 높은 전환 비용 |

### 3중 해자

```
해자 1: Taste Graph
  → 사용자 취향 데이터 축적. 기록 100개+ 시 대체 불가능한 개인화
  → 네이버 지도는 "어디에 갔다"는 알지만 "어떤 맛을 좋아하는지"는 모른다

해자 2: 특화 UX 깊이
  → 식음료 기록에 최적화된 수십 개의 마이크로 인터랙션
  → 범용 플랫폼이 "기능 하나"로 따라잡을 수 없는 깊이

해자 3: 버블 커뮤니티
  → 소규모 신뢰 그룹 기반의 관계
  → 이것은 제품이 아니라 관계이므로 복제 불가
```
