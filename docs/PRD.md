## 1. 제품 정의

### 1-1. 한 줄 정의

**Nyam은 식당, 와인, 직접 만든 요리 — 세 가지 식음료 경험을 기록하고, 즉시 활용하며, 신뢰할 수 있는 버블과 공유하는 "맛의 Second Brain"이다.**

### 1-2. 핵심 가치

#### 가치 1: 취향 기록 & 분석 — "기록할수록 나를 알아간다"

기록이 쌓이면 Nyam이 사용자의 취향을 두 가지 축으로 자동 분석한다.

**Taste DNA — 맛 그 자체의 취향**
- 모든 기록에는 **객관적 맛 특성**(매운맛 75, 감칠맛 60 등)과 **주관적 만족도**가 함께 기록된다.
- 객관적 맛 특성의 수집 방법은 유형별로 다르다: 식당/와인은 AI가 자동 산출, 요리는 사용자가 직접 입력.
- 이 둘을 교차하여 선호를 도출한다: 매운맛 75인 음식에 종합 만족도 90점 → 매운맛 선호 상승. 매운맛 75인데 종합 만족도 30점 → 매운맛 선호 하락.
- 레이더 차트로 시각화 (식당/요리 6축, 와인 WSET 7축). (상세: 4-2 참조)

**Style DNA — 맛 너머의 취향 패턴**
- 음식의 본질(맛)뿐 아니라, 내가 즐겨 찾는 경험의 외적 특성도 데이터로 쌓인다.
- 유형마다 고유한 경험 축으로 패턴을 추적한다:
  - **식당**: 장르(genre) + 지역(area) + 상황(scene)
  - **와인**: 품종(variety) + 산지(region) + 타입(type) + 상황(scene)
  - **요리**: 장르(genre) + 상황(scene)
- 각 축에서 기록 수 + 다양성 + 최근성 + 일관성으로 전문성 레벨이 올라간다. (상세: 4-4 참조)

**활용**: 기록이 충분히 쌓이면(콜드 스타트 해소) Taste DNA + Style DNA를 결합해 맞춤 추천을 제공한다. 기록이 적을 때는 순수한 개인 기록 도구로서의 가치에 집중한다.

#### 가치 2: 버블 네트워킹 — "신뢰할 수 있는 사람들의 진짜 정보"

버블(폐쇄형 그룹)은 Nyam의 소셜 단위이다. 개방형 리뷰 플랫폼과 근본적으로 다르다.

**공유 방식의 다양성**
- **양방향 버블**: 모든 멤버가 기록을 공유하고 서로의 기록에 리액션. 친구 모임, 동호회에 적합.
- **열람 전용 버블**: 운영자의 기록만 공개. 구독자는 보기만 가능. 전문가/인플루언서 채널에 적합.

**정보 품질 관리**
- 버블 가입/탈퇴로 정보 소스를 직접 관리한다.
- 정보 오염이 의심되면 해당 버블만 탈퇴하면 끝 — 플랫폼 전체를 버릴 필요가 없다.
- 향후 개인 팔로우 기능을 통해, 취향이 비슷한 사용자를 팔로우하거나 전혀 다른 전문 분야의 사용자를 팔로우해 경험의 폭을 넓힐 수 있다. (Phase 2+)

**다양성 = 재미**
- 수많은 버블이 각자의 관점으로 존재한다는 것 자체가 정보의 다양성이고, 곧 재미다.
- 하나의 식당도 "혼밥러 버블"과 "와인 동호회 버블"에서는 완전히 다른 평가를 받는다.

#### 가치 3 (보조): AI — 곳곳의 윤활유

AI는 핵심 가치라기보다 **모든 가치를 매끄럽게 만드는 도구**이다.

- **기록을 빠르게**: 사진 → 식당/메뉴/장르/태그 자동 완성 (5초 기록)
- **기록을 풍부하게**: 질문에 답하면 AI가 매거진 스타일 블로그를 자동 생성 (Phase 2)
- **데이터를 활용하게**: Taste DNA + Style DNA 데이터를 조합해 맞춤 추천, 궁합 매칭, Today's Pick 등 다양한 발견 기능을 만들어낸다.

### 1-3. 정체성 우선순위

```
1순위: 개인 기록 도구 — "나만의 맛 데이터베이스" (콜드 스타트에서도 가치)
2순위: 취향 분석 — "기록이 쌓이면 나의 Taste DNA가 선명해진다"
3순위: 신뢰 공유 — "내가 고른 사람들의 진짜 평가"
4순위: 발견 — "취향이 맞는 새로운 곳"
5순위: 크리에이터 경제 — "진짜 맛잘알이 인정받고 수익을 얻는 생태계"
```

5순위는 장기 비전이다. 꾸준히 양질의 기록을 쌓은 사용자가 유료 버블을 운영하거나, 전문 분야 큐레이터로서 팔로워를 모아 수익을 만들 수 있는 구조. 광고비를 쓰는 식당이 아니라, 진정성 있는 기록을 쌓은 개인이 보상받는 생태계를 만든다.

Nyam은 **리뷰 플랫폼이 아니다**. 다이어트 앱도 아니다. "기록할수록 똑똑해지는 경험 지식 도구"이다.

## 2. 문제 정의

### 2-1. 리뷰를 믿을 수 없다

| 문제 | 심각도 | 현황 |
|------|--------|------|
| 리뷰 신뢰도 붕괴 | 매우 높음 | Google이 2023년 한 해 1.7억 개 가짜 리뷰 삭제 |
| 광고-추천 이해충돌 | 매우 높음 | 상위 노출 = 광고비 많은 곳 |
| AI 가짜 리뷰 급증 | 높음 | 2026년 말 AI 리뷰가 인간 리뷰 초과 전망 |
| 별점 무의미화 | 높음 | 대부분 식당 4.0-4.5점대 집중, 변별력 소멸 |

### 2-2. 나에게 맞는 추천이 없다

| 문제 | 심각도 | 현황 |
|------|--------|------|
| 일반화된 추천 | 매우 높음 | 모든 사용자에게 동일한 "인기 맛집" 추천 |
| 취향 반영 불가 | 높음 | 과거 기록을 활용하지 않는 추천 시스템 |
| 상황 무시 | 중간 | 혼밥/데이트/비즈니스 구분 없는 추천 |

### 2-3. Nyam의 해법

```
문제: 리뷰를 믿을 수 없다
→ 해법: 버블(폐쇄형 그룹) 안에서만 공유. 가짜 리뷰 구조적 차단.

문제: 나에게 맞는 추천이 없다
→ 해법: 기록 → Taste DNA 자동 구축 → 개인화 추천. 기록할수록 정확해진다.

문제: 기록이 귀찮다
→ 해법: 사진 1장 → AI가 5초 안에 식당/메뉴/장르/태그 자동 완성.
```

## 4. 핵심 컨셉

### 4-1. 3-Phase 기록 시스템

기록의 깊이를 3단계로 나누어, 각 단계가 독립적으로 가치를 제공한다.

```
Phase 1 — 빠른 캡처 (사용자, 5초, +5 XP)
  사진 촬영/선택 + 만족도 평가(100점 슬라이더)
  → 사진 선택 시 AI가 식당/메뉴를 사전 추정(선택적 지원). 저장은 즉시.

  ↓ (백그라운드, 사용자 대기 없음)

AI 비동기 처리 (유형별 분기)
  🍽️ 식당:
    1) 사진/위치 분석 → 식당·메뉴 특정
    2) 식당 기본정보 조회 (카카오/네이버)
    3) 웹 리뷰 검색 → 맛 특성 6축 교차 검증
    4) DNA 도출: AI 맛 특성 × 사용자 만족도

  🍷 와인:
    1) 사진 분석 → 라벨 OCR → 와인 식별
    2) 와인 객관적 정보 수집 (빈티지, 산지, 품종, 적정가, 평점)
    3) WSET 기준 테이스팅 노트 7축 AI 산출
    4) AI 테이스팅 + 사용자 테이스팅 평균 → DNA 도출

  🍳 요리: 저장 전 AI 사진 분석(요리명/장르/태그 추출)은 실행. 저장 후 비동기 enrich/taste-profile은 불필요 (사용자가 맛 특성을 직접 입력). post-process(DNA 반영)만 비동기 실행.

Phase 2 — AI 블로그 리뷰 (사용자, 2-5분, +15 XP)
  AI가 이미 분석을 마친 상태.
  사용자는 AI 질문에 답변 → 매거진 스타일 블로그 자동 생성.

Phase 3 — 비교 게임 (사용자, 3분, +5 XP)
  유사 기록끼리 토너먼트 → Elo 기반 정밀 순위 산출.
```

**핵심 원칙**: Phase 1만으로도 완전한 기록. Phase 2, 3은 선택적 심화. AI는 Phase 1 저장 직후 백그라운드에서 모든 분석을 완료한다.

### 4-2. 기록 유형과 AI 분석

#### 2-카테고리 평가 구조

Taste DNA 도출에는 두 가지 평가 카테고리가 필요하다. 모든 유형(식당/와인/요리)에 동일 구조가 적용되며, 모두 100점 만점 스케일 바를 사용한다.

**카테고리 A: 종합 경험 평가 (주관적)**
사용자가 직접 기록하는 전체적인 경험 만족도.

| 유형 | 평가 항목 |
|------|----------|
| 🍽️ 식당 | 맛, 가성비, 서비스, 분위기, 청결, 양 |
| 🍷 와인 | 전체 맛 만족도, 전체 가성비 만족도 (2개: DNA용) |
| 🍳 요리 | 맛 균형, 맛 만족도, 난이도, 소요시간, 재현성, 플레이팅, 재료비 (7개: 맛 2개는 DNA용, 나머지 5개는 기록 메타) |

**카테고리 B: 맛 특성 평가 (객관적)**
해당 음식/와인의 세분화된 맛 특성을 0-100으로 기록. 수집 방법이 유형별로 다르다.
- **식당**: 사용자가 조정할 수 없다 — "객관적으로 짠 음식인데 맛있다"가 DNA의 본질.
- **요리**: AI 산출이 불가하므로 사용자가 직접 입력한다.
- **와인**: 테이스팅은 주관+객관 혼합 영역이므로, AI와 사용자가 **각각 독립 기록** → 평균으로 객관성 확보.

| 유형 | 맛 특성 항목 | 수집 방법 |
|------|------------|----------|
| 🍽️ 식당 | 매운맛, 단맛, 짠맛, 신맛, 감칠맛, 기름진맛 | AI가 웹 리뷰 교차 검증으로 자동 산출 (읽기 전용) |
| 🍷 와인 | WSET 기준 테이스팅 노트: 산미, 바디감, 타닌, 당도, 균형, 여운, 향 | AI가 산출 + 사용자도 기록 → 양쪽 평균으로 DNA 반영 |
| 🍳 요리 | 매운맛, 단맛, 짠맛, 신맛, 감칠맛, 기름진맛 | AI 산출 불가 → 사용자가 직접 기록 |

**DNA 도출 공식:**
```
객관적 맛 특성(B)이 강한 음식 + 주관적 경험(A)이 좋음 → 해당 맛 선호 상승
객관적 맛 특성(B)이 강한 음식 + 주관적 경험(A)이 나쁨 → 해당 맛 선호 하락

주관적 경험(A)은 유형별 전체 항목의 종합 점수(평균)를 사용:
- 식당: 맛, 가성비, 서비스, 분위기, 청결, 양 → 6개 평균 = 종합 만족도
- 와인: 전체 맛 만족도, 전체 가성비 만족도 → 2개 평균 = 종합 만족도
- 요리: 맛 균형, 맛 만족도 → 2개 평균 = 종합 만족도
```

**핵심**: AI는 "이 음식/와인이 객관적으로 어떤 맛인가"를 판단하고, 사용자는 "그 경험이 얼마나 좋았는가"를 판단한다. 둘의 교차가 Taste DNA를 만든다. (유형별 상세 흐름은 4-1 참조)

### 4-3. Taste DNA

사용자의 **맛 그 자체**에 대한 선호를 다차원 벡터로 자동 구축하는 시스템.

**카테고리별 Taste DNA 축**

🍽️ 식당 (6축):

| 축 | 스펙트럼 | 도출 방식 |
|---|---|---|
| 매운맛 | 순한 ↔ 매운 | AI 리뷰 분석 맛 특성 × 종합 만족도 (맛+가성비+서비스+분위기+청결+양 6개 평균) |
| 단맛 | 담백 ↔ 달콤 | 〃 |
| 짠맛 | 싱거운 ↔ 짠 | 〃 |
| 신맛 | — | 〃 |
| 감칠맛 | 깔끔 ↔ 감칠맛 | 〃 |
| 기름진맛 | 담백 ↔ 기름진 | 〃 |

🍷 와인 — WSET 기준 테이스팅 구조:

> 와인은 WSET(Wine & Spirit Education Trust) 체계적 테이스팅 기준을 따른다.

**와인 기록 구조:**

| 구분 | 항목 | 기록 주체 | 기록 화면(8-3) |
|------|------|----------|---------------|
| 객관적 정보 | 빈티지, 와이너리, 산지, 품종, 적정가/평점 | AI 자동 | AI 분석 결과 카드로 표시 |
| WSET 테이스팅 노트 | 산미, 바디감, 타닌, 당도, 균형, 여운, 향 | AI + 사용자 각각 기록 | 사용자 슬라이더 입력 가능 |
| 페어링 | 함께 먹은 음식 | AI 비동기 분석 후 자동 입력, Phase 2에서 사용자 확인/수정 | Phase 1에서 숨김. Phase 2(8-6) 또는 기록 수정(8-5)에서 표시 |
| 구입가 | 구입 가격 | AI 비동기 분석 후 자동 입력, Phase 2에서 사용자 확인/수정 | Phase 1에서 숨김. Phase 2(8-6) 또는 기록 수정(8-5)에서 표시 |
| 주관적 만족도 | 전체 맛 만족도, 전체 가성비 만족도 | 사용자 | 100점 슬라이더 |

**Taste DNA 도출:**
- WSET 테이스팅 노트 (AI 산출 + 사용자 기록의 **평균**) × 종합 만족도 (전체 맛 만족도 + 전체 가성비 만족도의 평균)
- 예: 산미 높고 바디 낮은 와인에 종합 만족도 90점 → "높은 산미 + 라이트 바디 선호" DNA 형성

| DNA 축 | 스펙트럼 |
|--------|---------|
| 산미 | 낮음 ↔ 높음 |
| 바디감 | 라이트 ↔ 풀바디 |
| 타닌 | 부드러운 ↔ 강한 |
| 당도 | 드라이 ↔ 스위트 |
| 균형 | 한쪽 치우침 ↔ 완벽 균형 |
| 여운 | 짧은 ↔ 긴 |
| 향 | 단순 ↔ 복합 |

**Style DNA**: 자주 마시는 품종, 산지, 와인 타입 누적 → style_dna_wine_varieties(품종) + style_dna_wine_regions(산지) + style_dna_wine_types(타입)

**Wine DNA 한줄 요약**: Taste DNA + Style DNA를 결합하여 자연어로 표현.
예: "부르고뉴 스타일의 라이트 바디, 높은 산미 와인 선호" / "나파 밸리 풀바디 카베르네 소비뇽 애호가"

🍳 요리 — Taste DNA (맛 선호, 식당과 동일 6축):

> 요리도 식당과 동일한 2-카테고리 구조를 따른다. 다만 AI 산출이 불가능하므로 맛 특성(카테고리 B)을 **사용자가 직접 수동 기록**한다.

| 축 | 스펙트럼 | 수집 방법 |
|---|---|---|
| 매운맛 | 순한 ↔ 매운 | 사용자 수동 입력 × (맛 균형 + 맛 만족도) |
| 단맛 | 담백 ↔ 달콤 | 〃 |
| 짠맛 | 싱거운 ↔ 짠 | 〃 |
| 신맛 | — | 〃 |
| 감칠맛 | 깔끔 ↔ 감칠맛 | 〃 |
| 기름진맛 | 담백 ↔ 기름진 | 〃 |

**요리 평가 3단계 흐름:**
1. **맛 특성** (카테고리 B) — 이 요리의 객관적 맛: 매운맛, 단맛, 짠맛, 신맛, 감칠맛, 기름진맛
2. **종합 맛 평가** (카테고리 A) — 맛 균형 + 맛 만족도 → 둘 다 DNA 도출에 사용 (맛 균형은 전체적 조화도 가중치, 맛 만족도는 개별 맛 선호 가중치)
3. **요리 경험 평가** — 난이도, 소요시간, 재현성, 플레이팅, 재료비 → DNA와 무관, 개별 기록 메타

**보조 태그** (기록 시 선택적): 식감(바삭, 쫄깃, 아삭, 부드러운), 온도(뜨거운/따뜻한/차가운)

**시각화**: 레이더 차트 (식당/요리 6축, 와인 7축) + 취향 코드 (예: "bold-savory")

### 4-4. Style DNA

사용자의 **맛 너머의 경험 패턴**을 기록으로 자동 구축하는 시스템.

유형마다 축의 의미가 다르므로, WHAT/WHERE/WHEN 프레임에 무리하게 맞추지 않고 각 유형의 고유한 경험 축으로 정의한다.

🍽️ **식당 Style DNA** — 3축 (`style_dna_restaurant_*`)
- **genre** (장르): 한식, 중식, 일식, 양식, 치킨, 피자, 버거, 분식, 족발/보쌈, 찜/탕/찌개, 돈까스/회, 고기/구이, 해산물, 아시안, 카페/디저트, 샐러드/건강식, 도시락/백반 (FOOD_CATEGORIES 기준)
- **area** (지역): 상권/동네 단위 — 성수, 홍대, 강남, 이태원 등 (GPS 자동)
- **scene** (상황): 혼밥, 데이트, 비즈니스, 가족, 친구모임, 술자리, 브런치, 간단점심

🍷 **와인 Style DNA** — 4축 (`style_dna_wine_*`)
- **variety** (품종): 카베르네 소비뇽, 피노 누아, 샤르도네, 리슬링 등
- **region** (산지): 프랑스/부르고뉴, 이탈리아/토스카나, 나파밸리, 칠레 등
- **type** (타입): 레드, 화이트, 스파클링, 로제, 오렌지, 내추럴
- **scene** (상황): 혼술, 데이트, 페어링, 파티, 테이스팅

🍳 **요리 Style DNA** — 2축 (`style_dna_cooking_*`)
- **genre** (장르): 한식, 양식, 중식, 일식, 베이킹, 디저트, 음료 (COOKING_GENRES 기준)
- **scene** (상황): 일상식사, 밀프렙, 손님초대, 도시락, 다이어트 (COOKING_SCENES 기준)

각 축에서 **기록 수 + 다양성 + 최근성 + 일관성**으로 전문성 레벨이 산출된다.
**활용**: 프로필 전문성 표시, 버블 참여 조건, 상호보완 매칭

### 4-5. 버블 네트워킹

신뢰 기반 폐쇄형 버블 시스템. 두 가지 독립 속성의 **조합**으로 버블을 정의한다.

**접근 방식 (access):**

| 값 | 설명 |
|------|------|
| Private | 초대 링크로만 진입 |
| Public | 조건 충족 시 누구나 참여 |

**공유 방식 (sharing):**

| 값 | 설명 |
|------|------|
| Interactive | 모든 멤버가 기록 공유 + 리액션 (양방향) |
| View-only | 운영자 기록만 공개, 구독자는 열람만 |

**과금 (paid):** 선택적 유료 구독 플래그 (어떤 조합에든 추가 가능)

**가능한 조합 예시:**
- Private + Interactive: 친구 모임, 동호회
- Private + View-only: 전문가 비공개 채널
- Public + Interactive: 오픈 커뮤니티
- Public + View-only: 인플루언서 공개 채널
- Public + View-only + Paid: 유료 구독 채널

**발견 흐름**: 기록 → 작성자 프로필 → 소속 버블 발견 (Bottom-up)

## 8. 페이지별 기능 정의

### 8-1. 로그인 (`/auth/login`)

8-1-1. `[BE]` 소셜 로그인 버튼 제공 (Google, Kakao, Naver, Apple) — Supabase Auth OAuth
8-1-2. 로그인 전 이용약관 및 개인정보처리방침 동의 체크
8-1-3. `[BE]` 이미 로그인된 사용자는 홈(`/`)으로 리다이렉트 — 세션 확인 (proxy.ts)

### 8-2. 홈 (`/`)

8-2-1. `[BE]` **Today's Pick 카드** — "오늘 뭐 먹지?" 추천. 사진 터치하면 다른 추천으로 랜덤 순환. **추천 이유를 표시**하며 "더보기" 버튼으로 해당 기록의 리뷰 페이지(8-4)로 이동. 콜드 스타트: 기록 5건 미만 시 온보딩 카드 표시.
  - **추천 알고리즘**: 시간대 + 추천 이유 프리셋 조합
    - 시간대 분기: 아침(~10시, 브런치/카페), 점심(10~14시, 한식/일식/분식), 오후(14~17시, 카페/디저트), 저녁(17~21시, 고기/양식/중식), 야식(21시~, 치킨/족발/라멘)
    - 추천 이유 프리셋 (랜덤 선택): "높은 만족도" (rating_overall ≥ 80), "이 장르 맛집" (genre 매칭), "가까운 곳" (거리 기반), "오래 안 갔던 곳" (2개월+ 미방문), "이 맛 좋아할걸" (Taste DNA 기반)
  - **표시**: 사진 배경 + 추천 이유 뱃지(좌상단) + 식당명 + 점수 + "더보기" 버튼
  — records + taste_dna_restaurant + restaurants (location) 기반
8-2-2. `[BE]` **프로필 카드** — 닉네임, 레벨(Lv.), XP, 통계(기록 수 / 장소 수 / 버블 수) 한눈에 표시 — users + user_stats
8-2-3. `[BE]` **Taste DNA 시각화** — Food(6축) / Wine(7축) 탭 전환. 레이더 차트로 맛 선호 표시. 기록 부족 시 안내 메시지. Cooking DNA는 홈에서 제외하고 프로필 페이지(8-13)에서만 표시 — 홈은 외식/와인 중심의 핵심 DNA에 집중하고, 요리 DNA는 기록 빈도가 상대적으로 낮아 프로필에서 심화 확인하는 구조 — taste_dna_restaurant + taste_dna_wine
8-2-4. `[BE]` **포토 캘린더** — 월별 달력에 기록한 날짜를 사진 썸네일로 표시. 월간 통계(기록 수, 방문 장소 수, 평균 점수) 하단 표시. 월 이동(이전/다음) 및 오늘로 이동 기능 — records + record_photos
8-2-5. **캘린더 날짜 팝업** — 날짜 탭 시: 기록 1건이면 상세 페이지로 이동, 2건 이상이면 팝업에 기록 목록 표시
8-2-6. `[BE]` **지도 섹션** — 네이버/카카오 지도 토글 전환. 내 기록과 버블 멤버 기록을 핀으로 표시. 필터(전체 / 내 기록 / 친구) 전환 — records + group_memberships + record_shares
8-2-7. `[BE]` **친구 피드** — 같은 버블 멤버들의 최근 기록을 카드 리스트로 표시 — group_memberships → record_shares → records

### 8-3. 빠른 기록 (`/record`)

8-3-1. **기록 유형 선택** — 식당 / 와인 / 요리 3개 탭 (record_type: 'restaurant' | 'wine' | 'cooking')
8-3-2. **사진 촬영/선택** — 카메라 촬영 또는 앨범에서 선택 (최대 8장). 선택 후 그리드로 미리보기. 개별 삭제 가능
8-3-3. `[BE]` **AI 사진 분석** — 사진 선택 시 자동으로 AI 분석 실행. 식당: 식당명, 메뉴, 장르, 맛/식감 태그 추출. 와인: 라벨 OCR → 와인명, 산지, 품종 식별. 요리: 요리명, 장르, 맛/식감 태그 추출 (맛 특성 6축은 AI 산출 불가 → 사용자 수동 입력). 분석 결과를 카드로 표시 — POST /api/analyze-visit + GET /api/restaurants/nearby
8-3-4. **종합 경험 평가** — 유형별 항목을 100점 슬라이더로 평가 (카테고리 A). 식당: 맛, 가성비, 서비스, 분위기, 청결, 양. 와인: 전체 맛 만족도, 전체 가성비 만족도. 요리: [맛 평가] 맛 균형, 맛 만족도 (→ DNA 도출에 사용) + [경험 평가] 난이도, 소요시간, 재현성, 플레이팅, 재료비 (→ 기록 메타, DNA 무관)
8-3-4b. `[BE]` **맛 특성 기록** — 세분화된 맛 특성을 100점 슬라이더로 기록 (카테고리 B). 식당: 기록 화면에서는 미표시, 저장 후 AI가 백그라운드에서 자동 산출하며 기록 상세(8-4-4b)에서 읽기 전용으로 확인 가능. 와인: AI가 WSET 7축을 산출 + 사용자도 기록 가능 → 평균으로 DNA 반영. 요리: 사용자가 직접 수동 입력 — record_taste_profiles
8-3-5. `[BE]` **저장** — 최소 저장 조건: 사진 1장 이상 + 종합 경험 평가(카테고리 A) 1개 이상 평가 시 저장 버튼 활성화. 저장 즉시 완료(Phase 1). 이후 유형별 비동기 파이프라인 실행:
- 🍽️ 식당: enrich(식당·메뉴 특정) → taste-profile(리뷰 교차 검증·맛 프로필 산출) → post-process(DNA 반영)
- 🍷 와인: enrich(와인 식별·객관적 정보) → taste-profile(WSET 7축 AI 산출 + 사용자 기록 평균) → post-process(DNA 반영)
- 🍳 요리: post-process만 실행 (맛 특성은 Phase 1에서 사용자가 직접 입력 완료)
— records + record_photos INSERT → POST /api/records/enrich → POST /api/records/taste-profile → POST /api/records/post-process

### 8-4. 리뷰 페이지 (`/records/[id]`)

> **핵심 원칙**: 리뷰 페이지는 해당 기록의 **모든 정보를 한 페이지에 통합** 표시한다. Phase 1 평가, Phase 2 블로그, 맛 프로필, 태그를 하나의 스크롤 페이지에서 확인할 수 있다.

8-4-0. `[BE]` **다회 방문 요약 배너** — 같은 식당(restaurant_id)에 기록이 2건 이상인 경우, 상단에 "N회 방문" 요약 배너를 표시. 방문 날짜 리스트를 보여주고, 탭하면 해당 날짜의 리뷰 페이지로 이동 — records WHERE restaurant_id GROUP BY
8-4-1. `[BE]` **사진 캐러셀** — 가로 스크롤. 1장이면 전체 너비, 2장 이상이면 85% 너비로 슬라이드 — record_photos
8-4-2. `[BE]` **기본 정보** — 메뉴 이름, 장르, 기록 날짜 — records
8-4-3. **종합 점수** — 전체 평균 점수 큰 숫자로 표시 (records.rating_overall)
8-4-4. **세부 평가** — 종합 경험 각 항목별 가로 바 차트로 점수 표시 (records.rating_*)
8-4-4b. `[BE]` **맛 특성** — 식당: AI 산출 6축 (읽기 전용). 와인: AI+사용자 WSET 7축. 요리: 사용자 입력 6축 — record_taste_profiles
8-4-5. **태그** — 맛 태그, 식감 태그를 색상별 칩으로 구분 표시 (records.flavor_tags, texture_tags)
8-4-6. **코멘트** — 사용자가 남긴 메모 표시 (records.comment)
8-4-6b. `[BE]` **Phase 2 블로그 리뷰** — 블로그 작성 완료(phase_status ≥ 3) 시 매거진 스타일 블로그를 인라인 표시 (사진+텍스트 교차). 미완료 시 "2단계 리뷰 작성하기" CTA 카드 표시 — record_journals
8-4-7. `[BE]` **Phase 2 진입** — AI 블로그 리뷰 미완료 시 CTA 카드 표시. 완료 시 블로그 인라인 표시 (8-4-6b) — records.phase_status
8-4-8. `[BE]` **리액션** — 좋아요, 댓글, 유용해요, 맛있겠다 리액션 버튼 — reactions
8-4-9. `[BE]` **액션 메뉴** — 공유, 북마크, 수정, 삭제 — record_shares + bookmarks + records DELETE
