1. 평가 시스템 & UI

설계 목표: 3초 안에 핵심 평가 완료 + 원하면 깊이 추가 + 평가 자체가 즐거운 경험

1-1. 평가 구조 — 점진적 확장 (Progressive Disclosure)

레벨이름수집 항목소요 시간
필수Quick Rate총점 (100점) + 상황 태그 (1개)~3초
선택 1Taste Card맛/분위기/서비스/가성비 4축 (또는 와인 4축)~5초
선택 2Story한줄평, 메뉴 별표, 사용팁, 사진, 함께 간 사람원하는 만큼

1-3. 와인 평가 특화

와인 Taste Card 축

  • 아로마 — 향의 복합성/매력
  • 맛 — 과일/산미/타닌/바디
  • 여운 — 피니시의 길이와 품질
  • 밸런스 — 전체적 조화

와인 상황 태그

  • 혼술
  • 데이트
  • 모임/파티
  • 식사 페어링
  • 선물/받은 것
  • 시음회/와인바

1-4. 평가 UI 추천 조합

추천: A안 (슬라이더 바이트)를 Quick Rate로,
Taste Card는 C안 (스와이프)의 카드 방식을 선택적 확장으로 결합.

Quick Rate (슬라이더 + 상황태그 3초) → "더 평가하기?" → Taste Card (스와이프 카드 4장) → Story (자유 입력)

2. 등록 시스템

2-1. 등록 진입점 3가지

A 검색 등록

식당/와인 이름 검색 → 선택 → 평가

가장 확실. 기억이 명확할 때.

B 사진 등록

사진 → AI 분석 → 식당/와인 추정 → 확인 → 평가

음식 사진 or 와인 라벨 사진

C 리스트 선택 등록

지역 기반 식당 리스트 → 체크 → 일괄 평가

온보딩 + 새 지역 방문 시. 콜드스타트 핵심.

2-2. 검색 시스템

[입력] 사용자가 가게명/와인명 타이핑
→ [자동완성] Nyam 캐시 DB 먼저 검색 (이미 다른 유저가 등록한 곳)
→ [외부 검색] 캐시에 없으면 카카오 + 네이버 + 구글 동시 검색
→ [교차 매칭] 3개 소스 결과를 가게명+주소로 fuzzy merge
→ [결과 리스트] 통합 결과 표시 (지도 미니맵 + 주소 + 장르 + 외부 평점)
→ [선택] 사용자가 식당 선택 → 평가 플로우로 이동

2-3. 사진 등록 플로우

[사진 선택] 갤러리에서 사진 선택 or 카메라 촬영
→ [EXIF 추출] GPS 좌표 + 촬영 날짜/시간
→ [Vision AI] Gemini Vision으로 음식/와인 라벨 분석
→ [후보 매칭]
  • GPS 있으면 → 반경 100m 식당 검색 + AI 분석 결과 교차
  • GPS 없으면 → AI 분석 결과 + 사용자 활동 지역에서 검색
  • 와인 라벨이면 → 와인 DB에서 매칭
→ [확인] "여기 맞나요?" → 사용자 확인 → 평가 플로우

3-2. 4가지 뷰 모드

뷰용도핵심 요소
리스트 뷰기본. 빠른 스캔썸네일 + 가게명 + 점수 + 상황태그 + 날짜

4. 추천 알고리즘

4-1. 설계 원칙

  • 인터넷 정보 기반 추천은 최소한 — 네이버/구글 크롤링 추천은 하지 않음
  • 나의 데이터 기반 — 내 기록이 쌓일수록 정확해지는 구조
  • 신뢰 기반 — 모르는 사람의 평가보다, 아는 사람(버블)의 평가 우선
  • 상황 인식 — "오늘 데이트인데"와 "오늘 혼밥인데"는 완전히 다른 추천

4-2. 추천 소스별 알고리즘

추천 탭알고리즘데이터 소스
"내 경험에서" 상황 필터 매칭: 현재 상황(사용자 선택 or 시간대 추정)에서 높은 점수를 준 곳
재방문 가치 점수: 총점 × 마지막 방문 이후 경과일 가중 (오래 안 간 고점수 식당)
계절/날씨 보정: 여름엔 냉면/빙수, 겨울엔 국물 등 (Phase 2+) 		내 기록
"새로운 도전" Wishlist 우선: 내가 찜해둔 곳
버블 인기: 내 버블에서 공유된 곳 중 내가 안 가본 곳 (Phase 2)
권위 리뷰: 미슐랭/블루리본 중 내가 안 가본 곳 (콜드스타트 용)
취향 유사 사용자: 나와 겹치는 기록이 많은 사용자가 좋게 평가한 곳 (Phase 3) 		Wishlist + 버블 + 외부 권위
"와인" 품종/산지 패턴: 내가 높게 평가한 와인들의 공통 특성 → 유사 와인 추천
가격대 매칭: 평소 마시는 가격대 기반
페어링 추천: 식당 기록에 와인 페어링 정보가 있으면 역으로 추천 		내 와인 기록

4-3. 콜드스타트 추천 (기록 부족 시)

기록 수추천 전략
0~5개온보딩 취향 카드 기반 + 지역 권위 리뷰(미슐랭/블루리본) + 네이버/구글 평점 상위
5~20개기록된 장르/지역 패턴 시작 + 권위 리뷰 보완
20개+본격 개인화 추천 가동. 상황별 취향이 통계적으로 의미 있어짐

6. 경험치 시스템

6-1. 경험치 구조

핵심: 하나의 통합 레벨이 아니라, 영역별 독립 경험치
"을지로 Lv.7" + "강남 Lv.3" + "보르도 와인 Lv.5" — 각자 다른 전문성.

식당 경험치 축

축기준예시
지역(동네)해당 지역 식당 기록 수 + 리뷰 깊이 + 활동 기간을지로 Lv.7, 성수 Lv.4
장르해당 장르 기록 수일식 Lv.5, 이탈리안 Lv.3

와인 경험치 축

축기준예시
산지해당 산지 와인 기록 수보르도 Lv.5, 부르고뉴 Lv.3
품종해당 품종 기록 수까베르네 소비뇽 Lv.4, 피노 누아 Lv.6

6-2. 레벨 체계

레벨이름기준 (대략)의미
1~2입문1~5개 기록시작했다
3~4탐험가6~15개 기록꽤 다녀봤다
5~6단골16~30개 기록 + 리뷰이 동네/장르를 잘 안다
7~8전문가30~60개 + 상세 리뷰 + 버블 기여추천을 믿을 수 있다
9~10마스터60개+ + 장기 활동 + 높은 커뮤니티 기여이 영역의 권위자

8. 소셜 시스템

8-1. 소셜 레이어 구조

레이어범위신뢰도기능
나 개인 최고 내 기록 관리, 취향 분석, 추천
버블 초대제 그룹 (5~50명) 높음 공유, 리뷰, 댓글, 찜, 통계
팔로우 1:N 관계 중간 공개 기록 피드, 리스트 구독
전체 Nyam 사용자 전체 낮음 리스트 탐색, 경험치 프로필 열람 (공개 설정 시)

8-2. 피드 시스템

버블 피드

  • 버블 내 공유된 기록 시간순
  • 좋아요/댓글 인라인
  • 새 공유 알림 뱃지
  • 핀 고정 (운영자가 중요 리뷰 상단 고정)

팔로우 피드

  • 팔로우한 사람의 공개 기록
  • 새 리스트 발행 알림
  • 좋아요로 반응 (댓글은 버블에서만)
  • 추천 팔로우 ("취향이 비슷한 사용자")

8-3. 인터랙션 시스템

액션설명범위
좋아요가볍게 동의/관심 표현. 카운트 공개.버블 + 팔로우
싫어요비공개 카운트 (리뷰어에게 안 보임). 추천 조정에만 사용.버블
찜내 Wishlist에 추가. 공유자에게 "1명이 찜했어요" 알림.버블 + 팔로우
리뷰같은 식당/와인에 대한 독립 리뷰 작성. 별도 ID.버블
댓글특정 리뷰에 대한 댓글. 토론/질문/정보 공유.버블
공유내 기록 → 특정 버블에 공유. 필터 적용 가능.버블

8-4. 알림 시스템

알림 유형설명푸시 여부
버블 새 공유버블에 새 식당/와인이 공유됨ON (설정 가능)
내 리뷰에 댓글누군가 내 리뷰에 댓글 달음ON
내 공유가 찜됨내가 공유한 식당을 누가 찜함ON
새 팔로워누군가 나를 팔로우ON
새 지역 도착새로운 동네 감지 → 맛집 리스트 제안ON (위치 동의 시)
경험치 레벨업"을지로 Lv.7 달성!"ON
위시리스트 근처찜한 식당 근처 도착ON (위치 동의 시)

8-5. 리스트 (큐레이션) 시스템

레터박스에서 가장 중독적인 기능. 성장 엔진.
요소설명
생성테마/제목 설정 → 내 기록에서 식당/와인 선택 → 순서 지정 → 설명 추가
공개 범위비공개 / 버블 공유 / 전체 공개 (선택)
반응좋아요, 저장(클론), 댓글
탐색전체 공개 리스트 검색. 지역/장르/태그 필터. 인기순/최신순 정렬
SEO/공유공개 리스트는 웹 URL 생성 → SNS 공유 → 외부 유입
