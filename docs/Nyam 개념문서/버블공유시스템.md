# 버블 공유 시스템 — 규칙 기반 자동 동기화

## 핵심 개념

버블에 가입할 때 **공유 규칙(필터)**을 설정하면,
기록을 생성/수정할 때마다 조건에 맞는 기록이 **자동으로** 해당 버블 피드에 반영된다.
수동 공유 없이, 홈 필터와 동일한 Notion식 FilterRule을 재사용한다.

```
기록 생성/수정
  → 내 모든 버블 멤버십의 shareRule 평가
  → 조건 매칭 → bubble_shares INSERT
  → 조건 불매칭 → bubble_shares DELETE (기존 공유 해제)
```

---

## 공유 규칙 (BubbleShareRule)

각 BubbleMember가 하나의 shareRule을 가진다.

```
┌──────────────────────────────────────────────────────┐
│  BubbleShareRule                                     │
├──────────────┬───────────────────────────────────────┤
│ mode         │ 'all' | 'filtered'                    │
│ rules        │ FilterRule[]  (mode=filtered일 때)    │
│ conjunction  │ 'and' | 'or'  (규칙 결합 방식)        │
└──────────────┴───────────────────────────────────────┘
```

| mode | 동작 |
|------|------|
| `all` | 내 모든 기록을 해당 버블에 자동 공유 |
| `filtered` | FilterRule 조건에 매칭되는 기록만 공유 |
| `null` (미설정) | 공유 안 함 |

### 예시

| 시나리오 | shareRule |
|---------|-----------|
| 모든 항목 공유 | `{ mode: 'all' }` |
| 강남 식당 90점 이상 | `{ mode: 'filtered', conjunction: 'and', rules: [{ attribute: 'area', operator: 'eq', value: '강남' }, { attribute: 'satisfaction', operator: 'eq', value: '90' }] }` |
| 이탈리안 또는 프렌치 | `{ mode: 'filtered', conjunction: 'or', rules: [{ attribute: 'genre', operator: 'eq', value: '이탈리안' }, { attribute: 'genre', operator: 'eq', value: '프렌치' }] }` |

## 동기화 트리거 & 흐름

### 트리거 1: 기록 생성/수정

```
saveRecord(record)
  │
  ├─ 내 active 버블 멤버십 조회 (shareRule 포함)
  │
  ├─ 각 멤버십에 대해:
  │   ├─ shareRule === null        → skip
  │   ├─ shareRule.mode === 'all'  → 무조건 공유
  │   └─ shareRule.mode === 'filtered'
  │       → matchesAllRules(record, rules, conjunction)
  │       → true  → bubble_shares UPSERT
  │       → false → bubble_shares DELETE (있었으면)
  │
  └─ 결과: 해당 기록의 버블별 공유 상태가 최신화
```

### 트리거 2: 공유 규칙 변경

```
updateShareRule(bubbleId, newRule)
  │
  ├─ 내 전체 기록 조회
  ├─ evaluateShareRule(records, newRule) → shouldShare[]
  ├─ 현재 bubble_shares 조회 → currentlyShared[]
  │
  ├─ diff 계산:
  │   ├─ toAdd    = shouldShare - currentlyShared
  │   └─ toRemove = currentlyShared - shouldShare
  │
  └─ bubble_shares 일괄 INSERT/DELETE (소급 적용)
```

### 트리거 3: 기록 삭제

```
deleteRecord(recordId)
  → 해당 record의 bubble_shares 전부 DELETE (CASCADE)
```

## 정보 공개 범위 (VisibilityOverride)

공유 규칙과 별개로, **어떤 정보를 공개할지**는 VisibilityOverride로 제어한다.

```
┌──────────────────────────────────────────────────┐
│  VisibilityOverride (7필드)                      │
├──────────┬───────────────────────────────────────┤
│ score    │ 만족도 점수                            │
│ comment  │ 한줄평                                 │
│ photos   │ 사진                                   │
│ level    │ 레벨 뱃지                              │
│ quadrant │ 맛 사분면                              │
│ bubbles  │ 소속 버블 목록                         │
│ price    │ 가격 정보                              │
└──────────┴───────────────────────────────────────┘
```

| 설정 | 동작 |
|------|------|
| 모두 공개 | `null` (오버라이드 없음, 전부 true) |
| 부분 공개 | 개별 필드 true/false 토글 |

> 공유 규칙 = **어떤 기록**을 공유할지
> 가시성 오버라이드 = 공유된 기록의 **어떤 정보**를 보여줄지

## DB 스키마

### bubble_members 테이블 변경

```sql
ALTER TABLE bubble_members
ADD COLUMN share_rule JSONB DEFAULT NULL;

-- share_rule 예시:
-- { "mode": "all" }
-- { "mode": "filtered", "conjunction": "and", "rules": [...] }
-- NULL = 공유 안 함
```

### bubble_shares 테이블 (기존 유지)

```
┌─────────────┬────────────────────────────────────┐
│ 컬럼        │ 설명                                │
├─────────────┼────────────────────────────────────┤
│ id          │ PK                                  │
│ record_id   │ FK → records                        │
│ bubble_id   │ FK → bubbles                        │
│ shared_by   │ FK → users                          │
│ shared_at   │ timestamp                           │
│ auto_synced │ boolean (자동 동기화 여부, 신규)     │
└─────────────┴────────────────────────────────────┘
UNIQUE(record_id, bubble_id)
```

`auto_synced` 컬럼으로 자동 동기화된 공유와 구분 가능 (디버깅/통계용).

## 자동 공유 + 수동 공유 공존

### 두 가지 공유 경로

| | 자동 공유 | 수동 공유 |
|---|---|---|
| 트리거 | 기록 생성/수정 시 shareRule 평가 | 사용자가 공유 버튼 클릭 |
| auto_synced | `true` | `false` |
| 규칙 변경 시 | 재평가 → 매칭 안 되면 삭제 | **삭제 안 됨** (사용자 의도 보존) |
| 용도 | 일상적 자동 갱신 | 규칙에 안 맞지만 공유하고 싶은 예외 항목 |

### 기록 저장 후 피드백 (토스트)

```
기록 저장 완료
  → sync 결과 반환: { sharedTo: ['맛집모임', '강남버블'], unsharedFrom: [] }
  → 토스트: "맛집모임, 강남버블에 공유됨"
  → 공유된 버블이 없으면: "공유된 버블 없음"
```

사용자에게 자동 공유 결과를 즉시 확인시켜 불안감 해소.

### 기록 카드 — 버블 스티커 + 공유 버튼

```
┌─────────────────────────────────────────┐
│  🍜 스시오마카세 하루                     │
│  강남 · 92점 · 2024.03.28               │
│                                         │
│  [맛집모임] [강남버블]         [+ 공유]   │
│   ↑ 버블 스티커 (공유 중)     ↑ 수동 공유 │
└─────────────────────────────────────────┘
```

- **버블 스티커**: 이 기록이 공유된 버블 목록 (아이콘+이름 칩)
- **공유 버튼**: 클릭 시 → 버블 선택 시트 (수동 공유) + 외부 공유 (카카오톡 등)

### 수동 공유 항목 관리

수동 공유(auto_synced=false)는 규칙 변경 시 자동 삭제되지 않음.
→ 사용자 설정에서 일괄 정리 가능:

```
설정 > 버블 공유 관리
  └─ [수동 공유 항목 정리] 버튼
     → "공유 규칙에 맞지 않는 수동 공유 N건을 삭제합니다"
     → 확인 시: auto_synced=false이면서 현재 shareRule에 매칭 안 되는 shares 삭제
```

### 기존 수동 공유 마이그레이션

auto_synced 컬럼 추가 시 기존 shares는 `false`(수동)로 초기화.
→ 사용자가 shareRule을 설정하면 자동 공유가 시작되고,
→ 기존 수동 공유는 그대로 유지 (삭제하고 싶으면 설정에서 정리)

## 물리화 vs 계산 — 왜 bubble_shares를 유지하는가

share_rule만 두고 피드 조회 시 계산하는 방식(계산)도 가능하지만, **물리화(bubble_shares 테이블)**를 선택.

```
피드 조회 = 하루 수십 회 × 멤버 수   ← 사용자가 매번 체감
기록 저장 = 일주일에 몇 번           ← fire-and-forget으로 체감 0
```

| | 물리화 (채택) | 계산 (미채택) |
|---|---|---|
| 피드 열기 | SELECT + 페이지네이션 → 즉시 | 멤버 N명 records 전부 fetch → 필터링 → 느림 |
| 기록 저장 | sync (비동기, 사용자 안 느낌) | 없음 |
| 댓글/리액션 | share row에 직접 연결 | 앵커 없음, 복합키 필요 |
| 통계 집계 | COUNT/AVG 단순 | 매번 재계산 |
| 정합성 | sync 누락 가능 (auto_synced로 추적) | 항상 최신 |

**결론: 읽기 빈도 >> 쓰기 빈도이므로 물리화가 체감 속도에서 압도적으로 유리.**

## 설계 원칙

1. **순수 필터 시스템**: 자동 공유는 규칙에 맞으면 공유, 안 맞으면 해제. 예외 없음
2. **소급 적용**: 규칙 설정/변경 시 기존 기록도 전부 재평가 (auto_synced=true만)
3. **양방향 동기화**: 기록 수정으로 조건 불매칭 시 자동 공유 해제
4. **마찰 제거**: 기록만 하면 끝. 버블 피드 갱신은 시스템이 처리
5. **물리화 우선**: bubble_shares에 실제 row를 유지하여 읽기 성능 보장, sync는 fire-and-forget
6. **수동 공유 보존**: auto_synced=false인 수동 공유는 규칙 변경에 영향 안 받음. 사용자 의도 존중
7. **투명한 피드백**: 기록 저장 시 토스트로 공유 결과 표시, 기록 카드에 버블 스티커 노출
