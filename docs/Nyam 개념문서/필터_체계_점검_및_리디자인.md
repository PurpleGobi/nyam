## 3. 진단: 잘 된 점

### 3-1. 구조적 강점

1. **SSOT 설계**: `filter-config.ts`에 모든 필터 속성이 중앙 정의됨. 옵션 추가/삭제가 한 곳에서 가능.
2. **서버/클라 분리**: `DB_FILTER_ATTRIBUTES` Set으로 서버 위임 가능한 필터를 명확히 구분. 성능 최적화됨.
3. **filter-matcher 순수 로직**: domain 레이어에 위치, R1 준수. 특수 속성(범위, 기간, 복합) 처리가 깔끔.
4. **칩 기반 UX**: 선택한 필터를 칩으로 표시 → 직관적. 편집/삭제 가능.
5. **뷰 필터 분리**: `view` 칩이 필터가 아니라 데이터 소스 결정자 → 역할 명확.

### 3-2. 식당 필터

- **상황(scene)**: 실제 사용 맥락을 잘 반영. "오늘 데이트인데 어디 가지?" → 바로 필터 가능.
- **명성(prestige)**: children-cascade로 "미슐랭 > 2스타" 같은 세분화 가능. 타입+등급 2레벨 모델 우수.
- **위치(location)**: 행정구역/생활권 2탭은 한국 외식 문화에 최적화된 설계.

### 3-3. 와인 필터

- **55개 품종**: WSET L2+L3 완전 커버리지. 와인 앱 수준의 전문성.
- **산미/당도/복합도**: 구조적 평가 요소를 필터로 제공 → 와인 학습 도구로도 기능.

### 3-4. 버블 필터

- **전문분야 축**: genre/area/wine_region/wine_variety 4축으로 버블의 전문성을 다각도 탐색 가능.

## 4. 진단: 문제점

### 4-1. 탭 간 불일관성

| 문제 | 식당 | 와인 | 버블 |
|------|:----:|:----:|:----:|
| **보기(view) 필터** | O | O | **X** (bubble_type으로 대체) |
| **점수 필터** | O | O | **X** |
| **시기 필터** | O | O (시음시기) | **X** |
| **소팅 방향** | 6종 (거리 포함) | 5종 (거리 미포함) | 5종 (완전히 다른 기준) |

- `view`와 `bubble_type`이 **같은 역할**(데이터 소스 결정)인데 이름이 다름.
- 식당/와인은 `satisfaction`으로 통일된 반면, 버블에는 점수 필터가 없음 (적합도 점수가 있는데도).

### 4-2. 과잉 필터

| 필터 | 문제 |
|------|------|
| **companion_count (동반자)** | 기록 시 입력률이 낮을 가능성 높음. 실사용 빈도 의문. |
| **menu_type (대표메뉴)** | "코스 vs 단품"을 기준으로 식당을 찾는 사용 패턴이 드뭄. |
| **purchase_price (와인 가격)** | range 타입이지만 **UI 구현이 되어있지 않음** (타입만 정의). 사실상 사용 불가. |
| **expertise_level / expertise_records** | 2개가 거의 같은 축을 다른 단위로 볼 뿐. 하나로 합치거나 하나를 없애야 함. |

### 4-3. 누락 필터

| 누락 | 탭 | 설명 |
|------|:--:|------|
| **식사시간(meal_time)** | 식당 | breakfast/lunch/dinner/snack이 기록에 있는데 필터로 제공 안 됨. scene보다 실용적일 수 있음. |
| **body_level (바디)** | 와인 | 와인의 핵심 특성인데 필터에 없음. 산미/당도와 같은 구조적 속성. |
| **tannin_level (타닌)** | 와인 | 레드 와인 필수 특성. body와 함께 제공되어야 자연스러움. |
| **적합도(similarity)** | 버블 | CF 적합도 점수가 UI에 표시되는데 필터로는 못 걸음. |
| **유형(focus_type)** | 버블(홈) | 식당/와인 버블 구분이 `BUBBLE_FILTER_ATTRIBUTES`엔 있는데 `HOME_BUBBLE_FILTER_ATTRIBUTES`엔 없음. |
| **멤버 수** | 버블(홈) | 탐색 필터엔 있는데 홈 필터엔 없음. |

### 4-4. UX 문제

| 문제 | 설명 |
|------|------|
| **품종 55개 flat list** | 스크롤이 너무 긺. White/Red 구분 없이 나열. 그룹핑 필요. |
| **prestige children 불일치** | `RESTAURANT_FILTER_ATTRIBUTES`에는 children 없음. `buildMapFilterAttributes`에만 있음. → 일반 뷰에서는 세부 등급 선택 불가? |
| **생활권 서울만** | `LOCATION_AREA_TREE`에 서울만 존재. 부산/제주 생활권 없음. 사용자가 부산 생활권을 찾으면 빈 결과. |
| **빈티지 하드코딩** | 2024~2018로 고정. 매년 업데이트 필요한 유지보수 부채. |
| **버블 지역 8개 하드코딩** | `BUBBLE_FILTER_ATTRIBUTES`의 area가 강남/을지로 등 8개 고정. 동적이어야 함. |

### 4-5. 구현 불일치

| 불일치 | 설명 |
|--------|------|
| **BUBBLE_FILTER_ATTRIBUTES vs HOME_BUBBLE_FILTER_ATTRIBUTES** | 같은 대상(버블)에 대해 2개의 완전히 다른 필터 세트. 어디서 어느 것을 쓰는지 혼란. |
| **prestige children** | 일반 뷰: children 없음 (type만 선택). 지도 뷰: children 있음 (grade까지). 동일 필터가 뷰에 따라 다른 UX. |
| **visit_date 의미** | 식당: "시기" (방문 시기). 와인: "시음시기". 같은 key인데 라벨만 다름. 일관성은 있으나 혼동 가능. |

## 5. 필터 체계 리디자인 제안

### 5-1. 설계 원칙

1. **3계층 필터 모델**: 뷰(데이터 소스) → 속성(대상 특성) → 기록(경험 메타)
2. **공통 필터 통일**: 3탭 공통 필터는 key/UX 통일
3. **실용성 기준**: 사용 빈도 낮은 필터는 "더보기"로 이동
4. **동적 옵션**: 하드코딩 옵션은 DB 또는 사용자 데이터 기반 동적 생성

### 5-2. 3계층 필터 모델

```
┌─────────────────────────────────────────────┐
│ Layer 1: 뷰 (데이터 소스)                      │
│   공통: mine / following / bubble             │
│   버블: mine / following / public             │
├─────────────────────────────────────────────┤
│ Layer 2: 속성 (대상 고유 특성)                  │
│   식당: 음식종류, 위치, 가격대, 명성            │
│   와인: 스타일, 품종, 산지, 빈티지              │
│   버블: 유형, 역할, 멤버 수                    │
├─────────────────────────────────────────────┤
│ Layer 3: 기록 (경험 메타데이터)                 │
│   공통: 점수, 시기                             │
│   식당: 상황, 식사시간                          │
│   와인: 페어링, 산미, 당도, 바디, 복합도        │
│   버블: 적합도, 활성도                         │
└─────────────────────────────────────────────┘
```

## 6. 소팅 체계 통일

### 현재 문제
- 식당/와인/버블 소팅이 완전히 다른 네이밍과 옵션
- 버블에 거리순이 있을 수 있는데(근처 버블) 없음

### 리디자인

| 소팅 | 식당 | 와인 | 버블 |
|------|:----:|:----:|:----:|
| 최신순 (latest) | O | O | — |
| 점수 높은순 (score_high) | O | O | — |
| 점수 낮은순 (score_low) | O | O | — |
| 이름순 (name) | O | O | O |
| 방문 많은순 (visit_count) | O | O | — |
| 거리순 (distance) | O | — | — |
| 기록 많은순 (records) | — | — | O |
| 회원 많은순 (members) | — | — | O |
| 활동량 많은순 (weekly_activity) | — | — | O |
| 최근 활동순 (activity) | — | — | O |

> 식당/와인은 통일 O. 버블은 대상 성격이 달라 별도 소팅 유지가 맞음.

## 7. UX 개선 사항

### 7-1. 품종 그룹핑

```
현재: 55개 flat list (스크롤 지옥)

개선안:
┌─────────────────┐
│ ← 품종           │
├─────────────────┤
│ ── White ──      │
│  뮈스카           │
│  리슬링           │
│  소비뇽 블랑      │
│  샤르도네         │
│  …               │
│ ── Red ──        │
│  피노 누아        │
│  메를로           │
│  카베르네 소비뇽   │
│  …               │
└─────────────────┘
```

FilterAttributeOption의 `group` 필드가 이미 존재하므로 옵션에 `group: 'White'` / `group: 'Red'` 추가만 하면 됨.

### 7-2. 빈티지 동적 생성

```typescript
// 현재: 하드코딩
options: [
  { value: '2024', label: '2024' },
  { value: '2023', label: '2023' },
  // ...매년 업데이트 필요
]

// 개선: 동적 생성 함수
function buildVintageOptions(): FilterAttributeOption[] {
  const currentYear = new Date().getFullYear()
  const years = Array.from({ length: 6 }, (_, i) => currentYear - i)
  return [
    ...years.map(y => ({ value: String(y), label: String(y) })),
    { value: `before_${currentYear - 6}`, label: `${currentYear - 6} 이전` },
  ]
}
```

### 7-3. Prestige children 통일

일반 뷰의 `RESTAURANT_FILTER_ATTRIBUTES`에도 children을 추가하여 지도 뷰와 동일한 세부 등급 선택 제공:

```typescript
{
  key: 'prestige',
  label: '명성',
  type: 'select',
  options: [
    { value: 'michelin', label: '미슐랭', children: [
      { value: '3_star', label: '★★★' },
      { value: '2_star', label: '★★' },
      { value: '1_star', label: '★' },
      { value: 'bib', label: '빕 구르망' },
    ]},
    { value: 'blue_ribbon', label: '블루리본', children: [
      { value: '3_ribbon', label: '3리본' },
      { value: '2_ribbon', label: '2리본' },
      { value: '1_ribbon', label: '1리본' },
    ]},
    { value: 'tv', label: 'TV출연', children: [
      { value: '흑백요리사', label: '흑백요리사' },
      // ...
    ]},
    { value: 'none', label: '수상없음' },
  ],
}
```

### 7-4. 생활권 확장

```
현재: 서울만
개선: 부산/제주 생활권 추가
```

## 10. 미결 사항

1. **range 타입 UI**: `purchase_price`를 삭제하면 range 타입이 코드에서 사라짐. 향후 필요 시 UI 컴포넌트부터 구현 필요.
2. **다중 선택 확장**: 현재 필터는 단일 선택(eq). "한식 OR 일식" 같은 다중 선택은 Advanced Filter로만 가능. 주요 필터에 multi-select 지원 검토.
3. **필터 조합 저장**: 자주 쓰는 필터 조합을 프리셋으로 저장하는 기능 (SavedFilter 엔티티는 이미 존재).
4. **빈 결과 가이드**: 필터 결과가 0건일 때 "이 조건의 기록이 없습니다" + 필터 완화 제안.
