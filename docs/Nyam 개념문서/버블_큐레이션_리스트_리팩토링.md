## 1. 왜 바꾸는가

### 기존 문제

버블의 멘탈 모델이 "동호회/커뮤니티"였다. 이 모델에서는:

- **기록이 있어야만** 버블에 공유할 수 있다 (찜만으로는 불가)
- **사람을 팔로우**하면 그 사람의 전체 기록이 보인다 (선택 불가)
- **개인 리스트**를 만들 방법이 없다 (찜은 flat, 분류 불가)

사용자가 실제로 원하는 것:

1. "2026 미슐랭" 같은 **목적별 리스트**를 만들고 하나씩 방문하며 점수 매기기
2. 다른 사람의 **특정 리스트만** 팔로우하기 (을지로 리스트만, 와인 리스트만)
3. 아직 안 가본 곳도 **리스트에 담아두기** (찜 상태로)

### 핵심 인사이트

> 버블은 모임이 아니라 **리스트**다.
> 혼자 관리하냐, 여럿이 관리하냐의 차이일 뿐.

## 2. 새로운 버블 = 큐레이션 리스트

### 정의

#### 왜 "리스트"가 아니라 "버블"인가

리스트는 콘텐츠(식당/와인 목록)다. 버블은 **그 리스트를 함께 공유하는 사람들의 모임**이다.

```
리스트 = 콘텐츠 (어떤 식당/와인이 담겨 있는가)
버블  = 리스트 + 사람들 (누가 이 리스트를 함께 만들고 보는가)
```

"리스트"라고 부르면 콘텐츠만 남고 사람이 빠진다. "버블"이라는 이름에는 **같은 맥락을 공유하는 사람들의 울타리**라는 뜻이 담겨 있다. 그래서 버블에는:
- 리스트(콘텐츠)가 있고
- 멤버(사람)가 있고
- 멤버들의 평가로 만들어지는 **맥락화된 점수**가 있다

1인 버블이어도 마찬가지다. 나 혼자만의 맥락 — 나만의 관점으로 큐레이션한 리스트다. 공개하면 다른 사람이 팔로우하거나 가입해서, 1인의 맥락이 여러 사람의 맥락으로 확장된다.

Spotify에 비유하면:
- 플레이리스트 = 리스트 (곡 목록)
- 플레이리스트 + 구독자 + 공동 편집자 = 버블 (곡 목록을 중심으로 모인 사람들)

#### 버블의 구조

```
버블 (bubble)
├── 리스트: 큐레이션된 식당/와인 목록 (콘텐츠)
├── 멤버: 이 리스트를 함께 만들고/보는 사람들 (사회적 맥락)
├── 점수: 멤버들의 평가로 산출되는 맥락화된 CF 점수 (의미)
└── 소통: 댓글, 리액션 (상호작용)
```

### 리스트 = 맥락(Context). 점수는 맥락 안에서 의미를 갖는다

버블 점수는 단순히 "더 많은 사람이 평가해서 더 신뢰할 수 있는 점수"가 아니다. **특정 맥락 안에서의 평가**다.

네이버 4.5점은 "모든 사람의 평균"이라 아무 맥락이 없다. Nyam 점수도 전체 유저 CF니까 맥락이 넓다. 하지만 버블 점수는 **이 리스트를 만든/관리하는 사람들의 관점**에서의 점수다.

```
같은 식당 "리스또란떼 에오":

  Nyam 점수    85  ← 전체 유저 CF. "대체로 좋은 곳"
  내추럴와인   91  ← 내추럴와인 리스트 멤버들의 평가. "내추럴와인 관점에서 훌륭"
  가성비       63  ← 가성비 리스트 멤버들의 평가. "가성비로는 별로"
  대학동기     90  ← 내 친구들의 평가. "우리끼리는 여기 좋아해"
```

**같은 식당인데 점수가 다르다.** 이것이 핵심이다. 점수가 다른 이유는 신뢰도 차이가 아니라 **관점이 다르기 때문**이다.

이 구조가 작동하는 이유:

1. **리스트의 성격이 멤버를 필터링한다.** "내추럴와인" 리스트에 가입하는 사람은 내추럴와인에 관심이 있는 사람이다. 이 사람들의 CF는 "내추럴와인 관점"이라는 맥락을 자연스럽게 반영한다.

2. **맥락이 다르면 같은 데이터도 다르게 해석된다.** 가성비파에게 5만원 코스는 비싸지만, 파인다이닝파에게는 합리적이다. 각 리스트의 멤버 구성이 이 관점 차이를 반영한다.

3. **사용자는 맥락을 선택할 수 있다.** "오늘 가성비 좋은 곳 갈까?" → 가성비 리스트 점수를 본다. "특별한 날 어디 갈까?" → 파인다이닝 리스트 점수를 본다. 하나의 절대적 점수가 아니라 **상황에 맞는 점수를 고를 수 있다.**

이것이 "리스트 = 맥락"의 본질이다. 버블이 동호회였을 때도 맥락화된 예측은 있었지만, 리스트 모델에서는 이 맥락이 더 자연스럽고 다양해진다:

| 리스트 유형 | 맥락 | 점수의 의미 |
|------------|------|------------|
| 장르/품종 | 관심사 전문성 | "이 관심사의 전문가들이 보는 평가" |
| 지역 | 생활권 | "이 동네를 잘 아는 사람들의 평가" |
| 인간관계 | 사회적 신뢰 | "내가 아는 사람들의 평가" |
| 가치관 | 평가 기준 | "같은 기준으로 평가하는 사람들의 평가" |
| 개인 목적 | 자기 관리 | (1인 리스트는 CF 대상 아님 — 내 점수만) |

> **한 문장 요약**: 버블 점수는 "더 정확한 점수"가 아니라 "다른 관점의 점수"다. 리스트가 맥락을 정의하고, 맥락이 점수에 의미를 부여한다.

### 리스트에 담기는 것

리스트의 단위는 **대상(식당/와인)**이다. 기록이 아니다.

```
리스트에 보이는 대상:
├── 자동: 필터 조건에 매칭된 기록의 대상
├── 자동: 필터 조건에 매칭된 찜의 대상
└── 수동: 직접 추가한 대상 (기록 없이 찜만 해도 가능)
```

### 세 가지 큐레이션 모드

| 모드 | 동작 | 예시 |
|------|------|------|
| **자동** | 필터 조건 설정 → 조건 맞는 기록/찜 자동 추가/제거 | "광화문 식당, 만족도 90+" |
| **수동** | 사용자가 직접 대상을 리스트에 추가/제거 | "2026 미슐랭"에 하나씩 담기 |
| **혼합** | 자동 규칙 + 수동 추가 공존 | 자동으로 채워지되, 수동으로도 추가 가능 |

### 자동 큐레이션의 확장

기존 자동 공유는 **기록만** 평가했다. 새 모델에서는 **찜도** 평가 대상이다.

```
기존: 기록 생성/수정 → 필터 평가 → bubble_shares INSERT/DELETE
신규: 기록 생성/수정 → 필터 평가 → bubble_items INSERT/DELETE
      찜 토글        → 필터 평가 → bubble_items INSERT/DELETE
```

### 수동 큐레이션

기록 없이도 대상을 리스트에 담을 수 있다.

```
식당/와인 상세 페이지 → "리스트에 추가" → 리스트 선택 → 추가
리스트 상세 페이지 → "대상 추가" → 식당/와인 검색 → 추가
기록 직후 → "리스트에 추가" → 리스트 선택
```

수동으로 추가된 항목은 자동 규칙과 **독립적**이다. 자동 규칙을 변경해도 수동 항목은 유지된다.

### 찜과의 관계

찜은 그대로 남는다. 찜 = "일단 저장" (원탭, 마찰 0). 리스트 = "정리된 큐레이션".

- 같은 식당이 찜이면서 동시에 3개 리스트에 있을 수 있다
- 찜은 리스트에 넣는 것과 독립적인 행위다
- 자동 모드에서는 찜도 필터 매칭 대상이 된다

## 3. 기존 모델 → 새 모델 변환

### 개념 매핑

| 기존 | 신규 | 변화 |
|------|------|------|
| 버블 = 동호회 | 버블 = 큐레이션 리스트 | 멘탈 모델 전환 |
| bubble_shares (record_id 필수) | bubble_items (target 단위, record 선택적) | 데이터 단위 변경 |
| 기록을 공유 | 대상을 큐레이션 | 행위 전환 |
| 기록만 자동 매칭 | 기록 + 찜 자동 매칭 | 범위 확장 |
| 수동 = 기록 선택 공유 | 수동 = 대상 직접 추가 | 유연성 증가 |
| UNIQUE(record_id, bubble_id) | UNIQUE(bubble_id, target_id, target_type) | 집계 단위 변경 |

### 바뀌지 않는 것

- 버블 생성/가입/공개설정/가입정책 — 그대로
- 멤버 역할 4종 (owner/admin/member/follower) — 그대로
- 데이터 접근 계층 5단계 — 그대로
- 댓글/리액션 — 그대로
- 경험치 기반 가입 조건 — 그대로
- CF 기반 버블 점수 — 그대로 (멤버 스코프 CF)
- 취향 유사도 매칭 — 그대로
- 주간 랭킹 — 그대로 (데이터 소스만 bubble_items로 변경)

## 4. 팔로우 모델의 변화

### 기존: 사람 팔로우 = 전체 기록

```
김영수를 팔로우 → 김영수의 모든 기록이 보인다
```

### 신규: 리스트 팔로우 = 선택적 구독

```
김영수의 "을지로 맛집" 리스트만 팔로우 → 해당 리스트의 대상만 보인다
김영수의 "와인바 투어" 리스트도 팔로우 → 추가로 보인다
김영수 전체를 개인 팔로우 → 여전히 가능 (기존 방식)
```

| | 개인 팔로우 | 리스트(버블) 팔로우 |
|---|---|---|
| 단위 | 사람 1명의 전체 기록 | 목적별 리스트 하나 |
| 선택적 구독 | 불가 (전체 기록) | 가능 (리스트 단위) |
| 핵심 목적 | 신뢰 부스트 (CF 가중치) | 특정 주제/맥락의 큐레이션 구독 |

## 5. 사용 시나리오

### A. 개인 목적별 리스트

```
"2026 미슐랭" — 1인, 비공개, 수동

→ 미슐랭 식당을 하나씩 리스트에 추가 (기록 없이 찜만으로도 가능)
→ 방문할 때마다 기록 → 리스트에서 진행 상황 한눈에
→ 순수 개인용
```

### B. 자동 필터 리스트

```
"광화문 90점 이상" — 1인, 공개, 자동

→ 필터 조건에 맞는 기록/찜이 자동으로 리스트에 추가/제거
→ 기록이 쌓일수록 리스트가 자동으로 풍성해짐
→ 팔로워들이 구독하여 참고
```

### C. 지인 공동 리스트

```
"직장 점심" — 5명, 비공개, 혼합

→ 자동 규칙으로 회사 근처 기록 자동 추가
→ "여기 가보자" 수동으로 찜만 해서 리스트에 담기도 가능
→ 댓글/리액션으로 소통
```

### D. 선택적 구독

```
김영수가 3개 리스트 운영:
  - "을지로 혼밥" (public, closed)
  - "와인바 투어" (public, closed)
  - "가족 외식" (private)

박소연은 "을지로 혼밥"만 팔로우
→ 김영수의 전체 기록이 아닌, 을지로 관련 큐레이션만 구독
```

## 6. 리팩토링 계획

### 6-1. DB 변경 — `bubble_items` 테이블

```sql
-- 새 테이블
CREATE TABLE bubble_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  bubble_id UUID NOT NULL REFERENCES bubbles(id) ON DELETE CASCADE,
  target_id UUID NOT NULL,
  target_type VARCHAR(10) NOT NULL CHECK (target_type IN ('restaurant', 'wine')),
  added_by UUID NOT NULL REFERENCES users(id),
  source VARCHAR(10) NOT NULL DEFAULT 'manual' CHECK (source IN ('auto', 'manual')),
  record_id UUID REFERENCES records(id) ON DELETE SET NULL,
  added_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(bubble_id, target_id, target_type)
);

-- 인덱스
CREATE INDEX idx_bubble_items_bubble_target
  ON bubble_items(bubble_id, target_type);              -- 리스트 뷰 조회 (식당/와인 탭)
CREATE INDEX idx_bubble_items_bubble_source
  ON bubble_items(bubble_id, source);                   -- auto 항목 일괄 재평가
CREATE INDEX idx_bubble_items_added_by
  ON bubble_items(added_by);                            -- 사용자별 큐레이션 조회

-- RLS
ALTER TABLE bubble_items ENABLE ROW LEVEL SECURITY;

-- SELECT: 버블 멤버(member 이상) + 팔로워(follower). 
--         private 버블은 멤버만. public 버블은 팔로워도 이름+점수 수준 열람.
-- INSERT: 멤버(member 이상). owner/admin은 물론, 일반 member도 자기 큐레이션 추가 가능.
--         follower는 INSERT 불가.
-- UPDATE: 없음 (immutable. 변경 시 DELETE → INSERT)
-- DELETE: source='manual' → 본인(added_by=current_user) 또는 admin/owner.
--         source='auto'  → 시스템만 (자동 규칙 재평가 시). admin/owner는 수동 삭제 가능.

-- 기존 데이터 이관 (manual 우선, 같은 소스면 최신순)
INSERT INTO bubble_items (bubble_id, target_id, target_type, added_by, source, record_id, added_at)
SELECT DISTINCT ON (bubble_id, target_id, target_type)
  bubble_id, target_id, target_type, shared_by,
  CASE WHEN auto_synced THEN 'auto' ELSE 'manual' END,
  record_id, shared_at
FROM bubble_shares
ORDER BY bubble_id, target_id, target_type,
  auto_synced ASC,    -- manual(false) 우선 → auto에 의해 덮어씌워지지 않음
  shared_at DESC;     -- 같은 소스 내에서는 최신 기록 선택

-- 이관 후 bubble_shares는 Phase 5까지 유지 (롤백 안전망)
```

**핵심 차이**: `bubble_shares`는 `UNIQUE(record_id, bubble_id)` — 기록 단위. `bubble_items`는 `UNIQUE(bubble_id, target_id, target_type)` — 대상 단위.

하나의 대상에 여러 기록이 있어도 bubble_items는 1행. 기록들은 records 테이블에서 target_id로 JOIN 조회.

**`added_by`의 의미**: source='manual'이면 직접 추가한 사용자. source='auto'이면 해당 기록의 작성자(records.user_id).

### 6-2. Phase별 구현 계획

#### Phase 1: DB + Domain 기반 (마이그레이션 + 엔티티 + 인터페이스)

| 작업 | 파일 | 규모 |
|------|------|------|
| `bubble_items` 테이블 + RLS + 데이터 이관 | `supabase/migrations/053_bubble_items.sql` | 신규 |
| `BubbleItem` 엔티티 추가 | `domain/entities/bubble.ts` | 수정 |
| `BubbleRepository`에 target 기반 메서드 추가 | `domain/repositories/bubble-repository.ts` | 수정 |

```typescript
// domain/entities/bubble.ts — 신규
interface BubbleItem {
  id: string;
  bubbleId: string;
  targetId: string;
  targetType: 'restaurant' | 'wine';
  addedBy: string;
  source: 'auto' | 'manual';
  recordId: string | null;
  addedAt: string;
}

// domain/repositories/bubble-repository.ts — 추가 메서드
addItem(bubbleId: string, targetId: string, targetType: string, source: 'auto' | 'manual'): Promise<void>;
```

#### Phase 2: 자동 큐레이션 확장 (기록 + 찜 매칭)

| 작업 | 파일 | 규모 |
|------|------|------|
| `bubble-share-sync.ts` → target 기반 동기화 | `domain/services/` | 대규모 |
| 찜도 자동 매칭 대상에 포함 | `bubble-share-sync.ts` | 수정 |
| Supabase 구현체 bubble_items 쿼리 | `infrastructure/supabase-bubble-repository.ts` | 대규모 |
| DI container 갱신 | `shared/di/container.ts` | 수정 |

**찜의 자동 매칭 방식**: 찜에는 satisfaction, visitDate 등 기록 필드가 없다. 매칭 가능한 속성은 대상의 메타데이터(지역, 장르, 와인 타입 등)로 제한된다. 구현 방식:

```
1. 찜 토글 시 → 대상의 restaurant/wine 엔티티 정보 조회
2. 해당 정보로 필터 규칙 평가 (genre, area, wine_type 등)
3. 매칭되면 bubble_items에 source='auto'로 INSERT
```

**share_rule 구조 확장**: 기존 `bubble_members.share_rule` JSONB에 `includeBookmarks` 필드 추가.

```json
{
  "mode": "filtered",
  "conjunction": "and",
  "includeBookmarks": true,
  "rules": [
    { "field": "area", "operator": "is", "value": "광화문" },
    { "field": "satisfaction", "operator": "gte", "value": 90 }
  ]
}
```

- `includeBookmarks: true` → 찜도 자동 매칭 대상에 포함 (대상 메타데이터로 필터 평가)
- `includeBookmarks: false` (기본값) → 기존과 동일, 기록만 매칭
- `mode: "all"` + `includeBookmarks: true` → 모든 기록 + 모든 찜 자동 포함

## 7. 리스크 & 주의사항

| 리스크 | 대응 |
|--------|------|
| bubble_shares → bubble_items 이관 시 데이터 손실 | manual 우선 DISTINCT ON (`auto_synced ASC, shared_at DESC`). 기존 테이블은 Phase 5까지 유지 (롤백 안전망) |
| 같은 대상에 auto+manual 혼재 시 이관 | manual 우선 선택 (auto_synced ASC). manual 항목이 살아남음 |
| 찜의 자동 매칭 속성 제한 | 찜은 대상 메타데이터(지역, 장르)로만 매칭. 만족도 등 기록 속성은 매칭 불가 → `includeBookmarks` 토글로 사용자 명시 선택 |
| 수동/자동 혼합 시 삭제 충돌 | source 필드로 구분. manual 항목은 자동 규칙 재평가에서 제외 |
| bubble_shares 참조 코드가 많음 (~20파일) | Phase별 점진 전환. infrastructure 4개 repository 포함 |
| 기존 trigger (011, 052) | bubble_items 대응 trigger로 교체. bubbles.record_count 갱신 로직 포함 |
| weekly-ranking-snapshot 재설계 | target_id 직접 JOIN으로 변경. 찜만 있는 항목(기록 없음)은 랭킹에서 자동 제외 |
| XP 어뷰징 (찜으로 대량 추가) | 찜만으로 리스트 추가 시 XP 0. 기록 기반 큐레이션만 XP 부여 |
