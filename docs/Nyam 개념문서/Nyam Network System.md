# Nyam Network System

> 기록이 신뢰를 만들고, 신뢰가 관계를 만들고, 관계가 더 나은 기록을 만든다.

Nyam의 소셜 체계는 세 시스템이 하나의 순환 고리를 이룬다.

```
기록 활동 ──→ XP/Level (신뢰 축적)
                │
         ┌──────┴──────┐
         ▼             ▼
      Follow         Bubble
    (1:1 관계)     (N:N 커뮤니티)
         │             │
         └──────┬──────┘
                ▼
         더 풍부한 기록
```

---

## 1. 세 시스템의 역할

| 시스템 | 핵심 질문 | 단위 |
|--------|----------|------|
| **XP/Level** | "이 사람이 얼마나 진짜 활동하는가?" | 개인 |
| **Follow** | "이 사람의 기록을 신뢰하고 보고 싶은가?" | 1:1 |
| **Bubble** | "이 그룹과 기록을 공유하고 싶은가?" | N:N |

### XP/Level — 신뢰의 척도

기록 활동을 정량화한 경험치. 레벨이 곧 "이 유저가 얼마나 진심인가"의 증거.

- **total_xp**: 누적 (절대 감소 안 함) → 레벨 산정
- **active_xp**: 최근 6개월 기록 기반 (자연 변동) → 버블 가입 기준
- XP의 96%+는 **기록 품질**에서 발생 (소셜 XP는 일일 상한 10)

| 기록 품질 | XP |
|----------|-----|
| 이름만 | 0 |
| + 사분면 점수 | +3 |
| + 점수 + 사진(EXIF GPS) | +8 |
| 풀 기록(점수+사진+코멘트+메뉴) | +18 |

→ 어뷰징(빈 기록 양산)으로는 레벨을 올릴 수 없다. 진짜 기록만 신뢰를 쌓는다.

### Follow — 개인 간 신뢰 관계

상대방의 기록을 보겠다는 의사 표현. 접근 권한의 기반.

- **상태**: pending → accepted / rejected
- **접근 레벨**: none(비관계) → follow(일방) → mutual(맞팔)
- 접근 레벨에 따라 프로필/기록 노출 범위가 달라짐

### Bubble — 그룹 기반 기록 공유

취향이 맞는 사람들이 모여 기록을 자동으로 공유하는 커뮤니티.

- 멤버별 **공유 규칙**(shareRule)으로 어떤 기록을 공유할지 필터링
- 기록 저장 시 규칙을 자동 평가하여 `bubble_shares`에 물리화
- 역할: owner > admin > member > follower (기록 공유는 member부터, follower는 피드 읽기만)

## 2. 세 시스템의 교차점

시스템들이 독립적으로 동작하지 않는다. 서로를 **조건**과 **트리거**로 참조한다.

### 2.1 XP → Follow: 팔로우 자격 조건

```
사용자 A가 B를 팔로우 요청
  ↓
B의 is_public 확인:
  is_public=true → follow_policy 무시, 누구나 자유 팔로우 (즉시 수락)
  ↓ (is_public=false일 때)
B의 follow_policy 확인:
  blocked       → 거절
  auto_approve  → 즉시 수락
  manual_approve → pending (B가 수동 승인)
  conditional   → A의 level과 record_count를 B가 설정한 기준과 비교
                   충족 → 자동 수락
                   미충족 → pending
```

`conditional` 정책의 조건값:
- `follow_min_level`: "최소 Lv.N 이상이어야 팔로우 가능"
- `follow_min_records`: "최소 N개 기록이 있어야 팔로우 가능"

→ "기록을 충분히 한 사람만 내 기록을 볼 수 있다"는 필터.

**역팔로우 자동 허용 원칙**: A→B 팔로우가 accepted 상태라면, B→A 역팔로우 시 A의 follow_policy를 무시하고 즉시 수락된다. 내가 먼저 다가갔으면 상대의 역팔로우를 막을 이유가 없다. (A→B가 pending 상태면 적용 안 됨 — 아직 관계가 성립 전)

### 2.2 Follow → XP: 소셜 경험치

```
팔로우 성립 시:
  일방 팔로우 → +1 XP (요청자)
  맞팔 성립  → +2 XP (스펙상 양쪽, 현재 코드는 요청자만 — 미구현 이슈)

제한: 소셜 XP 일일 상한 10
```

소셜 XP는 전체 XP의 극소 비율(~4% 미만). 팔로워 파밍으로는 레벨을 올릴 수 없다.

### 2.3 XP → Bubble: 가입 자격 조건

```
버블 가입 시도
  ↓
maxMembers 체크 (공통): 최대 인원 도달 시 가입 불가
  ↓
joinPolicy 확인:
  open          → 무조건 가입
  invite_only   → 초대 코드 필수 (없으면 불가)
  closed        → 가입 불가 (팔로우만)
  manual_approve → 신청 가능 (조건 미충족이어도 신청 자체는 가능, 오너가 판단)
  auto_approve  → minLevel AND minRecords 충족 시 자동 가입
                   미충족 시 가입 불가 (이유 표시)
```

→ 버블 오너가 "Lv.10 이상, 기록 20개 이상"처럼 커뮤니티 진입 기준을 설정.
→ 스펙상 총 XP(영구 레벨) AND/OR 활성 XP(최근 6개월 활동) 조합으로 기준 설정 가능. 현재 구현은 total_xp 기반 레벨만.

### 2.4 Bubble → 기록 → XP: 공유가 만드는 선순환

```
기록 저장
  ↓
XP 적립 (기록 품질 기반)
  ↓
활성 버블 멤버십 조회
  ↓
각 멤버십의 shareRule 평가
  mode='all'     → 무조건 공유
  mode='filtered' → 필터 조건 매칭 → 공유 여부 결정
  ↓
bubble_shares 테이블에 물리화
  ↓
버블 피드에 노출 → 다른 멤버들이 반응
  ↓
반응(좋아요/댓글) → 소셜 XP
```

### 2.5 Follow ↔ Bubble: RLS에서 만남

기록 접근 권한은 RLS에서 다중 경로로 판정:

```
기록 조회 요청
  ├─ 본인 기록 → 항상 허용
  ├─ 공개 프로필(is_public=true) → 허용
  ├─ bubble_shares 경유 → 같은 버블 멤버면 허용
  └─ follows 경유 → accepted 상태 + follow_policy != 'blocked' → 허용
```

Follow와 Bubble은 **독립된 접근 경로**. 팔로우 안 해도 같은 버블이면 기록을 볼 수 있고, 버블이 없어도 팔로우하면 볼 수 있다.

### 2.6 Follow → 홈 피드: 팔로잉 모드

홈에는 맞팔 친구의 기록을 모아보는 **팔로잉 피드**가 있다. 맞팔(mutual) 기록 + 버블 공유 기록을 통합하여 소스 필터(전체/버블/맞팔)로 구분 가능. Follow 시스템의 가장 직접적인 소비 지점.

## 3. 선순환 구조

### 신규 유저의 여정

```
Phase 1: 기록 축적
  └─ 기록 저장 → XP 축적 → 레벨 상승

Phase 2: 관계 형성
  └─ 다른 유저 팔로우 → 피드에서 기록 발견
  └─ 맞팔 → 서로의 기록 교환 → 소셜 XP

Phase 3: 커뮤니티 진입
  └─ 레벨 조건 충족 → 버블 가입
  └─ 공유 규칙 설정 → 기록 자동 공유
  └─ 버블 피드에서 인게이지먼트

Phase 4: 선순환
  └─ 버블 멤버의 기록을 보며 새 식당/와인 발견
  └─ 직접 방문 → 기록 → XP → 더 높은 레벨
  └─ 더 많은 팔로워 유치 → 더 좋은 버블 가입 가능
```

### 핵심 설계 원칙

**1. 기록이 모든 것의 시작점**
- XP는 기록에서 나온다 (96%+)
- 팔로우/버블의 가치는 기록이 공유될 때 발생
- 기록 없는 소셜 활동은 보상이 거의 없음

**2. XP는 신뢰의 대리인**
- 레벨이 높으면 "진짜 활동하는 사람"이라는 증거
- 팔로우/버블 모두 이 신뢰 지표를 조건으로 활용
- 어뷰징으로 신뢰를 위조할 수 없는 구조 (기록 품질 기반)

**3. Follow와 Bubble은 독립된 접근 경로**
- 둘 다 기록 접근을 제공하지만 성격이 다름
- Follow: "이 사람을 신뢰한다" (개인 판단)
- Bubble: "이 그룹에 속한다" (커뮤니티 소속)
- 하나가 없어도 다른 하나로 접근 가능

**4. 자동화로 마찰 최소화**
- 버블 공유: 기록 저장 시 shareRule 자동 평가
- 팔로우: auto_approve/conditional은 수동 승인 불필요
- XP: 기록 저장 시 자동 계산

## 4. 데이터 구조

```
users
├── total_xp, active_xp, active_verified  ← XP/Level (level은 total_xp에서 계산, 별도 필드 없음)
├── is_public, follow_policy            ← Follow 정책
├── follow_min_level, follow_min_records ← conditional 조건
├── follower_count, following_count     ← 비정규화 카운트
│
├── follows (1:N)
│   ├── follower_id, following_id
│   └── status (pending/accepted/rejected)
│
├── bubble_members (1:N)
│   ├── bubble_id, role (owner/admin/member/follower)
│   ├── status (pending/active/rejected)
│   └── share_rule (all/filtered + rules)
│
├── records (1:N)
│   └── → bubble_shares (N:N, 자동 물리화)
│
├── xp_log_changes (1:N)              ← XP 변동 이력
│   ├── xp_amount, reason
│   └── reason: record_score | record_photo | record_full | social_follow | social_mutual | milestone ...
│
└── xp_totals (1:N)                   ← 축별 XP 집계
    ├── axis_type: category | area | genre | wine_region | wine_variety
    └── axis_value, total_xp
```

## 5. 접근 레벨별 콘텐츠 노출

프로필 UI는 **accessLevel** 기반으로 노출을 제어한다. `is_public`은 RLS(데이터 접근)에만 영향을 주고, 프로필 UI 노출 범위는 팔로우 관계가 결정한다.

| accessLevel | 프로필 | 취향/통계 | Picks/최근기록 | 버블 컨텍스트 |
|-------------|--------|----------|---------------|-------------|
| none | "Lv.N 유저" + 기록수만 | X | X | X |
| follow | 전체 + 맛태그 + 팔로워수 | O | X | X |
| mutual | 전체 + 맛태그 + 팔로워수 | O | O | O (같은 버블 시) |

버블 피드는 별도 경로: `bubble_shares` 기반으로 공유된 기록만 버블 피드에서 열람.

> 공개 프로필(`is_public=true`)이어도 팔로우 없이는 accessLevel=none. RLS에서 기록 데이터는 접근 가능하지만, 프로필 UI는 팔로우해야 열린다.

## 6. 알림 연동

| 이벤트 | 알림 타입 | 수신자 | 액션 |
|--------|----------|--------|------|
| 팔로우 요청 (일방) | `follow_request` | 대상 | 승인/거절 (manual_approve 시) |
| 맞팔 성립 | `follow_accepted` | 양쪽 | 정보성 |
| 버블 초대 | `bubble_invite` | 초대 대상 | 수락/거절 |
| 버블 가입 신청 | `bubble_join_request` | 버블 오너 | 승인/거절 |
| 버블 가입 승인 | `bubble_join_approved` | 신청자 | 정보성 |

## 7. 관계 해제

| 이벤트 | 영향 |
|--------|------|
| 언팔로우 | accessLevel 즉시 변경, 피드에서 제외. **XP는 회수하지 않음** |
| 버블 탈퇴 | bubble_shares에서 본인 공유 삭제, 멤버 목록에서 제거 |
| 계정 삭제 | follows 양방향 삭제, 모든 버블 멤버십 삭제 (AUTH.md §5) |

## 8. UI 구조 방향

### 홈 탭 재구성

현재 홈 2탭(식당/와인) + 헤더 버블 버튼(별도 페이지)을 **홈 3탭**으로 통합한다.

```
현재:  [식당] [와인]  +  헤더 버블 버튼 → /bubbles [버블|버블러]
변경:  [식당] [와인] [버블]
```

- 기존 `/bubbles` 페이지 **완전 삭제**. 버블/버블러 탭 구분 없음.
- 헤더의 버블 버튼 제거.
- 홈 버블 탭에서 **내 버블 목록**을 카드로 나열. 카드 탭 → 버블 상세 페이지로 이동.

### 버블 상세 페이지 (신규)

기존 버블 상세 페이지(기록 리스트 중심)를 **SNS형 페이지**로 재설계한다.

```
기존: 버블 상세 = 공유된 기록 리스트 (식당/와인 탭 필터로 이미 가능 → 중복)
신규: 버블 상세 = 소셜 허브
```

버블 상세 페이지 구성:

```
버블 상세 (/bubbles/[id])
├── 버블 헤더 (아이콘, 이름, 설명, 멤버 수)
├── 멤버 목록 (아바타 나열, 탭 → 미니 프로필)
├── 버블 활동 요약 (주간 기록수, 인기 식당/와인 등)
├── 랭킹 (멤버별 기여도)
├── 댓글/리액션 피드 (소셜 인터랙션)
└── 버블 설정 (오너/관리자만)
```

기존 버블 상세의 기록 리스트 기능은 삭제한다. 이유:
- 식당/와인 탭에서 **버블별 필터**로 동일한 기록 확인 가능
- 버블 상세는 기록 나열이 아니라 **커뮤니티 소셜** 역할에 집중

### 아바타 메뉴

헤더 우측 아바타 드롭다운을 확장한다.

```
아바타 클릭
├── 프로필       (현행)
├── 팔로워/팔로잉  (신규)
└── 설정         (현행)
```

- **팔로워/팔로잉**: 전체 페이지로 이동. 탭으로 팔로워/팔로잉 전환. 목록 내 검색 + FollowButton.

### 상세 페이지 "모든 기록" 섹션 (구현 완료)

식당/와인 상세 페이지의 "버블 기록" → **"모든 기록"**으로 확장.

- **필터 칩**: 전체 / 버블 / 팔로잉 / 공개 — 데이터 소스별 구분
- **미니 프로필**: 닉네임/아바타 탭 → 미니 프로필 팝업 (타인이 볼 수 있는 프로필의 전부)
- 가시성은 기존 RLS 정책을 따름 (§2.5)

### 미니 프로필 (구현 완료)

타인의 프로필을 보는 유일한 방식. 별도 전체 프로필 페이지가 아닌 **팝업**으로 표시.

```
미니 프로필 팝업
├── 아바타 + 닉네임 + @핸들 + 레벨
├── 기록 통계 (식당 N / 평균 만족도, 와인 N / 평균 만족도)
├── 팔로워/팔로잉 수
├── 한줄 소개 + AI 취향 요약
├── 맛태그
├── 활동 버블 목록 (공개 버블, 탭 → 버블 상세)
├── 활동 지역 + 가입 시점
└── 팔로우 버튼
```

### 유저 발견 경로

별도 "사람" 탭이나 유저 검색 페이지를 만들지 않는다. 유저는 기존 맥락 안에서 자연스럽게 발견된다.

| 경로 | 설명 |
|------|------|
| **상세 페이지 "모든 기록"** | 식당/와인 상세에서 타인의 기록 → 아바타 탭 → 미니 프로필 → 팔로우 |
| **버블 상세 멤버 목록** | 버블 상세 → 멤버 아바타 탭 → 미니 프로필 → 팔로우 |
| **팔로워/팔로잉 목록** | 친구의 친구 발견 (아바타 메뉴 → 팔로워 목록 → 탭) |
| **버블 멤버 검색** | 버블 탭 검색에서 핸들(ID) 입력 시 전체 유저 대상 검색 |
| **알림** | 팔로우 요청 알림 → 미니 프로필 팝업 |

### 팔로워/팔로잉 페이지

**전체 페이지**로 구현한다.

```
/followers
├── [팔로워 N] [팔로잉 N] 탭
├── 검색바 (목록 내 필터링)
├── 팔로우 요청 (pending) 섹션 — 팔로워 탭 상단에 표시
│   └── [수락] [거절] 버튼
└── 유저 리스트
    └── 아바타 + 닉네임 + @핸들 + 레벨 + [팔로우/팔로잉/맞팔] 버튼
    └── 아바타 탭 → 미니 프로필 팝업
```

- 내 프로필: 아바타 메뉴에서 진입
- 타인: 미니 프로필의 팔로워/팔로잉 수 탭하여 진입

### 삭제 대상 (기존 페이지)

| 삭제 항목 | 이유 |
|-----------|------|
| `/bubbles` 페이지 (버블/버블러 탭) | 홈 버블 탭으로 흡수 |
| 기존 버블 상세의 기록 리스트 | 식당/와인 탭 버블 필터로 대체 (중복 제거) |
| 타인 전체 프로필 페이지 (`/users/[id]`) | 미니 프로필 팝업으로 대체 |
