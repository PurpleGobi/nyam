# 위치 필터 & 생활권 시스템

## 식당 테이블 위치 속성

`restaurants` 테이블에 위치 관련 컬럼이 5개 있다.

| 컬럼 | 타입 | 출처 | 설명 |
|------|------|------|------|
| `address` | TEXT | 카카오/네이버/구글 API 원본 | 전체 주소 ("서울 강남구 도곡로 408") |
| `country` | VARCHAR(50) | 기본값 '한국', 수동 입력 | 국가 단위 |
| `city` | VARCHAR(50) | 기본값 '서울', 수동 입력 | 도시 단위 |
| `district` | VARCHAR(50) | `address`에서 자동 파싱 | 구/군 단위 ("강남구") |
| `area` | VARCHAR(50) | `area_zones` 좌표 매칭 | 생활권 ("역삼", "홍대") |
| `lat` | FLOAT | 카카오/네이버/구글 API | 위도 |
| `lng` | FLOAT | 카카오/네이버/구글 API | 경도 |

### district 파싱 로직

식당 등록 API (`/api/restaurants`)에서 `address` 문자열의 토큰 중 `구/군/시`로 끝나는 것을 추출한다.

```
"서울 강남구 도곡로 408"     → "강남구"
"경기 성남시 분당구 판교로"   → "분당구"
"부산 해운대구 중동"         → "해운대구"
```

클라이언트가 `district`를 명시적으로 보내면 그대로 사용하고, 없으면 `address`에서 자동 추출한다.

## area_zones 테이블

### 왜 만들었나

사용자는 "강남", "홍대", "성수" 같은 생활권 이름에 익숙하다. 하지만 생활권은 행정구역과 1:1 대응이 안 된다.

- 강남구 안에 강남, 역삼, 삼성, 청담, 압구정, 신사 등 여러 생활권이 있고
- 하나의 생활권이 구 경계를 걸치기도 한다

동(洞) 이름으로 매핑하는 방법도 있지만, 도로명 주소에는 동이 없고 행정동/법정동 불일치 문제도 있다. 그래서 **좌표 기반**으로 판정한다.

### 테이블 구조

```sql
area_zones (
  name      VARCHAR(50)     -- 생활권 이름
  city      VARCHAR(50)     -- 소속 도시
  lat       DOUBLE PRECISION -- 중심 위도
  lng       DOUBLE PRECISION -- 중심 경도
  radius_m  INT              -- 반경 (미터)
)
```

### 작동 방식

식당의 `lat/lng`와 생활권의 중심 `lat/lng` 사이 거리를 haversine으로 계산하여, `radius_m` 이내이면 해당 생활권으로 판정한다.

```
haversine(restaurant.lat, restaurant.lng, zone.lat, zone.lng) <= zone.radius_m
```

여러 생활권의 반경이 겹칠 수 있으므로, 가장 가까운 것을 선택한다. 반경 내에 아무 생활권도 없으면 3km 이내 가장 가까운 곳으로 폴백하고, 그마저도 없으면 `null`로 남긴다.

### 현재 시드 데이터

서울 32개 생활권이 등록되어 있다 (강남, 역삼, 삼성, 청담, 압구정, 신사, 서초, 잠실, 송파, 홍대, 연남, 합정, 망원, 이태원, 한남, 녹사평, 을지로, 명동, 종로, 광화문, 성수, 건대, 왕십리, 여의도, 영등포, 신촌, 대학로, 북촌, 서촌, 상수, 한양대, 가로수길).

## 필터에서의 사용

필터 UI에서 위치 관련 속성은 두 가지를 제공한다.

| 필터 속성 | 매칭 방식 | 정확도 | 커버리지 |
|----------|----------|--------|---------|
| `district` (구) | `record.district === '강남구'` 문자열 비교 | 정확 | 전국 |
| `area` (생활권) | `record.area === '홍대'` 문자열 비교 (백필된 값) | 다소 부정확 (반경 기반이므로) | area_zones에 등록된 지역만 |

## 다른 지역/국가 확장

### 국내 다른 도시

`area_zones`에 행을 추가하면 된다. `city` 컬럼으로 구분한다.

```sql
INSERT INTO area_zones (name, city, lat, lng, radius_m) VALUES
  ('해운대',   '부산', 35.1631, 129.1635, 1500),
  ('서면',     '부산', 35.1579, 129.0596, 1200),
  ('제주시',   '제주', 33.4996, 126.5312, 2000),
  ('서귀포',   '제주', 33.2541, 126.5600, 2000);
```

### 해외

같은 구조를 그대로 쓸 수 있다. 생활권 개념은 전 세계 도시에 존재한다.

```sql
INSERT INTO area_zones (name, city, lat, lng, radius_m) VALUES
  ('Shibuya',    'Tokyo', 35.6580, 139.7016, 1200),
  ('Shinjuku',   'Tokyo', 35.6938, 139.7034, 1500),
  ('Le Marais',  'Paris', 48.8566, 2.3622, 800),
  ('Montmartre', 'Paris', 48.8867, 2.3431, 1000);
```

`district` 파싱은 국가별 주소 형식이 다르므로, 해외 확장 시에는 구 단위 대신 좌표 기반 `area`만 사용하는 것이 현실적이다.

### 생활권 추가/수정 시

1. `area_zones` 테이블에 행 추가/수정
2. 기존 식당 대상으로 백필 SQL 재실행 (좌표 기반 재매칭)
3. 신규 등록 식당은 INSERT 시점에 자동 매칭
