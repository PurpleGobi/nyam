# Nyam 콜드스타트 & 점진적 온보딩 기획서

> 작성일: 2026-03-19
> 참조: PRD v2.0 §3-1, §4-1~4-3, F01~F06, F13
> 핵심 목표: 온보딩은 30초로 끝내고, 나머지는 일상 속에서 자연스럽게

---

## 1. 문제 정의

### 1-1. 콜드스타트의 두 가지 층위

```
레벨 1: 앱이 텅 비어있다 (기록 0개 → 추천 불가, 사분면 비어있음)
레벨 2: 앱이 나를 모른다 (취향 모름 → 개인화 불가)
```

PRD v2의 기존 해법: "온보딩에서 취향 카드 + 지역 선택 + 가본 곳 체크 = 5분 안에 10~30개 기록 확보"

### 1-2. 기존 해법의 문제

**5분 온보딩은 너무 길다.**

| 레퍼런스 | 온보딩 소요 | 사용자 행동 |
|----------|-----------|-----------|
| TikTok | 0초 (입력 없음) | 스크롤하면 알고리즘이 학습 |
| Spotify | ~30초 (아티스트 3개 선택) | 바로 음악 재생 |
| Letterboxd | ~1분 (영화 검색+별점) | 바로 다이어리 시작 |
| Duolingo | ~1분 (언어+레벨 선택) | 바로 레슨 시작 |
| PRD v2 Nyam | ~5분 (카드+지역+체크+평가) | **너무 많은 걸 한꺼번에 시킴** |

**리서치 결론**: 온보딩에서 사용자에게 너무 많은 입력을 요구하면 **이탈률이 급증**. 특히 모바일에서는 **30초 이내**에 핵심 가치를 경험시켜야 한다.

### 1-3. 새로운 방향

```
온보딩: 30초. 최소한만 받고, 바로 앱을 쓰게 한다.
나머지: 일상 속에서, 틈틈이, 자연스럽게 정보가 쌓이게 한다.
```

## 2. 전략 개요: 3단계 점진적 프로파일링

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  Day 0: 즉시 온보딩 (30초)                       │
│  ────────────────────────                       │
│  가입 → 가본 곳 3~5개 체크 → 끝. 바로 홈 진입.    │
│                                                 │
│  Week 1~2: 컨텍스트 트리거 (수동 입력 최소화)      │
│  ────────────────────────                       │
│  위치 감지 → "여기 다녀오셨어요?" 넛지             │
│  사진 갤러리 → "이 사진 식당 기록으로 만들까요?"    │

│  버블 참여 → 더 기록하고 싶어짐                    │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 3. Day 0: 즉시 온보딩 (30초)

### 3-1. 플로우

```
Step 1: 가입 (10초)
  소셜 로그인 (카카오/구글/애플) → 닉네임 자동 설정
  별도 프로필 설정 없음. 나중에 변경 가능.

Step 2: "가본 곳 골라보세요" (20초)
  ┌──────────────────────────────┐
  │ 어디 자주 가세요?             │
  │                              │
  │ [광화문] [을지로] [강남]       │  현재 위치 기반 주변 동네 3~5개
  │ [성수] [홍대] [이태원]        │  + 서울 대표 동네
  │                              │  (위치 거부 시 서울 전체)
  │ 1개 이상 선택하면 다음 →       │
  └──────────────────────────────┘
  
  ┌──────────────────────────────┐
  │ 이 동네에서 가본 곳 있나요?    │
  │                              │
  │ ☐ 미진                       │  선택한 동네의 대표 식당
  │ ☐ 광화문 국밥                 │  미슐랭/블루리본 + 외부 평점 상위
  │ ☐ 토속촌                     │  + 유명 프랜차이즈
  │ ☐ 서울칼국수                  │
  │ ☐ 스시코우지                  │  10~15개 표시
  │ ...                          │
  │                              │
  │ ✓ 3개 선택됨                  │  체크만 하면 됨. 평가 안 시킴.
  │            [시작하기 →]       │
  └──────────────────────────────┘

Step 3: 홈 진입
  체크한 식당이 "나의 기록"에 바로 추가 (미평가 상태)
  홈에 기록이 보임 → 텅 빈 앱이 아님 → 성공
```

### 3-2. 핵심 결정: 온보딩에서 평가 안 시킴

PRD v2는 체크 후 바로 평가까지 시켰지만, **여기서는 체크만**.

```
이유:
  - 3~5개 식당에 각각 사분면 찍기 = 최소 15초 × 5 = 75초 추가
  - 온보딩 피로감 급증
  - "지금 당장 평가해야 해?" → 부담

대신:
  - 체크한 식당은 "가봤지만 아직 평가 안 함" 상태로 저장
  - 홈 타임라인에 표시: "미진 — 평가하기 →" 카드
  - 나중에 원할 때 탭해서 사분면 평가
```

### 3-3. 온보딩에서 안 하는 것

| 항목 | PRD v2 원안 | 변경 |
|------|-----------|------|
| 취향 카드 (2지선다) | 온보딩에서 | **안 함.** 기록 데이터에서 자연스럽게 파악 |
| 와인 취향 | 온보딩에서 (선택) | **안 함.** 와인 기록할 때 자연스럽게 |
| 상세 평가 | 온보딩에서 | **안 함.** 나중에 원할 때 |
| 지역 인증 | 온보딩에서 | **안 함.** 나중에 넛지로 유도 |

### 3-4. 체크한 식당의 초기 상태

```sql
-- 온보딩 체크로 생성된 기록
INSERT INTO records (
  user_id, target_id, target_type,
  status,        -- 'checked' = 가봤지만 미평가
  axis_x, axis_y, satisfaction, scene  -- 전부 NULL
)
```

status: `checked` → 평가하면 `rated`로 전환

홈 타임라인에서 표시:
```
┌────────────────────────────────┐
│ 미진                           │
│ 한식 · 광화문                   │
│                [평가하기 →]     │  탭하면 사분면 UI
└────────────────────────────────┘
```

## 4. Week 1~2: 컨텍스트 트리거

온보딩 이후 첫 2주가 리텐션의 골든타임. **사용자가 앱을 열지 않아도** 자연스럽게 기록이 쌓이게 만든다.

### 4-1. 위치 기반 넛지

**트리거**: 사용자가 식당 밀집 지역에 도착 → 식사 시간대(11:30~13:30, 17:30~21:00) 이후

```
푸시 알림 (식사 후 1~2시간):
  "광화문에서 식사하셨나요?"
  [기록 남기기] [아니요]
  
  "기록 남기기" 탭 시:
  → 현재 위치 주변 식당 리스트 (가까운 순)
  → 탭 → 사분면 평가 (선택적으로 한줄평/사진)
```

**빈도 제한**:
- 하루 최대 1회
- 같은 동네에서 연속 알림 안 함 (최소 3일 간격)
- "아니요" 3회 연속 → 해당 동네 알림 2주 중단
- 설정에서 완전 OFF 가능

**위치 권한 없을 때**: 이 기능 비활성. 대신 시간 기반 넛지(4-2)로 대체.

### 4-2. 시간 기반 넛지

**트리거**: 저녁 식사 시간대 이후 (20:00~21:00)

```
앱 내 알림 (앱 열었을 때):
  "오늘 저녁 어디서 드셨어요?"
  [기록하기] [건너뛰기]
```

- 위치 권한 없는 사용자용 대체 메커니즘
- 푸시가 아닌 **앱 내 배너**로만 (덜 침습적)
- 주 2~3회만. 매일은 아님.

### 4-3. 사진 갤러리 연동

**트리거**: 앱 진입 시 갤러리에 최근 음식 사진 감지

```
홈 상단 카드:
  ┌──────────────────────────────┐
  │ 📷 최근 사진에서 발견          │
  │                              │
  │ [사진 썸네일] 3/19 12:34      │
  │ 이 사진으로 기록할까요?        │
  │                              │
  │ [기록하기] [무시]              │
  └──────────────────────────────┘
  
  "기록하기" 탭 시:
  → EXIF GPS → 식당 자동 추정
  → EXIF 날짜/시간 → 방문일 자동 입력
  → 사진 자동 첨부
  → 사분면 평가만 하면 끝
```

**핵심**: 사용자가 **이미 찍은 사진**을 활용. 새로운 행동을 요구하지 않는다.

**구현 참고**: Google Photos가 EXIF GPS + 시간으로 자동 앨범을 만드는 것과 동일 원리. Vision AI로 음식 사진 판별 → GPS로 식당 매칭.

**프라이버시**:
- 갤러리 접근 권한 명시적 요청
- 사진은 기기에서만 분석 (서버 전송 없음)
- 음식으로 판별된 사진만 제안 (전체 갤러리 스캔 아님)

### 4-4. 미평가 기록 넛지

온보딩에서 체크만 한 식당들을 **틈틈이** 평가하도록 유도.

```
홈 타임라인 상단:
  ┌──────────────────────────────┐
  │ 아직 평가하지 않은 곳이 3곳    │
  │                              │
  │ 미진 — [평가하기]             │  하나만 보여줌. 전부 한꺼번에 X
  │                              │
  │ 나중에                        │
  └──────────────────────────────┘
```

**규칙**:
- 한 번에 1개만 제안 (전부 한꺼번에 보여주면 부담)
- 하루 1회만
- "나중에" 3회 → 더 이상 넛지 안 함 (해당 식당에 대해)
- 사용자가 직접 목록에서 선택해서 평가하는 건 항상 가능

### 4-5. 새 동네 진입 넛지 (PRD v2 F13)

```
푸시: "성수동에 오셨네요!"
  
  ┌──────────────────────────────┐
  │ 🗺 성수동 맛집                │
  │                              │
  │ ☐ 앵커드                     │
  │ ☐ 어니언                     │  대표 식당 리스트
  │ ☐ 카페 장                    │
  │ ☐ 성수 곱창                   │
  │                              │
  │ 가본 곳 체크하기               │  온보딩과 동일한 체크 UI
  │                              │
  │ 찜 목록에 추가하기             │  안 가본 곳은 찜 유도
  └──────────────────────────────┘
```

**이것이 PRD v2의 "지역 푸시"와 "가본 곳 체크"를 결합한 것.** 온보딩에서 안 했던 것을 자연스러운 맥락에서 하게 된다.

## 5. Month 1+: 자연 축적

### 5-1. 기록 습관 형성 — Hook 모델 적용

Nir Eyal의 Hook 모델을 식도락 기록에 적용:

```
Trigger (트리거):
  외적: 식사 후 넛지, 새 동네 진입, 사진 감지
  내적: "이 집 어땠지?" 궁금함 → 앱 열기

Action (행동):
  사분면 한 번 찍기 = 3초. 마찰 극소화.

Variable Reward (변동 보상):
  - 사분면에 새 점이 찍힌다 (시각적 보상)
  - 레벨업 알림 (예측 불가 — 언제 될지 모름)
  - 버블에서 반응 (좋아요/댓글 — 소셜 보상)

Investment (투자):
  - 기록이 쌓인다 = 내 데이터가 풍부해진다
  - 사분면이 채워진다 = 떠나기 아까워진다
  - 경험치 레벨 = 쌓아온 것을 잃고 싶지 않다
```

### 5-2. 기록이 쌓이면 자동으로 열리는 기능

기록 수에 따라 기능이 **점진적으로 해금**되어 가치가 커진다:

| 기록 수 | 해금되는 것 | 사용자 경험 |
|---------|-----------|-----------|
| 0~2 | 기본 홈 + 미평가 목록 | "아직 비어있지만 금방 채워질 거야" |
| 3~5 | 사분면에 점이 보이기 시작 | "오, 내 취향이 보이네" |
| 5~10 | 재방문 추천 작동 | "여기 다시 가볼까?" |
| 10~20 | 상황별 추천 작동 | "오늘 데이트인데..." |
| 20+ | 사분면 패턴 명확 | "나는 이런 타입이구나" |
| 50+ | 정교한 추천 + 프로필 정체성 | "을지로 Lv.5, 진짜 잘 아는 사람" |

**핵심**: 추천/분석 기능이 기록 수에 따라 자동으로 열리면서, "더 기록하면 더 좋아지는" 동기가 자연스럽게 형성.

### 5-3. 주간 리마인드 (가볍게)

```
주 1회 (월요일 오전):
  "지난 주에 3곳 다녀오셨네요! 기록해볼까요?"
  [기록하기]
  
  또는:
  "지난 주 사진 중 기록하지 않은 식사가 있어요"
  [사진 보기]
```

- 앱 미사용 7일 이후에만
- 푸시 1회 → 반응 없으면 2주 후 1회 더 → 그래도 없으면 중단
- "짜증나는 리마인드"가 아니라 "도움이 되는 알림"

## 6. 와인 콜드스타트

와인은 식당보다 **더 전문적**이라 별도 전략 필요.

### 6-1. 와인 기록 진입 시점

온보딩에서 와인을 묻지 않는다. 사용자가 **와인에 관심을 보일 때** 시작.

```
트리거:
  a. 사용자가 직접 "와인 기록" 시도
  b. 식당 기록 시 "와인도 마셨나요?" 선택지
  c. 와인 관련 버블 가입
  
첫 와인 기록 시:
  "와인은 처음이시네요! 간단하게 시작해볼까요?"
  → 사분면 (산미×바디) + 만족도만
  → 향 팔레트는 건너뛰기 가능 ("나중에 해볼래요")
```

### 6-2. 와인 체크리스트 (식당과 동일 패턴)

와인에 관심을 보인 사용자에게 **나중에** 제안:

```
"마셔본 와인 골라보세요"
  
  ☐ 모엣 샹동 (스파클링)
  ☐ 옐로우 테일 (호주)
  ☐ 1865 (칠레)
  ☐ 몬테스 알파 (칠레)
  ...
  
  한국에서 흔한 와인 위주로 리스트 구성
  체크만 → 미평가 상태로 기록
```

## 7. 넛지 시스템 아키텍처

### 7-1. 넛지 유형 & 우선순위

| 우선순위 | 넛지 | 트리거 | 빈도 | 침습도 |
|---------|------|--------|------|--------|
| 1 | 사진 감지 | 앱 진입 시 | 매 진입 | 낮음 (앱 내 카드) |
| 2 | 미평가 기록 | 앱 진입 시 | 1일 1회 | 낮음 (앱 내 카드) |
| 3 | 식사 후 넛지 | 위치+시간 | 1일 1회 | 중간 (푸시) |
| 4 | 새 동네 | 위치 진입 | 동네당 1회 | 중간 (푸시) |
| 5 | 주간 리마인드 | 미사용 7일+ | 주 1회 | 중간 (푸시) |
| 6 | 시간 기반 | 저녁 8~9시 | 주 2~3회 | 낮음 (앱 내 배너) |

### 7-2. 넛지 충돌 방지

```
규칙:
  - 하루 최대 푸시 1개 (모든 넛지 합산)
  - 앱 내 카드는 동시에 1개만 (가장 우선순위 높은 것)
  - 사용자가 "건너뛰기" 3회 → 해당 넛지 타입 2주 중단
  - 설정에서 넛지 유형별 ON/OFF
  - 23:00~08:00 푸시 없음
```

### 7-3. 넛지 피로 관리

```
넛지 피로도 점수 (내부):
  푸시 전송 = +3
  앱 내 카드 표시 = +1
  사용자가 무시 = +2
  사용자가 "아니요" = +1
  사용자가 기록 완료 = -5 (보상감 → 피로 리셋)
  
  피로도 > 10 → 48시간 넛지 중단
  피로도 > 20 → 1주 넛지 중단
  매일 -1 자연 감소
```

## 10. 데이터 구조 보완

```sql
-- 기록에 status 필드 추가
ALTER TABLE records ADD COLUMN status VARCHAR(10) DEFAULT 'rated';
-- 'checked' = 온보딩/체크로 생성됨 (미평가)
-- 'rated' = 사분면 평가 완료
-- 'draft' = 임시 저장

-- 넛지 이력
CREATE TABLE nudge_history (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id),
  nudge_type VARCHAR(30),  -- 'location', 'time', 'photo', 'unrated', 'new_area', 'weekly'
  target_id UUID,          -- 식당/와인 ID (해당 시)
  status VARCHAR(10),      -- 'sent', 'opened', 'acted', 'dismissed', 'skipped'
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 넛지 피로도
CREATE TABLE nudge_fatigue (
  user_id UUID PRIMARY KEY REFERENCES users(id),
  score INT DEFAULT 0,
  last_nudge_at TIMESTAMPTZ,
  paused_until TIMESTAMPTZ  -- 피로도 높으면 이 시점까지 넛지 중단
);
```

## 12. 핵심 결정 요약

| 결정 | 선택 | 이유 |
|------|------|------|
| 온보딩 시간 | 30초 (동네+체크만) | 5분은 너무 길다. 모바일 이탈 방지 |
| 온보딩 평가 | 안 함 (체크만) | 마찰 최소화. 나중에 넛지로 유도 |
| 취향 카드 | 삭제 | 기록 데이터에서 자연스럽게 파악 가능 |
| 와인 온보딩 | 삭제 | 관심 보일 때 자연스럽게 시작 |
| 넛지 핵심 | 사진 갤러리 연동 | 이미 찍은 사진 활용 = 새 행동 불필요 |
| 넛지 빈도 | 하루 푸시 1개, 앱 내 카드 1개 | 피로 방지 |
| 데이터 수집 | 점진적 (30초 → 2주 → 1개월+) | Progressive Profiling |
| 기능 해금 | 기록 수에 따라 자동 | "더 기록하면 더 좋아지는" 동기 |
