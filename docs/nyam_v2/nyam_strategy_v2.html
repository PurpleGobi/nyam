정체성 선언

"내가 아는 사람들의 진짜 맛집 기록으로 만드는, 상황별 맛집 비서"
기록은 개인의 것이지만, 모이면 동네의 맛집 지도가 된다.
낯선 동네에서는 그 동네 사람의 기록이 가장 믿을 만한 가이드가 된다.
핵심 가치 1: 최소 마찰 기록 핵심 가치 2: 상황별 취향 분석 핵심 가치 3: 신뢰 기반 로컬 추천

1. 문제 정의 — 왜 Nyam이 필요한가

1-1. 핵심 페인포인트

"나의 식도락 여정을 기록하고 싶다. 언제, 누구랑, 어디서, 뭘 먹었고, 어땠는지."
"그런데 기록은 너무 귀찮다."

1-2. 기존 솔루션이 해결 못하는 것

기존 서비스하는 것안 되는 것
네이버/카카오맵식당 검색, 리뷰 열람개인 기록, 취향 분석, 신뢰 기반 추천 없음. 광고/조작 리뷰
인스타그램맛집 사진 공유, 팔로우검색/필터 불가, 데이터 구조화 없음, 취향 분석 없음
메모앱/카톡간단 기록검색 어려움, 축적 가치 없음, 공유 구조 없음
캐치테이블예약, 일부 큐레이션개인 기록 도구 아님, 예약 가능 식당만 대상

1-3. Nyam만의 기회 — 식도락 도메인의 구조적 강점

축영화 (레터박스)식도락 (Nyam)Nyam 우위
빈도주 1~2회하루 2~3회데이터가 빠르게 축적
취향 차원"이 장르 좋아해" (1D)"이 상황에서 이런 걸 좋아해" (2D)더 정밀한 개인화 가능
상황 변수거의 없음혼밥/데이트/회식/가족 전부 다름상황별 추천이라는 신규 가치
로컬 가치전세계 동일 콘텐츠동네마다 다름현지인 추천 = 최고의 신뢰
맥락 데이터제한적누구랑, 예산, 대기, 메뉴, 분위기...풍부한 태그/분류 가능

2. 전략 프레임워크

2-1. 레터박스에서 가져오는 것

채택 최소 마찰 기록

레터박스: 영화 검색 → 별점 → 끝
Nyam: 식당 검색 → 별점 → 상황 태그 → 끝
탭 3번이면 기록 완료. 리뷰/사진은 선택사항.

채택 리스트 = 성장 엔진

유저가 만드는 리스트("을지로 혼밥 맛집", "강남 접대용 일식")가 SEO/공유/발견을 동시에 견인. 레터박스에서 가장 중독적인 기능.

채택 사람 기반 피드

알고리즘 추천이 아니라 "취향이 비슷한 사람"을 팔로우. 그 사람의 기록이 곧 내 추천. 식당 리뷰 신뢰 문제를 구조적으로 해결.

Phase 2 연말 결산 (Year in Review)

"올해 가장 많이 간 동네, 최애 장르, 단골 식당, 총 지출 추정" → 공유 → 바이럴. 기록의 보상.

2-2. 당근마켓에서 배우는 것

인사이트 지리적 제한 = 신뢰의 근거

당근: 전국 → 동네로 좁히니 사기가 줄고 신뢰가 생김
Nyam: 동네 사람의 기록 = 현지인 가이드. 지리적 한계가 핵심 가치.

인사이트 한 동네부터 시작

당근도 판교에서 시작. Nyam도 특정 동네 1~2곳에서 밀도 있게 시작하고, 그 동네 기록이 충분해지면 확장.

인사이트 기록보다 네트워크가 본질

당근의 핵심은 중고거래가 아니라 "동네 커뮤니티". Nyam도 "기록"이 수단이고, "동네 맛집 신뢰 네트워크"가 진짜 자산.

주의 즉각적 인센티브 부재

당근은 "물건 팔면 돈"이라는 즉각 보상이 있었음. Nyam은 기록의 장기 보상만으로 초기 구동이 어려울 수 있음. 대안 필요.

2-3. 레터박스와의 구조적 차이 — 솔직한 인정

문제레터박스는Nyam은대응 전략
DB 구축 TMDB에 모든 영화 완비 자체 식당 DB 불가능 카카오맵/네이버/구글 교차검증으로 위임. Nyam은 "관계 데이터"만 소유
콘텐츠 보편성 전세계가 같은 영화를 봄 내 동네 식당은 나만 관심 보편성 포기 → 로컬 밀도에 집중. "을지로의 레터박스"
통계적 의미 영화 1편에 수백만 평가 식당 1곳에 수십 평가 대중 평점 대신 "내가 아는 사람들의 평가"로 전환
기록 동기 영화 취향 = 정체성 식당 기록 동기가 약함 상황별 분석이라는 고유 가치 + 다른 사람에게 도움이 되는 보람
콜드스타트 인기 영화 체크로 시작 기록 쌓여야 가치 기존 데이터 import + 크리에이터 모델 (소수 기록, 다수 소비)

3. 기능 목록 — Phase별 로드맵

Phase 1: MVP — "기록이 쌓이면 나를 아는 맛집 비서"

목표: 혼자서도 가치 있는 개인 기록 도구. 소셜 없이도 쓸 이유가 있어야 함.

#기능설명레터박스 대응
1 Quick Log (빠른 기록) 식당 검색 → 별점(0.5~5) → 상황 태그 1개 → 끝 (탭 3번)
선택: 한줄평, 사진, 함께 간 사람, 메뉴, 가격대 		Log + Rating
2 상황 태그 (Scene Tag) 6개 고정 태그: 혼밥 / 데이트 / 친구 / 가족 / 회식·접대 / 술자리
기록할 때 하나만 탭. 이게 Taste DNA의 2차원 축이 됨. 		Nyam 고유 (없음)
3 식당 검색 & 매칭 카카오맵 + 네이버 + 구글 교차검색
가게명 입력 → 후보 리스트 → 선택. 레터박스의 TMDB 역할. 		TMDB 연동
4 다이어리 (타임라인) 날짜별 기록이 쌓이는 타임라인 뷰
"나의 식도락 일기" — 돌아보는 재미 		Diary
5 가고 싶은 곳 (Wishlist) 저장해두고, 방문하면 자동으로 리스트에서 제거
레터박스 Watchlist의 식도락 버전 		Watchlist
6 기본 통계 총 방문 수, 장르 분포, 상황별 선호 패턴, 동네별 분포
"혼밥 할 때 일식 선호도 4.3" 같은 인사이트 		Stats (Pro)
7 사진 기반 자동완성 (보조) 사진 선택 시 EXIF(GPS/날짜) + Vision AI로 식당/메뉴 자동 추정
기록 마찰을 더 줄이는 보조 수단. 필수 아님. 		Nyam 고유
8 기존 데이터 Import 카카오맵 저장 목록 / 네이버 마이플레이스 / 구글맵 리뷰 가져오기
콜드스타트 해결. 첫 사용 시 "와, 내가 이만큼 다녔네" WOW 모먼트. 		IMDb Import

Phase 2: 소셜 — "내 사람들의 맛집 지도"

목표: 개인 기록에서 신뢰 네트워크로 확장. 기록의 사회적 가치 부여.

9 팔로우 & 피드 취향 비슷한 사람 팔로우 → Activity 피드에 그 사람의 기록 노출

11 버블 (폐쇄형 그룹) 초대제 맛집 공유 그룹. 직장 동료 버블, 동네 맘 버블, 와인 모임 버블
통계가 부족한 식도락 도메인에서 "아는 사람의 추천"이 대안 		Nyam 고유

13 상황별 추천 "오늘 데이트인데 을지로에서 어디 갈까?" → 데이트 상황에서 높은 평가를 받은 을지로 식당 추천

목표: 수익화 + 가치 사슬 확장. Phase 1~2 검증 후.

4. 콜드스타트 전략 — 닭이 먼저

4-1. 문제

기록이 쌓여야 가치 → 가치 없으면 기록 안 함 → 기록이 안 쌓임

4-2. 해결 접근

1. Import로 첫날부터 데이터 있게

카카오맵 저장 / 네이버 마이플레이스 / 구글맵 리뷰 → 한 번에 수십~수백 개 기록 생성

첫 사용 시 "내가 3년간 이만큼 다녔구나" WOW 모먼트

2. 크리에이터 모델 (1%가 기록, 99%가 소비)

위키피디아/인스타와 같은 구조. 동네별로 "맛잘알" 10~20명만 확보하면 동네 데이터가 만들어짐

맛잘알의 동기: 자기표현 + 팔로워 + (Phase 3) 수익

3. 기록 전에 "질문"부터

"을지로에서 혼밥 어디 가?" → 기존 기록에서 답을 줌 → 가치 체감 → 그 다음에 기록

기록하라고 하지 말고, 물어보게 만들기

4. 한 동네부터 밀도 있게

당근이 판교에서 시작한 것처럼. 특정 동네(을지로? 성수?)의 밀도를 먼저 확보

얇게 전국보다, 깊게 한 동네가 먼저

5. UX 핵심 설계

5-1. 기록 플로우 (최소 마찰)

[1단계] 식당 이름 검색 (자동완성)
[2단계] 별점 탭 (★★★★☆)
[3단계] 상황 태그 탭 (혼밥 / 데이트 / 친구 / 가족 / 회식 / 술자리)
[완료] 기록 저장 ✓

━━ 여기까지가 필수. 아래는 선택 ━━

[선택] 한줄평
[선택] 사진 추가 (EXIF로 식당/날짜 자동 추정)
[선택] 함께 간 사람 태그
[선택] 메뉴, 가격대, 대기시간

5-2. 습관 형성 설계 (Hook 모델)

단계Nyam 구현
Trigger (트리거) • 점심/저녁 시간대 푸시: "오늘 뭐 드셨어요?" (짧고 가볍게)
• 팔로우한 사람의 새 기록 알림
• 위시리스트 식당 근처 갈 때 알림 (Phase 2)
Action (행동) • 탭 3번 기록 (초저마찰)
• 리스트 만들기 (창작 욕구)
Variable Reward (보상) • 통계가 채워지는 쾌감 ("혼밥 때 일식 4.3, 중식 3.1")
• 피드에서 맛집 발견
• 내 리스트에 좋아요/팔로우
Investment (투자) • 다이어리가 쌓일수록 이탈 비용 증가
• 팔로우 그래프가 "나의 맛집 신문"이 됨
• 상황별 DNA가 정교해질수록 추천 정확도 상승

5-3. "아하 모먼트" 설계

첫 번째 Import 후 타임라인

"카카오맵 저장 목록을 가져왔더니, 2년간 내 식도락 히스토리가 한눈에..."

두 번째 상황별 취향 발견

10~20개 기록 후: "혼밥은 일식, 데이트는 이탈리안, 회식은 고기... 나 이렇구나"

세 번째 낯선 동네에서 도움

"출장 가서 을지로인데... Nyam에서 을지로 맛잘알의 회식 추천 보고 갔더니 대성공"

7. 리스크 & 미해결 과제

#리스크심각도대응 방향
1 기록 동기 부족 높음 Import로 초기 데이터 확보 + 상황별 분석이라는 고유 보상 + 크리에이터 모델
2 콜드스타트 (닭과 달걀) 높음 한 동네 집중 전략 + 맛잘알 시딩 + "질문 먼저" 접근
3 네이버/카카오 진입 시 방어 불가 높음 대기업이 못 하는 것에 집중: 폐쇄형 신뢰 네트워크, 상황별 추천, 커뮤니티 문화
4 리뷰 조작/악용 중간 버블 + 팔로우 기반이라 오픈 리뷰보다 구조적으로 강함. 추가 모더레이션 정책 필요
5 외부 식당 DB 의존 중간 카카오/네이버/구글 교차검증으로 단일 의존 회피. 캐싱 전략
6 즉각적 인센티브 없음 높음 미해결. "돈"이 아닌 다른 즉각 보상 필요.
후보: 상황별 취향 분석의 재미? 맛잘알 인정? 기록 자체의 게임화?

8. 데이터 아키텍처 — Nyam이 소유해야 할 것

Nyam이 소유하지 않는 것

  • 식당 마스터 DB (카카오/네이버/구글에 위임)
  • 메뉴/가격 실시간 데이터
  • 예약/결제 시스템
vs

Nyam이 소유하는 것

  • 사용자의 기록 (언제, 누구랑, 어떤 상황, 평가)
  • 상황별 취향 프로필 (Taste DNA × Scene)
  • 소셜 그래프 (팔로우, 버블 관계)
  • 리스트/큐레이션 콘텐츠 (UGC)
  • 동네별 기록 밀도 데이터

9. 포지셔닝 맵

Nyam네이버/카카오맵인스타 맛집캐치테이블망고플레이트
목적개인 기록 + 신뢰 추천검색/정보자기표현/공유예약리뷰/평점
신뢰아는 사람 기반광고 혼재팔로워 기반예약 검증오픈 리뷰
개인화상황별 2D없음없음없음제한적
기록 도구핵심부수적게시물없음부수적
소셜팔로우+버블없음강력없음제한적

10. 핵심 메시지 — 엘리베이터 피치

For 식도락을 즐기는 사람들
"탭 3번이면 기록, 쌓이면 나를 아는 맛집 비서"
식당 이름 + 별점 + 상황 태그. 기록은 이게 전부.
쌓이면 내 취향이 보이고, 아는 사람들의 기록이 모이면
낯선 동네에서도 현지인 추천을 받을 수 있다.
