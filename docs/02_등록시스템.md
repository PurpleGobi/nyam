## 1. 시스템 개요

### 설계 원칙

| 원칙 | 설명 |
|------|------|
| **등록 ≠ 평가** | 등록은 "무엇을"만 특정. 평가는 별도 단계 (01_평가시스템) |
| **최소 입력** | 사용자가 직접 입력하는 건 최소. 나머지는 자동 완성 |
| **틀려도 괜찮다** | AI 추정이 틀리면 수정하면 됨. 완벽보다 속도 |
| **2가지 방법** | 검색 등록 + 사진 등록. 상황에 맞게 선택 |

### 등록의 범위

등록 시스템이 하는 것:
- 식당 또는 와인을 **특정**한다 (무엇을 먹었는가 / 마셨는가)
- 기본 정보를 **자동으로 채운다** (외부 DB 활용)
- **평가 플로우로 연결**한다

등록 시스템이 하지 않는 것:
- 평가 (→ 01_평가시스템)
- 리뷰 작성 (→ 평가의 확장 단계)
- 버블 공유 (→ 버블 시스템)

## 2. 등록 진입점

### 진입 경로

| 경로 | 설명 | 주 사용 상황 |
|------|------|------------|
| **홈 하단 + 버튼** | 중앙 FAB 버튼 → 등록 모드 선택 | 가장 일반적 |
| **상세 페이지 "다시 방문"** | 기존 식당/와인 상세에서 재방문 기록 | 재방문 등록 |
| **홈 추천 카드에서** | 추천된 식당/와인에서 "다녀왔어요" | 추천 → 등록 전환 |

### + 버튼 탭 후 모드 선택

```
    ┌─────────────────────────┐
    │                         │
    │    🔍 검색으로 등록       │
    │    식당/와인 이름 검색    │
    │                         │
    ├─────────────────────────┤
    │                         │
    │    📷 사진으로 등록       │
    │    음식/와인 라벨 사진    │
    │                         │
    └─────────────────────────┘
```

> 두 가지만. 심플하게. 선택지가 많으면 오히려 마찰.

## 3. 검색 등록 — 식당

### 3-1. 플로우

```
[입력] 사용자가 식당 이름 타이핑
  ↓
[자동완성] Nyam 캐시 DB 먼저 검색
  → 이미 다른 유저가 등록한 식당이면 바로 후보 표시
  → 없으면 외부 검색으로 넘어감
  ↓
[외부 검색] 카카오 + 네이버 + 구글 동시 검색
  ↓
[교차 매칭] 3개 소스 결과를 가게명 + 주소로 fuzzy merge
  → 같은 식당이면 하나로 합침
  → 다른 식당이면 각각 후보로 표시
  ↓
[결과 리스트] 후보 표시
  → 가게명 / 주소 / 장르 / 대표사진(있으면) / 외부 평점
  → 지도 미니맵으로 위치 확인 가능
  ↓
[선택] 사용자가 식당 선택
  ↓
[평가 플로우] → 01_평가시스템으로 이동
```

### 3-2. 검색 UI 상세

**검색 입력 화면:**

```
┌──────────────────────────────┐
│ 🔍 식당 이름을 입력하세요       │ ← 텍스트 입력
├──────────────────────────────┤
│                              │
│  스시코우지                    │ ← 자동완성 후보 1
│  일식 · 교대역 · ★4.6         │
│  ──────────────────────────  │
│  스시코우지 을지로점            │ ← 자동완성 후보 2
│  일식 · 을지로3가 · ★4.3      │
│  ──────────────────────────  │
│  스시 코지                    │ ← 자동완성 후보 3
│  일식 · 압구정 · ★4.1         │
│                              │
│  결과가 없나요? [직접 등록]     │ ← 폴백
│                              │
└──────────────────────────────┘
```

### 3-3. 검색 결과 카드 표시 정보

| 항목 | 소스 | 표시 방식 |
|------|------|----------|
| 가게명 | 카카오/네이버/구글 교차 | 볼드 텍스트 |
| 주소 (간략) | 교차 취합 | "교대역", "을지로3가" 등 주요 랜드마크 기준 |
| 장르 | 카카오 카테고리 기반 | "일식", "한식", "이탈리안" 등 |
| 대표 사진 | 카카오/구글 (있으면) | 좌측 썸네일 |
| 외부 평점 | 네이버/카카오/구글 중 최고 | ★4.6 |
| 권위 인증 | 미슐랭/블루리본 (해당 시) | 배지 |
| 거리 | GPS 사용 시 | "현재 위치에서 1.2km" |

### 3-4. 검색 정확도 향상

**자동완성 순서:**
1. Nyam 캐시 DB (이미 등록된 식당) — 가장 빠름
2. 사용자 활동 지역 내 결과 우선 — 내가 주로 다니는 동네 식당이 위로
3. 외부 API 결과 — 캐시에 없으면

**fuzzy matching 규칙:**
- 띄어쓰기 무시 ("스시 코우지" = "스시코우지")
- 점 이름의 '점/역/호' 접미사 무시 ("을지로점" = "을지로")
- 한영 혼용 허용 ("sushi" → "스시" 매칭 시도)

### 3-5. 직접 등록 (검색 결과 없을 때)

검색에서 식당을 못 찾았을 때:

```
[직접 등록] 탭
  ↓
가게명 입력 (필수)
주소/위치 입력 (선택 — 지도에서 핀 드롭 or 텍스트)
장르 선택 (선택 — 한식/일식/중식/양식/카페/바 등)
  ↓
등록 완료 → 평가 플로우
```

> 직접 등록된 식당은 외부 DB 매칭이 안 되므로 기본 정보가 비어있음.
> 추후 다른 사용자가 같은 식당을 검색 등록하면 자동 매칭 시도.

## 4. 검색 등록 — 와인

### 4-1. 플로우

```
[입력] 사용자가 와인 이름 타이핑
  ↓
[자동완성] Nyam 캐시 DB → 없으면 와인 DB API 검색
  ↓
[결과 리스트] 후보 표시
  → 와인명 / 생산자 / 산지 / 빈티지 / 품종 / 타입 / 라벨 사진
  ↓
[선택] + 빈티지 확인/변경 (같은 와인이라도 빈티지 별로 다른 경험)
  ↓
[평가 플로우] → 01_평가시스템 (와인 모드)
```

### 4-2. 와인 검색 결과 카드 표시 정보

| 항목 | 소스 | 표시 방식 |
|------|------|----------|
| 와인명 | 와인 DB | 볼드 텍스트 |
| 생산자 (와이너리) | 와인 DB | 서브텍스트 |
| 산지 | 와인 DB | "Bordeaux, France" |
| 품종 | 와인 DB | "Cabernet Sauvignon, Merlot" |
| 타입 | 와인 DB | 레드 / 화이트 / 로제 / 스파클링 |
| 빈티지 | 사용자 선택/확인 | 연도 선택 드롭다운 |
| 라벨 사진 | 와인 DB (있으면) | 좌측 라벨 이미지 |
| 평균 평점 | 와인 DB / Nyam 데이터 | ★ 표시 |

### 4-3. 빈티지 처리

같은 와인이라도 빈티지가 다르면 다른 경험:
- 검색 결과 선택 후 **빈티지 확인 화면** 표시
- 최근 연도가 기본 선택
- "빈티지 모름"도 선택 가능
- 빈티지별로 독립된 기록 (같은 와인 다른 빈티지 = 다른 기록)

### 4-4. 와인 DB 소스 (검토 필요)

| 후보 | 장점 | 단점 |
|------|------|------|
| Vivino API | 가장 방대한 데이터 | 비공식 API, 불안정 가능 |
| Wine-Searcher | 가격/유통 정보 풍부 | API 비용 |
| LCBO/기타 공공 DB | 무료 | 범위 제한 |
| 자체 축적 | 사용자 입력 기반 | 초기 데이터 부족 |

> Phase 1에서는 자체 캐시 + 하나의 외부 API로 시작. 확장은 사용 패턴 보고 결정.

## 5. 사진 등록 — 식당

### 5-1. 플로우

```
[사진 선택] 갤러리에서 선택 or 카메라 촬영
  ↓
[EXIF 추출]
  → GPS 좌표 (있으면)
  → 촬영 날짜/시간
  ↓
[Vision AI 분석] (Gemini Vision)
  → 음식 사진인가? (음식 / 식당 외관 / 메뉴판 / 기타)
  → 음식 종류 추정 (일식, 파스타, 스테이크 등)
  → 메뉴명 추정 가능하면 추정
  ↓
[식당 매칭]
  ├─ GPS 있을 때:
  │   → 반경 100m 내 식당 검색 (카카오/네이버/구글)
  │   → AI 분석 결과(장르)와 교차 → 최적 후보 상위 표시
  │   → "스시코우지에서 드셨나요?" 형태로 제안
  │
  └─ GPS 없을 때:
      → AI 분석 결과 + 사용자 활동 지역에서 검색
      → 후보 여러 개 표시
      → 또는 "어디서 드셨나요?" 검색 유도
  ↓
[확인] 사용자가 맞는 식당 선택 (또는 검색으로 전환)
  ↓
[날짜 확인] EXIF 날짜가 있으면 자동 채움. 없으면 "오늘"로 기본.
  ↓
[평가 플로우] → 01_평가시스템 (사진은 자동 첨부)
```

### 5-2. EXIF 데이터 활용

| EXIF 항목 | 활용 | 없을 때 |
|-----------|------|---------|
| GPS 좌표 | 반경 100m 식당 검색 | 사용자 활동 지역 기반 검색 or 수동 검색 |
| 촬영 일시 | 방문 날짜 자동 채움 | "오늘" 기본. 사용자가 변경 가능 |
| 카메라 모델 | 사용 안 함 | - |

**EXIF 유실 주의:**
- 카톡/인스타에서 공유받은 사진은 EXIF가 날아가는 경우가 많음
- 직접 촬영한 사진이 가장 정확
- EXIF 없으면 자연스럽게 검색 등록으로 폴백

### 5-3. Vision AI 분석 상세

| 분석 항목 | 신뢰도 | 활용 |
|----------|--------|------|
| 음식 사진 여부 | 높음 | 음식이 아니면 "음식 사진을 선택해주세요" 안내 |
| 음식 장르 (한식/일식/양식 등) | 중간 | 검색 범위 축소에 활용 |
| 메뉴명 추정 | 낮음~중간 | 참고 정보로만. "이거 초밥인가요?" 형태 |
| 가격대 추정 | 낮음 | 사용 안 함 (부정확) |

### 5-4. GPS + AI 교차 매칭 로직

```
GPS 반경 100m 내 식당 목록:
  1. 스시코우지 (일식)
  2. 교대 짬뽕 (중식)  
  3. 스타벅스 교대역점 (카페)

AI 분석: "일식 / 초밥류 추정"

→ 장르 매칭: 스시코우지 (일식) = 최우선 후보
→ "스시코우지에서 드셨나요?" [예] [아니요, 검색할게요]
```

**매칭 우선순위:**
1. GPS 근접 + 장르 일치 → 최우선
2. GPS 근접 + 장르 불일치 → 후보 리스트에 포함 (순위 낮게)
3. GPS 없음 → 검색으로 폴백

## 6. 사진 등록 — 와인

### 6-1. 플로우

```
[사진 선택] 와인 라벨 사진 선택/촬영
  ↓
[Vision AI 분석]
  → 라벨 텍스트 OCR (와인명, 생산자, 빈티지, 산지)
  → 라벨 이미지 자체로 와인 DB 매칭 시도
  ↓
[와인 매칭]
  → OCR 텍스트로 와인 DB 검색
  → 라벨 이미지 유사도 매칭 (가능한 경우)
  → 후보 리스트 표시
  ↓
[확인] 사용자가 맞는 와인 선택 + 빈티지 확인
  ↓
[평가 플로우] → 01_평가시스템 와인 모드 (라벨 사진 자동 첨부)
```

### 6-2. 와인 라벨 OCR 추출 항목

| 항목 | 추출 난이도 | 활용 |
|------|-----------|------|
| 와인명 (Château, Domaine 등) | 중간 | 와인 DB 검색 키워드 |
| 생산자 | 중간 | 검색 보조 |
| 빈티지 (연도) | 높음 (숫자 인식) | 빈티지 자동 채움 |
| 산지 (Appellation) | 중간 | 검색 범위 축소 |
| 품종 | 낮음 (라벨에 없는 경우 많음) | 보조 |
| 알코올 도수 | 높음 (숫자) | 기본 정보 채움 |

### 6-3. 라벨 인식 실패 시

```
라벨 인식 실패
  → "와인을 찾지 못했어요"
  → [이름으로 검색] ← 검색 등록으로 자연스럽게 전환
  → [직접 등록] ← 와인명/품종/산지 수동 입력
```

## 7. 등록 후 평가 연결

### 식당 등록 → 평가

```
식당 확정
  ↓
자동 채워지는 정보:
  - 가게명, 주소, 장르 (외부 DB)
  - 방문 날짜 (EXIF or 오늘)
  - 사진 (사진 등록이었으면 자동 첨부)
  ↓
01_평가시스템 진입:
  → 사분면 (가격% × 분위기%)
  → 크기 (맛 만족도)
  → 상황 태그
  → (선택) 확장 평가
  ↓
기록 저장
```

### 와인 등록 → 평가

```
와인 확정 + 빈티지 확인
  ↓
자동 채워지는 정보:
  - 와인명, 생산자, 산지, 품종, 타입 (와인 DB)
  - 날짜 (EXIF or 오늘)
  - 라벨 사진 (사진 등록이었으면)
  ↓
01_평가시스템 와인 모드 진입:
  → 사분면 (산미% × 바디%)
  → 향 팔레트 (영역 그리기)
  → (선택) 복합성, 여운, 균형
  → 만족도 확정
  → 상황 태그
  → (선택) 확장 평가
  ↓
기록 저장
```

## 8. 재방문 / 재음용 등록

### 이미 등록된 식당/와인에 대한 새 기록

**진입점:**
- 식당/와인 상세 페이지의 **"다시 방문"** / **"다시 마셨어요"** 버튼
- 검색 등록 시 이미 기록이 있는 식당이면 "이전에 방문하셨네요! 새 기록 추가?" 안내

**플로우:**
```
"다시 방문" 탭
  ↓
식당/와인 정보는 이미 있음 (검색 단계 스킵)
  ↓
날짜 확인 (기본: 오늘)
  ↓
바로 평가 플로우 진입
  → 이전 평가의 사분면 위치를 반투명으로 표시 ("지난번에는 여기였어요")
  → 새로운 위치를 찍을 수 있음
  ↓
기록 저장 (이전 기록과 독립된 새 기록)
```

**재방문의 가치:**
- 같은 식당이라도 상황/경험이 다를 수 있음
- "혼밥 95점, 회식 70점, 데이트 85점" — 상황별 경험 차이가 쌓임
- 이전 기록과 비교하는 재미 ("지난번보다 맛이 떨어졌네?")

## 9. 예외 처리

### 9-1. 검색 관련

| 상황 | 처리 |
|------|------|
| 동명 식당 (같은 이름, 다른 위치) | 주소/지도로 구분. 카드에 위치 강조 표시 |
| 프랜차이즈 (같은 브랜드, 다른 지점) | "○○ 강남점", "○○ 홍대점" 각각 별도 식당으로 등록 |
| 폐업한 식당 | 외부 DB 업데이트 시 감지. 기록은 아카이브 유지, 신규 등록은 경고 |
| 오타/불완전 입력 | fuzzy matching으로 최대한 커버. "혹시 이 식당인가요?" 형태 제안 |
| 외부 API 장애 | Nyam 캐시 DB만으로 검색. 없으면 직접 등록 안내 |

### 9-2. 사진 관련

| 상황 | 처리 |
|------|------|
| 음식 사진이 아닌 사진 | "음식 사진을 선택해주세요" 안내. 재선택 유도 |
| EXIF GPS 없음 | 검색으로 자연스럽게 폴백. "어디서 드셨나요?" |
| EXIF 날짜가 오래됨 (1개월+) | "3개월 전 사진이네요. 날짜가 맞나요?" 확인 |
| 여러 장 선택 | 첫 번째 사진으로 분석. 나머지는 기록에 첨부 |
| 와인 라벨 인식 실패 | 검색 등록으로 전환 or 직접 등록 |

### 9-3. 매칭 관련

| 상황 | 처리 |
|------|------|
| 같은 식당인데 카카오/네이버 이름이 다름 | fuzzy merge. 주소 기반 최종 판단 |
| 신규 오픈 (외부 DB에 아직 없음) | 직접 등록. 차후 외부 DB 갱신 시 자동 매칭 시도 |
| 와인 빈티지 DB에 없음 | 가장 가까운 빈티지 제안 or "빈티지 모름"으로 등록 |

## 10. 데이터 흐름

### 식당 등록 데이터 흐름

```
사용자 입력 (검색어 or 사진)
  │
  ├─ 검색: 텍스트 → Nyam 캐시 → 외부 API (카카오/네이버/구글)
  │                              → fuzzy merge
  │                              → 후보 리스트
  │
  └─ 사진: 이미지 → EXIF 추출 → Vision AI → GPS 기반 검색 → 매칭
  │
  ↓
식당 선택/확정
  │
  ├─ Nyam 캐시에 있음 → 기존 데이터 사용
  └─ Nyam 캐시에 없음 → 외부 데이터 캐싱 (새로 저장)
      → 기본 정보 (가게명, 주소, 장르, 사진, 영업시간)
      → 외부 평점 (네이버/카카오/구글)
      → 권위 인증 (미슐랭/블루리본 해당 시)
      → 2주 후 자동 업데이트 대기열에 등록
  │
  ↓
평가 플로우 → Rating 엔티티 생성
```

### 와인 등록 데이터 흐름

```
사용자 입력 (검색어 or 라벨 사진)
  │
  ├─ 검색: 텍스트 → Nyam 캐시 → 와인 DB API
  │
  └─ 사진: 라벨 이미지 → OCR → 와인 DB 검색 → 매칭
  │
  ↓
와인 선택/확정 + 빈티지 확인
  │
  ├─ Nyam 캐시에 있음 → 기존 데이터 사용
  └─ Nyam 캐시에 없음 → 와인 DB 데이터 캐싱
      → 기본 정보 (와인명, 생산자, 산지, 품종, 타입, 알코올)
      → 라벨 이미지
      → 2주 후 업데이트 대기열 등록
  │
  ↓
평가 플로우 (와인 모드) → Rating 엔티티 생성
```

### 캐싱 & 업데이트

| 항목 | 정책 |
|------|------|
| **최초 캐싱** | 사용자가 처음 등록할 때. 외부 API 호출 → Nyam DB에 저장 |
| **갱신 주기** | 2주 간격 |
| **갱신 방식** | 첫 등록 시점 기준 2주+ 시간차로 분산. 대기열 방식 |
| **범위** | 사용자가 등록한 식당/와인만. 전체 크롤링 안 함 |
| **폐업 감지** | 갱신 시 외부 DB에서 삭제/폐업 확인 → 사용자에게 알림 |
