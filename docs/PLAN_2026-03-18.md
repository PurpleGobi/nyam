## Feature 3: AI 파이프라인 5단계 재구성

### 현재 코드 분석 (정확한 현황)

현재 `enrich/route.ts`가 하는 일:
1. Gemini에 사진을 보내 식당명/와인을 **식별** + 메뉴 추정 + 태그 추출을 **한 번의 호출로 통합** 처리
2. `record_ai_analyses` INSERT + `records` UPDATE

현재 `enrich/route.ts`가 **하지 않는** 일:
- 카카오 API 호출 (현재 카카오 연동은 `use-create-record.ts`에서 클라이언트가 `selectedPlace`를 직접 전달)
- GPS 교차 검증

따라서 5단계 분리는 "역할 축소"가 아니라 **역할 재정의**:
- identify: Gemini 식별 (기존 enrich에서 추출) + GPS 교차 검증 (**신규**)
- enrich: 카카오 API 상세 수집 (**완전 신규** — 현재 코드에 없는 기능)
- analyze-photos: Gemini 사진 분석 (기존 enrich에서 추출, 확정 맥락 기반으로 정확도 향상)

### 3-1. 신규 API: `POST /api/records/identify`

**신규 파일**: `src/app/api/records/identify/route.ts`

**역할**: 식당/와인을 특정하는 것이 유일한 목표. 메뉴 추정, 태그 추출 등은 하지 않음.

**Input**:
```typescript
{ recordId: string }
// photoUrls → 내부에서 record_photos 테이블 조회
// location → 내부에서 record.location_lat/location_lng 조회
```

> **설계 결정**: photoUrls와 location을 input으로 받지 않고 DB에서 읽음.
> 이유: retryAnalysis, reanalyze 등 재실행 시에도 동일하게 동작하려면 DB가 SSOT이어야 함.
> 사진은 파이프라인 시작 전에 이미 `record_photos`에 업로드 완료된 상태.

**공통 흐름**:
```typescript
export async function POST(request: NextRequest) {
  // 1. auth 체크
  // 2. recordId로 record 조회 (record_type, location_lat, location_lng)
  // 3. record_photos에서 photo_url 목록 조회
  // 4. prepareImageParts()로 이미지 변환
  // 5. record_type별 분기
}
```

**식당 로직**:
```
1. GPS 교차 검증용 주변 식당 목록 수집
   - record.location_lat/lng가 있으면:
     searchNearbyRestaurants(lat, lng, 500) 호출 (기존 kakao-local.ts)
   - 없으면: nearbyPlaces = [] (OCR만으로 식별 시도)

2. Gemini 호출 — 식당 식별 전용 프롬프트 (TECH_SPEC Step 1)
   - input: imageParts + nearbyPlaces 목록
   - output: { restaurantName, matchedPlaceId, confidence, ocrTexts }

3. 식당 확정
   - confidence > 0.7 + matchedPlaceId 있음:
     → nearbyPlaces에서 해당 place 찾기
     → restaurants UPSERT (name, address, external_id, lat, lng, source='kakao')
     → records.restaurant_id 연결
   - confidence 0.3~0.7:
     → identified_restaurant에 후보 저장 (사용자 수정 가능)
   - confidence < 0.3:
     → 미특정 (사용자 직접 검색 유도)

4. record_ai_analyses INSERT:
   - identified_restaurant: { name, matchedPlaceId, confidence }
   - raw_response: Gemini 원본
   - confidence_score

5. records UPDATE:
   - ai_recognized: true
```

**와인 로직**:
```
1. Gemini 호출 — 와인 식별 전용 프롬프트 (TECH_SPEC Step 1 와인)
   - 기존 handleWineEnrich() 프롬프트 확장
   - output: wine_info { name, vintage, winery, origin, variety, estimated_price_krw, critic_score }
   - 부가: 영수증 → 구입가, 음식 사진 → 페어링 음식

2. record_ai_analyses INSERT:
   - wine_info, pairing_food, raw_response, confidence_score

3. records UPDATE:
   - menu_name (와인명, 비어있으면)
   - purchase_price, pairing_food (있으면)
   - ai_recognized: true
```

**요리**: 즉시 `{ success: true, skipped: true }` 반환 (특정할 외부 대상 없음)

### 3-2. 기존 API 재작성: `POST /api/records/enrich`

**파일**: `src/app/api/records/enrich/route.ts` — **전면 재작성**

**역할 재정의**: identify에서 확정된 대상의 **외부 데이터 수집**. Gemini 호출 없음. 카카오 API 호출은 **이 API에서 새로 추가**되는 기능.

> **현재 코드와의 차이**: 현재 enrich는 Gemini 사진 분석이 핵심. 새 enrich는 Gemini 대신 카카오 API가 핵심. 완전히 다른 역할.

**Input**: `{ recordId: string }`

**식당 로직**:
```
1. record + record_ai_analyses에서 최신 identified_restaurant 조회
   - restaurant_id가 없거나 identified_restaurant.matchedPlaceId가 없으면 → skip

2. restaurants 테이블에서 현재 데이터 조회
   - external_id(카카오 place ID) 확인

3. 카카오 API로 추가 정보 수집
   - searchRestaurantsByKeyword(restaurant.name, restaurant.latitude, restaurant.longitude)
   - 결과에서 external_id 매칭하여 최신 정보 추출
   - (카카오에는 place ID 단건 조회 API가 없으므로, 이름+좌표로 재검색 후 ID 매칭)

4. restaurants 테이블 UPDATE:
   - address, phone (최신화)
   - region: extractArea(matched.addressName)으로 상권명 추출
   - genre: categoryName → FOOD_CATEGORIES key 매핑
   - synced_at: now()

5. records UPDATE:
   - genre: FOOD_CATEGORIES 매핑 (비어있으면)
```

**와인/요리**: skip — `{ success: true, skipped: true }` 반환

```typescript
// 새 구조
import { searchRestaurantsByKeyword } from "@/infrastructure/api/kakao-local"
import { extractArea } from "@/shared/constants/areas"
import { FOOD_CATEGORIES } from "@/shared/constants/categories"

export async function POST(request: NextRequest) {
  // auth 체크
  const { recordId } = await request.json()

  // record 조회
  const record = ...
  if (record.record_type !== "restaurant") {
    return NextResponse.json({ success: true, skipped: true })
  }
  if (!record.restaurant_id) {
    return NextResponse.json({ success: true, skipped: true, reason: "no restaurant linked" })
  }

  // restaurants + identified_restaurant 조회
  const restaurant = ...
  if (!restaurant.external_id || restaurant.source !== "kakao") {
    return NextResponse.json({ success: true, skipped: true, reason: "no kakao external_id" })
  }

  // 카카오 키워드 검색 → external_id 매칭
  const places = await searchRestaurantsByKeyword(
    restaurant.name,
    restaurant.latitude,
    restaurant.longitude,
  )
  const matched = places.find((p) => p.externalId === restaurant.external_id)

  if (!matched) {
    // 매칭 실패 — 식당명 변경 등. 기존 데이터 유지.
    return NextResponse.json({ success: true, skipped: true, reason: "kakao match not found" })
  }

  // restaurants UPDATE (카카오 최신 정보)
  const region = extractArea(matched.addressName) ?? restaurant.region
  const genreKey = mapCategoryToGenre(matched.categoryName)  // 카카오 카테고리 → FOOD_CATEGORIES

  await admin.from("restaurants").update({
    address: matched.address,
    phone: matched.phone || restaurant.phone,
    region,
    synced_at: new Date().toISOString(),
  }).eq("id", restaurant.id)

  // records.genre 업데이트 (비어있으면)
  if (!record.genre && genreKey) {
    await admin.from("records").update({ genre: genreKey }).eq("id", recordId)
  }

  return NextResponse.json({ success: true, updated: true })
}
```

  const updateData: Record<string, unknown> = {}

  if (data.identifiedRestaurant !== undefined) updateData.identified_restaurant = data.identifiedRestaurant
  if (data.orderedItems !== undefined) updateData.ordered_items = data.orderedItems

  if (data.wineInfo !== undefined) updateData.wine_info = data.wineInfo

    .from("record_ai_analyses")
    .select("id")
    .eq("record_id", recordId)
    .order("created_at", { ascending: false })
    .limit(1)
    .single()

  if (!latest) throw new Error("No AI analysis found for record")

  const { error } = await this.supabase
    .from("record_ai_analyses")
    .update(updateData)
    .eq("id", latest.id)

### 2-4. 갱신 API (서버 사이드)

**신규 파일**: `src/app/api/restaurants/[id]/refresh/route.ts`

stale 체크 + 카카오 재호출 + 폐업 감지를 서버에서 처리:

```typescript
import { NextRequest, NextResponse } from "next/server"
import { createClient } from "@/infrastructure/supabase/server"
import { createAdminClient } from "@/infrastructure/supabase/admin"
import { searchRestaurantsByKeyword } from "@/infrastructure/api/kakao-local"
import { extractArea } from "@/shared/constants/areas"

const STALE_TTL_MS = 14 * 24 * 60 * 60 * 1000  // 14일

export async function POST(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  const { id } = await params
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ error: "Unauthorized" }, { status: 401 })

  const admin = createAdminClient()
  const { data: restaurant } = await admin
    .from("restaurants")
    .select("*")
    .eq("id", id)
    .single()

  if (!restaurant) return NextResponse.json({ error: "Not found" }, { status: 404 })

  // 이미 폐업 확인된 식당
  if (restaurant.is_closed) {
    return NextResponse.json({ fresh: true, isClosed: true })
  }

  // stale 체크: 14일 미경과면 갱신 불필요
  const syncedAt = restaurant.synced_at ? new Date(restaurant.synced_at).getTime() : 0
  if (Date.now() - syncedAt < STALE_TTL_MS) {
    return NextResponse.json({ fresh: true, isClosed: false })
  }

  // 카카오 소스가 아니면 갱신 불가
  if (!restaurant.external_id || restaurant.source !== "kakao") {
    return NextResponse.json({ fresh: true, isClosed: false })
  }

  // 카카오 키워드 검색 → external_id 매칭
  try {
    const places = await searchRestaurantsByKeyword(
      restaurant.name,
      restaurant.latitude,
      restaurant.longitude,
    )
    const matched = places.find((p) => p.externalId === restaurant.external_id)

    if (!matched) {
      // 검색 결과에 없음 → 폐업 처리
      await admin.from("restaurants").update({
        is_closed: true,
        closed_at: new Date().toISOString(),
        synced_at: new Date().toISOString(),
      }).eq("id", id)
      return NextResponse.json({ fresh: true, isClosed: true })
    }

    // 정보 갱신
    await admin.from("restaurants").update({
      name: matched.name,
      address: matched.address,
      phone: matched.phone || restaurant.phone,
      latitude: matched.latitude,
      longitude: matched.longitude,
      region: extractArea(matched.addressName) ?? restaurant.region,
      synced_at: new Date().toISOString(),
      is_closed: false,
      closed_at: null,
    }).eq("id", id)

    return NextResponse.json({ fresh: true, isClosed: false, updated: true })
  } catch {
    // API 실패 시 기존 데이터 유지 (synced_at 갱신 안 함 → 다음 조회 시 재시도)
    return NextResponse.json({ fresh: false, error: "Kakao API failed" })
  }
}
```

### 2-5. 파이프라인 연동: enrich에서 stale 체크

Feature 3의 새 `enrich/route.ts`에 stale 체크를 통합. enrich가 카카오 API를 호출하는 역할이므로 자연스럽게 통합됨:

```typescript
// enrich/route.ts 식당 분기 내부

  // 14일 이내에 이미 갱신됨 → 카카오 재호출 불필요
  return NextResponse.json({ success: true, skipped: true, reason: "restaurant data is fresh" })

```
