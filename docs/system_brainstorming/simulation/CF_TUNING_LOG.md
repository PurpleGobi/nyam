### 근본 원인 분석

**1. T1 분리도 WEAK — 클러스터 오프셋이 너무 작다**
- 현재 오프셋: ±1.0~1.5 (10점 만점 기준)
- 개인 noise(0.5~1.5)와 bias(±2)가 오프셋을 압도
- 결과: 클러스터 간 차이가 노이즈에 묻힘

**2. T5 부스트 무효 — 팔로우가 같은 클러스터에 집중되기 때문**
- 같은 클러스터 팔로우 70% → 적합도가 이미 높은 사람들에게 부스트
- 부스트 유무와 무관하게 가중치 순서가 동일 → MAE 차이 없음
- 분리도가 높아지면 부스트 효과도 드러날 것

**3. T9-B 측정 오류 — 기준선(clean)과 오염 데이터의 seed가 다른 데이터를 생성**
- 오염 유저 추가 시 record 생성 순서가 바뀜 → holdout 마킹도 달라짐
- 같은 유저의 같은 식당 예측이지만 기준값 자체가 달라져서 비교 불가

**4. T8 확신도 약함 — 확신도 공식의 요소별 가중치가 부적절**
- rater_factor(0.4) + weight_factor(0.3) + agreement(0.3)
- 대부분의 예측이 확신도 0.5~0.7에 몰려있어 분별력 부족

**5. T7 버블 < 글로벌 — 버블 멤버 수가 적어서 CF 데이터가 부족**
- 8~15명 버블 vs 100명 전체 → 당연히 데이터가 많은 쪽이 정확
- 이건 버블의 구조적 한계이지 버그가 아님 (§3.6 참조)

### V2 핵심 발견

**좋은 소식:**
1. **CF vs 단순평균 개선율 63.7%** — V1의 37%에서 대폭 상승. 클러스터 분리가 강해지니 CF의 이점이 뚜렷해짐

3. **콜드스타트 데이터 확보** — 8개 이상이면 MAE 0.5 이하로 쓸 만함
4. **확신도 단조 달성** — V1에서 단조 깨졌던 것이 해결

**남은 문제:**
1. **T9-B 측정이 여전히 음수** — "clean 유저만의 평균"이 CF 예측과 다른 기준을 측정하고 있음. CF는 개인화된 예측이므로 단순평균과 같은 기준으로 비교하면 안 됨
2. **T1 분리도 여전히 WEAK** — D 값이 작아야(D=2) 분리가 뚜렷한데, D=2에서 적합도 자체가 너무 낮음 (0.3~0.4)
3. **T5 부스트 무효** — 구조적 문제. 적합도가 높은 사람 = 이미 팔로우하는 사람이라 부스트가 순서를 바꾸지 못함
4. **T8 확신도 상관 약함** — 범위가 0~0.49로 좁아짐. 공식 스케일링 필요

### V3 핵심 발견

**T1 해결**: 절대 gap은 작지만 t-test에서 **p < 0.005**로 모든 D값, 식당/와인 모두 통계적으로 유의미. 클러스터 분리가 잘 동작하고 있다는 의미. 다만 gap이 작은 이유는 겹치는 기록 수가 많아질수록 평균으로 수렴하기 때문이며, 이는 CF의 정상 동작이다.

**T8 개선**: 3개 구간에 데이터 분포 (낮음 265, 보통 368, 높음 443). 단조 감소 유지. 상관계수는 -0.137로 아직 강하지 않지만, 확신도의 실용적 가치(높을수록 MAE가 낮다)는 입증됨.

**T9-B 구조적 문제**: 오염 유저를 추가하면 generate_records의 random 순서가 바뀌어 **같은 clean 유저라도 다른 식당을 방문하고 다른 holdout이 마킹**된다. 해결책: clean 유저 기록을 먼저 고정 생성한 뒤, 오염 유저 기록만 별도 추가.

### V4 핵심 발견

**T5 부스트의 구조적 한계가 명확해짐:**
- 평균 평가자 중 팔로우는 **1.2명 (9%)**, 비팔로우는 **12.2명 (91%)**
- 100명 유저에서 5~15명 팔로우 → 한 식당 기록자 중 팔로우가 포함될 확률이 낮음
- **부스트가 효과를 보려면 팔로우가 많거나 유저풀이 작아야 함**
- 결론: 부스트는 "소수의 신뢰하는 사람 의견을 살짝 더 반영"하는 정도의 역할. **극적인 차이를 만드는 기능은 아님**. 이건 합리적이다.

**T8 확신도 상관이 약한 근본 원인:**
- CF 자체가 이미 잘 동작 (MAE 0.42~0.43)하기 때문에 예측 오차의 분산이 작음
- 확신도가 낮든 높든 대부분 비슷하게 정확 → 상관이 약해질 수밖에 없음
- **단조 감소는 달성**했고, 이것이 실용적으로 더 중요 (높으면 더 정확하다는 보장)

| **확신도 공식** | **V4 (유효N + 가중분산 + 집중도)** | 단조 감소 달성, 4구간 분포 |

### 부스트 비율 변경 사유 (2.0/1.5/1.0 → 1.5/1.2/1.0)

- 시뮬레이션에서 MAE 최적이 1.5/1.2/1.0
- 2.0/1.5와 극적 차이는 없지만, 높은 부스트는 오염 유저 팔로우 시 피해를 키움 (T9-D)
- **보수적 부스트가 안전**: 오염 방어와 예측 정확도 모두에서 더 나음
- 부스트 효과가 미미한 이유: 평가자 중 팔로우 비율이 9%뿐 (1.2명/13.4명). 유저풀이 커지면 비율은 더 낮아질 것. 부스트는 "소수의 신뢰하는 사람 의견을 미세 조정"하는 역할.

### 확신도에 대한 결론

**상관계수 -0.3 목표는 달성하지 못했다.** 근본 원인:

1. **CF 자체가 이미 정확** (MAE 0.43) → 오차 분산이 작아서 상관이 약할 수밖에 없음
2. **소수 평가자 구간이 불안정** — 2~5명일 때 우연에 따라 오차가 크게 흔들림
3. 상관계수보다 **단조 감소가 실용적으로 더 중요** — "확신도가 높으면 더 정확하다"는 보장

**실용적 결론**: 확신도는 "이 점수를 보여줘도 되는가"의 필터로 사용:
- 확신도 < 0.3: **점수 숨김** (데이터 부족)
- 확신도 0.3~0.5: **"참고용" 표시**
- 확신도 ≥ 0.5: **정상 표시**

## V6 — 속도 최적화 전략 검증 (2026-04-08)

### 목표

| 지표 | 목표 |
|------|------|
| 단건 예측 | < 10ms |
| 20건 피드 | < 200ms |
| 정확도 손실 | < 0.5점 (100점 만점) |

### T11-A 결과: 정확도 손실

```
설정                   MAE(×10)  손실    커버리지
기준선                    4.3     —      99.9%
겹침 ≥1 (0 드랍)          4.3    0.0점    99.9%
겹침 ≥3                  4.3    0.0점    92.3%
겹침 ≥5                  4.1   -0.2점    78.3%   ← 오히려 개선!
겹침 ≥3 + Top-10          4.3    0.0점    92.3%
겹침 ≥5 + Top-20          4.1   -0.2점    78.3%
```

**핵심 발견**: 정확도 손실이 **0**이거나 오히려 개선됨. 겹침이 적은 유저는 노이즈에 가까워서 제거하면 예측이 더 정확해지는 것. 평균 유효 평가자도 8~13명이라 Top-K가 발동할 일이 거의 없었음.

### T11-B 결과: 속도 — 병목은 필터링이 아니라 적합도 계산

```
            필터 없음    겹침≥3     겹침≥3+Top30
200명         8.9ms     14.8ms     14.2ms
500명        55.4ms     93.9ms     79.5ms
1,000명     218.6ms    365.0ms    299.1ms
```

**문제**: 필터링 적용이 오히려 느려졌다! 이유:
- 필터링을 위해 `overlapping_items()`를 **모든 기록자에 대해** 호출
- 이 함수가 두 유저의 기록을 교집합 계산하는데, 이것 자체가 비용
- "필터링 비용 > 절감 비용"이 되는 구조

### V6 핵심 결론

**필터링 자체는 정확도에 무해하지만, 현재 구현에서는 속도 개선 효과가 없다.**

진짜 해결책은 필터링이 아니라 **적합도 사전 계산(pre-compute)**:

```
전략: 적합도 매트릭스 캐시 + 실시간 예측

1. 배치(크론): 유저 쌍별 적합도를 미리 계산해서 DB에 저장
   - 새 기록 추가 시 해당 유저의 적합도만 갱신
   - 100명: 1.5초, 500명: 72초, 1,000명: ~5분

2. 실시간: 예측 시 적합도는 DB에서 조회 (O(1))
   - 기록자 목록 조회 → 각 기록자의 적합도 조회(캐시) → 가중 예측
   - 적합도 계산이 없으므로 기록자 수에만 비례
   - 기록자 100명이어도 < 5ms 예상

3. 런타임 필터: 겹침 ≥3 + Top-50 (적합도 조회 후 빠르게 필터링)
   - 사전 계산된 적합도 기반이므로 overlapping_items 호출 불필요
```

이 전략은 시뮬레이션이 아니라 **실제 Supabase 구현에서 검증**해야 함 (materialized view, Edge Function 등).

### 최종 속도 전략

| 단계 | 방법 | 시점 |
|------|------|------|
| **사전 계산** | 유저 쌍별 적합도 매트릭스 → DB 테이블 | 크론 (기록 추가 시 or 주기적) |
| **런타임 필터** | 겹침 ≥3 (사전 계산 시 겹침 0~2는 저장 안 함) | 예측 시 |
| **Top-K** | 적합도×신뢰도 상위 50명만 | 예측 시 (적합도 조회 후) |

**정확도 영향**: 손실 0점 (시뮬 검증 완료). 커버리지: 92.3% (겹침 ≥3 기준).

### 현실적 규모 테스트 (5,000명 × 5,000 식당)

```
식당별 기록자: 평균 37명 (max 54)
나와 겹치는 유저: 1,083명 (21.7%)
특정 식당 기록자 중 겹치는: ~8명

단건 (캐시 없음): 119ms           ⚠️
20건 피드 (캐시 없음): 2,435ms    ❌
20건 피드 (적합도 캐시): 297ms     ⚠️ 경계
적합도 1건 계산: 10ms
```

### 최종 아키텍처: 이벤트 기반 증분 업데이트

**크론잡의 문제**: 5,000명이면 1,250만 쌍 (2시간), 실시간성 포기.

**해결**: 기록 추가 이벤트 → 관련 유저만 적합도 갱신 (백그라운드)

```
[유저가 기록 추가]
  → Supabase trigger / Edge Function (백그라운드)
  → 해당 식당/와인의 다른 기록자와의 적합도만 갱신
  → 평균 37건 × 10ms = 370ms (유저는 기다리지 않음)

[예측 요청 시]
  → DB에서 사전계산된 적합도 조회 (O(1))
  → 겹침 ≥3 필터 + Top-50 → 가중 예측
  → 20건 피드: ~300ms
```

| 이벤트 | 작업 | 비용 | 유저 대기 |
|--------|------|------|----------|
| 기록 추가 | 해당 아이템 기록자와 적합도 갱신 | ~370ms | 없음 (백그라운드) |
| 팔로우 변경 | 관계 부스트 갱신 (DB 필드만) | ~1ms | 없음 |
| 점수 조회 | 캐시된 적합도로 예측 | ~15ms/건 | 있음 (실시간) |
| 피드 로드 | 20건 예측 | ~300ms | 있음 (실시간) |

**크론잡은 보조용**으로만: 주 1회 전체 적합도 정합성 체크 + 누락 쌍 보정.
