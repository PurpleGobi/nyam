# CF 점수 체계 시뮬레이션 계획

> 작성: 2026-04-08

## 1. 목적

PRD §3에서 정의한 Nyam 점수 체계(User-based Collaborative Filtering)의 **파라미터를 검증하고 튜닝**한다. 머릿속 가정을 합성 데이터로 돌려보고, 실제로 동작하는 수치를 확정한다.

### 최우선 검증 질문

> **"오염된 유저(광고, 랜덤, 악의적)가 섞여도 정직한 유저의 점수가 보호되는가?"**

이것이 Nyam에 CF가 필요한 이유다. 네이버/구글의 단순 평균은 오염에 무방비다 — 한 명의 극단적 점수가 전체 평균을 움직인다. CF는 "나와 취향이 다른 사람(오염원 포함)의 영향력을 자동으로 낮추는" 구조적 방어벽이다. 이 방어벽이 실제로 작동하는지가 시뮬레이션의 핵심이다.

## 3. 합성 데이터 설계

### 3.1 사분면 데이터 구조

Nyam의 평가는 1D 점수가 아니라 **2D 사분면 좌표**이며, 식당과 와인의 축이 다르다.

| 카테고리 | X축 | Y축 | 만족도 |
|---------|-----|-----|--------|
| **식당** | 음식 퀄리티 (0~10) | 경험 만족도 (0~10) | (X+Y)/2 자동 계산 |
| **와인** | 구조·완성도 (0~10) | 경험 만족도 (0~10) | (X+Y)/2 자동 계산 |

**CF 계산에서의 핵심 차이**: 적합도를 계산할 때 1D 거리가 아니라 **2D 유클리드 거리**를 사용한다. 이는 같은 만족도(예: 7점)라도 (X=9, Y=5)와 (X=5, Y=9)는 취향이 다르다는 것을 포착한다.

```
적합도 비교 = mean-centered 2D 좌표 간 유클리드 거리
  거리 = √((Xᵢ - Xⱼ)² + (Yᵢ - Yⱼ)²)   ← 1D와의 핵심 차이
```

**식당 적합도와 와인 적합도는 별개로 산출**: 한 사람과 식당 취향은 맞는데 와인 취향은 안 맞을 수 있다. 따라서:
- 식당 적합도: 겹치는 **식당** 기록의 2D 좌표로 계산
- 와인 적합도: 겹치는 **와인** 기록의 2D 좌표로 계산
- 통합 적합도: 필요시 가중 평균 (식당 기록 수 : 와인 기록 수 비율)

### 3.2 유저 프로필

100명의 합성 유저를 생성한다. **식당 취향 클러스터**와 **와인 취향 클러스터**를 독립적으로 부여.

**식당 취향 클러스터** (X=음식 퀄리티, Y=경험 만족도):

| 클러스터 | 비율 | 특성 | 평가 패턴 |
|----------|------|------|----------|
| 미식파 | 20% | 음식 퀄리티 중시, 경험 만족도 덜 중시 | X축에 후하고 Y축에 박함 |
| 분위기파 | 20% | 경험 만족도 중시, 음식은 보통이면 OK | X축에 박하고 Y축에 후함 |
| 균형파 | 30% | 음식과 경험 모두 중시 | X, Y 편차 작음 |
| 까다로운파 | 15% | 전반적으로 점수가 낮음 (박한 유저) | X, Y 모두 낮은 편향 |
| 후한파 | 15% | 전반적으로 점수가 높음 (후한 유저) | X, Y 모두 높은 편향 |

**와인 취향 클러스터** (X=구조·완성도, Y=경험 만족도):

| 클러스터 | 비율 | 특성 | 평가 패턴 |
|----------|------|------|----------|
| 구조파 | 20% | 밸런스·복합미 중시 | X축에 후하고 Y축에 박함 |
| 감성파 | 20% | 즐거움·접근성 중시 | X축에 박하고 Y축에 후함 |
| 균형파 | 35% | 둘 다 중시 | X, Y 편차 작음 |
| 까다로운파 | 10% | 전반적으로 낮게 줌 | X, Y 모두 낮은 편향 |
| 후한파 | 15% | 전반적으로 높게 줌 | X, Y 모두 높은 편향 |

> 한 유저가 식당은 "미식파"인데 와인은 "감성파"일 수 있다. 두 클러스터는 독립.

각 유저 파라미터:
- `restaurant_cluster`: 식당 취향 클러스터
- `wine_cluster`: 와인 취향 클러스터
- `bias_x`, `bias_y`: 축별 개인 편향 (-2 ~ +2, 후한/박한 정도)
- `noise`: 개인 노이즈 수준 (평가의 일관성. 0.3~1.0) ← V2에서 축소

### 3.3 식당/와인 데이터

**식당 200개** — 각 식당은 "객관적 특성"을 가진다:

```python
Restaurant(
    true_food_quality=7.5,   # X축: 음식 퀄리티 (객관적 기준선)
    true_experience=6.0,     # Y축: 경험 만족도 (객관적 기준선)
    genre="일식",            # 장르 (16종 중)
    area="강남",             # 지역
)
```

**와인 150개** — 와인도 "객관적 특성"을 가진다:

```python
Wine(
    true_structure=6.5,      # X축: 구조·완성도 (객관적 기준선)
    true_pleasure=7.0,       # Y축: 경험 만족도 (객관적 기준선)
    wine_type="red",         # 타입 (7종 중)
    region="부르고뉴",        # 산지
)
```

### 3.4 기록 생성

각 유저는 무작위로 **식당 15~60개** + **와인 5~30개**를 기록한다 (와인 기록이 식당보다 적은 현실 반영).

```python
# 유저 i가 식당 j를 평가할 때 (2D)
x_score = restaurant.true_food_quality + user.restaurant_cluster_offset_x + user.bias_x + gauss(0, noise)
y_score = restaurant.true_experience   + user.restaurant_cluster_offset_y + user.bias_y + gauss(0, noise)

# 유저 i가 와인 k를 평가할 때 (2D, 축 의미 다름)
x_score = wine.true_structure + user.wine_cluster_offset_x + user.bias_x + gauss(0, noise)
y_score = wine.true_pleasure  + user.wine_cluster_offset_y + user.bias_y + gauss(0, noise)

# 모두 0~10 범위로 클램프
# 만족도는 자동 계산: satisfaction = (x + y) / 2
```

같은 클러스터 유저는 비슷한 `cluster_offset`을 공유 → 자연스러운 취향 유사도 형성.

### 3.4 관계 설정

- 각 유저는 5~15명을 팔로우
- 맞팔 비율 ~40%
- 같은 클러스터 내 팔로우 확률 50%, 다른 클러스터 50% (V4: 현실 반영 — 친구/지인 팔로우)

### 3.5 버블 설정

5개의 버블 생성:

| 버블 | 멤버 구성 | 맥락 |
|------|----------|------|
| 일식 마니아 | 미식파 위주 + 일부 균형파 | 장르 전문성 |
| 강남 맛집 | 지역 기반 혼합 | 지역 |
| 가성비 클럽 | 까다로운파 + 일부 균형파 | 가치관 |
| 대학동기 | 랜덤 5~10명 | 인간관계 |
| 와인 모임 | 혼합 | 관심사 |

## 4. 테스트 시나리오

### T1: 적합도 정확성 (2D + 식당/와인 분리)

**질문**: 같은 클러스터 유저 간 적합도가 실제로 높게, 다른 클러스터 간에는 낮게 나오는가?

```
식당 적합도:
- 같은 식당 클러스터 쌍 100개 → 평균 적합도
- 다른 식당 클러스터 쌍 100개 → 평균 적합도
- 분리가 명확한가? (예: 같은 클러스터 >70%, 다른 <40%)

와인 적합도:
- 같은 와인 클러스터 쌍 100개 → 평균 적합도
- 다른 와인 클러스터 쌍 100개 → 평균 적합도

교차 검증:
- 식당 클러스터 같고 와인 클러스터 다른 쌍 → 식당 적합도 높고 와인 적합도 낮은가?
- 2D 거리가 1D 만족도 거리보다 취향 차이를 더 잘 포착하는가?
  (예: 만족도 같은 7인데 (9,5) vs (5,9) → 2D에서는 차이 포착, 1D에서는 동일)
```

**D 값 탐색**: 2D 유클리드 거리의 최대값은 √(10²+10²) ≈ 14.1이지만, mean-centering 후 실제 분포는 더 작다. D를 2, 4, 6, 8, 10으로 변경하며 분리도 확인.

### T2: 신뢰도 곡선 (λ 튜닝)

**질문**: 겹치는 기록이 몇 개일 때 적합도가 안정화되는가?

```
- 특정 유저 쌍을 잡고 겹치는 기록을 1, 2, 3, ..., 30개씩 늘려가며
- 각 단계의 적합도 값이 최종(전체 겹침)과 얼마나 차이나는지 측정
- λ = 3, 5, 7, 10, 15로 변경하며 신뢰도 곡선이 현실을 잘 반영하는지 확인
```

### T3: Mean-centering 효과

**질문**: 후한 유저와 박한 유저의 편향이 보정되는가?

```
- Mean-centering 적용 전/후 적합도 비교
- 후한 유저(bias +2) ↔ 박한 유저(bias -2)가 같은 클러스터일 때
  - 보정 전: 적합도 낮게 나옴 (점수 차이 커서)
  - 보정 후: 적합도 높게 나와야 함 (패턴이 같으니까)
```

### T4: Nyam 점수 예측 정확도 (핵심)

**질문**: 예측 점수와 실제 점수의 오차가 허용 범위 내인가?

2D 사분면이므로 **X축, Y축 각각의 MAE + 만족도 MAE**를 측정한다.

```
- 각 유저에 대해 기록의 20%를 숨기고 (hold-out)
- 나머지 80%로 CF 예측을 수행 (X축, Y축 별도 예측)
- 숨긴 20%에 대해 예측 vs 실제 비교
- 측정 지표:
  - MAE_X: X축 예측 오차
  - MAE_Y: Y축 예측 오차
  - MAE_satisfaction: 만족도((X+Y)/2) 예측 오차
  - 2D 유클리드 오차: √((예측X-실제X)² + (예측Y-실제Y)²) 평균
- 목표: MAE_satisfaction < 1.0 (10점 만점 기준)
```

**식당/와인 별도 측정**: 식당 예측 MAE와 와인 예측 MAE를 따로 측정. 와인 기록이 적으므로 와인 쪽 MAE가 더 클 수 있음 → 허용 범위 차등 설정 가능.

**비교군**:
- (a) 전체 유저 CF (관계 부스트 없음) → baseline
- (b) 전체 유저 CF + 관계 부스트 → Nyam 점수
- (c) 팔로잉만 CF → 기존 PRD 방식
- (d) 단순 평균 (네이버 방식) → 비교 기준
- (e) 1D 만족도만으로 CF → 2D의 이점 확인용

### T5: 관계 부스트 민감도

**질문**: 부스트 비율(2.0/1.5/1.0)이 예측 정확도에 미치는 영향은?

```
- 부스트 조합을 변경하며 T4의 MAE 측정:
  - (3.0, 2.0, 1.0) — 팔로잉 강하게
  - (2.0, 1.5, 1.0) — 현재 가정
  - (1.5, 1.2, 1.0) — 팔로잉 약하게
  - (1.0, 1.0, 1.0) — 부스트 없음 (순수 CF)
- 어떤 조합이 MAE를 가장 낮추는가?
```

### T6: 콜드스타트 성능

**질문**: 기록이 적은 유저에게도 유의미한 예측이 나오는가?

```
- 기록 수별 그룹: 1~3개, 4~7개, 8~15개, 16~30개, 31+개
- 각 그룹의 평균 MAE
- 팔로잉 0명 vs 5명 vs 10명일 때 MAE 차이
- 최소 기록 몇 개부터 MAE < 1.5인가?
```

### T7: 버블 점수 맥락 효과

**질문**: 버블 점수가 글로벌 Nyam 점수와 다른 유의미한 값을 내는가?

```
- 일식 마니아 버블: 일식 식당에 대해 글로벌 점수 vs 버블 점수 비교
  - 버블 점수가 일식 식당에 대해 더 정확해야 함 (MAE 더 낮아야)
- 강남 맛집 버블: 강남 식당에 대해 글로벌 vs 버블
- 버블 멤버 수 최소 몇 명이어야 유의미한 버블 점수가 나오는가?
```

### T8: 예측 확신도 검증

**질문**: 예측 확신도가 실제 예측 정확도와 상관관계가 있는가?

```
- 모든 예측에 대해 확신도 계산
- 확신도 구간별 (낮음/보통/높음) 실제 MAE 측정
- 확신도 높음 → MAE 낮아야, 확신도 낮음 → MAE 높아야
- 단조 감소 + 음의 상관 목표 (→ 결과: 단조 ✅, 상관 -0.144 — 구조적 한계 §8.2 참조)
```

### T9: 오염 내성 (★ 최우선 — Nyam에 CF가 필요한 이유)

> Nyam의 존재 이유는 "오염된 정보를 걸러내는 것"이다. CF가 이 역할을 못 하면 네이버 평점과 다를 바 없다. 이 테스트가 시뮬레이션에서 가장 중요하다.

#### T9-A: 오염 유형별 영향 측정

5가지 현실적 오염 시나리오를 테스트한다:

| 오염 유형 | 시뮬레이션 방법 | 현실 대응 |
|----------|---------------|----------|
| **랜덤 노이즈** | 완전 랜덤 점수 (일관성 0) | 무성의한 유저, 아무 점수나 누르는 사람 |
| **광고성 조작** | 특정 식당 5곳만 극단적 고평가 (9~10), 나머지는 정상 | 업주/대행사가 점수 올리기 |
| **악의적 폄하** | 특정 식당 5곳만 극단적 저평가 (0~1), 나머지는 정상 | 경쟁 업체 깎아내리기 |
| **집단 조작** | 오염 유저 10명이 동일 패턴으로 특정 식당 고평가 | 조직적 별점 테러 |
| **은밀한 조작** | 대부분 정상이지만 특정 장르만 살짝 편향 (+2) | 가장 탐지 어려운 유형 |

각 유형별로 오염 유저 비율을 변경하며 테스트:
- 오염 5% (100명 중 5명)
- 오염 10% (100명 중 10명)
- 오염 20% (100명 중 20명) — 극한 시나리오

#### T9-B: CF vs 단순 평균 방어력 비교 (핵심)

**같은 오염 데이터**에 대해 네 가지 방식의 점수 영향을 비교한다:

```
(a) 단순 산술평균 (네이버 방식) — 오염에 무방비 기준선
(b) 전체 유저 CF (부스트 없음) — CF의 순수 방어력
(c) 전체 유저 CF + 관계 부스트 — Nyam 점수
(d) 팔로잉만 CF — 네트워크 방어력
```

측정 지표:
- **오염 침투율**: 오염 유저 포함 전/후 정직한 유저의 예측 점수 변화량
- **MAE 변화**: 오염 전 MAE vs 오염 후 MAE (작을수록 방어력 높음)
- **타겟 식당 점수 왜곡**: 광고/폄하 대상 식당의 예측 점수가 얼마나 움직이는가

**당초 목표**: CF 방식에서 오염 침투율이 단순 평균 대비 70% 이상 감소.
**실제 결과**: T9-B 전용 측정은 holdout 구조 한계로 정량화 실패. 대신 T4에서 CF가 단순 평균 대비 63.9% 정확 (MAE 0.426 vs 1.179) — 오염이 섞인 전체 데이터에서도 CF가 압도적. T9-A/C/E에서 오염 내성 별도 확인 (§8.2 참조).

#### T9-C: 오염원 적합도 자동 필터링

CF가 오염원을 자연스럽게 걸러내는 메커니즘을 검증한다:

```
- 정직한 유저 ↔ 오염 유저 간 적합도 분포
  → 적합도가 자연스럽게 낮아지는가? (낮아야 자동 감쇠됨)
- 오염 유저 간 적합도
  → 랜덤 유저끼리는 적합도가 높아지지 않는가? (높아지면 서로 강화됨)
- 은밀한 조작 유저의 적합도
  → 대부분 정상이라 적합도가 높게 나오면? (CF의 한계 지점)
```

#### T9-D: 팔로우/언팔의 방어 효과

```
- 오염 유저를 실수로 팔로우한 상태에서의 점수 왜곡
- 언팔 후 점수 복원 (즉시 반영되는가?)
- 맞팔(×2.0 부스트)된 오염 유저의 피해 규모
  → 부스트가 오히려 오염을 증폭하지 않는가?
```

#### T9-E: 오염 비율 한계점

```
- 오염 비율을 5% → 10% → 20% → 30% → 50%로 올려가며
- CF의 방어력이 무너지는 지점(tipping point)은 어디인가?
- 무너지는 시점에서 팔로잉 부스트가 추가 방어력을 제공하는가?
```

#### T9-F: 버블 내 오염 격리

```
- 버블 멤버 20명 중 오염 유저 3명이 섞였을 때
- 글로벌 Nyam 점수 vs 오염된 버블 점수 비교
- 오염된 버블에서 탈퇴하면 글로벌 점수에 영향 없는지 확인
- 소규모 버블(5명)에서 1명이 오염이면? → 버블 점수의 최소 멤버 수와 연결
```

### T10: 스케일 & 속도 테스트 (★ 중요)

식당/와인 하나의 Nyam 점수를 산출하려면: **해당 아이템 기록자 전원과의 적합도 조회 → 신뢰도 계산 → 관계 부스트 적용 → 가중 예측**을 해야 한다. 유저가 늘면 이 연산이 선형으로 증가한다.

#### T10-A: 단건 예측 속도

**질문**: 한 식당/와인의 Nyam 점수 1건을 산출하는 데 걸리는 시간은?

```
연산 분해:
  1. 해당 아이템의 기록자 목록 조회
  2. 각 기록자와 나의 적합도 조회 (겹치는 기록 비교)
  3. 각 기록자와의 신뢰도 계산
  4. 관계 부스트 적용
  5. 가중 예측 산출

스케일별 측정:
  - 100명 유저, 기록자 10명 → ?ms
  - 1,000명 유저, 기록자 50명 → ?ms
  - 10,000명 유저, 기록자 200명 → ?ms

목표: 단건 < 100ms (상세 페이지 로드 시)
```

#### T10-B: 목록 페이지 속도

**질문**: 홈 피드에 식당 20개를 표시할 때, 20건의 Nyam 점수를 동시에 산출하는 시간은?

```
- 20건 직렬 계산 vs 병렬 계산
- 목표: 20건 < 500ms (피드 로드 시)
```

#### T10-C: 사전 계산(pre-compute) 전략

실시간이 느리면 사전 계산이 필요하다:

```
전략 A: 적합도 매트릭스 캐시
  - 유저 쌍별 적합도를 미리 계산해서 저장
  - 새 기록 추가 시 해당 쌍만 갱신
  - 저장 공간: N명 유저 → N×N/2 쌍 (1,000명 = 50만 쌍)

전략 B: Nyam 점수 캐시
  - 유저별 × 아이템별 예측 점수를 미리 계산
  - 새 기록/팔로우 변경 시 해당 유저 점수만 갱신
  - 저장 공간: 유저 × 아이템 수

전략 C: 하이브리드
  - 적합도 매트릭스는 캐시 (변동 적음)
  - 예측 점수는 실시간 계산 (적합도 조회만 빠르면 됨)

각 전략의 갱신 빈도, 저장 비용, 정합성 trade-off 측정
```

#### T10-D: Supabase 환경 현실 테스트

```
- Edge Function에서 CF 계산 시 cold start 포함 응답 시간
- PostgreSQL에서 적합도 매트릭스를 materialized view로 관리할 때 성능
- RPC 호출 vs 클라이언트 계산 비교
```

### 8.2 핵심 검증 결과

#### CF vs 단순 평균 (T4)

```
         MAE_sat  개선율
단순 평균   1.179    —
CF(Nyam)   0.426   63.9%
1D CF      0.440    —
```

- 식당: 66.5% 개선, 와인: 57.3% 개선
- **유저마다 다른 예측점수**를 제공하는 CF가 "모두에게 같은 점수"인 단순 평균보다 압도적으로 정확

#### 오염 방어 (T9)

| 항목 | 결과 | 판정 |
|------|------|------|
| 전체 MAE 영향 (10명 오염) | +0.016 | ✅ 미미 |
| 50명 오염(전체의 33%) | +0.012 | ✅ 내성 강함 |
| 정직↔오염 적합도 gap | 0.48 (정직 0.78 vs 오염 0.30) | ✅ 자동 감쇠 |
| 맞팔 오염 피해 → 언팔 후 | 즉시 복원 | ✅ |

**CF의 오염 방어 메커니즘**: 오염 유저는 일관된 취향 패턴이 없으므로 정직한 유저와의 적합도가 자연스럽게 낮아진다 (0.30 vs 0.78). 가중치가 자동으로 줄어들어 점수에 미치는 영향이 미미. **누가 오염인지 판별할 필요 없이** 시스템이 자동 방어한다.

#### 적합도 분리 (T1)

| D값 | 식당 gap | p-value | 와인 gap | p-value |
|-----|---------|---------|---------|---------|
| 2 | 0.058 | <0.001 | 0.064 | 0.003 |
| 4 | 0.029 | <0.001 | 0.036 | 0.002 |
| 6 | 0.019 | <0.001 | 0.024 | 0.002 |
| 8 | 0.015 | <0.001 | 0.018 | 0.002 |

모든 D값에서 식당/와인 모두 **통계적으로 유의미한 클러스터 분리** (p<0.005). gap 절대값이 작은 이유: 겹치는 기록이 많을수록 평균으로 수렴하는 CF의 정상 동작.

#### 예측 확신도 (T8)

```
확신도 공식 = n_factor × 0.50 + agreement × 0.35 + quality × 0.15

  n_factor  = effective_n / (effective_n + 7)     ← 유효 평가자 수 (가중치>0.05)
  agreement = 1 - weighted_std(편차) / 2          ← mean-centered 편차의 가중 분산
  quality   = avg_weight / (avg_weight + 0.3)     ← 평균 가중치 품질

구간           MAE    N     
낮음 (0~0.3)   0.591    4   → 점수 숨김
보통 (0.3~0.5) 0.530  113   → "참고용" 표시
높음 (0.5~0.7) 0.418  932   → 정상 표시
매우높음 (0.7+) 0.395   27   → 정상 표시

단조 감소: ✅  |  상관계수: -0.144 (약함, 구조적 한계)
```

상관계수가 목표(-0.3)에 미달인 이유: CF 자체가 정확(MAE 0.43)하여 오차 분산이 작음. 확신도가 높든 낮든 대부분 비슷하게 정확해서 상관이 약해질 수밖에 없는 구조적 한계. **단조 감소 달성**이 실용적으로 더 중요.

#### 관계 부스트 (T5)

```
부스트 조합            MAE_sat
5.0/3.0/1.0 (극강)     0.447
3.0/2.0/1.0 (강)       0.444
2.0/1.5/1.0 (PRD초안)  0.444
1.5/1.2/1.0 (확정)     0.443  ← 최적
1.0/1.0/1.0 (없음)     0.443
2.0/1.5/0.5 (비팔↓)    0.446
3.0/2.0/0.5 (비팔↓강)  0.449
```

차이가 미미한 이유: 평가자 중 팔로우 비율이 **9%** (1.2명/13.4명). 부스트는 극적 차이를 만드는 기능이 아니라 "소수의 신뢰하는 사람 의견을 미세 조정"하는 역할. 높은 부스트(2.0+)는 오염 유저 팔로우 시 피해가 커지므로 보수적 값(1.5/1.2/1.0)이 안전.

#### 콜드스타트 (T6)


##### 확정 아키텍처: 이벤트 기반 증분 업데이트

크론잡으로 전수 계산하는 대신, **기록이 추가될 때마다 관련 쌍만 갱신**한다. 서비스 시작부터 기록마다 증분 계산하므로 "10만 명을 한꺼번에 계산하는 시점"은 존재하지 않는다.

```
[유저가 기록 추가]
  → 백그라운드 (Supabase trigger / Edge Function)
  → 해당 아이템의 다른 기록자와의 적합도만 갱신
  → 겹침 ≥3 필터 적용 (겹침 < 3인 쌍은 스킵)

[예측 요청]
  → DB에서 사전계산된 적합도 조회 (O(1))
  → Top-50 필터 → 가중 예측
```

| 이벤트 | 작업 | 비용 | 유저 대기 |
|--------|------|------|----------|
| 기록 추가 | 해당 아이템 기록자와 적합도 갱신 | 아래 표 참조 | 없음 (백그라운드) |
| 팔로우 변경 | 관계 부스트 갱신 | ~1ms | 없음 |
| 점수 조회 | 캐시 적합도로 예측 | < 20ms/건 | 있음 (실시간) |
| 피드 로드 | 20건 예측 | < 400ms | 있음 (실시간) |

##### 기록 추가 이벤트 비용 (100,000명 기준, 겹침≥3 필터 적용)

| 식당 기록자 수 | 겹침≥3 후 | 이벤트 비용 | 판정 |
|--------------|----------|-----------|------|
| 20명 (소규모) | 5명 | 0.1초 | ✅ |
| 100명 (보통) | 25명 | 0.5초 | ✅ |
| 500명 (인기) | 125명 | 2.2초 | ✅ |
| 1,000명 (매우 인기) | 250명 | 4.5초 | ✅ |
| 5,000명 (극인기) | 1,250명 | 22.5초 | ⚠️ 비동기 큐 분산 필요 |

> 기록자 5,000명급 극인기 식당은 비동기 작업 큐(예: Supabase Edge Function + pg_cron)로 분산 처리하거나, delta 기반 갱신(적합도 변화량이 임계값 이하면 스킵)으로 최적화.

크론잡은 보조용: 주 1회 전체 적합도 정합성 체크 + 누락 쌍 보정.

**필터링 전략** (정확도 손실 0점 검증 완료):
- 겹침 ≥3 필터: 커버리지 92.3%, MAE 손실 0
- Top-50: 사전계산 적합도 기반으로 적용 (추가 비용 0)
