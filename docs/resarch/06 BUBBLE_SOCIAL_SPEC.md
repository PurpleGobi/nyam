## 1. 소셜의 철학

### 1-1. Nyam 소셜이 아닌 것

```
인스타그램: 모든 사람에게 보여주기 (공개 자기 표현)
네이버 리뷰: 불특정 다수에게 정보 제공 (오픈 리뷰)
카카오톡 그룹: 채팅 중심 커뮤니케이션

Nyam: "내가 고른 사람들"의 "구조화된 기록"을 공유하고 활용
```

### 1-2. 핵심 원칙

1. **기록이 먼저, 소셜은 그 다음** — 소셜 없이도 Nyam은 유용하다. 소셜은 기록의 가치를 증폭시키는 레이어
2. **폐쇄가 신뢰를 만든다** — 모두에게 열려있으면 오염된다. 광고 리뷰, 가짜 평가가 구조적으로 불가능한 환경
3. **공유는 선택적** — 내 기록 전체가 아닌, 필터링한 기록만 특정 버블에 공유. 통제권은 나에게
4. **소셜 압박 최소화** — 좋아요 수 비교, 팔로워 수 경쟁 지양. 기록의 깊이와 질이 가치

### 1-3. 레퍼런스 모델 분석

| 서비스 | 소셜 구조 | 차용할 것 | 피할 것 |
|--------|----------|----------|---------|
| **Letterboxd** | 팔로우 기반 + 리스트 공유 | 리뷰 문화, 리스트 큐레이션 | 공개 리뷰 중심 (신뢰 문제) |
| **Strava** | 클럽 + 팔로우 + 피드 | 클럽 구조, 활동 피드, 리더보드 | 공개 프로필 기본 (프라이버시) |
| **Untappd** | 친구 + 뱃지 + 체크인 공유 | 게이미피케이션, 체크인 피드 | 공개 체크인 (식당은 민감할 수 있음) |
| **Telegram** | 채널(1→N) + 그룹(N↔N) | 채널/그룹 이원 구조 | 채팅 중심 (Nyam은 기록 중심) |
| **Goodreads** | 그룹 + 서가 + 리뷰 | 그룹 서가 개념, 읽기 챌린지 | UI 구식, 소셜 기능 약함 |
| **Discord** | 서버 + 채널 + 역할 | 역할/권한 체계, 초대 시스템 | 너무 복잡 (Nyam은 심플) |

## 2. 버블 시스템

### 2-1. 버블이란

```
버블 = 초대제 폐쇄형 식도락 공유 그룹

- 멤버만 기록을 보고 공유할 수 있다
- 초대 링크 또는 직접 초대로만 가입
- 광고 리뷰, 가짜 평가가 구조적으로 불가능
- 내가 원하는 기록만 선택해서 공유
```

### 2-2. 버블 유형

| 유형 | 구조 | 사용 예시 | 멤버 권한 |
|------|------|----------|----------|
| **양방향 버블** | 모든 멤버가 공유 + 반응 | 직장 맛집 모임, 동네 맘, 와인 동호회 | 공유, 리뷰, 댓글, 찜, 리액션 |
| **열람 전용 버블** | 운영자만 공유. 구독자는 보기만 | 맛잘알 큐레이션, 소믈리에 추천 | 보기, 찜, 리액션 (리뷰/댓글 불가) |

### 2-3. 버블 생성 플로우

```
1. [+] 버블 만들기
2. 기본 설정
   - 이름 (필수, 20자 이내)
   - 설명 (선택, 100자 이내)
   - 유형: 양방향 / 열람 전용
   - 아이콘: 기본 제공 아이콘 선택 or 사진
3. 초대 설정
   - 초대 링크 생성 (만료 기간: 1일/7일/30일/무제한)
   - 직접 초대 (Nyam 사용자 검색)
   - 가입 승인 필요 여부 (ON/OFF)
4. 완료 → 버블 페이지로 이동
```

### 2-4. 버블 역할 & 권한

단순하게. Discord처럼 복잡한 역할 시스템은 과도하다.

| 역할 | 권한 | 인원 |
|------|------|------|
| **만든이** | 모든 관리 + 버블 삭제 | 1명 |
| **관리자** | 멤버 관리 + 공지 | N명 (만든이가 지정) |
| **멤버** | 공유 + 리뷰 + 댓글 + 리액션 | N명 |
| **구독자** | 보기 + 찜 + 리액션 (열람 전용 버블만) | N명 |

### 2-5. 버블 설정

| 설정 | 옵션 | 기본값 |
|------|------|--------|
| 가입 방식 | 초대 링크 / 직접 초대 / 승인제 | 초대 링크 |
| 멤버 상한 | 10 / 30 / 100 / 무제한 | 100 |
| 기록 가시성 | 멤버만 / 링크 있으면 누구나 | 멤버만 |
| 익명 리뷰 | 허용 / 금지 | 허용 |
| 알림 | 새 공유 시 / 내 기록에 반응 시 | 둘 다 ON |

### 2-6. 멤버 관리

- **초대**: 링크 공유 or 사용자 검색 → 초대
- **승인**: 승인제 ON일 때, 가입 요청 → 관리자 승인/거절
- **추방**: 관리자가 멤버 추방 (기록은 유지, 접근 차단)
- **탈퇴**: 멤버 자발적 탈퇴 (본인 공유 기록 유지 or 삭제 선택)
- **차단**: 특정 사용자를 영구 차단 (재가입 불가)

## 3. 공유 시스템

### 3-1. 공유의 단위

```
공유 단위 = 내 기록 (Record)
  - 1개 식당/와인에 대한 1회 방문/음용 기록
  - 사분면 좌표 + 만족도 + 상황 태그 + 한줄평 + 사진 등
  - 즉, 내가 평가한 기록 자체가 공유됨
```

### 3-2. 공유 플로우

```
방법 1: 기록 직후 공유
  기록 완료 화면 → "버블에 공유하기" 버튼 → 버블 선택 → 공유

방법 2: 기존 기록에서 공유
  내 기록 목록 → 기록 선택 → 공유 아이콘 → 버블 선택 → 공유

방법 3: 필터 일괄 공유
  내 기록 → 필터 적용 (지역 + 만족도 범위) → 선택된 기록들 → 버블에 일괄 공유
  예: "광화문 + 80점 이상" → 5개 기록을 "직장 맛집" 버블에 한 번에 공유
```

### 3-3. 공유 시 포함되는 정보

| 항목 | 포함 여부 | 비고 |
|------|----------|------|
| 식당/와인 이름 | 항상 | 기본 정보 |
| 사분면 좌표 | 항상 | 가격%/분위기% 또는 산미%/바디% |
| 만족도 (점 크기) | 항상 | 핵심 평가 |
| 상황 태그 | 항상 | 혼밥/데이트/회식 등 |
| 한줄평 | 있으면 포함 | 선택 입력이니 없을 수 있음 |
| 사진 | 있으면 포함 | |
| 추천 메뉴 | 있으면 포함 | |
| 사용팁 | 있으면 포함 | |
| 방문 날짜 | 항상 | |
| 가격 | 공유자 선택 | 민감할 수 있으므로 ON/OFF |
| 함께 간 사람 | 공유자 선택 | 프라이버시 보호 |

### 3-4. 공유 취소

- 공유한 기록은 언제든 버블에서 철회 가능
- 철회하면 버블 피드에서 사라지지만, 이미 달린 리뷰/댓글은 유지 (고아 상태)
- 내 기록 자체는 삭제되지 않음 (개인 기록은 별개)

## 4. 리뷰 & 댓글 구조

### 4-1. 리뷰 vs 기록 vs 댓글

```
기록 (Record): 내가 직접 방문/음용한 경험. 사분면 + 만족도 + 상황.
                → 개인 소유. 공유 선택 가능.

리뷰 (Review): 버블에 공유된 기록에 대해 다른 멤버가 쓰는 반응.
               "나도 거기 갔는데..." 형태.
               → 같은 식당/와인에 대한 본인의 독립된 평가.

댓글 (Comment): 특정 기록이나 리뷰에 대한 짧은 반응/토론.
               "거기 A코스 진짜 맛있지!" 형태.
               → 기록/리뷰에 종속.
```

### 4-2. 데이터 구조

```
식당/와인 (1)
  └── 기록 Record (N) — 각 사용자의 독립 평가
        ├── 리뷰 Review (N) — 다른 멤버의 반응 (버블 내에서)
        │     └── 댓글 Comment (N per Review)
        └── 댓글 Comment (N) — 기록 자체에 대한 댓글
              
리액션은 기록/리뷰/댓글 모두에 가능 (좋아요/찜)
```

### 4-3. 리뷰 작성 플로우

```
버블 피드에서 공유된 기록 보기
  → "나도 기록 남기기" 탭
  → 사분면 평가 (같은 식당/와인에 대한 내 평가)
  → 자동으로 해당 기록에 리뷰로 연결
  → 버블 피드에 표시: "○○님이 △△식당에 대한 기록을 남겼습니다"

즉, 리뷰 = 같은 대상에 대한 또 다른 기록. 별도 UI가 아니라 기록 플로우 자체.
```

### 4-4. 댓글

- 최대 300자
- 텍스트만 (사진 첨부 없음 — 댓글은 가볍게)
- 익명 옵션 (버블 설정에서 허용한 경우)
- 삭제: 본인만 가능

### 4-5. 리액션

| 리액션 | 대상 | 의미 |
|--------|------|------|
| **좋아요** | 기록, 리뷰, 댓글 | "공감" / "도움됐어" |
| **찜** | 기록의 식당/와인 | 내 Wishlist에 추가 |

- 리액션 종류는 2개면 충분. 이모지 리액션은 오버킬.
- 좋아요 수는 표시하되, 랭킹/정렬에 가볍게만 활용 (소셜 압박 최소화)

## 5. 버블 피드 & 페이지

### 5-1. 버블 피드 (홈화면 하단)

PRD v2 홈화면 구조에서 내 기록 아래에 버블 피드가 이어지는 형태.

```
┌─────────────────────────────┐
│ 내 식도락 (기록 타임라인)      │ ← Phase 1 홈
│ ...                          │
├─────────────────────────────┤
│ ── 버블 ──────────────────  │ ← Phase 2 추가
│                             │
│ [아바타] 김영수 · 직장 맛집   │
│ 미진 — 93점 · 혼밥           │
│ "메밀국수 진짜 맛있다"         │
│ ♡ 4  💬 2  🔖               │
│                             │
│ [아바타] 박소연 · 와인 모임   │
│ Penfolds Grange — 96점      │
│ ♡ 8  💬 5  🔖               │
│                             │
│ [모든 버블 보기 →]            │
└─────────────────────────────┘
```

**피드 정렬**: 최신순 (시간 역순). 알고리즘 피드 없음 — 신뢰 기반 커뮤니티에서 알고리즘이 무엇을 보여줄지 선택하면 신뢰가 깨진다.

**피드 필터**:
- 전체 버블 / 특정 버블 선택
- 식당만 / 와인만
- 상황 태그 필터

### 5-2. 버블 상세 페이지 (`/groups/[id]`)

```
┌──────────────────────────────────────┐
│ [←] 직장 맛집 버블                     │
│ 멤버 8명 · 기록 47개                    │
│                                       │
│ ── 헤더 ────────────────────────────  │
│ 버블 이름 + 설명 + 멤버 아바타 (가로)    │
│ [초대하기] [설정]                       │
│                                       │
│ ── 탭 ──────────────────────────────  │
│ [피드] [식당 목록] [와인 목록] [통계]     │
│                                       │
│ ─── 피드 탭 (기본) ─────────────────  │
│ 최신 공유 기록 리스트 (시간순)            │
│ 각 카드: 공유자 + 식당/와인 + 만족도     │
│         + 사분면 좌표 + 상황 + 한줄평    │
│         + 리액션 수 + 댓글 수            │
│                                       │
│ ─── 식당 목록 탭 ───────────────────  │
│ 이 버블에서 공유된 모든 식당              │
│ 정렬: 최신/만족도순/기록수순/찜수순       │
│ 필터: 지역, 장르, 상황, 만족도 범위      │
│ 각 식당: 이름 + 평균 만족도 + 기록 N개   │
│ 탭 → 식당 상세 (버블 기록 포함)          │
│                                       │
│ ─── 와인 목록 탭 ───────────────────  │
│ 동일 구조, 와인 버전                    │
│                                       │
│ ─── 통계 탭 ────────────────────────  │
│ 공유 랭킹: 누가 가장 많이 공유했는지      │
│ 인기 식당: 기록이 가장 많은 식당 Top 5    │
│ 장르 분포: 버블 전체 장르별 기록 수       │
│ 이번 달 핫: 최근 30일 기록 트렌드        │
│ 사분면 분포: 버블 전체 기록의 사분면 산포  │
└──────────────────────────────────────┘
```

### 5-3. 버블 사분면 분포

버블 통계의 핵심 시각화. 멤버 전체 기록을 하나의 사분면 위에 표시.

```
"우리 버블은 저렴+캐주얼 쪽에 점이 몰려있네 — 동네 맛집 위주"
"회식 버블은 고가+포멀 쪽에 큰 점이 많아 — 비싼 곳을 잘 찾아냄"
```

- 각 멤버별 색상 구분 (아바타 색)
- 점 크기 = 만족도
- 전체 기록 수 표시

## 6. 식당/와인 상세페이지 — 소셜 레이어

### 6-1. 상세페이지 내 버블 기록 섹션

DETAIL_PAGE_SPEC과 연동. 식당/와인 상세페이지에서 **내 기록 아래에 버블 기록 섹션**이 추가된다.

```
── 버블 기록 ──────────────────────

[필터: 전체 버블 / 직장 맛집 / 와인 모임]

┌────────────────────────────────┐
│ [아바타] 김영수 · 직장 맛집      │
│ 만족도 90 · 회식 · 3/15         │
│ 가격 75% · 분위기 80%           │ ← 사분면 좌표
│ "여기 코스A 진짜 미침"           │
│ ♡ 4  💬 2                      │
│                                │
│   └ 박소연: 거기 런치도 괜찮아    │ ← 댓글 미리보기
│   └ [댓글 더보기]                │
├────────────────────────────────┤
│ [아바타] 박소연 · 와인 모임      │
│ 만족도 78 · 모임 · 2/20         │
│ 가격 80% · 분위기 70%           │
│ "음식은 좋은데 와인 리스트 별로"  │
│ ♡ 1  💬 0                      │
└────────────────────────────────┘
```

### 6-2. 표시 규칙

| 규칙 | 설명 |
|------|------|
| 정렬 | 최신순 |
| 최대 표시 | 5개 (더보기로 확장) |
| 버블 필터 | 복수 버블 소속 시 특정 버블만 보기 가능 |
| 비멤버 | 해당 버블에 속하지 않은 기록은 안 보임 |
| 내 기록 구분 | 내 기록은 "나의 기록" 섹션에, 버블 멤버 기록은 "버블 기록" 섹션에 |
| 사분면 표시 | 각 기록의 좌표를 텍스트로 표시 (가격 75% · 분위기 80%) |

### 6-3. 상세페이지 사분면 차트에 버블 기록 추가

식당 상세의 사분면 차트에 **버블 멤버의 점도 함께** 표시.

```
내 기록: 불투명, 상황별 테두리 색
버블 기록: 반투명 (30% opacity), 멤버 아바타 색
외부 레퍼런스: 매우 연한 (15% opacity)
```

한 사분면에 내 점과 버블 멤버의 점이 함께 보이면:
- "나는 여기가 고가+포멀이라고 느꼈는데, 김영수는 좀 더 캐주얼하게 느꼈네"
- "전체적으로 만족도가 높은 점이 많으니 좋은 곳이구나"

## 9. 팔로우 시스템 (P2+)

### 9-1. 버블과의 차이

```
버블: 그룹 단위. 초대제. 모든 멤버의 기록이 공유.
팔로우: 개인 단위. 일방향. 상대의 공개 기록만 보임.
```

### 9-2. 팔로우 기능

- 다른 사용자를 팔로우 → 그 사람의 **공개 기록**이 내 피드에 노출
- 공개 범위: 프로필 설정에 따름 (전체 공개 선택한 사용자만 팔로우 가능)
- 팔로우는 **승인 불필요** (공개 프로필이면 누구나 팔로우)
- 팔로워 수 표시는 하되, 프로필에서 크게 강조하지 않음

### 9-3. 팔로우 피드 vs 버블 피드

```
홈화면 피드 구성:
  탭 1: 버블 (기본)
  탭 2: 팔로잉
  탭 3: 전체 (버블 + 팔로잉 통합)
```

## 10. 경험치 & 소셜 신뢰

### 10-1. 경험치의 소셜 활용

PRD v2 §8의 경험치 시스템은 개인 기록의 깊이를 나타내지만, 소셜에서는 **신뢰의 근거**가 된다.

```
"을지로 Lv.7인 사람이 을지로 식당을 추천하면 무게가 다르다"
"내추럴 와인 Lv.5인 사람의 와인 리뷰는 참고할 만하다"
```

### 10-2. 버블 내 경험치 표시

- 각 멤버의 기록/리뷰 옆에 **관련 경험치 레벨만** 표시
  - 을지로 식당 기록 → "을지로 Lv.7" 뱃지
  - 보르도 와인 기록 → "보르도 Lv.3" 뱃지
- 전체 레벨 목록은 프로필에서만 보임
- 경험치에 따른 기록 정렬 옵션: "경험자 순" (해당 지역/종류 레벨 높은 순)

### 10-3. 경험치가 추천에 미치는 영향

버블 추천에서 경험치는 **가중치**가 아닌 **필터/정렬 옵션**으로만 활용.

```sql
-- "이 동네 경험 많은 사람의 기록만 보기"
SELECT r.*, u.experience_level
FROM records r
JOIN user_experience u ON r.user_id = u.user_id 
  AND u.area = :target_area
WHERE r.restaurant_id = :restaurant_id
ORDER BY u.experience_level DESC, r.satisfaction DESC
```

경험치로 자동 가중치를 주면 초보자의 기록이 묻힌다 → 공정하지 않음. 사용자가 원할 때 "경험자 순" 정렬로 선택하는 게 맞다.

## 11. 데이터 구조

### 11-1. 핵심 테이블

```sql
-- 버블
CREATE TABLE bubbles (
  id UUID PRIMARY KEY,
  name VARCHAR(20) NOT NULL,
  description VARCHAR(100),
  type ENUM('bidirectional', 'readonly') DEFAULT 'bidirectional',
  icon_url TEXT,
  created_by UUID REFERENCES users(id),
  invite_code VARCHAR(20) UNIQUE,
  invite_expires_at TIMESTAMPTZ,
  requires_approval BOOLEAN DEFAULT false,
  max_members INT DEFAULT 100,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 버블 멤버
CREATE TABLE bubble_members (
  bubble_id UUID REFERENCES bubbles(id),
  user_id UUID REFERENCES users(id),
  role ENUM('owner', 'admin', 'member', 'subscriber') DEFAULT 'member',
  joined_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (bubble_id, user_id)
);

-- 공유 (기록 → 버블 연결)
CREATE TABLE bubble_shares (
  id UUID PRIMARY KEY,
  record_id UUID REFERENCES records(id),
  bubble_id UUID REFERENCES bubbles(id),
  shared_by UUID REFERENCES users(id),
  shared_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (record_id, bubble_id)  -- 같은 기록을 같은 버블에 중복 공유 방지
);

-- 댓글
CREATE TABLE comments (
  id UUID PRIMARY KEY,
  target_type ENUM('record', 'review') NOT NULL,
  target_id UUID NOT NULL,  -- record_id or review_id
  bubble_id UUID REFERENCES bubbles(id),
  user_id UUID REFERENCES users(id),
  content VARCHAR(300) NOT NULL,
  is_anonymous BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 리액션
CREATE TABLE reactions (
  id UUID PRIMARY KEY,
  target_type ENUM('record', 'comment') NOT NULL,
  target_id UUID NOT NULL,
  reaction_type ENUM('like', 'bookmark') NOT NULL,
  user_id UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (target_type, target_id, reaction_type, user_id)
);

-- 알림
CREATE TABLE notifications (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id),
  type VARCHAR(20) NOT NULL,  -- 'share', 'reaction', 'comment', 'invite', 'review'
  actor_id UUID REFERENCES users(id),
  target_type VARCHAR(20),
  target_id UUID,
  bubble_id UUID REFERENCES bubbles(id),
  is_read BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 팔로우
CREATE TABLE follows (
  follower_id UUID REFERENCES users(id),
  following_id UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (follower_id, following_id)
);
```

### 11-2. RLS (Row Level Security) 정책

```sql
-- 버블 기록은 멤버만 볼 수 있음
CREATE POLICY bubble_share_read ON bubble_shares
  FOR SELECT USING (
    bubble_id IN (
      SELECT bubble_id FROM bubble_members 
      WHERE user_id = auth.uid()
    )
  );

-- 자기 기록만 공유 가능
CREATE POLICY bubble_share_insert ON bubble_shares
  FOR INSERT WITH CHECK (
    shared_by = auth.uid() AND
    record_id IN (SELECT id FROM records WHERE user_id = auth.uid())
  );
```

## 13. 핵심 설계 결정 요약

| 결정 | 선택 | 이유 |
|------|------|------|
| 피드 정렬 | 시간순 (알고리즘 없음) | 신뢰 커뮤니티에서 알고리즘이 선별하면 신뢰 깨짐 |
| 리액션 종류 | 좋아요 + 찜 (2개만) | 이모지 리액션은 노이즈. 심플하게 |
| 리뷰 = 기록 | 리뷰 전용 UI 없음, 같은 대상에 기록 남기기 | 평가 문법 통일 (사분면), 별도 UI 불필요 |
| 프로필 기본값 | 버블 멤버에게만 공개 | 프라이버시 우선 |
| 경험치 소셜 활용 | 필터/정렬 옵션 (자동 가중치 아님) | 초보자 기록이 묻히면 안 됨 |
| 댓글에 사진 | 불가 | 댓글은 가벼운 반응. 사진은 기록에서 |
| 익명 리뷰 | 버블 설정에서 허용/금지 선택 | 솔직한 의견 vs 책임감 — 버블 운영자가 결정 |
| 공유 단위 | 기록 1개씩 (+ 일괄 공유 옵션) | 통제감. 내가 뭘 공유하는지 알아야 |
