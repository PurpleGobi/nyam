# Nyam 추천 알고리즘 기획서

> 작성일: 2026-03-19
> 참조: PRD v2.0 (F12, §4-2, §3-2), 홈화면 딥리서치, DETAIL_PAGE_SPEC
> 범위: 식당 추천 + 와인 추천

---

## 1. 추천 철학

### 1-1. Nyam이 아닌 것

Nyam의 추천은 네이버/카카오처럼 **"모든 사람에게 맞는 인기 맛집"**을 보여주는 게 아니다.

```
네이버/카카오: "여기 인기 많아" (대중 인기도)
Vivino:       "비슷한 사람들이 좋아해" (Collaborative Filtering)
Nyam:         "네가 이런 상황에서 이런 걸 좋아하잖아" (Context-Aware Personal)
```

### 1-2. Nyam 추천의 3원칙

1. **내 데이터가 먼저** — 내 기록이 충분하면 외부 데이터는 배경으로 밀린다
2. **상황이 핵심** — 같은 사람이라도 혼밥/데이트/회식에서 원하는 게 다르다
3. **신뢰가 필터** — 알고리즘보다 "내가 아는 사람의 경험"이 우선

### 1-3. 추천이 기록을 침범하지 않는다

홈화면 딥리서치에서 확정한 원칙: **추천은 접혀있다가 필요할 때만 펼침**. 홈의 주인공은 내 기록이지, 추천이 아니다. 추천은 "오늘 뭐 먹지?" 탭을 열었을 때 작동한다.

## 2. 추천 유형 분류

### 2-1. 전체 추천 맵

| 유형 | 트리거 | 데이터 소스 | Phase |
|------|--------|------------|-------|
| **재방문 추천** | 홈 "오늘 뭐 먹지?" | 내 기록 | P1 |
| **상황별 추천** | 상황 필터 선택 시 | 내 기록 × 상황 태그 | P1 |
| **찜 리마인드** | 홈 / 지역 진입 시 | Wishlist | P1 |
| **권위 추천** | 콜드스타트 / 새 지역 | 미슐랭/블루리본/외부 평점 | P1 |
| **와인 페어링 추천** | 홈 "와인 고르기" | 내 와인 기록 + 음식 메타 | P1 |
| **버블 추천** | 버블 피드 | 버블 멤버 기록 | P2 |
| **취향 유사 추천** | 발견 탭 | Collaborative Filtering | P2+ |
| **AI 발견** | 발견 탭 | 전체 기록 + 컨텐츠 기반 | P3 |

### 2-2. Phase별 복잡도

```
Phase 1: 규칙 기반 (Rule-Based) — SQL 쿼리로 충분
Phase 2: 소셜 그래프 기반 — 버블/팔로우 관계 활용
Phase 3: ML 기반 — Collaborative Filtering + Content-Based Hybrid
```

## 3. Phase 1 추천 알고리즘 (규칙 기반)

### 3-1. 재방문 추천 — "다시 가고 싶은 곳"

**트리거**: 홈 → "오늘 뭐 먹지?" expand

**알고리즘:**

```sql
-- 기본 로직: 높은 점수 + 최근 안 간 곳 + 상황 매칭
SELECT r.restaurant_id, 
       AVG(r.rating_overall) as avg_score,
       MAX(r.created_at) as last_visit,
       COUNT(*) as visit_count
FROM records r
WHERE r.user_id = :current_user
  AND r.type = 'restaurant'
  AND r.rating_overall >= 80  -- 높은 평가만
GROUP BY r.restaurant_id
ORDER BY 
  -- 스코어링: 점수 × 시간 감쇠 × 재방문 보너스
  (AVG(r.rating_overall) * 0.6)                    -- 60%: 내 평가
  + (EXTRACT(DAY FROM NOW() - MAX(r.created_at)) 
     * 0.002 * 100) * 0.3                          -- 30%: 오래 안 갔을수록 ↑
  + (LEAST(COUNT(*), 5) * 4) * 0.1                 -- 10%: 재방문 많을수록 ↑
  DESC
LIMIT 10
```

**핵심 요소:**
- **점수 가중치 60%** — 내가 높게 평가한 곳이 기본
- **시간 감쇠 30%** — 최근에 간 곳은 추천 하위로 (어제 간 곳을 오늘 또 추천하지 않음)
- **재방문 보너스 10%** — 여러 번 갔으면 그만큼 좋아하는 곳

### 3-2. 상황별 추천 — "오늘 데이트인데..."

**트리거**: 재방문 추천에서 상황 필터 선택

**알고리즘:**

```sql
-- 상황 태그 매칭 추가
SELECT r.restaurant_id,
       AVG(r.rating_overall) as avg_score,
       r.scene_tag
FROM records r
WHERE r.user_id = :current_user
  AND r.type = 'restaurant'
  AND r.scene_tag = :selected_scene  -- '데이트', '혼밥', '회식' 등
  AND r.rating_overall >= 75
GROUP BY r.restaurant_id, r.scene_tag
ORDER BY AVG(r.rating_overall) DESC
LIMIT 10
```

**상황 태그 목록 (식당):**
| 태그 | 설명 | 추천 시 강조 요소 |
|------|------|------------------|
| 혼밥 | 혼자 먹기 | 가성비, 빠른 서비스 |
| 데이트 | 연인과 | 분위기, 맛 |
| 친구 | 친구 모임 | 맛, 양, 접근성 |
| 가족 | 가족 식사 | 분위기, 메뉴 다양성 |
| 회식·접대 | 비즈니스 | 분위기, 서비스, 와인 |
| 술자리 | 술 중심 | 안주, 분위기, 주류 |

**핵심 인사이트**: 같은 식당이라도 상황에 따라 점수가 다를 수 있다. "도쿄등심: 회식 92점, 혼밥 83점" → 회식 상황에서는 상위 추천, 혼밥에서는 중위.

### 3-3. 사분면 수치를 필터로 활용

사분면(가격×분위기, 산미×바디)은 **사용자가 직관적으로 파악하기 위한 시각화 도구**이다. 추천 알고리즘이 좌표 클러스터링이나 유사도 계산을 할 필요 없다. 그냥 **수치를 필터/정렬 조건으로** 쓰면 충분하다.

**알고리즘:**

```sql
-- "오늘 데이트인데 고급스러운 곳 가고 싶어"
SELECT r.restaurant_id, AVG(r.satisfaction) as avg_sat
FROM records r
WHERE r.user_id = :current_user
  AND r.scene = '데이트'
  AND r.satisfaction >= 80
  AND r.axis_x >= 60        -- 가격: 고가 쪽
  AND r.axis_y >= 60        -- 분위기: 포멀 쪽
GROUP BY r.restaurant_id
ORDER BY avg_sat DESC

-- "혼밥으로 가성비 좋은 곳"
SELECT r.restaurant_id, AVG(r.satisfaction) as avg_sat
FROM records r  
WHERE r.user_id = :current_user
  AND r.scene = '혼밥'
  AND r.satisfaction >= 75
  AND r.axis_x <= 40        -- 가격: 저렴 쪽
GROUP BY r.restaurant_id
ORDER BY avg_sat DESC
```

**핵심**: 사분면 좌표는 추천의 **필터 조건**일 뿐이다. 상황 태그 + 만족도 + 좌표 범위. SQL WHERE절이면 끝.

### 3-4. 찜(Wishlist) 리마인드

**트리거**: 홈 "새로운 도전" 영역 / 새 지역 진입 푸시

**알고리즘:**

```sql
SELECT w.restaurant_id, w.created_at,
       ST_Distance(r.location, :user_location) as distance
FROM wishlists w
JOIN restaurants r ON w.restaurant_id = r.id
WHERE w.user_id = :current_user
  AND w.visited = false
ORDER BY 
  -- 가까운 곳 + 오래된 찜 우선
  (1 / (ST_Distance(r.location, :user_location) + 1)) * 0.6
  + (EXTRACT(DAY FROM NOW() - w.created_at) * 0.01) * 0.4
LIMIT 5
```

- 현재 위치에서 가까운 찜 식당 우선
- 오래 찜해둔 곳 리마인드 ("3개월 전에 찜한 곳이에요")

### 3-5. 권위 추천 — 콜드스타트/새 지역

**트리거**: 기록 < 5개 / 새 지역 진입

**알고리즘:**

```
1순위: 미슐랭/블루리본 선정 식당 (권위 필터)
2순위: 외부 평점 상위 (네이버 ≥ 4.3 AND 카카오 ≥ 4.0 AND 구글 ≥ 4.2)
3순위: Nyam 전체 사용자 평균 점수 상위

표시 방식:
  - "미슐랭 선정" / "블루리본" 뱃지
  - "네이버 4.5 · 카카오 4.3 · 구글 4.6" 외부 점수 투명하게 노출
  - "Nyam 사용자 87점 (14명)" 있으면 표시
```

**원칙**: 외부 데이터는 **참고용으로만** 투명하게 제시. "Nyam이 추천합니다"가 아니라 "이 동네에서 평가가 높은 곳이에요" 톤.

### 3-6. 지역 푸시 추천

**트리거**: 새 지역 geofence 진입

**알고리즘:**

```
1. 현재 지역(동 단위) 식별
2. 해당 지역의 식당 리스트 구성:
   a. 내가 이 지역에서 높게 평가한 곳 (있으면)
   b. 내 찜 목록 중 이 지역의 식당
   c. 버블 멤버가 이 지역에서 높게 평가한 곳 (P2)
   d. 권위 추천 (미슐랭/블루리본/외부 평점)
3. 푸시: "○○동에 오셨네요! 이 동네 맛집 둘러볼까요?"
4. 가본 곳 체크 유도 → 기록 확장
```

## 4. 와인 추천 알고리즘

### 4-1. 와인 추천의 특수성

식당과 와인은 추천 로직이 근본적으로 다르다:

| 차이점 | 식당 | 와인 |
|--------|------|------|
| 반복 소비 | 같은 곳 여러 번 | 같은 와인 재구매 드뭄 |
| 탐색 동기 | "어디 가지?" (장소 탐색) | "뭐 마시지?" (제품 탐색) |
| 컨텍스트 | 위치 + 상황 | 음식 페어링 + 상황 |
| 객관적 축 | 장르/가격대 정도 | 품종/산지/빈티지/스타일 (구조화됨) |
| 취향 정밀도 | 상대적 낮음 | 매우 높음 (산미 좋아하면 확실) |

### 4-2. 와인 사분면 수치 + 메타데이터 필터

식당과 동일한 원칙. 사분면 좌표(산미×바디)는 필터 조건으로 활용하고, 향 팔레트는 사용자가 직접 보고 판단하는 시각화 도구이지 알고리즘이 색상 거리를 계산할 필요 없다.

```sql
-- "풀바디 레드 중 만족도 높았던 것과 비슷한 와인"
SELECT w.id, w.name, w.variety, w.region
FROM wines w
WHERE w.type = 'red'
  AND w.body >= 70           -- 바디: 풀바디 쪽 (와인 DB 메타)
  AND w.id NOT IN (          -- 이미 마신 와인 제외
    SELECT target_id FROM records 
    WHERE user_id = :user AND target_type = 'wine'
  )
ORDER BY w.popularity DESC   -- 또는 Nyam 전체 평균 만족도
LIMIT 10

-- "내가 좋아하는 산미 범위의 화이트"
-- 내 높은 만족도 와인의 산미 범위 파악 후 필터
SELECT AVG(r.axis_x) as preferred_acidity
FROM records r
WHERE r.user_id = :user AND r.target_type = 'wine' AND r.satisfaction >= 80
-- → preferred_acidity = 45 라면
-- → 와인 DB에서 산미 35~55 범위 필터
```

**향 팔레트는 추천에 활용하지 않는다.** 향은 같은 와인이라도 사람마다 다르게 느끼는 주관적 데이터. 추천 필터로 쓰기엔 부정확하고, 사용자가 사분면 위의 색깔 분포를 보고 자기 취향을 직관적으로 파악하는 용도로 충분하다.

추천에 활용하는 와인 속성은 **와인 DB의 객관적 메타데이터**(품종, 산지, 타입, 바디, 산미 등급)이다.

### 4-3. 음식 페어링 추천 — "오늘 메뉴에 어울리는 와인"

**트리거**: 홈 "와인 고르기" → 메뉴 입력

**알고리즘:**

```
1. 음식 카테고리 매핑
   "스테이크" → heavy_protein, red_meat, grilled
   "해산물" → light_protein, seafood, raw
   "파스타" → carb, tomato_based OR cream_based

2. 페어링 규칙 (WSET 기반)
   heavy_protein + grilled → 높은 타닌, 풀바디 (Cabernet, Malbec)
   light_protein + seafood → 높은 산미, 라이트바디 (Sauvignon Blanc, Chablis)
   cream_based → 미디엄 바디, 중간 산미 (Chardonnay)

3. 내 기록에서 필터링
   - 내가 마셔본 와인 중 페어링 조건에 맞는 것 우선
   - "이 와인은 전에 스테이크와 마셨을 때 94점 줬어요"
   - 없으면 페어링 규칙 기반 일반 추천

4. 출력
   a. 내 와인 기록에서 매칭 (있으면 최우선)
   b. 와인 DB에서 페어링 매칭 + Wine DNA 유사도
```

**홈화면 목업의 "와인 고르기" expand와 연동:**
```
┌─────────────────────────────┐
│ 메뉴 입력: [스테이크]        │
│                             │
│ 내 와인에서 찾기:            │
│ Ch. Margaux 2018  94점      │
│ "스테이크와 환상 궁합"       │
│                             │
│ 새로운 시도:                 │
│ Penfolds Bin 389  (추천)    │
│ 바로사 밸리 · 풀바디          │
└─────────────────────────────┘
```

### 4-4. 빈티지/성장 추천 (P2+)

```
"Cabernet Sauvignon을 좋아하시네요. 
 이번엔 같은 품종의 다른 산지를 시도해볼까요?"

로직:
1. 사용자가 가장 많이/높게 평가한 품종 식별
2. 해당 품종의 다른 산지/스타일 중 미경험 와인 추천
3. "경험 확장" 프레임 — 취향의 폭을 넓히는 방향
```

## 5. 콜드스타트 전략

### 5-1. 사용자 콜드스타트 (기록 0~5개)

PRD v2의 온보딩 "가본 곳 체크" 전략으로 **구조적 해결**:

```
Step 1: 취향 카드 (2지선다 이미지)
  → 초기 선호 프로필 생성
  → "새로운 도전 vs 익숙한 맛" → novelty_preference
  → "분위기 vs 맛" → atmosphere_weight vs taste_weight
  → "가성비 vs 프리미엄" → price_range

Step 2: 지역 선택
  → 해당 지역 대표 식당 리스트 제공

Step 3: 가본 곳 체크 + 간단 평가
  → 10~30개 기록 일괄 생성
  → 즉시 추천 가능 상태

콜드스타트 해소 기준:
  식당: 5개 이상 기록 (상황 태그 2종 이상)
  와인: 3개 이상 기록 (타입 1종 이상)
```

### 5-2. 아이템 콜드스타트 (새 식당/와인)

```
새 식당:
  - 외부 데이터(네이버/카카오/구글 평점)로 초기 정보 구성
  - Nyam 기록이 쌓이면 점진적으로 내부 데이터로 전환

새 와인:
  - 와인 DB의 메타데이터(품종/산지/스타일)로 DNA 추정
  - 실제 기록이 쌓이면 추정 → 실측으로 업데이트
```

### 5-3. 소셜 콜드스타트 (P2)

```
버블 0개 상태:
  - "친구 초대" CTA
  - 내 기록에서 자동 공유 추천: "이 기록을 공유하면 버블이 활성화돼요"
  - 공개 버블 탐색 (지역/관심사 기반)
```

## 6. Phase 2: 소셜 그래프 추천

### 6-1. 버블 기반 추천

```
알고리즘:
1. 내가 속한 버블의 멤버들이 높게 평가한 식당/와인 수집
2. 내가 아직 안 가본/안 마셔본 것만 필터링
3. 스코어링:
   bubble_rec_score = 
     AVG(member_ratings) * 0.4           -- 평균 점수
     + COUNT(unique_members) * 0.3       -- 여러 명이 평가한 곳 우선
     + trust_weight * 0.3                -- 경험치 높은 멤버 가중치

4. 경험치 가중치 (PRD v2 §8):
   "을지로 Lv.7"인 멤버의 을지로 식당 추천 > "을지로 Lv.1"인 멤버
```

### 6-2. 취향 유사도 추천 (P2+)

같은 식당에 대해 비슷한 만족도를 준 사용자 = 취향이 비슷한 사람. 전통적인 CF.

```sql
-- 나와 겹치는 식당에서 만족도 차이가 작은 사용자 찾기
SELECT other.user_id,
       AVG(ABS(my.satisfaction - other.satisfaction)) as avg_diff,
       COUNT(*) as overlap_count
FROM records my
JOIN records other ON my.target_id = other.target_id 
  AND my.target_type = other.target_type
  AND other.user_id != :current_user
WHERE my.user_id = :current_user
GROUP BY other.user_id
HAVING overlap_count >= 3       -- 최소 3개 겹침
ORDER BY avg_diff ASC           -- 차이 작은 순
LIMIT 20

-- 그 사람들이 높게 평가한 곳 중 나는 안 가본 곳
```

사분면 좌표를 CF에 끼워넣을 필요 없다. 만족도 패턴이 비슷하면 취향이 비슷한 거다. 단순하게.

### 6-3. 와인 소셜 추천

```
와인 동호회 버블 특화:
- 멤버들의 Wine DNA 평균 → "우리 버블은 산미 높은 화이트를 좋아하는 그룹"
- 그룹 취향과 다른 와인을 가져오면 → "이건 우리 취향과 좀 다른데, 새로운 시도!"
- 그룹 테이스팅 노트 비교 → 같은 와인에 대한 다른 관점
```

## 7. Phase 3: ML 기반 추천

### 7-1. Hybrid Recommender

```
최종 추천 점수 = 
  α × Content-Based Score       -- 사분면/DNA 기반
  + β × Collaborative Score     -- 유사 사용자 기반
  + γ × Context Score           -- 상황/위치/시간 기반
  + δ × Social Score            -- 버블/팔로우 기반

가중치는 데이터 충분도에 따라 동적:
  기록 < 10: α=0.6, β=0.0, γ=0.3, δ=0.1
  기록 10~50: α=0.4, β=0.2, γ=0.3, δ=0.1
  기록 50+: α=0.3, β=0.3, γ=0.2, δ=0.2
```

### 7-2. Context-Aware Factors

```
시간: 점심(11-14) vs 저녁(17-21) vs 야식(21+)
요일: 평일 vs 주말
날씨: 비 오는 날 → 따뜻한 국물 계열 ↑
계절: 여름 → 냉면/초밥 ↑, 겨울 → 찌개/전골 ↑
위치: 현재 위치 반경 내 우선
동행: 태그한 사람의 취향도 고려 (교집합)
```

### 7-3. 와인 ML 추천

```
Vivino 접근법 참고 (5개 이상 평가 시 작동):
  - Matrix Factorization (SVD) 기반
  - 사용자-와인 행렬의 잠재 요인 추출
  - Nyam 차별화: 잠재 요인 + 명시적 Wine DNA 7축 결합

데이터 규모 현실:
  - Vivino: 200M+ 평가 → CF 충분
  - Nyam 초기: 수천~수만 평가 → CF 불충분
  - 대안: Content-Based (Wine DNA) + 소셜 (버블) 우선
  - 10만+ 평가 쌓이면 CF 전환
```
