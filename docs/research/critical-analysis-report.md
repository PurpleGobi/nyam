# 냠(Nyam) 프로젝트 비판적 분석 보고서

> 작성일: 2026-03-16
> 목적: PRD 및 현재 설계의 현실적 문제점 진단 + 대안 제시

---

## Executive Summary

냠(Nyam)의 핵심 아이디어 — "프롬프트 브릿지로 사용자가 직접 AI 검증" — 는 리뷰 신뢰도 붕괴라는 진짜 문제를 겨냥하고 있으나, **해결 방식 자체에 구조적 결함**이 있다. PRD가 사용자 심리를 정교하게 분석했음에도 불구하고, 정작 핵심 UX에서 사용자 행동의 현실을 무시하고 있다.

**3가지 치명적 문제:**

1. **프롬프트 복사-붙여넣기 UX는 비현실적** — 8단계 마찰, 예상 완료율 5% 미만
2. **AI가 맛집 정보를 정확히 모른다** — 폐업 식당 추천, 존재하지 않는 식당 생성
3. **대형 플랫폼이 이미 같은 문제를 풀고 있다** — Google Ask Maps, 카카오 AI메이트, 네이버 AI 브리핑

이 보고서는 문제를 진단하고, 딥리서치 기반의 현실적 대안을 제시한다.

---

## Part 1: 현실적 문제점 진단

### 1. 프롬프트 브릿지 — "영리한 아이디어, 잘못된 UX"

PRD의 핵심 플로우를 다시 보자:

```
앱에서 "AI 검증하기" 탭 → 프롬프트 미리보기 → 복사하기
→ ChatGPT 앱 열기 → 붙여넣기 → 전송 → 응답 대기
→ 결과 읽기 → 다시 Nyam으로 돌아오기
```

**문제의 규모:**

| 단계 | 이탈 예상 | 남은 사용자 |
|------|----------|------------|
| 1. "AI 검증하기" 탭 | - | 1,000명 |
| 2. 프롬프트 미리보기 | 30% | 700명 |
| 3. 복사 | 20% | 560명 |
| 4. ChatGPT 앱 전환 | 40% | 336명 |
| 5. 붙여넣기+전송 | 20% | 269명 |
| 6. 응답 대기(5~30초) | 15% | 228명 |
| 7. 결과 해석 | 10% | 206명 |
| 8. Nyam으로 복귀+결과 공유 | 70% | **62명** |

**1,000명 중 62명만 전체 루프를 완료한다.** 검증 완료율 목표 50%는 환상이다.

비교하면:
- Amazon 1-Click → 전환율 18% 향상. 1단계 줄이는 것만으로.
- Shazam → "1탭, 5초, 결과." 이게 맛집 앱의 기준이어야 한다.
- 카카오 AI메이트 → "대화창에 타이핑하면 끝." 앱 전환 없음.

**PRD 자체도 이 리스크를 인지하고 있다** (섹션 12: "사용자가 프롬프트 실행을 귀찮아함 — 심각도: 높음"). 하지만 대응이 "1탭 UX, 가치 체감 온보딩"으로는 근본 문제를 해결하지 못한다. 문제는 1탭이 아니라 **8탭**이라는 것이다.

> **판정: 프롬프트 브릿지는 기술적으로 영리하지만, UX적으로 실패할 구조다.**

---

### 2. AI 맛집 추천의 정확도 문제 — "인기도 ≠ 맛"

PRD는 "AI의 발전에 자동으로 편승"한다고 했지만, 현재 AI의 맛집 추천 능력은 **심각하게 제한적**이다.

**Food & Wine (2026.01) 테스트 결과:**
- ChatGPT는 "Popular does not mean good"의 함정에 빠진다
- 이미 폐업한 식당을 자신있게 추천
- 존재하지 않는 가게를 만들어내는 hallucination 발생
- 특정 동네의 맥락(분위기, 최근 변화)을 이해하지 못함

**구조적 한계:**

| 문제 | 원인 | 영향 |
|------|------|------|
| 폐업 식당 추천 | 학습 데이터 시점 한계 | 헛걸음, 신뢰 붕괴 |
| 가짜 식당 생성 | Hallucination | 앱 전체 신뢰도 파괴 |
| 인기도 편향 | 리뷰 볼륨 = 품질로 오인 | "진짜 맛집" 발견 실패 |
| 지역 맥락 무시 | 실시간 로컬 데이터 부재 | 부정확한 추천 |
| 주관적 취향 미반영 | 개인 데이터 없음 | 획일적 추천 반복 |

**PRD의 "IKEA 효과"가 오히려 역효과:**
PRD는 "사용자가 직접 AI 검증하면 더 신뢰한다"고 했다. 하지만 AI가 틀린 정보를 줬을 때, 사용자는 **"내가 직접 확인했는데 틀렸다"** → 앱에 대한 분노가 더 커진다. 노력을 투입한 만큼 실망도 비례한다.

> **판정: AI에게 맛집 판단을 위임하는 구조 자체가 현재 AI 기술로는 신뢰할 수 없다.**

---

### 3. 사용자 데이터 확보 — "자발적 참여의 환상"

PRD의 데이터 플라이휠 설계:

```
프롬프트 복사 → AI 검증 → 결과를 Nyam에 공유 → 데이터 축적
```

**왜 작동하지 않는가:**

**90-9-1 법칙**: 온라인 커뮤니티에서 콘텐츠를 만드는 사용자는 1%, 가볍게 기여하는 사용자 9%, 90%는 소비만 한다 (Nielsen Norman Group). 이 구조에서 "AI 결과를 다시 앱에 공유"는 **가장 높은 수준의 능동적 기여**에 해당한다.

**PRD의 참여 퍼널 현실:**

| 단계 | PRD 예상 | 현실 예상 |
|------|---------|----------|
| 프롬프트 복사 | 높음 | 중간 (호기심 1~2회) |
| AI에서 결과 확인 | 높음 | 낮음 (앱 전환 마찰) |
| 결과를 Nyam에 공유 | 중간 | 매우 낮음 (<5%) |
| 4축 평가 입력 | 높음(P0) | 극히 낮음 |
| 반복 검증 | 중간 | 거의 없음 |

**Beli가 증명한 것**: 90-9-1을 깬 앱은 **"남을 위한 리뷰"가 아니라 "나를 위한 기록"**으로 리프레이밍했다. Beli 사용자는 "리뷰를 쓴다"고 생각하지 않는다. "내 맛집 랭킹을 정리한다"고 생각한다. 그 부산물이 커뮤니티 데이터가 된다.

냠의 현재 설계는 정반대다: "AI로 검증하고, 결과를 커뮤니티에 공유하세요." 이것은 **기여를 명시적으로 요청하는 구조**이고, 대부분의 사용자는 이를 무시한다.

> **판정: 데이터 확보 전략이 "자발적 참여"에 의존하는 한, 냉닭 문제(Cold Start)를 돌파할 수 없다.**

### 4. 경쟁 환경 — "3일 전에 세상이 바뀌었다"

2026년 3월 13일, Google이 **"Ask Maps"**를 발표했다.

| 기능 | Nyam | Google Ask Maps |
|------|------|-----------------|
| 자연어 맛집 추천 | 프롬프트 복사 → 외부 AI | 지도 앱 내 대화형 추천 |
| 데이터 소스 | 사용자 검증(미확보) | 300만+ 장소, 5억 기여자 리뷰 |
| 개인화 | 프롬프트 변수 수동 입력 | 저장 장소, 검색 이력, 선호도 자동 반영 |
| 정보 정확도 | AI 모델에 의존(hallucination) | 실시간 Google 데이터 + Gemini |
| UX 마찰 | 8단계 | 1단계 (질문 입력) |

**한국 시장:**
- 카카오맵 AI메이트 로컬 (2025.07~): 이미 서비스 중. "애견 동반 브런치집" 같은 자연어 추천.
- 네이버 AI 브리핑: 리뷰 요약으로 체류시간/클릭률 두 자릿수 증가 확인.

**냠이 직면한 경쟁 현실:**
- 네이버/카카오는 이미 **수억 건의 리뷰 데이터**를 보유
- 두 회사 모두 **자체 AI 모델(HyperCLOVA X, Kanana-2)**을 지도에 통합 완료
- 사용자는 **이미 사용하는 앱(카카오맵, 네이버지도) 안에서** AI 맛집 추천을 받을 수 있음
- 별도의 앱을 설치할 이유가 없다

> **판정: 대형 플랫폼이 이미 "AI + 맛집"을 자체 해결하고 있어, 독립 맛집 앱의 존재 이유가 위협받고 있다.**

### 5. 게이미피케이션과 리텐션 — "배지로는 습관이 안 만들어진다"

PRD의 게이미피케이션 설계는 정교하다 (배지, 지역 마스터, 스트릭, 리더보드). 하지만:

**근본적 문제: 맛집 검색은 상황적(episodic) 니즈다.**
- 사람들은 새 식당을 매일 찾지 않는다 (보통 주 1~2회)
- "순수 맛집 추천 앱으로 DAU 습관을 만든 사례는 거의 없다" (리서치 결론)
- 배달 앱(배민/쿠팡이츠)이 DAU를 가지는 건 "주문"이라는 거래 행위 때문이지 "추천" 때문이 아니다

**BeReal의 교훈:**
- 2022년 폭발적 성장 → 2024년 23M으로 급락
- "매번 기록하세요" 같은 의무적 프롬프트는 참신함이 사라지면 귀찮아짐
- "진정성 강요"는 오히려 이탈을 부름

**냠의 "검증 스트릭"은 BeReal과 같은 위험을 갖고 있다.** "연속 검증 일수"는 처음엔 재미있지만, 맛집 검색이 없는 날에도 억지로 검증하게 만들면 피로감만 쌓인다.

> **판정: 게이미피케이션은 기존에 가치 있는 행동을 강화하는 도구이지, 무의미한 행동에 의미를 부여하는 마법이 아니다.**

---

### 6. 수익 모델 — "광고 없는 것은 좋지만, 대안이 약하다"

| 수익 모델 | 현실성 평가 |
|----------|------------|
| 프리미엄 프롬프트 팩 | 기본 프롬프트가 무료인데 유료 프롬프트에 돈 낼 사람? PromptBase도 니치에 머무름 |
| 식당 검증 리포트 | 검증 데이터가 충분히 쌓이기 전까지 가치 없음. 닭이 먼저냐 달걀이 먼저냐 |
| 지역 가이드북 | 검증 데이터 필요 + PDF 가이드 시장 자체가 축소 중 |
| B2B API | 데이터가 충분해야 의미 있음. 최소 수년 뒤 이야기 |

**네이버/카카오와의 차이**: 이들은 광고 수익으로 무료 서비스를 운영하면서 데이터를 축적한다. 냠은 "광고 없음"을 차별점으로 내세우지만, 그 대가로 **수익 엔진이 매우 약하다.**

> **판정: 현재 수익 모델은 "데이터가 충분히 쌓인 미래"를 전제하는데, 그 미래에 도달할 자원이 불충분하다.**

## Part 2: 딥리서치 기반 대안 제시

### 대안 1: "프롬프트 브릿지" → "앱 내 AI 완결"

**문제**: 8단계 마찰, 5% 완료율
**해결**: 앱 안에서 AI 결과가 바로 나타나야 한다

**구체적 방안:**

```
현재: 프롬프트 복사 → ChatGPT 열기 → 붙여넣기 → 결과 → 돌아오기 (8단계)
대안: "추천받기" 탭 → 결과 표시 (1단계)
```

- **RAG (Retrieval-Augmented Generation)** 적용: Kakao Local API / Google Places API의 실시간 데이터를 AI에 주입 → hallucination 대폭 감소
- 비용 우려: 경량 모델(GPT-4o-mini, Claude Haiku) 사용 시 건당 $0.001 미만
- 프롬프트 브릿지는 "오프라인 모드" 또는 "고급 기능"으로 남겨두되, 기본 경험은 앱 내 완결

**Shazam 원칙 적용:**
- 앱 열면 **"지금 여기서 뭐 먹을까?"**가 바로 보임
- 1탭이면 AI가 현위치 + 시간대 + 과거 취향 기반으로 3곳 추천
- 결과 화면에서 바로 지도/전화/예약 연결

---

### 대안 2: "리뷰 공유" → "나만의 맛집 리스트" (Beli 모델)

**문제**: 자발적 데이터 기여율 극히 낮음
**해결**: "남을 위한 리뷰"가 아닌 "나를 위한 기록"으로 리프레이밍

**Beli의 성공 공식:**
- 2024년 리뷰 30M건 (Yelp 21M건 추월)
- 사용자 80%가 35세 미만
- 핵심: **"내가 가본 식당을 랭킹으로 정리한다"** → 이게 곧 데이터

**냠 적용:**

| 현재 설계 | 대안 |
|----------|------|
| "AI 검증 결과를 커뮤니티에 공유하세요" | "가본 곳에 별표 하나만 찍으세요" |
| 4축 슬라이더 평가 (맛/가성비/서비스/분위기) | 좋았다/괜찮다/별로 (3단계, 1탭) |
| 한 줄 코멘트 작성 | 선택형 태그 (#혼밥맛집 #가성비 #데이트) |
| 검증 결과 텍스트 붙여넣기 | 사진 한 장 찍기 (자동 태깅) |

**데이터 수집 3계층:**
1. **Passive (자동)**: 위치 기반 방문 감지, 체류 시간, 검색 패턴
2. **Micro-active (1탭)**: 좋아요/저장/별표
3. **Full-active (선택)**: 사진, 코멘트, 상세 평가 — 하지 않아도 됨

**핵심 전환:**
```
"검증해주세요" (이타적 행동 요구) 
→ "당신의 맛집 지도를 만드세요" (이기적 동기 활용)
```

---

### 대안 3: "검증 커뮤니티" → "친구 기반 추천" (Social Graph)

**문제**: 익명 커뮤니티의 검증 데이터는 결국 네이버 리뷰와 같은 신뢰 문제를 겪는다
**해결**: 친구/지인의 추천이 가장 강력한 신뢰 신호

**연구 근거:**
- Gen Z의 맛집 발견은 **"또래 추천"** 중심 (TikTok, Instagram)
- "Who matters more than What" — 누가 추천하느냐가 뭘 추천하느냐보다 중요
- Beli 성공의 핵심도 "전체 평점"이 아닌 "내 친구가 좋아하는 곳"

**구체적 방안:**
- 카카오톡/인스타 소셜 그래프 연동
- "내 친구 중 3명이 좋아한 곳" → 이게 "47명이 AI로 검증"보다 훨씬 강력한 신호
- 친구가 새 맛집을 저장하면 피드에 알림
- "오늘 점심 뭐먹지?" → 친구들이 최근 간 곳 중심 추천

---

### 대안 4: Cold Start 돌파 — "Single-Player Mode First"

**문제**: 사용자 없으면 데이터 없고, 데이터 없으면 사용자 없다
**해결**: Andrew Chen의 "Come for the Tool, Stay for the Network"

**Phase 0 (커뮤니티 데이터 없이도 유용한 도구):**
- AI 맛집 추천 자체가 도구적 가치 (RAG 기반, 실시간 데이터)
- 개인 맛집 리스트 관리 (Notion 맛집 버전)
- 맛집 비교 도구 (2~3곳 나란히 비교)

**Phase 1 (초기 데이터 시딩):**
- Kakao Local API + Google Places API로 기본 식당 데이터 확보
- 운영팀이 핵심 지역(강남, 홍대, 이태원) 100곳을 직접 검증
- 10명의 "파운딩 푸디" 모집 → 고품질 초기 콘텐츠

**Phase 2 (네트워크 효과):**
- 친구 초대 → "친구의 맛집 리스트 볼 수 있음"이 초대 동기
- 지역 밀도가 임계점 넘으면 자연 성장 시작

### 대안 5: 포지셔닝 재정의

현재 PRD는 냠을 **"맛집 검증 플랫폼"**으로 정의한다. 하지만 이 포지션은 네이버/카카오/Google과 정면 경쟁이다.

**대안 포지셔닝 옵션:**

| 옵션 | 설명 | 차별점 | 경쟁 강도 |
|------|------|--------|----------|
| A. 소셜 맛집 리스트 | "친구들의 맛집 지도" (Beli 한국판) | 소셜 그래프 | 중간 |
| B. 상황별 추천 특화 | "오늘 이 상황에 딱 맞는 1곳" | 컨텍스트 이해 | 낮음 |
| C. 그룹 의사결정 도구 | "팀 점심/모임 장소 함께 고르기" | 그룹 UX | 낮음 |
| D. 개인 맛집 DB + AI | "나만의 맛집 기록 + AI 비서" | 개인화 깊이 | 낮음 |

**추천: B + D 조합**
- "오늘 이 순간에 딱 맞는 추천" (상황별 AI)
- "내가 가본 곳, 가고 싶은 곳을 기록하면 추천이 점점 정확해진다" (개인화 플라이휠)
- 네이버/카카오가 잘 못하는 영역: **깊은 개인화 + 상황 맥락 이해**

## Part 4: 최종 판단

### 냠이 살아남으려면

1. **"프롬프트 브릿지"에 미련을 버려라.** 영리한 아이디어이지만 사용자가 쓰지 않을 UX다. 앱 안에서 끝나야 한다.

2. **"검증"이라는 프레임을 바꿔라.** 사용자는 검증자가 되고 싶지 않다. **"내 맛집 리스트를 정리하는 사람"**이 되고 싶다. 그 부산물이 커뮤니티 데이터가 되어야 한다.

3. **네이버/카카오와 다른 싸움을 해라.** 검색 데이터 볼륨으로는 절대 이길 수 없다. 이길 수 있는 곳: **친구 기반 추천, 상황 맥락 이해, 깊은 개인화.**

4. **"식사 시간의 1초"를 잡아라.** 앱을 열면 바로 "오늘 여기서 이거 어때?"가 나와야 한다. 30초가 걸리면 이미 졌다.

5. **데이터 플라이휠을 "사용자 몰래" 돌려라.** 앱을 쓰는 것 자체가 데이터를 만드는 구조. "공유해주세요"라고 부탁하는 순간 진다.

### 냠의 진짜 기회

문제 인식은 정확하다. 맛집 앱 시장은 진짜로 망가져 있다. 리뷰는 못 믿겠고, 별점은 의미 없고, 광고가 추천을 오염시킨다. 이 분노의 에너지를 **올바른 해결책**으로 연결하면 기회는 분명히 있다.

핵심은 "AI가 대신 검증해준다"가 아니라, **"내가 신뢰하는 사람들의 취향이 나를 안내한다"**는 것이다. AI는 이 과정을 매끄럽게 만드는 도구일 뿐, 주인공이 아니다.
