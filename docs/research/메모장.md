데이터 관리를 위한 백앤드 스키마 계획이 development_docs/systems/DATA_MODEL.md  여기에 있긴 한데... 초기에 작업하고 목업이 계속 쉊ㅇ되면서 follow up 이 안됐어,목업을 기준으로 DB schema 가 적절한지 꼼꼼하게 수정할 필요가 있어. 목업 데이터 가 너무 방대해서 한번에 정합성을 확일할수느 없고, 목업파일 하나씩 보면서 수정해야 할 것 같아. 부족한게 있거나 틀린게 있으면 수정을 하면서 주석을 달아놓고, 순차자적으로 다음 목업 파일과 정합성을 보명서 또 수정하는 수밖에 없을 것 같아. 주석을 잘 봐. 이전 수정을 다시 수정해야 할수도 있으니 필요한 경우 이전 수정을 재수정하면서 이전 목과의 정합성도 유지할 수 있도록 해. 중요한건 필요한 모든 DB schema 를 구축하되 중복없이 효율적이면서도 확장성도 적절히 고려한 설계여야 해. 중복 테이블 같은건 단연히 있어서는 안될 일이고. 속성값도 신경 많이 써야 하고. 전체를 관통하면서 직관적이고 중복걱정없는 체이블 및 속성  네이밍으로 관리도 편해야 해. 모든 기능들을 꼼꼼하게 봐야해. 필터엔진 특히 신경쓰고, 탭별로 데이터 뭐가 필요한지도 보고, 팝업이나 슬라이드 시트, 드로어 같은게 있으면 거기 필요한 데이터가 뭔지도 파악해야 해. 모든 검토는 정렬로 진행해.

우선 다음 문서와의 정합성을 확인하자. 완료되면 한번 더 모든 기능 빠뜨리지 않고 검토했는지 정말 꼼꼼하게 확인해봐.

다음 목업 파일이 가장 최신의 상태의 구현하고자 하는 모습이야. 모든 경우의 수를 담지는 못했지만 디자인과 레이아웃, 작동원리 등을 파악할 수 있을 거야. 이 목업 파일과 코드 구현 시 각 페이지의 UI 구조, 컴포넌트, 데이터 흐름, 상태 관리 등을 정의하는 SSOT(Single Source of Truth) 문서인 그 다음 문서와의 정합성을 검토해 필요하면 md 문서를 수정해. 완전한 제품구축을 목표로 작성하도록 해.

  development_docs/prototype/00_onboarding.html \                                                                                               
  development_docs/pages/00_ONBOARDING.md        

과거 기록 의존하지 말고 새롭게 정합성 일관성 완정성 전면 검토 다시 해보고 md 문서 꼼꼼하게 수정해. 억지로 수정하지는 말고, 꼭 필요하다 판단되면 수정해. (임시방편이나 다음에 검토하자는 식으로 처리하면  안된다) /pages 폴더내 md문서만 수정하도록 해.
