# BUBBLE — 버블 시스템

> depends_on: DATA_MODEL, AUTH, XP_SYSTEM, DESIGN_SYSTEM
> affects: HOME, PROFILE, RESTAURANT_DETAIL, WINE_DETAIL
> route: /bubbles, /bubbles/create, /bubbles/[id], /bubbles/[id]/settings, /bubbles/invite/[code], /users/[id]?bubble=[bubbleId]
> prototype: `prototype/04_bubbles.html`, `prototype/04_bubbles_detail.html`, `prototype/04_bubbler_profile.html`

## 1. 핵심 원칙

- **버블 = 큐레이션 리스트** — 개인 또는 공동으로 관리하는 식당/와인 리스트. 동호회가 아니라 Spotify 플레이리스트에 가까운 개념
- **기록이 먼저, 소셜은 그 다음** — 소셜 없이도 유용
- **신뢰가 네트워크의 핵심** — 기록 수 + 레벨 기반 가입 조건으로 데이터 오염 방지
- **큐레이션은 자동+수동** — 필터 조건에 매칭되는 기록/찜은 자동 추가, 직접 담기도 가능
- **리스트 단위로 팔로우** — 사람 전체가 아닌 특정 리스트만 선택적 구독 가능
- **피드는 시간순** — 알고리즘 피드 없음 (신뢰 보호)
- **버블은 가변적** — 멤버 탈퇴/제거로 데이터 오염 정화 가능
- **기여도에 비례한 접근** — 더 많이 기여할수록 더 깊은 정보에 접근 가능

### 핵심 비전

nyam은 개인 기록용 앱이지만, **버블(리스트)을 통한 큐레이션과 신뢰할 만한 맛집/와인 추천이 핵심 가치**다.
홈 화면 추천에도 버블 내 데이터가 내부적으로 반영된다 (→ RECOMMENDATION.md 참조).

버블은 **뉴스 피드가 아니다.** 실시간 타임라인을 소비하는 소셜 미디어가 아니라, 내가 어디 갈지 고민할 때 꺼내보는 **목적별 식당/와인 큐레이션 리스트**다. 혼자 관리하는 개인 리스트(1인 버블)일 수도, 여럿이 함께 채워가는 공동 리스트일 수도 있다.

## 2. 데이터 접근 계층 모델

### 2-1. 설계 원칙

> **기여도에 비례한 접근 권한** — 더 많이 기여할수록 더 깊은 정보에 접근 가능

### 2-2. 버블에서의 5단계 접근

| 단계 | 역할 | 조건 | 보이는 것 | 할 수 있는 것 |
|------|------|------|----------|-------------|
| **탐색** | 비팔로워 | public 버블이면 누구나 | 버블 이름, 설명, 멤버 수, 통계 요약 | 검색, 열람 |
| **팔로우** | follower | XP 불필요, 누구나 가능 | + 식당/와인 **이름, 평균 점수, 지역** | 검색, 피드 목록 열람 |
| **멤버** | member | 가입 승인 필요 | + **개별 리뷰, 사진, 팁, 누가 몇 점인지** | + 기록 공유, 댓글, 리액션 |
| **관리자** | admin | 오너 지정 | 전체 | + 멤버 관리, 초대 |
| **오너** | owner | 버블 생성자 | 전체 | + 설정 변경, 역할 변경, 삭제 |

**팔로워 = 맛보기.** 식당 이름과 평균 점수는 알 수 있지만, "왜 좋은지, 뭘 먹어야 하는지"는 멤버만 볼 수 있다.

**복제 방어:** 이름+점수만으로는 가치가 제한적. Spotify 플레이리스트 곡 제목만 복사하는 것과 같음 — 진짜 가치는 지속적인 큐레이션과 상세 맥락.

### 2-3. 개인 팔로우에서의 3단계 접근

| 단계 | 관계 | 조건 | 보이는 것 |
|------|------|------|----------|
| **비팔로우** | 없음 | — | 공개 프로필만 (레벨, 기록 수, 팔로워/팔로잉 수) |
| **일방 팔로우** | 내가 팔로우 | 상대가 public 프로필 | + 식당/와인 **이름, 점수, 지역** (맛보기) |
| **맞팔로우** | 상호 팔로우 | 양쪽 모두 팔로우 | + **풀 액세스** (리뷰, 사진, 팁, 상세 기록) |

**핵심: 맞팔로우 = 기브앤테이크의 구현**

- "이 사람 기록 제대로 보고 싶다" → 팔로우
- 상대가 "이 사람도 볼만하네" → 맞팔
- 맞팔 조건 = 내 기록의 질이 좋아야 상대가 팔로우할 동기가 생김
- → 기록 품질 향상 동기 자연스럽게 발생

### 2-4. 버블 멤버 → 개인 팔로우 연결

```
버블 가입
  → 멤버 목록 열람 가능
  → 특정 멤버의 기록이 좋다 → 개인 팔로우
  → 상대도 나를 팔로우 → 맞팔 = 버블 밖에서도 풀 액세스
  → 버블을 넘어선 개인 네트워크 형성
```

이를 통해 **개인 팔로우의 차별점**이 명확해진다:
- **버블 팔로우** = 특정 주제/목적의 리스트를 구독 (여러 사람의 큐레이션이 하나의 리스트로 모임)
- **개인 팔로우** = 특정 개인의 전체 취향을 깊게 추적 (리스트에서 발견한 취향 맞는 사람)
- **버블이 개인 팔로우의 발견 채널** 역할

## 3. 버블 생성

- **누구나 자유롭게, 무한정 생성 가능** (단톡방처럼)
- 1인 버블 가능 — 외부 공유용 큐레이션 채널로 활용
- 여러 명이 아니어도 됨

```
1. [+] 버블 만들기 (FAB)
2. 아이콘 선택 (50+ 루씨드 아이콘 from BUBBLE_ICON_MAP)
   - 6개 카테고리: 음식·음료 / 여행·장소 / 사람·소셜 / 자연·날씨 / 활동·엔터 / 오브젝트
   - 배경 색상 12종 선택 (기본 #F97316)
   - 사진 업로드도 가능
3. 이름 (20자) + 설명 (100자, 선택)
4. 공개 설정: 비공개 (lock) | 공개 (globe)
   - 비공개: "초대받은 사람만 접근"
   - 공개: "누구나 검색 팔로우 가능"
5. 초대 링크 만료: 1일 | 7일 | 30일(기본) | 무제한
6. (공개일 때) 가입 정책 4종 (아이콘+설명 카드 형태):
   - 팔로우만 (eye) — "가입 안 받음. 팔로워는 이름+점수만 열람"
   - 승인 가입 (shield-check, 기본) — "가입 신청 → 프로필 보고 승인/거절"
   - 자동 승인 (zap) — "검증 기록 N개 이상이면 자동 가입"
   - 자유 가입 (door-open) — "누구나 바로 가입 가능"
7. (auto_approve일 때) 가입 조건: 최소 기록 수 / 최소 레벨 / 최대 인원
8. 내 공유/프라이버시 설정 (접힘 패널):
   - 자동 공유 규칙 (ShareRuleEditor): 모든 기록 / 필터 기반
   - 정보 공개 범위 (VisibilityOverride 7키): 전체 공개 / 일부만 공개
     - score(점수), comment(한줄평), photos(사진), level(레벨), quadrant(사분면), bubbles(소속 버블), price(가격)
9. 완료 → "버블 만들기" CTA → 버블 상세
```

> **생성 시 미포함, 설정에서 변경 가능한 항목:**
> - 버블 포커스/정체성 (focus_type) → §4-6
> - 공개 수위 (content_visibility) → §4-5
> - 권한 설정 (댓글/외부공유) → §4-4
> - 검색 노출 설정 (isSearchable, searchKeywords) → §12-6

## 4. 버블 속성

### 4-1. 공개 여부 (visibility)

| 값 | 검색 | 사용 예 |
|------|------|--------|
| `private` | 불가 | 직장 맛집, 가족 (초대받은 사람만 접근) |
| `public` | 가능 | 누구나 검색/팔로우 가능 |

- 기존 3단계(`private|members|public`)에서 2단계로 단순화
- "누가 볼 수 있는가"와 "누가 가입할 수 있는가"를 분리 (→ 4-2 join_policy)

### 4-2. 가입 정책 (join_policy)

| 값 | UI 레이블 | 아이콘 | 설명 | 적용 시점 |
|------|----------|--------|------|----------|
| `invite_only` | — | — | 초대받은 사람만 가입 | private 버블의 암묵적 기본값 (UI에 노출 안 됨) |
| `closed` | 팔로우만 | eye | 가입 안 받음. 팔로워는 이름+점수만 열람 | public 생성 시 선택 |
| `manual_approve` | 승인 가입 | shield-check | 가입 신청 → 프로필 보고 승인/거절 | public 생성 시 기본값 |
| `auto_approve` | 자동 승인 | zap | 검증 기록 N개 이상이면 자동 가입 | public 생성 시 선택 |
| `open` | 자유 가입 | door-open | 누구나 바로 가입 가능 | public 생성 시 선택 |

- private 버블은 join_policy 선택 UI가 표시되지 않고 `invite_only`로 자동 설정
- public 버블은 4가지 옵션(closed/manual_approve/auto_approve/open) 중 선택

#### 가입 조건 설정 (버블 설정 페이지에서 관리)

| 설정 | 설명 | 기본값 | UI 형태 |
|------|------|--------|--------|
| `min_records` | 최소 기록 수 | 0 | 컴팩트 숫자 필드 |
| `min_level` | 최소 레벨 | 0 | 컴팩트 숫자 필드 |
| `max_members` | 최대 인원 | null(무제한) | 컴팩트 숫자 필드 |

- 발견 팝업에서 체크마크 리스트로 표시: "관리자 승인 필요", "최소 기록 N개 이상", "최소 Lv.N 이상", "현재 N/N명 (여유 N자리)"
- `joinPolicy` 변경은 JoinPolicySelector 컴포넌트로 직접 선택

#### auto_approve 자동 승인 기준

`auto_approve` 정책일 때 `min_records` + `min_level` 조건 **모두 충족**(AND) 시 자동 승인.
`manual_approve`일 때는 조건 미충족이어도 신청 자체는 가능 (오너가 프로필 보고 판단).
`max_members` 도달 시 모든 정책에서 가입 불가.

### 4-3. 양방향 프리뷰 (가입 시)

기존 "XP 레벨 5" 같은 숫자만으로 판단하던 것을 **기록의 질적 프로필**로 대체.

**가입 희망자가 보는 것 (가입 전 미리보기):**

| 정보 | 예시 |
|------|------|
| 멤버 수 | 12명 |
| 총 기록 | 234개 |
| 주요 지역 | 을지로, 광화문 |
| 평균 점수 | 87 |
| 나와 겹치는 곳 | 5개 |
| 취향 유사도 | 78% |

→ "이 버블에 들어갈 가치가 있겠다"

**버블 오너가 보는 것 (가입 신청 시):**

| 정보 | 예시 |
|------|------|
| 검증 기록 (EXIF) | 28개 |
| 단순 등록 | 5개 |
| 주요 지역 | 을지로, 성수 |
| 취향 유사도 | 78% |
| 최근 활동 | 3일 전 |

→ "이 사람을 받아도 되겠다" or "거절"

### 4-4. Owner 권한 설정

| 설정 | 설명 | 기본값 |
|------|------|--------|
| `allow_comments` | 댓글 허용 여부 | true |
| `allow_external_share` | 외부 공유 허용 여부 | false |

### 4-5. 공개 수위 (content_visibility)

**버블에 공유된 기록이 비멤버에게 어떻게 보이는가** (→ SETTINGS.md §4-2 상세 매트릭스)

| 값 | 버블 멤버에게 | 비멤버에게 (상세 페이지 L9) |
|------|------------|-------------------------|
| `rating_only` | 점수 + 한줄평 + 사진 + 메뉴 전체 | 점수만 (아바타 + Lv + 만족도 숫자) |
| `rating_and_comment` | 점수 + 한줄평 + 사진 + 메뉴 전체 | 점수 + 한줄평 |

- `records.comment` 한줄평이 공유 텍스트 — 별도 "버블 전용 코멘트" 필드 없음
- 한줄평 없이 공유하면 점수만 표시
- 멤버에게는 공유 시점에 노출 동의한 것으로 간주

### 4-6. 버블 포커스 (focus_type)

| 값 | 의미 |
|------|------|
| `all` | 식당 + 와인 모두 |
| `restaurant` | 식당 중심 |
| `wine` | 와인 중심 |

- 정체성을 정한다고 **제한이 생기지는 않음** — 노출되는 라벨일 뿐
- 언제든 변경 가능

## 5. 버블 사용 시나리오 (중요도 순)

1. **나만의 목적별 리스트 관리** — 개인 큐레이션 ("2026 미슐랭", "데이트 코스", "와인바 투어")
2. **지인 간 맛집/와인 리스트 공동 관리** — 직장 동료, 친구, 가족
3. **다른 사람의 리스트 선택적 구독** — 사람 전체가 아닌 관심 리스트만 팔로우
4. **개인 큐레이션 채널 운영** — 1인 버블로 팔로워 대상 추천 리스트 제공
5. **콜드스타트 유저의 생태계 진입** — 팔로우로 즉시 가치 체감

### 시나리오 A: 개인 목적별 리스트

```
"2026 미슐랭" — 1인, 비공개
  visibility: private
  curation: 수동

  → 미슐랭 식당을 하나씩 찜하면서 리스트에 추가
  → 방문할 때마다 기록 → 리스트에서 진행 상황 한눈에
  → 가봤는지/안 가봤는지, 점수는 몇 점이었는지 리스트로 관리
  → 순수 개인용, 공유 불필요
```

### 시나리오 B: 자동 필터 리스트

```
"광화문 90점 이상" — 1인, 공개
  visibility: public
  join_policy: closed
  curation: 자동 (지역=광화문, 만족도≥90)

  → 필터 조건에 맞는 기록/찜이 자동으로 리스트에 추가/제거
  → 팔로워들이 이름+점수 리스트로 참고
  → 기록이 쌓일수록 리스트가 자동으로 풍성해짐
  → 관리 부담: 없음 (자동 큐레이션)
```

### 시나리오 C: 지인 공동 리스트

```
"직장 점심" — 5명
  visibility: private
  join_policy: invite_only
  curation: 자동+수동 혼합

  → 초대받은 멤버만 접근
  → 각자 기록하면 자동 공유 규칙에 의해 리스트에 추가
  → 수동으로 "여기 가보자" 찜만 해서 리스트에 담기도 가능
  → 댓글/리액션으로 소통
  → 관리 부담: 없음 (소규모 + 아는 사람)
```

### 시나리오 D: 오픈 공동 큐레이션

```
"강남 와인바" — 30명
  visibility: public
  join_policy: manual_approve

  → 누구나 팔로우 (이름+점수 열람)
  → 가입 신청 → 오너가 프로필 보고 승인
  → 멤버: 풀 액세스 + 리스트에 기록/찜 추가 + 댓글/리액션
  → 멤버 목록에서 개인 팔로우 → 맞팔로 확장
  → 관리 부담: 있음 (가입 승인, 오염원 관리). 대신 리텐션 강화
```

### 시나리오 E: 콜드스타트 유저 여정

```
가입 직후 (기록 0, XP 0)

  1. 온보딩에서 리스트 1개 생성 (내 맛집)
  2. "리스트 탐색"으로 공개 리스트 둘러보기
  3. 취향 유사도 높은 리스트 3~5개 팔로우 (무료, XP 불필요)
  4. 홈에서 팔로잉 리스트의 이름+점수 즉시 활용

  8. 멤버 목록에서 취향 맞는 개인 팔로우
  9. 맞팔 성사 → 리스트 밖에서도 풀 액세스

### 5-1. 취향 유사도 매칭 (public 버블 탐색 시)

공개 버블을 탐색할 때, 내 기록과 버블 멤버들의 기록을 비교하여 **취향 유사도**를 표시한다.

#### 원리 (설계 의도)

- 내가 기록한 식당/와인과 **같은 대상을 리뷰한 버블**이 비교 대상
- 겹치는 대상이 많고 점수 오차가 작을수록 유사도 높음

#### 현재 구현

**버블 상세 리스트** (`aggregateByTarget` 내):
```
동일 targetId에 내 기록이 있을 때:
tasteMatchPct = max(0, 100 - abs(mySatisfaction - avgSatisfaction))
내 기록이 없으면 null (미표시)
```
내 점수와 버블 평균의 차이를 100에서 뺀 값. 점수가 가까울수록 높음.

**멤버 그리드**: `bubble_members.taste_match_pct` 컬럼 값을 그대로 사용.

> `bubble_members.taste_match_pct`는 **뷰어 상대적 값**이므로 테이블에 직접 저장하면 정합성 문제가 있음. 현재 컬럼은 유지하되 정확한 갱신 로직은 미완성.

#### 표시

```
┌─────────────────────────────────┐
│ 을지로 맛탐정 클럽               │
│ 멤버 23명 · 기록 156개           │
│ 취향 유사도 78%                  │
│ [팔로우] [가입 신청]              │
└─────────────────────────────────┘
```

- public 버블에서만 표시 (private은 검색 불가이므로 해당 없음)

### 5-2. 버블 검색 + 필터

#### 검색 UI
- **버블 목록 페이지**: 스티키 탭 우측에 검색(Search) 아이콘 → 클릭 시 인라인 텍스트 입력으로 전환 ("버블 이름·지역으로 검색"). 필터는 ConditionFilterBar로 별도 관리.
- **탐색 시트 (BubbleDiscoverSheet)**: 헤더에 닫기(X) 버튼만. 탭(추천/근처/인기/새로운)으로 필터링. 별도 검색/필터 UI 미구현.

#### 버블 목록 필터 (구현 완료 — §11-2 참조)

BUBBLE_FILTER_ATTRIBUTES: role, focus_type, area, join_policy, member_count, activity.

#### 탐색 시트 필터 (미구현)

**설계 스펙 — 버블 탐색 필터 (5종)**:

## 6. 핵심 구동 원칙

### 6-1. 경험치 기반 데이터 신뢰

- 버블 가입 조건 = **기록 수(recordCount) + 레벨(level)** 기반 (bubble-join-service.ts). JoinApplicantProfile에 activeXp/activeVerified 필드도 있으나 현재 미사용 → 데이터 오염 상당 부분 방지
- 이름만 등록(0 XP) vs 풀 기록(+18 XP) — 15배 차이로 품질 기록 유도 (→ XP_SYSTEM.md 참조)
- 평균 점수 산출: **Trimmed Mean (양끝 5%)** → 트롤/극단 점수 자동 제거
- 문제 확인 시 **해당 멤버를 제거하면 오염 정화** 가능
- 데이터는 고정이 아니라 **개인별로 가변적** — 멤버 추가/제거에 따라 버블 데이터 동적 변화
- 개인이 **능동적으로 버블 탈퇴**함으로써도 오염 정화 가능

### 6-2. 오염 감지 (추후 구현)

- bubble owner에게 abuser/오염원을 **자동 경고**하는 기능 예정
- 이상 패턴 감지: 과도한 고평점, 특정 식당 집중 공유 등

### 6-3. 홈 추천과의 연동

- 홈 화면의 추천 식당/와인은 표면적으로 기본 노출 데이터이나
- **내부 로직에서 리스트(버블) 내 큐레이션 데이터 중 평이 좋은 것을 반영**
- `private` 리스트의 경우 **리스트 존재 자체가 노출되지 않으면서** 추천에만 영향

## 7. 버블 공개/팔로우 동기 구조

```
왜 리스트를 만드나?
├── 나만의 목적별 식당/와인 정리 (개인 리스트)
├── 지인과 공동 큐레이션 (공동 리스트)
├── 자동 필터로 기록/찜 자동 분류
└── 방문 진행 상황 추적 (체크리스트)

왜 리스트를 공개하나?
├── 팔로워 획득 → XP (+1/팔로워)
├── 좋아요/찜 수신 → XP (+1/건)
├── 자기 만족 (큐레이션 인정)
└── XP 축적 → 더 나은 리스트 가입 자격 확보

왜 리스트를 팔로우하나?
├── 사람 전체가 아닌, 관심 있는 주제/지역의 리스트만 구독
├── "어디 갈지" 고민할 때 참고
├── 취향 유사도 높은 리스트 발견
└── 무료 (XP 불필요)

왜 맞팔하나? (개인)
├── 특정 개인의 상세 기록/팁에 접근
├── 리스트 밖에서도 지속적 관계
├── 상대의 기록 품질을 확인한 후 결정
└── 양쪽 모두 XP +2

선순환 (전체 플라이휠):
  나를 위한 기록 (이기적 동기)
    → 개인 리스트로 정리 (이기적 가치)
    → 리스트 공개 → 팔로워 유입
    → 좋은 멤버 합류 → 리스트 가치 상승
    → 개인 팔로우/맞팔로 네트워크 확장
    → 더 많은 신뢰 데이터 접근
    → 더 나은 기록 동기
    → (반복)

콜드스타트 진입점:
  리스트 팔로우 (무료) → 맛보기 → 가치 체감 → 기록 시작 → 생태계 합류
```

## 8. 큐레이션 플로우

리스트(버블)에 대상이 담기는 방식은 자동/수동/혼합 세 가지다.

### 8-1. 자동 큐레이션 (기존 자동 공유 확장)

> `privacy_records = 'private'`이면 자동 큐레이션 비활성. 상세 규칙: SETTINGS.md §3~§4

```
ShareRuleEditor에서 필터 조건 설정
→ 조건에 매칭되는 기록 자동 추가/제거
→ 조건에 매칭되는 찜 자동 추가/제거
→ 규칙 변경 시 기존 데이터도 소급 재평가
```

예: "광화문 식당, 만족도 90+" 규칙 → 조건 맞는 기록과 찜이 자동으로 리스트에 포함

### 8-2. 수동 큐레이션

```
방법 1: 식당/와인 상세 페이지 → "리스트에 추가" → 리스트 선택
방법 2: 리스트 상세 페이지 → "대상 추가" → 식당/와인 검색 → 추가
방법 3: 기록 직후 → "리스트에 추가" → 리스트 선택
```

수동으로 담은 대상은 기록이 없어도 가능 (찜 상태로 리스트에 존재).

### 8-3. 혼합 큐레이션

자동 규칙 + 수동 추가 공존. 자동으로 채워지되, 수동으로도 추가 가능.
수동 추가 항목은 자동 규칙과 독립 — 규칙 변경 시에도 수동 항목은 유지.

### 8-4. 공유 데이터

**리스트에 포함 시**: 식당/와인명, 사분면 좌표, 만족도, 상황, 한줄평, 사진, 메뉴, 팁, 날짜
**리스트 내에서만 공개**: 가격 (프로필 방문자에게는 비공개)
**항상 비공개**: 동반자 이름 (`records.companions`, 나만 열람, 외부 노출 절대 금지). 동반자 수(`companion_count`)는 별개 — 필터/통계용 활용 가능
**제거**: 언제든 제거 가능 (달린 댓글은 고아 상태로 유지)
**리스트 간 격리**: 기록 A를 리스트 X에 담아도 리스트 Y에는 보이지 않음

### 12-2. 리스트 (기본 서브페이지)

리스트에 담긴 대상(자동 큐레이션 + 수동 추가 + 기록)을 **targetId별로 집계**하여 CompactListItem으로 표시한다. 개별 기록 카드가 아닌 식당/와인 단위의 큐레이션 뷰.

#### 집계 로직 (aggregateByTarget)

동일한 targetId의 공유를 그룹핑하여 다음을 계산:
- 평균 만족도 (rated 멤버 평균)
- 평가한 멤버 수
- 최신 공유일 / 최신 방문일
- 사분면 점(dot) 배열 (axisX, axisY, satisfaction, avatarColor, nickname)
- 내 기록 존재 여부 + 내 점수와 버블 평균의 차이로 tasteMatchPct 계산

#### 리스트 뷰

```
┌──────────────────────────────┐
│ [1] 📷 │ 미진               │
│  순위   │ 한식 · 을지로      │
│         │ 평균 93 · 3명 평가  │
│         │ 사분면 dot 미리보기  │
└──────────────────────────────┘
```

- CompactListItem: 순위 + 썸네일 + 이름 + 메타(카테고리·지역) + 평균점수 + 멤버 dot
- 클릭 → 식당/와인 상세 페이지 (`?bubble=[bubbleId]` 파라미터 포함)
- 페이지당 8개, ConditionFilterBar 인라인 페이저
- 빈 상태: UtensilsCrossed/Wine 아이콘 + "공유된 식당/와인이 없어요"

#### 필터 (ConditionFilterBar)
- 탭별 독립: 식당 탭은 RESTAURANT_FILTER_ATTRIBUTES, 와인 탭은 WINE_FILTER_ATTRIBUTES
- matchesAllRules로 조건 필터 적용 (genre, area, satisfaction, member_count 등)
- 탭 전환 시 칩 자동 초기화

#### 정렬
- 최신순 (기본), 점수 높은순, 점수 낮은순, 이름순, 방문 많은순

### 12-3. 랭킹 (서브페이지)

랭킹 아이콘(Trophy) 클릭 시 전환. 현재 활성 탭(식당/와인)에 따라 자동으로 targetType 결정.
useBubbleRanking 훅으로 이번 주 vs 지난 주 비교 후 delta(up/down/new/same) 계산.

#### 포디움 뷰 (Top 3)

```
         ┌────┐
       ┌─┤ 1  ├─┐          ← 1위: 110px, 왕관 아이콘, 금색
       │ └────┘ │
     ┌──────┐ ┌──────┐
     │  2   │ │  3   │     ← 2위: 88px 은색, 3위: 76px 동색
     └──────┘ └──────┘
```

- 각 아이템: 사진 배경 + 순위 배지(좌상단) + 점수 오버레이(우하단)
- 아래: 이름 + 메타 (카테고리 · 지역)

#### 리스트 뷰 (4위~)

```
│ 순위 │ 썸네일 │ 이름 + 메타 │ 점수 + 변동 │
│  4   │ 📷    │ 이코이      │ 88   ▲3    │
│  5   │ 📷    │ 파스타바르바리│ 84   NEW   │
```

- 변동 표시: ▲N(초록), ▼N(빨강), NEW(회색)

- 빈 상태: Trophy 아이콘 + "아직 랭킹이 없어요" + "기록이 쌓이면 주간 랭킹이 생겨요"

#### 랭킹 데이터
- useBubbleRanking(bubbleId, targetType) 훅으로 로드
- 이번 주 vs 지난 주 getPreviousRankings로 비교
- delta 계산: up(▲N), down(▼N), new(NEW), same(-)
- targetType은 현재 활성 탭(식당/와인)에 연동

### 12-4. 멤버 (서브페이지)

│  ✏️ 72개  📍 8곳   │  ✏️ 61개  📍 5곳   │
│  [팔로잉]          │  [팔로우]          │
└────────────────────┴────────────────────┘
```

| 요소 | 상세 |

#### 리스트 뷰 (viewMode='list', 코드 존재하나 전환 UI 미노출)

## 13. 버블러 프로필 페이지 (/users/[id]?bubble=[bubbleId])

멤버 카드 클릭 시 진입 (`/users/${userId}?bubble=${bubbleId}`). FabBack으로 뒤로가기.

### 13-0. 접근 수준 (Access Level)

| 수준 | 조건 | 보이는 섹션 |
|------|------|-----------|
| `none` | 팔로우 관계 없음 | 히어로만 (닉네임 대신 "Lv.N 유저", 핸들/아바타 숨김) |
| `follow` | 내가 상대를 팔로우 (일방) | + 탭, 취향 프로필, 활동 히트맵 |
| `mutual` | 양방향 맞팔로우 or 본인 프로필 | + 버블 컨텍스트 카드(bubbleId 있을 때), 강력 추천, 최근 기록 |

> 접근 수준은 순수하게 팔로우 관계(getAccessLevel)로 결정. 같은 버블 멤버여도 팔로우가 없으면 `none`.

### 13-1. 레이아웃 (위→아래)

1. **앱 헤더** — AppHeader 공통 컴포넌트
2. **FabBack** — 뒤로가기 플로팅 버튼
3. **프로필 히어로** — BubblerHero
4. **버블 컨텍스트 카드** — BubbleContextCard (mutual only, bubbleId가 있을 때)
5. **스티키 탭** — 식당(food) | 와인(wine) (follow+ only)
6. **취향 프로필** — TasteProfile (follow+ only)
7. **강력 추천 (Picks)** — PicksGrid (mutual only)
8. **최근 기록** — RecentRecords (mutual only)
9. **활동 (Activity)** — ActivitySection (follow+ only)

### 13-2. 프로필 히어로 (BubblerHero)

| 요소 | 상세 | 접근 수준 |
|------|------|----------|
| 아바타 | 원형, 그라디언트 배경, 이니셜 | 항상 (none일 때 색상만) |
| 레벨 배지 | "Lv.N 타이틀" | 항상 |
| 이름 | 닉네임 (none일 때 "Lv.N 유저"로 대체) | 항상 |
| 핸들 | @username | follow+ |
| 통계 | 기록 N / 팔로워 N / 팔로잉 N | 항상 |
| 맛 태그 | 필(pill) 리스트 — 상위 태그 강조 | follow+ |

## 14. 홈화면 버블 피드

버블의 기본 뷰는 **리스트 뷰**(대상 목록)이지만, 홈 피드에서는 **기록 시간순 피드**로 표시한다. 리스트 = 큐레이션된 콘텐츠, 피드 = 최근 활동 알림.

```
── 버블 ──
[아바타] 김영수 · 직장 맛집
미진 — 93점 · 혼밥
"메밀국수 진짜 맛있다"
♡ 4  💬 2  🔖

[아바타] 박소연 · 와인 버블
Penfolds Grange — 96점
♡ 8  💬 5  🔖

[모든 버블 보기 →]
```

- 정렬: 시간순 (알고리즘 없음)
- 필터: 전체/특정 버블, 식당/와인, 상황 태그
- **리스트 뷰 vs 피드 뷰**: 버블 상세(§12)는 대상 단위 리스트 뷰가 기본. 홈 피드는 기록 단위 시간순 (최근 활동 파악용)

## 16. 경험치 & 소셜 신뢰

### 기록 XP (XP_SYSTEM.md v2 기준)

| 기록 수준 | XP | 비고 |
|---|---|---|
| 이름만 등록 | **0** | XP 없음. 어뷰징 무력화 |
| + 사분면 점수 | **+3** | 최소한의 평가 |
| + 사진 (EXIF GPS 검증) | **+8** | 실제 방문 증거 |
| + 풀 기록 (리뷰+상황+메뉴) | **+18** | 핵심 보상. 가장 효율적 |
| 카테고리 XP (지역/장르) | **+5** (별도) | 종합 XP에 미포함, 카테고리 전용 |

### 소셜 XP

| 활동 | XP | 비고 |
|------|-----|------|
| 버블에 공유 | +1 | — |
| 좋아요/찜 받음 | +1 | — |
| 팔로워 획득 (버블/개인) | +1 | — |
| 맞팔 성사 | +2 | 양쪽 모두에게 |

> **소셜 XP 합산 일일 상한: 10 XP/일** (어떤 조합이든)

### XP 이원화

- **총 XP** = 누적 (절대 안 줄어듦). 레벨 표시에 사용. "이 사람의 역사"
- **활성 XP** = 최근 6개월 기록 기반 (자연 변동). 설계상 버블 진입 기준 의도이나 현재 코드는 recordCount + level만 체크 (§6-1 참조). "이 사람의 현재"
- 레벨(총 XP 기반)은 절대 안 깎임 → 손실 회피 심리 회피

### 신뢰 표시

- 기록 옆에 **관련 레벨만** 표시 (을지로 기록 → 을지로 Lv.7)
- "경험자 순" 정렬 = 필터/정렬 옵션. 자동 가중치 아님
- **EXIF 검증 기록 수** = 프로필의 핵심 신뢰 지표

## 18. 팔로우

> 핵심 기능. 버블과 함께 1.0 런칭에 포함.

### 18-1. 개인 팔로우 — 양방향 가능 (맞팔로우)

- 개인 단위, 양방향 가능 (일방 팔로우 + 맞팔로우)
- 공개 프로필만 팔로우 가능

| 관계 | 보이는 것 | 할 수 있는 것 |
|------|----------|-------------|
| 일방 팔로우 | 이름, 점수, 지역 (맛보기) | 좋아요, 찜 |
| 맞팔로우 | 풀 액세스 (리뷰, 사진, 팁, 상세 기록) | 좋아요, 찜, 댓글 |

### 18-2. 팔로우 발견 경로

- **버블 멤버 목록**에서 취향 맞는 사람 발견 → 개인 팔로우
- 상대도 나를 팔로우 → 맞팔 = 버블 밖에서도 풀 액세스
- 버블이 개인 팔로우의 **발견 채널** 역할

### 18-3. 피드

- 팔로우 피드 vs 버블 피드 탭 분리
- 팔로워 수 표시하되 크게 강조 안 함

## 19. 역할 & 권한

### 권한 매트릭스 (bubble-permission-service.ts 기준)

| 권한 (BubblePermissions) | follower | member | admin | owner |
|-------------------------|----------|--------|-------|-------|
| canReadFeed | O | O | O | O |
| canReadFullFeed | X | O | O | O |
| canShare | X | O | O | O |
| canReact | X | O | O | O |
| canComment | X | *allow_comments | O | O |
| canManageMembers | X | X | O | O |
| canInvite | X | X | O | O |
| canRemoveMember | X | X | X | O |
| canEditSettings | X | X | X | O |
| canChangeRoles | X | X | X | O |
| canDeleteBubble | X | X | X | O |

> *멤버의 댓글 권한은 `bubble.allowComments` 설정에 따라 결정
> 비멤버(role=null)의 canReadFeed는 `joinPolicy !== 'invite_only'`일 때만 true

### 역할 우선순위

follower(0) < member(1) < admin(2) < owner(3)

### 역할 변경 규칙 (canChangeRole)
- owner만 역할 변경 가능
- 자기 자신의 역할은 변경 불가
- owner 역할로의 변경 불가 (transferOwnership 별도 로직)

### 팔로워의 데이터 접근 범위
- 식당/와인 **이름, 평균 점수(trimmed mean), 지역** 열람 가능 (canReadFeed=true)
- 개별 리뷰, 사진, 팁, 개인별 점수는 **멤버 이상**만 열람 가능 (canReadFullFeed)
- 리액션, 공유, 댓글 모두 불가 (canReact=false, canShare=false, canComment=false)
