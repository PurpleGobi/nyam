## 1. 핵심 원칙

- **기록이 먼저, 소셜은 그 다음** — 소셜 없이도 유용
- **신뢰가 네트워크의 핵심** — 활성 XP + EXIF 검증 기반 가입 조건으로 데이터 오염 방지
- **공유는 선택적** — 원하는 기록만 특정 버블에
- **피드는 시간순** — 알고리즘 피드 없음 (신뢰 보호)
- **버블은 가변적** — 멤버 탈퇴/제거로 데이터 오염 정화 가능
- **기여도에 비례한 접근** — 더 많이 기여할수록 더 깊은 정보에 접근 가능

### 핵심 비전

nyam은 개인 기록용 앱이지만, **버블의 네트워크 기능이 신뢰할 만한 맛집/와인 추천을 이어가는 것이 핵심 가치**다.
홈 화면 추천에도 버블 내 공유 데이터가 내부적으로 반영된다 (→ RECOMMENDATION.md 참조).

버블은 **뉴스 피드가 아니다.** 실시간 타임라인을 소비하는 소셜 미디어가 아니라, 내가 어디 갈지 고민할 때 꺼내보는 **신뢰할 수 있는 맛집 DB(공유 데이터베이스)**에 가깝다.

## 2. 데이터 접근 계층 모델

### 2-1. 설계 원칙

> **기여도에 비례한 접근 권한** — 더 많이 기여할수록 더 깊은 정보에 접근 가능

### 2-2. 버블에서의 4단계 접근

| 단계 | 역할 | 조건 | 보이는 것 | 할 수 있는 것 |
|------|------|------|----------|-------------|
| **탐색** | 비팔로워 | public 버블이면 누구나 | 버블 이름, 설명, 멤버 수, 통계 요약 | 검색, 열람 |
| **팔로우** | follower | XP 불필요, 누구나 가능 | + 식당/와인 **이름, 평균 점수, 지역** | + 좋아요, 찜 |
| **멤버** | member | 가입 승인 필요 | + **개별 리뷰, 사진, 팁, 누가 몇 점인지** | + 기록 공유, 댓글, 리액션 |
| **운영** | admin/owner | 오너 지정 | 전체 | + 멤버 관리, 설정 변경 |

**팔로워 = 맛보기.** 식당 이름과 평균 점수는 알 수 있지만, "왜 좋은지, 뭘 먹어야 하는지"는 멤버만 볼 수 있다.

**복제 방어:** 이름+점수만으로는 가치가 제한적. Spotify 플레이리스트 곡 제목만 복사하는 것과 같음 — 진짜 가치는 지속적인 큐레이션과 상세 맥락.

### 2-3. 개인 팔로우에서의 3단계 접근

| 단계 | 관계 | 조건 | 보이는 것 |
|------|------|------|----------|
| **비팔로우** | 없음 | — | 공개 프로필만 (레벨, 기록 수, 버블 수) |
| **일방 팔로우** | 내가 팔로우 | 상대가 public 프로필 | + 식당/와인 **이름, 점수, 지역** (맛보기) |
| **맞팔로우** | 상호 팔로우 | 양쪽 모두 팔로우 | + **풀 액세스** (리뷰, 사진, 팁, 상세 기록) |

**핵심: 맞팔로우 = 기브앤테이크의 구현**

- "이 사람 기록 제대로 보고 싶다" → 팔로우
- 상대가 "이 사람도 볼만하네" → 맞팔
- 맞팔 조건 = 내 기록의 질이 좋아야 상대가 팔로우할 동기가 생김
- → 기록 품질 향상 동기 자연스럽게 발생

### 2-4. 버블 멤버 → 개인 팔로우 연결

```
버블 가입
  → 멤버 목록 열람 가능
  → 특정 멤버의 기록이 좋다 → 개인 팔로우
  → 상대도 나를 팔로우 → 맞팔 = 버블 밖에서도 풀 액세스
  → 버블을 넘어선 개인 네트워크 형성
```

이를 통해 **개인 팔로우의 차별점**이 명확해진다:
- **버블** = 그룹 단위의 맥락적 큐레이션 (여러 사람의 기록이 하나의 주제로 모임)
- **개인 팔로우** = 특정 개인의 취향을 깊게 추적 (버블 가입 후 발견한 취향 맞는 사람)
- **버블이 개인 팔로우의 발견 채널** 역할

## 3. 버블 생성

- **누구나 자유롭게, 무한정 생성 가능** (단톡방처럼)
- 1인 버블 가능 — 외부 공유용 큐레이션 채널로 활용
- 여러 명이 아니어도 됨

```
1. [+] 버블 만들기 (FAB)
2. 이름 (20자) + 설명 (100자, 선택)
3. 공개 설정: 비공개 (lock) | 공개 (globe)
   - 비공개: "초대받은 사람만 접근"
   - 공개: "누구나 검색 팔로우 가능"
4. 초대 링크 만료: 1일 | 7일 | 30일(기본) | 무제한
5. (공개일 때) 가입 정책 4종 (아이콘+설명 카드 형태):
   - 팔로우만 (eye) — "가입 안 받음. 팔로워는 이름+점수만 열람"
   - 승인 가입 (shield-check, 기본) — "가입 신청 → 프로필 보고 승인/거절"
   - 자동 승인 (zap) — "검증 기록 N개 이상이면 자동 가입"
   - 자유 가입 (door-open) — "누구나 바로 가입 가능"
6. 완료 → "버블 만들기" CTA → 버블 상세
```

> **생성 시 미포함, 설정에서 변경 가능한 항목:**
> - 버블 포커스/정체성 (focus_type) → §4-6
> - 공개 수위 (content_visibility) → §4-5
> - 권한 설정 (댓글/외부공유) → §4-4
> - 가입 조건 상세 (최소 기록, 최소 레벨, 최대 인원) → §12-6

## 4. 버블 속성

### 4-1. 공개 여부 (visibility)

| 값 | 검색 | 사용 예 |
|------|------|--------|
| `private` | 불가 | 직장 맛집, 가족 (초대받은 사람만 접근) |
| `public` | 가능 | 누구나 검색/팔로우 가능 |

- 기존 3단계(`private|members|public`)에서 2단계로 단순화
- "누가 볼 수 있는가"와 "누가 가입할 수 있는가"를 분리 (→ 4-2 join_policy)

### 4-2. 가입 정책 (join_policy)

| 값 | UI 레이블 | 아이콘 | 설명 | 적용 시점 |
|------|----------|--------|------|----------|
| `invite_only` | — | — | 초대받은 사람만 가입 | private 버블의 암묵적 기본값 (UI에 노출 안 됨) |
| `closed` | 팔로우만 | eye | 가입 안 받음. 팔로워는 이름+점수만 열람 | public 생성 시 선택 |
| `manual_approve` | 승인 가입 | shield-check | 가입 신청 → 프로필 보고 승인/거절 | public 생성 시 기본값 |
| `auto_approve` | 자동 승인 | zap | 검증 기록 N개 이상이면 자동 가입 | public 생성 시 선택 |
| `open` | 자유 가입 | door-open | 누구나 바로 가입 가능 | public 생성 시 선택 |

- private 버블은 join_policy 선택 UI가 표시되지 않고 `invite_only`로 자동 설정
- public 버블은 4가지 옵션(closed/manual_approve/auto_approve/open) 중 선택

#### 가입 조건 설정 (버블 설정 페이지에서 관리)

| 설정 | 설명 | 기본값 | UI 형태 |
|------|------|--------|--------|
| `require_approval` | 가입 승인 필요 여부 | true | 토글 |
| `min_records` | 최소 기록 수 | 5 | 숫자 행 |
| `min_level` | 최소 레벨 | Lv.3 | 숫자 행 |
| `max_members` | 최대 인원 | 20 | 숫자 행 |

- 발견 팝업에서 체크마크 리스트로 표시: "관리자 승인 필요", "최소 기록 N개 이상", "최소 Lv.N 이상", "현재 N/N명 (여유 N자리)"

#### auto_approve 자동 승인 기준

`auto_approve` 정책일 때 위 조건(min_records, min_level) 충족 시 자동 승인. 추가 세밀 제어:

| 설정 | 설명 | 기본값 |
|------|------|--------|
| `approve_criteria_type` | `and` 또는 `or` — 조건 조합 방식 | `and` |
| `approve_min_total_xp` | 총 XP 최소 기준 | 0 |
| `approve_min_active_verified` | 최근 6개월 검증(EXIF) 기록 최소 수 | 10 |

```
예시:
  criteria_type = and, min_total_xp = 100, min_active_verified = 10
  → "총 XP 100 이상 AND 최근 6개월 검증 기록 10개 이상"이면 자동 승인

  criteria_type = or, min_total_xp = 500, min_active_verified = 5
  → "총 XP 500 이상 OR 최근 6개월 검증 기록 5개 이상"이면 자동 승인
```

### 4-3. 양방향 프리뷰 (가입 시)

기존 "XP 레벨 5" 같은 숫자만으로 판단하던 것을 **기록의 질적 프로필**로 대체.

**가입 희망자가 보는 것 (가입 전 미리보기):**

| 정보 | 예시 |
|------|------|
| 멤버 수 | 12명 |
| 총 기록 | 234개 |
| 주요 지역 | 을지로, 광화문 |
| 평균 점수 | 87 |
| 나와 겹치는 곳 | 5개 |
| 취향 유사도 | 78% |

→ "이 버블에 들어갈 가치가 있겠다"

**버블 오너가 보는 것 (가입 신청 시):**

| 정보 | 예시 |
|------|------|
| 검증 기록 (EXIF) | 28개 |
| 단순 등록 | 5개 |
| 주요 지역 | 을지로, 성수 |
| 취향 유사도 | 78% |
| 최근 활동 | 3일 전 |

→ "이 사람을 받아도 되겠다" or "거절"

### 4-4. Owner 권한 설정 (AND 조건으로 조합)

| 설정 | 설명 | 기본값 |
|------|------|--------|
| `allow_comments` | 댓글 허용 여부 | true |
| `allow_external_share` | 외부 공유 허용 여부 | false |

### 4-5. 공개 수위 (content_visibility)

**버블에 공유된 기록이 비멤버에게 어떻게 보이는가** (→ SETTINGS.md §4-2 상세 매트릭스)

| 값 | 버블 멤버에게 | 비멤버에게 (상세 페이지 L9) |
|------|------------|-------------------------|
| `rating_only` | 점수 + 한줄평 + 사진 + 메뉴 전체 | 점수만 (아바타 + Lv + 만족도 숫자) |
| `rating_and_comment` | 점수 + 한줄평 + 사진 + 메뉴 전체 | 점수 + 한줄평 |

- `records.comment` 한줄평이 공유 텍스트 — 별도 "버블 전용 코멘트" 필드 없음
- 한줄평 없이 공유하면 점수만 표시
- 멤버에게는 공유 시점에 노출 동의한 것으로 간주

### 4-6. 버블 포커스 (focus_type)

| 값 | 의미 |
|------|------|
| `all` | 식당 + 와인 모두 |
| `restaurant` | 식당 중심 |
| `wine` | 와인 중심 |

- 정체성을 정한다고 **제한이 생기지는 않음** — 노출되는 라벨일 뿐
- 언제든 변경 가능

## 5. 버블 사용 시나리오 (중요도 순)

1. **지인 간 맛집/와인 경험 공유** — 직장 동료, 친구, 가족
2. **생소한 동네에서 믿을 만한 맛집 추천 받기** — public 버블 팔로우
3. **개인 큐레이션 채널 운영** — 1인 버블로 팔로워 대상 추천 리스트 제공
4. **콜드스타트 유저의 생태계 진입** — 팔로우로 즉시 가치 체감

### 시나리오 A: 개인 큐레이션 채널

```
"을지로 맛집 가이드" — 1인 운영
  visibility: public
  join_policy: closed

  → 팔로워들이 이름+점수 리스트로 참고
  → 좋아요/찜 → 오너 XP 상승
  → 오너는 성실하게 기록 → 팔로워 증가 → 선순환
  → 관리 부담: 최소 (가입 안 받으니 관리할 멤버 없음)
```

### 시나리오 B: 지인 모임 (관리 최소)

```
"직장 점심" — 5명
  visibility: private
  join_policy: invite_only

  → 초대받은 멤버만 접근
  → 외부에 노출 안 됨
  → 내부에서 자유롭게 기록 공유 + 댓글/리액션
  → 관리 부담: 없음 (소규모 + 아는 사람)
```

### 시나리오 C: 오픈 커뮤니티 (카페형, 소셜 강화)

```
"강남 와인바 모임" — 30명
  visibility: public
  join_policy: manual_approve

  → 누구나 팔로우 (이름+점수 열람)
  → 가입 신청 → 오너가 프로필(검증 기록 수, 취향 유사도) 보고 승인
  → 멤버: 풀 액세스 + 기록 공유 + 댓글/리액션
  → 멤버 목록에서 개인 팔로우 → 맞팔로 확장
  → 관리 부담: 있음 (가입 승인, 오염원 관리). 대신 리텐션 강화
```

### 시나리오 D: 콜드스타트 유저 여정

```
가입 직후 (기록 0, XP 0)

  1. 온보딩에서 버블 1개 생성 (내 모임)
  2. "버블 탐색"으로 공개 버블 둘러보기
  3. 취향 유사도 높은 버블 3~5개 팔로우 (무료, XP 불필요)
  4. 홈에서 팔로잉 버블의 이름+점수 리스트 즉시 활용
  5. "여기 괜찮겠다" → 방문 → 사진+리뷰 풀 기록 (+18 XP)
  6. 검증 기록 쌓임 → 프로필 신뢰도 상승
  7. 멤버 가입 신청 → 오너 승인 → 풀 액세스 (상세 리뷰, 사진, 팁)
  8. 멤버 목록에서 취향 맞는 개인 팔로우
  9. 맞팔 성사 → 버블 밖에서도 풀 액세스
```

### 5-1. 취향 유사도 매칭 (public 버블 탐색 시)

공개 버블을 탐색할 때, 내 기록과 버블 멤버들의 기록을 비교하여 **취향 유사도**를 표시한다.

#### 원리

- 내가 기록한 식당/와인과 **같은 대상을 리뷰한 버블**이 비교 대상
- 동일 대상의 **점수 오차 ±5% 이내**면 "취향 일치"로 판정
- 겹치는 대상이 많고 점수 오차가 작을수록 유사도 높음

#### 유사도 계산

```
겹치는 대상 수 = 내 기록 ∩ 버블 기록 (같은 식당/와인)
일치 수 = 겹치는 대상 중 점수 오차 ±5% 이내
유사도 = 일치 수 / 겹치는 대상 수 × 100%
```

#### 표시

```
┌─────────────────────────────────┐
│ 을지로 맛탐정 클럽               │
│ 멤버 23명 · 기록 156개           │
```

- **겹치는 대상이 3개 미만**이면 유사도 미표시 ("데이터 부족")
- public 버블에서만 표시 (private은 검색 불가이므로 해당 없음)

#### taste_match_pct 구현 참고

`bubble_members.taste_match_pct`는 **뷰어 상대적 값**이므로 테이블에 직접 저장할 수 없다 (뷰어 A↔멤버 B ≠ 뷰어 C↔멤버 B). 구현 시 다음 중 택 1:
1. **RPC 온디맨드 계산** — `get_taste_match(viewer, target)` 함수로 필요 시 계산. 결과를 `taste_match_cache(user_a, user_b, pct, computed_at)` 테이블에 캐시. 일간 크론으로 갱신.
2. **버블 평균 대비 매치로 재정의** — "이 멤버가 버블 평균과 얼마나 비슷한가"로 변경하면 per-row 저장 가능. 단, UI 문구 "나와 취향 일치도"와 불일치.

→ 구현 시점(S6)에 결정. 현재는 컬럼 유지하되 의미 정의만 보류.
- 유사도순 정렬 옵션 제공

## 6. 핵심 구동 원칙

### 6-1. 경험치 기반 데이터 신뢰

- 버블 가입 조건 = **활성 XP + EXIF 검증 기록 수** → 데이터 오염 상당 부분 방지
- 이름만 등록(0 XP) vs 풀 기록(+18 XP) — 15배 차이로 품질 기록 유도 (→ XP_SYSTEM.md 참조)
- 평균 점수 산출: **Trimmed Mean (양끝 5%)** → 트롤/극단 점수 자동 제거
- 문제 확인 시 **해당 멤버를 제거하면 오염 정화** 가능
- 데이터는 고정이 아니라 **개인별로 가변적** — 멤버 추가/제거에 따라 버블 데이터 동적 변화
- 개인이 **능동적으로 버블 탈퇴**함으로써도 오염 정화 가능

### 6-2. 오염 감지 (추후 구현)

- bubble owner에게 abuser/오염원을 **자동 경고**하는 기능 예정
- 이상 패턴 감지: 과도한 고평점, 특정 식당 집중 공유 등

### 6-3. 홈 추천과의 연동

- 홈 화면의 추천 식당/와인은 표면적으로 기본 노출 데이터이나
- **내부 로직에서 버블 내 공유 데이터 중 평이 좋은 것을 반영**
- `private` 버블의 경우 **버블 존재 자체가 노출되지 않으면서** 추천에만 영향

## 7. 버블 공개/팔로우 동기 구조

```
왜 버블을 공개하나?
├── 팔로워 획득 → XP (+1/팔로워)
├── 좋아요/찜 수신 → XP (+1/건)
├── 자기 만족 (큐레이션 인정)
└── XP 축적 → 더 나은 버블 가입 자격 확보

왜 팔로우하나? (버블)
├── 신뢰할 수 있는 맛집 리스트 확보
├── "어디 갈지" 고민할 때 참고
├── 취향 유사도 높은 버블 발견
└── 무료 (XP 불필요)

왜 맞팔하나? (개인)
├── 특정 개인의 상세 기록/팁에 접근
├── 버블 밖에서도 지속적 관계
├── 상대의 기록 품질을 확인한 후 결정
└── 양쪽 모두 XP +2

선순환 (전체 플라이휠):
  나를 위한 기록 (이기적 동기)
    → XP 축적 (노력이 자산)
    → 버블 생성/공개 → 팔로워 유입
    → 좋은 멤버 합류 → 버블 가치 상승
    → 개인 팔로우/맞팔로 네트워크 확장
    → 더 많은 신뢰 데이터 접근
    → 더 나은 기록 동기
    → (반복)

콜드스타트 진입점:
  팔로우 (무료) → 맛보기 → 가치 체감 → 기록 시작 → 생태계 합류
```

## 8. 공유 플로우

> `privacy_records = 'private'`이면 공유 버튼 비활성. 상세 규칙: SETTINGS.md §3~§4

```
방법 1: 기록 직후 → "버블에 공유하기" → 버블 선택 → 공유
방법 2: 기존 기록 → 공유 아이콘 → 버블 선택
방법 3: 필터 일괄 → "광화문 + 80점 이상" → 5개 한번에 공유
```

**공유 시 포함**: 식당/와인명, 사분면 좌표, 만족도, 상황, 한줄평, 사진, 메뉴, 팁, 날짜
**버블 내에서만 공개**: 가격 (프로필 방문자에게는 비공개)
**항상 비공개**: 동반자 이름 (`records.companions`, 나만 열람, 외부 노출 절대 금지). 동반자 수(`companion_count`)는 별개 — 필터/통계용 활용 가능
**공유 취소**: 언제든 철회 가능 (달린 댓글은 고아 상태로 유지)
**버블 간 격리**: 기록 A를 버블 X에 공유해도 버블 Y에는 보이지 않음

## 16. 경험치 & 소셜 신뢰

### 기록 XP (XP_SYSTEM.md v2 기준)

| 기록 수준 | XP | 비고 |
|---|---|---|
| 이름만 등록 | **0** | XP 없음. 어뷰징 무력화 |
| + 사분면 점수 | **+3** | 최소한의 평가 |
| + 사진 (EXIF GPS 검증) | **+8** | 실제 방문 증거 |
| + 풀 기록 (리뷰+상황+메뉴) | **+18** | 핵심 보상. 가장 효율적 |
| 카테고리 XP (지역/장르) | **+5** (별도) | 종합 XP에 미포함, 카테고리 전용 |

### 소셜 XP

| 활동 | XP | 비고 |
|------|-----|------|
| 버블에 공유 | +1 | — |
| 좋아요/찜 받음 | +1 | — |
| 팔로워 획득 (버블/개인) | +1 | — |
| 맞팔 성사 | +2 | 양쪽 모두에게 |

> **소셜 XP 합산 일일 상한: 10 XP/일** (어떤 조합이든)

### XP 이원화

- **총 XP** = 누적 (절대 안 줄어듦). 레벨 표시에 사용. "이 사람의 역사"
- **활성 XP** = 최근 6개월 기록 기반 (자연 변동). 버블 진입 기준에 사용. "이 사람의 현재"
- 레벨(총 XP 기반)은 절대 안 깎임 → 손실 회피 심리 회피

### 신뢰 표시

- 기록 옆에 **관련 레벨만** 표시 (을지로 기록 → 을지로 Lv.7)
- "경험자 순" 정렬 = 필터/정렬 옵션. 자동 가중치 아님
- **EXIF 검증 기록 수** = 프로필의 핵심 신뢰 지표

## 18. 팔로우

> 핵심 기능. 버블과 함께 1.0 런칭에 포함.

### 18-1. 개인 팔로우 — 양방향 가능 (맞팔로우)

- 개인 단위, 양방향 가능 (일방 팔로우 + 맞팔로우)
- 공개 프로필만 팔로우 가능

| 관계 | 보이는 것 | 할 수 있는 것 |
|------|----------|-------------|
| 일방 팔로우 | 이름, 점수, 지역 (맛보기) | 좋아요, 찜 |
| 맞팔로우 | 풀 액세스 (리뷰, 사진, 팁, 상세 기록) | 좋아요, 찜, 댓글 |

### 18-2. 팔로우 발견 경로

- **버블 멤버 목록**에서 취향 맞는 사람 발견 → 개인 팔로우
- 상대도 나를 팔로우 → 맞팔 = 버블 밖에서도 풀 액세스
- 버블이 개인 팔로우의 **발견 채널** 역할

### 18-3. 피드

- 팔로우 피드 vs 버블 피드 탭 분리
- 팔로워 수 표시하되 크게 강조 안 함

## 19. 역할 & 권한

| 역할 | 데이터 열람 | 좋아요/찜 | 댓글 | 기록 공유 | 멤버 관리 | 버블 삭제 |
|------|-----------|----------|------|----------|----------|----------|
| **오너** (owner) | 전체 | O | O | O | O | O |
| **관리자** (admin) | 전체 | O | O | O | O | X |
| **멤버** (member) | 전체 | O | O* | O | X | X |
| **팔로워** (follower) | 제한적 (이름+점수+지역) | O | X | X | X | X |

> *댓글은 `allow_comments` 설정에 따라
> 기존 `subscriber` → `follower`로 명칭 변경

팔로워의 데이터 접근 범위:
- 식당/와인 **이름, 평균 점수(trimmed mean), 지역** 열람 가능
- 개별 리뷰, 사진, 팁, 개인별 점수는 **멤버 이상**만 열람 가능
- 좋아요/찜은 가능, 댓글/공유는 불가
