# Nyam PRD (Product Requirements Document)

> 최종 갱신: 2026-04-02

---

## 1. 문제 정의

### Pain Point

> "제대로 된, 신뢰할 만한 맛집 정보를 찾기 어렵다."

기존 맛집 플랫폼(네이버, 망고플레이트, 캐치테이블 등)의 정보는 **출처를 알 수 없기 때문에 신뢰할 수 없다.** 별점 조작, 광고성 리뷰, 불특정 다수의 취향 평균화 — 정보가 많을수록 오히려 노이즈가 커지는 구조다.

사람들이 실제로 맛집을 고를 때 가장 의존하는 건 **아는 사람의 추천**이다. 하지만 지인 네트워크만으로는 정보량이 턱없이 부족하다.

### 핵심 허들

| 허들 | 설명 |
|------|------|
| **정보량 부족** | 지인끼리만 공유하면 커버할 수 있는 범위가 너무 적다 |
| **이타적 기록의 한계** | 불특정 다수를 위해 반복적으로 기록하는 행위는 지속되지 않는다 |

대부분의 리뷰 플랫폼은 이 문제를 보상(쿠폰, 포인트)으로 풀려다 실패했다. 외재적 보상은 기록의 질을 떨어뜨리고, 보상이 사라지면 기록도 멈춘다.

## 2. Solution Logic — Nyam의 해법

### 핵심 전제

> "이타적 기록은 지속되지 않는다. 동기 자체를 이기적으로 설계해야 한다."

기록의 시작점은 **남을 위한 리뷰가 아니라, 나 자신을 위한 기록**이어야 한다. 공유는 그 기록의 부산물이다.

### 3단계 플라이휠

```
[1단계] 이기적 동기 — 기록의 시작
│
│  나를 위한 기록 + 내 모임(버블)을 위한 공유
│  → 기록의 출발점은 철저히 자기 자신을 위한 것
│  → 가본 식당을 잊지 않기 위해, 내 취향을 파악하기 위해
│  → 내 모임(버블) 사람들에게 공유하기 위해
│
▼
[2단계] 축적이 자산이 된다 — 경험치
│
│  기록이 쌓이면 경험치(XP)로 남는다
│  → 노력이 휘발되지 않고 "자산"이 된다
│  → 경험치 = 이 사람이 얼마나 진지하게 기록해왔는지의 증거
│  → 단순 숫자가 아닌, 신뢰도의 정량적 지표
│
▼
[3단계] 버블 생태계 확장 + 팔로우 네트워크 — 네트워크 효과
│
│  무수히 많은 버블이 탄생한다
│  → 다른 버블에 대한 호기심이 발생한다
│  → "저 사람/버블의 기록을 보고 싶다"는 욕구
│  → 기브앤테이크: 내가 공유할 만한 게 있어야 타인도 공유해준다
│  → 경험치가 진입 허들이 되어 정보 오염을 방지한다
│  → 더 나은 버블에 들어가기 위해 더 성실히 기록한다
│
│  팔로우/맞팔로우로 개인 네트워크 확장
│  → 팔로우 (무료) → 이름+점수 맛보기 확인
│  → 맞팔로우 → 풀 액세스 (리뷰, 사진, 팁)
│  → 버블 멤버 목록에서 개인 팔로우 발견
│  → 개인 네트워크 → 버블 밖에서도 신뢰 데이터 접근
│
▼
[선순환] 기록 → 경험치 → 팔로우/맞팔/버블 → 더 넓은 네트워크 접근 → 더 많은 기록 동기
```

### 신뢰 메커니즘

| 요소 | 역할 |
|------|------|
| **경험치(XP)** | 신뢰의 정량적 지표. 기록 성실도의 증거이자 생태계 진입 자격 |
| **버블** | 신뢰의 단위. 취향이 맞는 사람들의 클러스터 |
| **경험치 기반 진입 허들** | 버블 간 연결, 개인 간 맞팔로우의 자격 조건. 노력 없이 정보만 취하는 것을 방지 |
| **팔로우** | 버블의 맛보기 진입점. XP 없이도 이름+점수 접근 가능. 콜드스타트 해결 |
| **맞팔로우** | 기브앤테이크의 구현. 상호 기록 품질이 풀 액세스의 조건 |
| **연결 해지** | 오염된 인물/버블 감지 시 즉시 끊을 수 있는 유동적 구조 |

### 한 문장 요약

> **"자기 자신을 위해 시작한 기록이, 버블이라는 신뢰 단위를 거쳐, 경험치라는 자격 증명을 통해 더 넓은 맛집 정보 생태계로 확장되는 구조."**

## 3. 제품 정의

**Nyam은**: 나의 식도락 경험을 기록하고, 그 기록이 신뢰 기반 네트워크를 통해 가치를 확장해가는 앱

**Nyam이 아닌 것**:
- 네이버/카카오 같은 공개 리뷰 플랫폼이 아님 — 불특정 다수의 리뷰가 아니라 신뢰할 수 있는 사람의 기록
- 캐치테이블 같은 예약 서비스가 아님
- 배달/요리 앱이 아님
- **범위: 식당 + 와인** (배달, 요리, 카페 제외)

### 핵심 가치

| 순서 | 가치 | 설명 |
|------|------|------|
| 1 | **기록** | 3초 사분면 평가로 식당/와인 경험을 구조화. 나를 위한 기록 |
| 2 | **축적** | 기록이 경험치로 쌓여 자산이 된다. 노력이 사라지지 않는다 |
| 3 | **신뢰** | 버블(그룹) 안에서 아는 사람의 경험만 공유. 경험치가 진입 자격 |
| 4 | **확장** | 버블 간 연결을 통해 신뢰할 수 있는 정보 네트워크가 넓어진다 |

### 시그니처 기능: 사분면 평가

- 2D 좌표(특성 2축) + 자동 계산 만족도 + 상황 태그
  - 식당: X축=음식 퀄리티, Y축=경험 가치
  - 와인: X축=구조·완성도, Y축=즐거움·감성
  - 만족도 = (X + Y) / 2 자동 계산 (별도 입력 아님)
  - 점 크기는 고정 20px
- 기록이 쌓이면 사분면 위에 "나만의 취향 지도" 자동 생성 (참조 점 opacity 0.3, 최대 8~12개)

## 4. 차별점

| 기존 앱 | 문제 | Nyam의 접근 |
|---------|------|-------------|
| 네이버/카카오 | 출처 불명의 공개 리뷰, 광고 오염 | 경험치로 검증된 사람의 기록만 공유 |
| Vivino | 와인만, 불특정 다수 평점 | 식당+와인 통합, 신뢰 기반 네트워크 |
| 인스타그램 | 비구조화, 검색 불가, 과시 중심 | 구조화된 기록, 활용 가능한 데이터 |
| 망고플레이트 | 리뷰어 신뢰도 불투명, 광고 혼재 | 경험치 = 투명한 신뢰 지표 |

**근본적 차이**: 기존 플랫폼은 "리뷰를 모아서 보여주는 것"이 목적이다. Nyam은 "나를 위한 기록"이 목적이고, 그 기록이 자연스럽게 신뢰 네트워크로 흘러가는 구조다.

## 5. 타겟 유저

**Primary**: 25-40세, 주 2회 이상 외식, 기록 습관 있거나 만들고 싶은 사람
**Secondary**: 와인 입문~중급, 와인 모임 참여자

**타겟 유저의 전형적 행동**:
- 맛집 갈 때 인스타/블로그/지인에게 물어봄 → 신뢰할 만한 정보를 찾기 어려움
- 좋은 곳 발견해도 기록하지 않으면 잊어버림
- 모임에서 "거기 어땠어?" 물어보면 기억이 흐릿함
- 와인은 더 심함 — 라벨 사진만 남기고 맛 기억은 사라짐

## 6. Phase별 범위

### Phase 1 + Phase 2 통합 (현재 개발 범위)

> S1에서 P2 테이블 포함 전체 스키마 생성. 10 스프린트(68 태스크)로 진행.

**개인 기록**:
- 온보딩 (지역 선택 → 식당 등록 → 버블 생성/탐색 → 완료)
- 식당/와인 검색 및 등록 (텍스트 + 사진 + AI 인식)
  - 카메라 3모드: 개별(individual), 진열대(shelf), 영수증(receipt)
  - AI 이미지 인식 + OCR (와인 라벨, 진열대, 영수증)
  - EXIF GPS 검증
- 사분면 평가
  - 식당: X축=음식 퀄리티, Y축=경험 가치
  - 와인: X축=구조·완성도, Y축=즐거움·감성
  - 만족도 = (X + Y) / 2 자동 계산
- 와인 아로마 휠 (WSET Level 3 기준, 16섹터 3링: 1차향 9섹터 + 2차향 4섹터 + 3차향 3섹터)
- 와인 품질 평가 BLIC (Balance, Length/Finish, Intensity, Complexity) + autoScore
- 상세 페이지 (식당/와인) — 리뷰, 사분면, 사진, 버블 점수
- 홈 — 4종 뷰 (카드/리스트/캘린더/지도) + 필터/소팅 + 통계 차트
  - 식당 서브탭: 방문/위시리스트/팔로잉
  - 와인 서브탭: 시음/위시리스트/셀러
- 리스트 상태 4종: visited(방문), wishlist(위시), cellar(셀러), tasted(시음)
- 식당 장르 16종 + 장르 대분류 7개
- 와인 7종 타입 (레드/화이트/로제/스파클링/오렌지/주정강화/디저트)
- 페어링 8종 카테고리 (적색육/백색육/어패류/치즈/채소/매운·발효/디저트/샤퀴트리)
- 상황 태그 — 식당 6종(혼밥/데이트/친구/가족/회식/술자리), 와인 7종(혼술/데이트/모임/페어링/선물/테이스팅/디캔팅)
- 식당·와인 연결 기록 (linkedRestaurantId/linkedWineId)
- Discover (지역별 AI 분석/추천/랭킹 기반 식당 탐색)
- 경험치/레벨 (5축: category, area, genre, wine_variety, wine_region)
  - 기록 품질 XP (recordQualityXp)
  - 소셜 XP (공유/좋아요/팔로우/맞팔 — 일일 제한)
  - 보너스 XP (온보딩/첫 기록/첫 버블/첫 공유)
  - 마일스톤 시스템
- Wrapped (연간 결산) — 카테고리별 통계, 레벨, Top 아이템, 버블 활동
- 프라이버시 — 프로필 가시성(public/bubble_only/private), 기록 가시성(all/shared_only)
  - 가시성 설정 7개 키: score, comment, photos, level, quadrant, bubbles, price
- 알림 10종: level_up, bubble_join_request/approved, follow_request/accepted, bubble_invite, bubble_new_record, bubble_member_joined, reaction_like, comment_reply
- Edge Functions 3종 (주간 랭킹 스냅샷, 활성 XP 갱신, 계정 삭제 처리)

**소셜 (버블)**:
- 버블 생성/가입/공유 (누구나 무한 생성, 1인 버블 가능)
- 가입 정책 5종: invite_only, closed, manual_approve, auto_approve, open
- 경험치 기반 가입 조건 설정 (최소 기록 수 + 최소 레벨)
- 버블 피드, 댓글(익명 옵션), 리액션 5종(좋아요/저장/가고싶다/다녀왔다/불꽃)
- 자동 공유 시스템 — 필터 규칙 기반(all/filtered 모드), 소급 적용 + 수정 반영
- 콘텐츠 가시성 (rating_only/rating_and_comment)
- 멤버별 가시성 오버라이드 (7개 키)
- 멤버 역할 4종: owner, admin, member, follower
- 초대 코드 + 만료 기한
- 주간 랭킹 스냅샷 (크론)
- 맛 궁합 (tasteMatchPct)
- 팔로우/맞팔로우 (맞팔 = 풀 액세스, 콜드스타트 해결의 핵심)
  - 팔로우 (무료, XP 불필요) → 이름+점수 맛보기
  - 맞팔로우 → 풀 액세스 (리뷰, 사진, 팁, 상세 기록)
  - 버블 → 개인 팔로우 발견 채널
- 프로필 (미식 정체성, 레벨, 히트맵, 통계 차트, 버블 활동)
- 버블러 프로필 (타인 프로필 + 맛 궁합)
- 버블 간 연결/해지
- 알림 시스템

### Phase 3: 수익화 (추후)
- 프리미엄 통계/대시보드
- 크리에이터 채널 (유료 버블)
- 로컬 파트너십

> 수익화는 제품-시장 적합성 확인 이후 문제. 현재는 핵심 루프에 집중.

## 7. 기술 스택

- **Frontend**: Next.js (App Router) + TypeScript + Tailwind + shadcn/ui
- **Backend**: Supabase (PostgreSQL + Auth + RLS + Edge Functions + Storage)
- **AI/LLM**: Gemini Vision (사진 분석, OCR, 와인 검색/상세, 식당 분석/추천) — provider-agnostic 라우팅
- **외부 API**: 카카오맵, 네이버 지도, 구글 Places (식당), 구글 이미지 검색, Tavily (웹 검색)
- **모바일**: PWA 우선, 네이티브 전환 검토 (Phase 3+)

## 8. 성공 지표

| 지표 | Phase 1+2 목표 | 의미 |
|------|---------------|------|
| 온보딩 완료율 | > 80% | 진입 장벽이 낮은가 |
| D7 리텐션 | > 25% | 기록 습관이 형성되는가 |
| D30 기록 수 | >= 10개 | 지속적으로 기록하는가 |
| 평균 기록 시간 | < 10초 | 기록이 충분히 쉬운가 |
| 버블 생성률 | > 30% (활성 유저 중) | 소셜 루프가 작동하는가 |
| 버블 내 공유 기록 비율 | > 50% | 기록이 자연스럽게 공유로 이어지는가 |

## 9. 핵심 설계 원칙

1. **동기는 이기적으로**: 모든 기능은 "나에게 먼저 가치를 주는가?"가 최우선 기준. 공유는 부산물
2. **경험치는 신뢰의 증거**: 단순 활동량이 아닌, 기록의 성실도와 진정성을 반영해야 한다
3. **오염 방지가 생존 조건**: 경험치 허들 + 연결 해지로 정보 품질을 유지. 한 번 오염되면 플랫폼 가치가 붕괴한다
4. **연결은 유동적으로**: 버블 간 연결은 직관적이면서 언제든 해지 가능해야 한다. 경직된 구조는 신뢰를 떨어뜨린다
5. **기록이 쉬워야 모든 게 시작된다**: 기록에 10초 이상 걸리면 플라이휠이 돌지 않는다
6. **콜드스타트는 팔로우로 해결**: 기록 0, XP 0인 신규 유저도 공개 버블 팔로우 → 이름+점수 즉시 확인 가능. 기록하면 맞팔/멤버 승격으로 상세 데이터 접근
