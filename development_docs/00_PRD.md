# Nyam PRD (Product Requirements Document)

> 최종 갱신: 2026-04-13

---

## 1. 문제 정의

### Pain Point

> "신뢰할 만한 맛집/와인 리뷰가 없다."

기존 맛집 플랫폼(네이버, 망고플레이트, 캐치테이블 등)의 정보는 **출처를 알 수 없기 때문에 신뢰할 수 없다.** 별점 조작, 광고성 리뷰, 불특정 다수의 취향 평균화 — 정보가 많을수록 오히려 노이즈가 커지는 구조다.

사람들이 실제로 맛집을 고를 때 가장 의존하는 건 **아는 사람의 추천**이다. 하지만 지인 네트워크만으로는 정보량이 턱없이 부족하다.

### 왜 기존 방식은 실패하는가

| 문제 | 설명 |
|------|------|
| **하나의 거대 DB는 반드시 오염된다** | 플랫폼이 커질수록 광고, 악의적 리뷰, 무성의한 평점이 유입된다. 이 오염을 막을 구조적 방법이 없다 |
| **취향은 주관적이다** | 수만 명의 평균 별점은 "아무도의 취향"이다. 4.5점 식당이 나에겐 실망일 수 있다 |
| **이타적 기록은 지속되지 않는다** | 불특정 다수를 위해 반복적으로 기록하는 행위는 지속되지 않는다. 외재적 보상(쿠폰, 포인트)은 기록의 질을 떨어뜨린다 |

## 2. Solution Logic — Nyam의 해법

### 핵심 전제

> "거대 DB를 정화하는 건 불가능하다. 각자의 신뢰 네트워크를 구축해야 한다."

중앙화된 리뷰 DB가 아니라, **개인이 자기 기록을 쌓고, 취향이 맞는 사람을 선택적으로 팔로우**하는 구조. 내가 직접 검증한 사람들의 리스트만 모아서 나만의 신뢰 네트워크를 만든다.

### 핵심 메커니즘: 취향 적합도

두 사람이 같은 식당/와인을 각각 평가했을 때, 그 점수 패턴을 비교하면 **취향이 얼마나 비슷한지** 알 수 있다.

- **적합도(Similarity)**: 겹치는 기록의 평가 패턴이 얼마나 유사한가 (0~100%)
- **신뢰도(Confidence)**: 이 적합도 수치를 얼마나 믿을 수 있는가 (겹치는 기록 수에 비례)

겹치는 기록이 많을수록 적합도의 신뢰도가 올라간다. → **기록을 더 할수록 내 추천 네트워크가 정교해진다.**

### 플라이휠

```
[1단계] 나를 위한 기록
│
│  가본 식당, 마신 와인을 기록한다
│  → 시작은 철저히 나 자신을 위한 것
│  → 잊지 않기 위해, 내 취향을 파악하기 위해
│
▼
[2단계] 겹치는 기록 → 적합도 계산 → 팔로우로 신뢰 부스트
│
│  기록이 쌓이면 다른 유저와 겹치는 기록이 자연스럽게 생긴다
│  → 전체 유저와의 취향 적합도가 자동 산출된다
│  → 적합도가 높은 사람을 팔로우하면 신뢰 부스트가 적용된다
│  → 팔로우 = 내가 검증한 신뢰를 더하는 행위
│
▼
[3단계] 기록 → 네트워크 정밀도 → 더 나은 추천 → 더 많은 기록
│
│  내가 기록을 더 할수록 겹치는 데이터가 늘어난다
│  → 적합도의 신뢰도가 올라간다
│  → 추천이 더 정확해진다
│  → 더 정확한 추천을 받기 위해 더 기록한다
│
▼
[선순환] 기록 → 겹치는 데이터 증가 → 적합도 정확도 상승 → 더 좋은 추천 → 더 많은 기록
```

### 신뢰 메커니즘

| 요소 | 역할 |
|------|------|
| **취향 적합도** | 신뢰의 핵심 지표. 겹치는 기록을 대조해서 산출하는 취향 유사도 |
| **신뢰도** | 적합도의 정확성 지표. 겹치는 기록의 양(수량)과 질(다양성) 모두 반영 |
| **팔로우** | 내가 직접 선택하는 신뢰 네트워크. 적합도 기반으로 확장/정리 |
| **언팔로우** | 오염 방지의 핵심. 특정인이 오염원이면 즉시 끊으면 끝. 시스템이 자정됨 |
| **버블** | 큐레이션 리스트. 개인 또는 공동으로 관리하며, 독립적으로 팔로우 가능. 맥락화된 예측 레이어 (§7 참조) |
| **경험치(XP)** | 기록 성실도의 증거. 버블 가입 자격, 프로필 신뢰 시그널 |

### 오염 방지

기존 플랫폼은 오염을 **플랫폼 차원**에서 막으려다 실패한다. Nyam은 오염 방지를 **개인 차원**으로 내린다:

- 내 네트워크는 내가 직접 구성한다 (팔로우/언팔)
- 시스템이 적합도 변화를 추적하여 이상 신호를 감지한다
- 오염원을 끊으면 내 추천에 즉시 반영된다
- 거대 DB 전체를 정화할 필요가 없다 — 각자의 네트워크만 건강하면 된다
- **겹침 다양성 보정**: 프랜차이즈에서만 겹치는 유저의 신뢰도를 자동 감쇠 — 프랜차이즈 대량 리뷰로 적합도를 부풀린 공격을 구조적으로 차단

### 한 문장 요약

> **"각자의 기록이 겹치는 데이터를 통해 취향 적합도를 계산하고, 적합도가 높은 사람들의 리스트를 모아 나만의 신뢰할 수 있는 맛집/와인 네트워크를 구축하는 구조."**

## 3. 점수 체계

Nyam은 세 가지 층위의 점수를 제공한다.

| 층위 | 이름 | 성격 | 산출 방식 |
|------|------|------|----------|
| **1** | 내 점수 | 절대적, 주관적 사실 | 사분면 좌표 → 만족도 자동 계산 |
| **2** | Nyam 점수 | 개인화된 예측 | 전체 유저 CF (팔로잉 부스트) |
| **3** | 버블 점수 | 맥락화된 예측 | 버블 멤버 CF (팔로잉 부스트) |

- **내 점수**: 내가 직접 경험하고 평가한 점수. 있으면 무조건 최우선 표시
- **Nyam 점수**: 아직 가보지 않은 곳의 "나에게 맞을 확률" 예측. **유저마다 다른 점수**가 나옴
- **버블 점수**: 특정 버블 멤버들로만 산출되는 점수. 같은 식당이라도 어떤 버블이냐에 따라 다름 (§7 참조)

> 네이버/구글 평점은 모든 유저의 산술평균이므로 누구에게나 같은 숫자가 보인다. Nyam은 Collaborative Filtering(CF) 기반이므로 **같은 식당이라도 유저마다 다른 예측점수**가 나온다. 이것이 근본적 차이다.

### 3.1 용어 정의

| 용어 | 대상 | 의미 |
|------|------|------|
| **적합도** (similarity) | 나 ↔ 특정 유저 | 겹치는 기록의 평가 패턴이 얼마나 유사한가 (0~100%) |
| **신뢰도** (confidence) | 나 ↔ 특정 유저 | 적합도 수치를 얼마나 믿을 수 있는가 (겹치는 기록 수에 비례) |
| **예측 확신도** (prediction confidence) | 나 → 특정 식당/와인 | 이 예측점수를 얼마나 믿을 수 있는가 |

### 3.2 적합도 (Taste Similarity)

겹치는 기록의 **사분면 2D 좌표(X, Y, 0~100)**를 비교한다. Mean-centering 필수 (시뮬레이션 검증: +19점 개선).

```
1. Mean-centering: 각 유저의 평균 점수를 빼서 편향 제거
   (후한 사람 vs 박한 사람 차이를 보정 → 상대적 취향 패턴만 비교)

2. 보정된 좌표 간 2D 유클리드 거리 계산 (겹치는 각 식당/와인마다)

3. 평균 거리를 0~100% 적합도로 변환
   적합도 = max(0, 1 - 평균거리 / D)     (D = 60)
```

**식당 적합도와 와인 적합도는 별개 산출**: 식당 취향은 맞는데 와인 취향은 안 맞을 수 있다.

> D=60은 시뮬레이션에서 클러스터 분리 p<0.005 + 예측 MAE 최적으로 확정 (100점 스케일 기준).

### 3.3 신뢰도 (Confidence)

겹치는 기록의 **양(수량)**과 **질(다양성)**을 모두 반영한다.

```
신뢰도 = 기본 신뢰도 × 겹침 다양성 비율

기본 신뢰도 = n / (n + λ)     (λ = 7)
겹침 다양성 비율 = 니치 겹침 수 / 전체 겹침 수
```

- **기본 신뢰도**: 겹치는 기록이 많을수록 비교 결과를 믿을 수 있다 (데이터 신뢰도)
- **겹침 다양성 비율**: 프랜차이즈(기록자 전체 유저의 10% 초과)에서만 겹치면 취향 판별 가치가 낮다. 니치 식당에서 겹쳐야 진짜 취향이 비슷한 것이다 (취향 신뢰도)

| 상대 유저 | 전체 겹침 | 니치 겹침 | 다양성 비율 | 기본 신뢰도 | 최종 신뢰도 |
|---|---|---|---|---|---|
| 니치 다양 | 20개 | 15개 | 75% | 74% | **56%** |
| 프랜차이즈만 | 50개 | 0개 | 0% | 88% | **0%** |
| 대부분 프랜차이즈 | 30개 | 3개 | 10% | 81% | **8%** |

> 시뮬레이션 검증: 공격자 500명(기록자의 95%)이 프랜차이즈만 정직 리뷰 + 타겟 식당 100점 조작 시, 겹침 다양성 보정 적용으로 점수 부풀림 +12.25 → +0.20으로 완전 무력화. 일반 예측 정확도 부작용 없음 (MAE 변화 -0.1%).

### 3.4 Nyam 점수 산출 (User-based Collaborative Filtering)

팔로잉뿐 아니라 **전체 유저**를 대상으로 CF를 수행한다. 팔로잉은 가중치 부스트로 반영된다.

```
가중치ᵢ = 적합도ᵢ × 신뢰도ᵢ × 관계 부스트ᵢ
예측 = 내 평균 + Σ(가중치ᵢ × (상대 점수ᵢ - 상대 평균ᵢ)) / Σ|가중치ᵢ|
```

| 관계 | 관계 부스트 | 이유 |
|------|-----------|------|
| 내가 팔로우 (맞팔 포함) | ×1.5 | 내가 선택한 신뢰 |
| 비팔로우 | ×1.0 | 순수 알고리즘 |

**팔로잉의 역할**: 팔로잉은 추천의 전제조건이 아니라 **신뢰 부스트 레이어**다. 팔로잉 0명이어도 전체 유저 CF로 점수가 산출된다. 팔로잉이 많을수록 내 네트워크 비중이 자연스럽게 커져서 예측이 더 정확해진다. 단, 부스트 효과는 미세 조정 수준이다 — 평가자 중 팔로우 비율이 ~10%이므로, 극적 차이가 아니라 "내 사람 의견을 살짝 더 반영"하는 역할. 맞팔 여부로 부스트를 나누지 않는 이유: 팔로우는 "나는 이 사람의 취향을 신뢰한다"는 일방적 선언이며, 상대가 나를 팔로백한다고 해서 상대의 점수가 나에게 더 유용해지는 건 아니다.

> 높은 부스트(2.0+)는 오염 유저를 실수로 팔로우했을 때 피해가 커지므로 보수적 값(1.5/1.0)으로 확정.

### 3.5 예측 확신도 (Prediction Confidence)

Nyam 점수가 "얼마나 믿을 만한가"를 나타내는 지표.

```
확신도 = n_factor × 0.50 + agreement × 0.35 + quality × 0.15

  n_factor  = 유효평가자수 / (유효평가자수 + 7)    ← 가중치 > 0.05인 평가자만
  agreement = 1 - 가중편차표준편차 / 2              ← mean-centered 편차의 일치도
  quality   = 평균가중치 / (평균가중치 + 0.3)       ← 적합도×신뢰도 품질
```

**표시 기준** — 점수는 항상 보여준다. 숨기지 않는다. 확신도를 **%로 투명하게 표시**하여 유저가 직접 판단:

```
Nyam 82  확신 87%          ← 높으면 신뢰
Nyam 74  확신 12%          ← 낮으면 유저가 알아서 감안
```

라벨 없이 % 숫자만으로 충분. 유저가 "87% vs 12%"를 보고 스스로 판단하게 한다. 점수를 숨기면 유저가 판단할 기회 자체를 잃는다.

> 시뮬레이션에서 단조 감소 확인 (확신도↑ → 오차↓). 확신 87%일 때 MAE ~4점, 확신 12%일 때 MAE ~6점.

### 3.6 버블 점수

**Nyam 점수와 동일한 CF 엔진이지만, 스코프와 파라미터가 다르다.** 버블 가입 자체가 "이 멤버들을 신뢰한다"는 의사 표시이므로, 공격 방어 로직 없이 적은 데이터에서도 점수를 산출한다.

| | Nyam 점수 | 버블 점수 |
|---|---|---|
| 대상 | 전체 유저 | 버블 멤버만 |
| 다양성 보정 | 적용 (공격 방어) | **미적용** |
| 최소 겹침 | 3개 | **0개** (겹침 불필요) |
| 최소 기록자 | 3명 | **1명** |
| 팔로우 부스트 | ×1.5 / ×1.0 | **×1.0 균등** (멤버 간 차등 없음) |
| 산출 방식 | CF 가중 예측 | 1명이면 그대로, 2명+이면 CF |

**멤버 1명만 기록한 경우:** 해당 멤버의 점수를 그대로 표시. 확신도는 낮지만 점수는 보여준다.

같은 식당이라도 버블마다 다른 점수가 나온다 — 이것이 버블 점수의 핵심 가치다 (§7 참조):
```
와인동호회  87  확신 72%  5명 ← "내추럴와인 관점에서 훌륭"
가성비버블  63  확신 58%  12명 ← "가성비로는 별로"
대학동기    91  확신 45%  3명 ← "친구들은 좋아함"
```

> 버블 점수는 글로벌 Nyam 점수보다 MAE가 높다 (데이터 적으므로 당연). 버블의 가치는 정확도가 아니라 **맥락** 제공.

### 3.7 유저에게 보여주는 방식

3종 점수 카드(나/Nyam/버블)를 보여주고, 확신도를 %로 투명하게 공개한다.

```
┌──────┐  ┌──────────┐  ┌──────────┐
│  나   │  │   Nyam   │  │   버블    │
│  85  │  │  82      │  │  78      │
│ 2회   │  │ 확신 87% │  │ 확신 64% │
└──────┘  └──────────┘  └──────────┘
```

**Nyam 카드 클릭 → 점수 근거 펼침:**
```
평가자 23명 기반

팔로잉 (가중 ×1.5)
  @kim  적합도 91% → 85점
  @lee  적합도 84% → 80점

유사 유저
  21명 평균 적합도 68% → 81점
```

**버블 카드 클릭 → 버블별 점수 펼침:**
```
와인동호회  87  확신 72%  5명
강남맛집    79  확신 58%  12명
대학동기    91  확신 45%  3명
```

- 버블 카드에는 확신도 최고 버블의 점수를 대표로 표시
- 가입한 버블 중 기록 1건+ 있는 것만 표시
- 적합도/신뢰도도 투명 공개: "적합도 87% (공통 기록 12개)"
- 신뢰도가 낮으면 → "더 기록하면 정확해진다"는 메시지 → 자연스러운 기록 동기

## 4. 제품 정의

**Nyam은**: 내 식도락 기록을 바탕으로, 취향이 맞는 사람들의 네트워크를 구축하여 신뢰할 수 있는 맛집/와인 추천을 받는 앱

**Nyam이 아닌 것**:
- 네이버/카카오 같은 공개 리뷰 플랫폼이 아님 — 하나의 거대 DB가 아니라 각자의 신뢰 네트워크
- 캐치테이블 같은 예약 서비스가 아님
- 배달/요리 앱이 아님
- **범위: 식당 + 와인** (배달, 요리, 카페 제외)

### 핵심 가치

| 순서 | 가치 | 설명 |
|------|------|------|
| 1 | **기록** | 사분면 평가로 식당/와인 경험을 구조화. 나를 위한 기록 |
| 2 | **적합도** | 겹치는 기록을 대조하여 취향이 맞는 사람을 찾는다 |
| 3 | **신뢰 네트워크** | 적합도 높은 사람들을 팔로우하여 나만의 추천 소스를 구축 |
| 4 | **추천** | 전체 유저 CF + 팔로잉 신뢰 부스트로 아직 가보지 않은 곳의 점수를 예측 |

### 시그니처 기능: 사분면 평가

- 2D 좌표(특성 2축) + 자동 계산 만족도 + 상황 태그
  - 식당: X축=음식 퀄리티, Y축=경험 만족도
  - 와인: X축=구조·완성도, Y축=경험 만족도
  - 만족도 = (X + Y) / 2 자동 계산 (별도 입력 아님)
  - 점 크기는 고정 20px
- 기록이 쌓이면 사분면 위에 "나만의 취향 지도" 자동 생성 (참조 점 opacity 0.3, 최대 8~12개)
- **적합도 계산의 핵심 데이터**: 이 2D 좌표가 취향 비교의 기반이 된다

## 5. 차별점

| 기존 앱 | 문제 | Nyam의 접근 |
|---------|------|-------------|
| 네이버/카카오 | 하나의 거대 DB, 출처 불명, 오염 불가피 | 각자의 신뢰 네트워크. 오염되면 언팔로 즉시 정리 |
| Vivino | 불특정 다수 평점, 와인만 | 식당+와인 통합, 취향 적합도 기반 추천 |
| 인스타그램 | 비구조화, 검색 불가, 과시 중심 | 구조화된 기록, 적합도 계산 가능한 데이터 |
| 망고플레이트 | 리뷰어 신뢰도 불투명, 광고 혼재 | 겹치는 기록으로 직접 검증된 취향 유사도 |

**근본적 차이**: 기존 플랫폼은 "리뷰를 하나의 DB에 모아서 평균내는 것"이 구조다. Nyam은 "각자의 기록을 유지하고, 취향이 맞는 사람의 리스트만 선택적으로 참조하는 것"이 구조다. 오염 방지를 플랫폼이 아니라 개인이 통제한다.

## 6. 타겟 유저

**Primary**: 25-40세, 주 2회 이상 외식, 기록 습관 있거나 만들고 싶은 사람
**Secondary**: 와인 입문~중급, 와인 모임 참여자

**타겟 유저의 전형적 행동**:
- 맛집 갈 때 인스타/블로그/지인에게 물어봄 → 신뢰할 만한 정보를 찾기 어려움
- 좋은 곳 발견해도 기록하지 않으면 잊어버림
- 모임에서 "거기 어땠어?" 물어보면 기억이 흐릿함
- 와인은 더 심함 — 라벨 사진만 남기고 맛 기억은 사라짐
- 네이버 4.5점 보고 갔는데 실망한 경험이 반복됨

## 7. 버블의 역할

### 위치: 큐레이션 리스트 + 맥락화된 예측

버블은 Nyam의 **큐레이션 리스트**이다. 혼자 관리하는 개인 리스트일 수도, 여럿이 함께 채워가는 공동 리스트일 수도 있다. Spotify 플레이리스트처럼 누구나 자유롭게 만들고, 다른 사람의 리스트를 팔로우할 수 있다.

**버블은 동호회가 아니다.** 실시간 타임라인을 소비하는 소셜 미디어가 아니라, 내가 어디 갈지 고민할 때 꺼내보는 **목적별 식당/와인 큐레이션**이다.

| 역할 | 설명 |
|------|------|
| **개인 큐레이션** | 나만의 목적별 리스트 (1인 버블) — "2026 미슐랭", "을지로 혼밥", "와인바 투어" |
| **공동 큐레이션** | 여럿이 함께 채우는 리스트 — "우리팀 점심", "와인 버블" |
| **구독** | 다른 사람의 리스트를 팔로우 — 전체 팔로우 대신 관심 리스트만 선택적 구독 |
| **사람 발견** | 리스트 멤버/기여자 구경 → 취향 적합도 확인 → 개인 팔로우 |
| **맥락화된 예측** | 리스트 멤버로 스코프를 한정한 CF 점수 → 특정 맥락에서의 예측 (§3.6 참조) |

### 리스트에 담기는 것

리스트(버블)에는 **대상(식당/와인)**이 담긴다.

```
리스트에 보이는 대상:
├── 자동: 필터 조건에 매칭된 기록
└── 수동: FAB "버블에 추가"로 직접 추가한 대상
```

| 모드 | 동작 | 예시 |
|------|------|------|
| **자동** | 필터 조건 설정 → 조건 맞는 기록 자동 추가/제거 | "광화문 식당, 만족도 90+" |
| **수동** | FAB → 버블에 추가 → 아이템 선택 → 버블 선택 | "2026 미슐랭"에 하나씩 담기 |
| **혼합** | 자동 규칙 + 수동 추가 공존 | 자동으로 채워지되, 수동으로도 추가 가능 |

자동 모드는 기존 홈의 filter-matcher 로직을 재사용한다. 소급 적용 + 조건 변경 시 자동 정리.

### 찜(bookmark) → 버블 전환

**찜 기능은 제거**됨. 찜의 역할은 버블이 대체한다:
- 개인 큐레이션 = 비공개 버블 (1인 버블)
- 목적별 분류 = 버블 이름으로 자연스럽게 해결 ("점심 맛집", "미슐랭 모음" 등)
- FAB Speed Dial "버블에 추가" = 기존 찜의 저장 행위를 대체 (2탭: 아이템 선택 → 버블 선택)
- 버블 포크(복제) = 다른 사람의 리스트를 내 것으로 가져오기

### 버블 = 맥락(Context)

버블의 성격에 따라 반영하는 맥락이 다르다:

| 버블 유형 | 반영하는 맥락 | 예시 |
|-----------|-------------|------|
| 장르/품종 | 관심사·전문성 | 내추럴와인, 오마카세, 이탈리안 |
| 지역 | 생활권 | 성수동맛집, 제주도민 |
| 인간관계 | 사회적 신뢰 | 대학동기, 회사팀 |
| 가치관 | 평가 기준 | 가성비파, 분위기파, 비건 |
| 개인 목적 | 자기 관리 | 2026 미슐랭, 데이트 코스, 와인바 투어 |

같은 식당이라도 버블마다 다른 점수가 나온다 (공동 리스트인 경우):
- 내추럴와인 버블 9.1 → "내추럴와인 관점에서 여긴 훌륭하다"
- 가성비 버블 6.3 → "가성비 관점에선 별로"
- 대학동기 버블 9.0 → "내 친구들은 여길 좋아한다"

### 팔로우 vs 버블

| | 개인 팔로우 | 버블 팔로우 |
|---|---|---|
| 단위 | 사람 1명의 전체 기록 | 목적별 리스트 하나 |
| 핵심 목적 | 신뢰 부스트 (추천 정확도 향상) | 특정 주제/맥락의 큐레이션 구독 |
| 데이터 활용 | CF 가중치 부스트 | 멤버로 스코프를 한정한 CF |
| 핵심 가치 | 내가 검증한 신뢰 네트워크 | 목적별 큐레이션, 사람 발견, 맥락별 점수 |
| 선택적 구독 | 불가 (전체 기록) | 가능 (리스트 단위로 골라서 팔로우) |

### 버블 기능

- 생성/가입/공유 (누구나 무한 생성, 1인 버블 가능)
- 가입 정책 5종: invite_only, closed, manual_approve, auto_approve, open
- 경험치 기반 가입 조건 설정 (최소 기록 수 + 최소 레벨)
- 리스트 뷰 (대상 목록) + 피드 뷰 (기록 시간순)
- 댓글(익명 옵션), 리액션 5종(좋아요/저장/가고싶다/다녀왔다/불꽃)
- 자동/수동 큐레이션 — 자동: 필터 규칙 기반(기록 매칭), 수동: FAB "버블에 추가"로 대상 직접 추가
- 콘텐츠 가시성 (rating_only/rating_and_comment)
- 멤버별 가시성 오버라이드 (7개 키)
- 멤버 역할 4종: owner, admin, member, follower
- 초대 코드 + 만료 기한
- 주간 랭킹 스냅샷 (크론)
- 맛 궁합 (tasteMatchPct)

## 8. Phase별 범위

### Phase 1 + Phase 2 통합 (현재 개발 범위)

> S1에서 P2 테이블 포함 전체 스키마 생성. 10 스프린트(68 태스크)로 진행.

**개인 기록**:

  - AI 이미지 인식 + OCR (와인 라벨, 진열대, 영수증)
  - EXIF GPS 검증
- 사분면 평가
  - 식당: X축=음식 퀄리티, Y축=경험 만족도
  - 와인: X축=구조·완성도, Y축=경험 만족도
  - 만족도 = (X + Y) / 2 자동 계산
- 와인 아로마 휠 (WSET Level 3 기준, 16섹터 3링: 1차향 9섹터 + 2차향 4섹터 + 3차향 3섹터)
- 와인 품질 평가 BLIC (Balance, Length/Finish, Intensity, Complexity) + autoScore
- 상세 페이지 (식당/와인) — 리뷰, 사분면, 사진, 적합도 기반 점수
- 홈 — 4종 뷰 (카드/리스트/캘린더/지도) + 필터/소팅 + 통계 차트
  - 보기 필터: 내 기록(기본 ON) / 팔로잉 / 버블 (멀티셀렉, 미선택=전체)
  - FAB Speed Dial: 기록 추가 / 버블에 추가
- 식당 장르 16종 + 장르 대분류 7개
- 와인 7종 타입 (레드/화이트/로제/스파클링/오렌지/주정강화/디저트)
- 페어링 8종 카테고리 (적색육/백색육/어패류/치즈/채소/매운·발효/디저트/샤퀴트리)
- 상황 태그 — 식당 6종(혼밥/데이트/친구/가족/회식/술자리), 와인 7종(혼술/데이트/모임/페어링/선물/테이스팅/디캔팅)
- 식당·와인 연결 기록 (linkedRestaurantId/linkedWineId)
- Discover (지역별 AI 분석/추천/랭킹 기반 식당 탐색)
- 경험치/레벨 (5축: category, area, genre, wine_variety, wine_region)
  - 기록 품질 XP (recordQualityXp)
  - 소셜 XP (공유/좋아요/팔로우/맞팔 — 일일 제한)
  - 보너스 XP (온보딩/첫 기록/첫 버블/첫 공유)
  - 마일스톤 시스템
- Wrapped (연간 결산) — 카테고리별 통계, 레벨, Top 아이템, 버블 활동
- 프라이버시 — 프로필 공개(is_public) + 팔로우 정책(4종)
  - 가시성 설정 7개 키: score, comment, photos, level, quadrant, bubbles, price
- 알림 10종: level_up, bubble_join_request/approved, follow_request/accepted, bubble_invite, bubble_new_record, bubble_member_joined, reaction_like, comment_reply
- Edge Functions 3종 (주간 랭킹 스냅샷, 활성 XP 갱신, 계정 삭제 처리)

**소셜 — 팔로우 네트워크 (핵심)**:
- 팔로우/맞팔로우 (Nyam의 핵심 소셜 축)
  - 팔로우 → 이름 + 적합도 점수 확인
  - 맞팔로우 → 풀 액세스 (리뷰, 사진, 팁, 상세 기록)
  - 겹치는 기록 대조 → 취향 적합도 자동 산출
  - 신뢰도(겹치는 기록 수) 투명 공개
  - 시스템이 적합도 변화 추적 → 이상 신호 감지
- CF 기반 추천 엔진
  - User-based Collaborative Filtering (전체 유저 대상, 팔로잉 부스트)
  - Mean-centered Euclidean distance (2D 사분면 데이터)
  - 3-tier 점수: 내 점수 / Nyam 점수 / 버블 점수
  - 예측 확신도 (prediction confidence)
- 프로필 (미식 정체성, 레벨, 히트맵, 통계 차트, 버블 활동)
- 버블러 프로필 (타인 프로필 + 취향 적합도)

**소셜 — 버블 (큐레이션 리스트)**:
- 버블 생성/가입/공유 (누구나 무한 생성, 1인 버블 = 개인 리스트)
- 가입 정책 5종: invite_only, closed, manual_approve, auto_approve, open
- 경험치 기반 가입 조건 설정 (최소 기록 수 + 최소 레벨)
- 자동/수동 큐레이션 — 자동: 필터 규칙 기반(기록 매칭), 수동: FAB "버블에 추가"로 대상 직접 추가
- 리스트 뷰 (대상 목록) + 피드 뷰 (기록 시간순)
- 댓글(익명 옵션), 리액션 5종
- 멤버 역할 4종: owner, admin, member, follower
- 초대 코드 + 만료 기한
- 주간 랭킹 스냅샷 (크론)
- 맛 궁합 (tasteMatchPct)
- 버블 간 연결/해지
- 알림 시스템

## 11. 핵심 설계 원칙

1. **신뢰는 적합도에서 나온다**: 겹치는 기록을 대조해서 검증된 취향 유사도만이 진짜 신뢰다. 뱃지, 별점, 리뷰 수가 아니다
2. **오염 방지는 개인이 통제한다**: 거대 DB를 정화하려 하지 않는다. 각자가 자기 네트워크를 팔로우/언팔로 관리한다
3. **기록은 내 네트워크를 정교하게 하는 행위다**: 기록할수록 겹치는 데이터가 늘고, 적합도 신뢰도가 올라가고, 추천이 정확해진다
4. **동기는 이기적으로**: 모든 기능은 "나에게 먼저 가치를 주는가?"가 최우선 기준. 공유는 부산물
5. **기록이 쉬워야 모든 게 시작된다**: 기록에 10초 이상 걸리면 플라이휠이 돌지 않는다
6. **연결은 유동적으로**: 팔로우/언팔은 자유롭고 즉각적이어야 한다. 오염원 감지 시 끊으면 끝
7. **콜드스타트는 전체 유저 CF로 해결**: 팔로잉 0명이어도 전체 유저 CF로 예측 가능. 팔로잉이 늘수록 신뢰 부스트로 정확도 상승. 버블은 폴백이 아니라 큐레이션 + 맥락화된 예측 레이어
