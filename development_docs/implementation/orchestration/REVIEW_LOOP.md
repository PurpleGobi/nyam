# Review Loop

> 검증 루프 + 품질 게이트. 모든 태스크 완료 시, 스프린트 완료 시 이 루프를 실행한다.

---

## 1. 태스크 완료 검증 (매 태스크마다)

### 체크리스트

```
□ 빌드 통과          pnpm build 에러 없음
□ 린트 통과          pnpm lint 경고 0개
□ 타입 체크          TypeScript strict, any/as any/@ts-ignore 없음
□ 레이어 규칙        R1~R5 위반 없음 (아래 검증 방법 참조)
□ 문서 정합성        구현이 SSOT 문서와 일치
□ 목업 정합성        UI가 prototype/*.html과 일치
□ 보안 점검          RLS 우회 없음, 키 노출 없음
□ 모바일 확인        360px에서 레이아웃 깨짐 없음
```

### 레이어 규칙 검증 방법

```bash
# R1: domain에 React/Supabase/Next import 없는지
grep -r "from 'react\|from '@supabase\|from 'next" src/domain/ && echo "❌ R1 위반" || echo "✅ R1 통과"

# R2: infrastructure가 domain 인터페이스를 구현하는지 (수동 확인)
# 각 infrastructure/repositories/*.ts에 implements XxxRepository가 있어야 함
grep -rL "implements" src/infrastructure/repositories/ && echo "⚠️ R2 확인 필요: implements 없는 파일" || echo "✅ R2 통과"

# R3: application에서 infrastructure 구현체 직접 import 없는지
grep -r "from '.*infrastructure" src/application/ && echo "❌ R3 위반" || echo "✅ R3 통과"

# R4: presentation에서 Supabase/infrastructure 직접 호출 없는지
grep -r "from '@supabase\|from '.*infrastructure" src/presentation/ && echo "❌ R4 위반" || echo "✅ R4 통과"

# R5: app/ 페이지가 Container 렌더링만 하는지
# 수동 확인 — page.tsx에 비즈니스 로직이 없어야 함
```

## 2. 스프린트 완료 검증 (각 스프린트 완료 시마다)

> 웨이브 병렬 실행 시: 같은 웨이브 내 여러 스프린트가 병렬 완료될 수 있다.
> 각 스프린트는 **개별 완료 시점에** 이 검증을 실행한다.
> 웨이브 내 모든 스프린트 완료 후 → 다음 웨이브 시작 전에 **통합 회귀 테스트** 추가 실행.

### 통합 체크리스트

```
□ 모든 태스크 검증 통과
□ pnpm build 성공 (프로덕션 빌드)
□ 스프린트 산출물 전체 확인 (MASTER_TRACKER의 "검증" 컬럼 기준)
□ 이전 스프린트 기능 회귀 없음 (병렬 세션의 머지 후 재확인)
□ DECISIONS_LOG에 주요 결정 기록됨
□ MASTER_TRACKER 갱신 완료
```

### 회귀 테스트 범위

| 현재 스프린트 | 회귀 확인 대상 |
|-------------|--------------|
| S2 완료 후 | S1: 인증 로그인/로그아웃, 디자인 토큰 적용 |
| S3 완료 후 | S2: 기록 생성 플로우 정상 동작 |
| S4 완료 후 | S2: 기록 저장, S3: 검색→선택 |
| S5 완료 후 | S4: 상세 페이지 접근, S2: 기록 플로우 |
| S6 완료 후 | S2: 기록→XP 적립 연결 |
| S7 완료 후 | S6: XP/프로필 표시 |
| S8 완료 후 | S4: 상세 L1~L8, S5: 홈 탭, S7: 버블 기능 |
| S9 완료 후 | **전체 E2E 플로우** |

## 3. 정합성 검증 루프 (수시)

### 문서-코드 정합성

```
1. 현재 구현 중인 기능의 SSOT 문서를 읽는다
2. 코드가 문서의 스펙과 일치하는지 대조한다
3. 불일치 발견 시:
   a. 문서가 맞고 코드가 틀림 → 코드 수정
   b. 문서가 명백히 틀림 → 사용자에게 알리고 확인 후 수정
   c. 문서에 없는 기능 → 구현하지 않음 (스코프 크립 방지)
```

### 목업-코드 정합성

```
1. prototype/*.html에서 해당 페이지 목업을 확인한다
2. UI 요소, 레이아웃, 인터랙션이 목업과 일치하는지 대조한다
3. 컬러, 간격, 폰트 등 디자인 토큰 사용 확인
4. 불일치 시 코드를 목업에 맞춘다
```

## 4. 크리티컬 게이트 (절대 통과 조건)

> 이 조건 중 하나라도 실패하면 다음 태스크로 넘어가지 않는다.

| 게이트 | 조건 | 검증 방법 |
|--------|------|----------|
| **빌드** | `pnpm build` 성공 | CLI 실행 |
| **타입** | `any` 사용 0개 | grep 검색 |
| **보안** | RLS 우회 없음 | 코드 리뷰 |
| **레이어** | R1~R5 위반 없음 | grep + 수동 확인 |
| **디자인** | 하드코딩 컬러 없음 | grep `bg-white`, `text-black` 등 |

## 5. 검증-수정 반복 루프 (최대 10회)

적절한 규모의 구현이 완료되면 아래 루프를 **문제가 0건이 될 때까지** 반복한다.

> **핵심 원칙: 매 회차마다 이전 검토 결과를 신뢰하지 않는다.**
> 수정이 다른 곳에 부작용을 만들 수 있으므로, 매 회차를 처음 검토하듯 전체를 다시 확인한다.
> "이전 회차에서 통과했으니 이번에는 스킵" 하지 않는다.

```
[회차 1] 크리티컬 게이트 전체 실행
  │
  ├── §1 체크리스트 8항목 실행
  ├── SSOT 문서 다시 읽고 → 구현 누락/불일치 대조
  ├── prototype/*.html → UI 차이 대조
  │
  ├── 문제 발견 시:
  │   ├── 빌드/타입 에러 → 즉시 수정
  │   ├── 레이어 위반 → 올바른 레이어로 이동
  │   ├── SSOT 불일치 → 코드를 문서에 맞춤 (문서가 틀리면 사용자 확인)
  │   ├── 목업 불일치 → 코드를 목업에 맞춤
  │   ├── 보안 이슈 → 즉시 수정 (최우선)
  │   └── 회귀 → 원인 파악, 영향 범위 확인 후 수정
  │
  ├── → [회차 2~N] 처음부터 전체 재검토 (이전 통과 항목도 다시 실행)
  ├── → 반복...
  │
  ├── 문제 0건 → 루프 종료 → 태스크 done
  └── 10회 도달 → 사용자에게 잔여 이슈 목록 보고
```

### 루프 트리거 시점

| 시점 | 루프 범위 |
|------|----------|
| 태스크 1개 완료 | 해당 태스크의 산출물 |
| 관련 태스크 묶음 완료 (예: 2.2~2.8) | 묶음 전체 산출물 |
| 스프린트 완료 | 스프린트 전체 + 회귀 테스트 |
| 웨이브 완료 (병렬 세션 머지 후) | 머지된 코드 전체 통합 검증 |
