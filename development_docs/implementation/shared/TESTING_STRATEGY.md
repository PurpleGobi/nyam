# Testing Strategy

> 스프린트별 테스트 전략과 검증 체크포인트.

---

## 테스트 레벨

### Level 1: 빌드 검증 (매 태스크)
```bash
pnpm build    # 프로덕션 빌드 성공
pnpm lint     # 경고 0개
```

### Level 2: 수동 검증 (매 태스크)
- 브라우저에서 `localhost:7911` 접속
- 해당 기능 직접 동작 확인
- 360px 모바일 뷰포트에서 확인
- 콘솔 에러 없음 확인

### Level 3: 통합 검증 (스프린트 완료 시)
- 이전 스프린트 기능 회귀 테스트
- 플로우 간 연결 확인 (예: 검색→기록→상세→홈)
- RLS 정책 동작 확인

### Level 4: E2E 검증 (S9 최종)
- 온보딩→기록→공유→프로필 전체 시나리오
- 다중 사용자 시나리오 (버블, 팔로우)

---

## 스프린트별 검증 시나리오

### S1 검증
```
□ pnpm dev → localhost:7911 접속 성공
□ Supabase 연결 확인 (DB 쿼리 응답)
□ 소셜 로그인 → 세션 생성 → 로그아웃
□ RLS: 인증 없이 데이터 접근 차단됨
□ 디자인 토큰: --accent-food, --accent-wine, --bg 등 CSS 변수 적용됨 (DESIGN_SYSTEM.md §1)
□ 모바일 레이아웃: 360px에서 깨짐 없음
```

### S2 검증
```
□ 사분면 드래그 → 좌표 저장 → 조회 시 같은 위치
□ 만족도 게이지 1~100 범위 제한 동작
□ 아로마 팔레트 15섹터 3링 선택/해제
□ 씬태그 단일 선택 (식당 6종: solo/romantic/friends/family/business/drinks)
□ Phase 1 저장 → Phase 2 추가 정보 저장
□ 저장된 기록 DB 확인
```

### S3~S9
> 세부 지침 작성 시 각 스프린트별 시나리오를 구체화한다.

---

## 검증 실패 대응

| 유형 | 대응 | 우선순위 |
|------|------|---------|
| 빌드 실패 | 즉시 수정, 다음 작업 금지 | P0 |
| 타입 에러 | domain/entities 수정 또는 infrastructure에서 변환 | P0 |
| RLS 우회 | 즉시 수정, SECURITY DEFINER 금지 | P0 |
| UI 깨짐 | props/null 처리, 디자인 토큰 확인 | P1 |
| 회귀 | 원인 파악, 영향 범위 확인 후 수정 | P1 |
