# 이식 지향 목업 구현 원칙

> 프로덕션 이식을 전제로 한 HTML 목업 제작 원칙.
> 플랫폼 무관 (Swift, React, Flutter 등 어디든 이식 가능한 구조).

---

## 대원칙

```
목업을 만든다 → 프로덕션으로 이식한다
```

목업은 **빠르게 수정 가능한 설계도**다. 프로덕션 코드가 아니다.
단, 이식 시 **모든 의도가 누락 없이 전달**되어야 한다.

---

## Phase 1. 공통 기반 구축 — `shared.css`

모든 목업 파일이 공유하는 **변하지 않는 껍데기**만 분리한다.

### 포함 대상

| 요소 | 설명 |
|------|------|
| **디자인 토큰** | `:root` 변수 — 컬러, 타이포, 간격, 라운딩, 그림자 |
| **디바이스 프레임** | 목업 폰 외형, 내부 컨테이너, 다이나믹 아일랜드, 상태바 |
| **화면 전환 시스템** | screen wrapper, active/hidden/slide-out 트랜지션 |
| **스크롤 영역** | 공통 스크롤 컨테이너, 스크롤바 숨김 |
| **리셋** | `*`, `body` 기본 스타일 |

### 포함하지 않는 것

- 바텀 내비게이션 (페이지마다 활성 탭이 다름)
- 헤더 (페이지마다 구성이 다름)
- 페이지별 컴포넌트 전부

### 사용법

```html
<!-- 모든 목업 페이지 상단 -->
<link rel="stylesheet" href="shared.css">
```

> `shared.css`는 `file://` 프로토콜에서도 동작한다. 로컬 서버 불필요.

## Phase 2. 페이지 레이어 구성

목업 파일을 **접근 깊이**에 따라 레이어로 나눈다.

### Layer 1 — 상시 접근 페이지

홈, 네비게이션 바에서 **직접 진입**할 수 있는 페이지.

```
예시: (소셜 앱 기준)
  01_home.html
  02_explore.html
  03_community.html
  04_profile.html
```

**특징**:
- 앱의 뼈대. 이 파일들이 전체 구조를 결정한다
- 바텀 내비게이션 또는 탭으로 상호 전환
- **가장 먼저 만들고, 가장 자주 수정한다**

### Layer 2 — 진입점이 여러 곳인 페이지

Layer 1에서 **한 단계 더 들어가야** 도달하는 페이지.

```
예시: (소셜 앱 기준)
  05_detail_item.html          ← 홈, 탐색, 프로필에서 진입
  06_detail_user.html          ← 홈, 커뮤니티, 검색에서 진입
  07_settings.html             ← 프로필에서 진입
  08_create_flow.html          ← FAB, 상세에서 진입
  09_notifications.html        ← 헤더에서 진입
```

**특징**:
- 진입점이 다양하므로, **어디서 왔는지**에 따라 헤더/뒤로가기 동작이 달라질 수 있다
- 이를 목업에서 주석으로 명시한다:

```html
<!-- @entry: home(카드 탭), explore(검색결과 탭), profile(목록 탭), community(공유 링크 탭) -->
<!-- @back: 진입점으로 돌아감 (push 네비게이션) -->
```

### 제작 순서

```
Layer 1 전체 완성 → Layer 2 제작 (Layer 1의 진입점이 확정된 후)
```

## Phase 3. 네비게이션 원칙

### Layer 1 네비게이션

```
┌─────────────────────────────────────┐
│  모든 Layer 1 페이지는 어디서든     │
│  바텀 내비 또는 탭으로 접근 가능    │
└─────────────────────────────────────┘
```

- Layer 1 간 전환은 **탭 전환** (fade 또는 즉시 교체)
- 각 Layer 1 페이지는 독립된 스크롤 상태를 유지
- 바텀 내비는 Layer 1에서 항상 보인다

### Layer 2 네비게이션

```
┌─────────────────────────────────────┐
│  Layer 2는 반드시 "돌아갈 곳"이     │
│  명확해야 한다                      │
└─────────────────────────────────────┘
```

| 동작 | 설명 | 목업 주석 |
|------|------|----------|
| **Push** | 오른쪽에서 슬라이드 인 | `<!-- @nav: push -->` |
| **Modal** | 아래에서 올라옴, 전체/반 화면 | `<!-- @nav: modal -->` |
| **Sheet** | 바텀 시트, detent 지정 | `<!-- @nav: sheet, detents: 280px/50%/90% -->` |
| **Replace** | 현재 화면을 대체 (뒤로가기 없음) | `<!-- @nav: replace -->` |

### Layer 2 → Layer 2 전환

Layer 2에서 또 다른 Layer 2로 이동하는 경우도 명시:

```html
<!-- @nav-depth: detail → related-detail (push, 스택 쌓임) -->
<!-- @nav-depth: detail → create (modal, 스택 위에 오버레이) -->
```

### 바텀 내비 표시 규칙

```
Layer 1: 바텀 내비 보임
Layer 2 (push): 바텀 내비 숨김
Layer 2 (modal/sheet): 바텀 내비 숨김 (오버레이 아래에 깔림)
```

## Phase 4. 공통 컴포넌트 디자인 파일

페이지 목업을 만들기 전에, **자주 쓰이는 UI 요소를 별도 파일에 정의**해둔다.

### 파일 구성

```
prototype/
├── shared.css
├── 00_design_system.html     ← 공통 컴포넌트 카탈로그
├── 01_home.html
├── 02_discover.html
└── ...
```

### `00_design_system.html`에 들어가는 것

| 카테고리 | 요소 |
|---------|------|
| **버튼** | primary, secondary, ghost, danger / sm, md, lg |
| **입력** | text, search, textarea, select, date picker |
| **헤더** | Layer 1용 (로고+액션), Layer 2용 (뒤로가기+제목+액션) |
| **바텀 내비** | 탭 구성, 활성/비활성 상태 |
| **카드** | 엔티티별 카드 (도메인 주요 객체) |
| **리스트 아이템** | 기본, 썸네일 포함, 스와이프 액션 |
| **필터/소팅** | 필터 칩, 소팅 드롭다운, 탭 필터 |
| **바텀 시트** | detent별 상태, 헤더, 콘텐츠 영역 |
| **모달/다이얼로그** | 확인, 선택, 입력 |
| **피드백** | 토스트, 스낵바, 빈 상태, 에러 상태, 스켈레톤 |
| **뱃지/태그** | 상태 표시, 카테고리, 레벨 |
| **아바타** | 사이즈별, 뱃지 포함 |

### 사용 방법

```
실제 재사용(import)하지 않는다.
디자인 파일에서 필요한 요소를 찾아 → 복사 → 페이지 목업에 붙여넣기.
```

**이유**:
- JS import 없이 `file://`에서 동작
- 페이지마다 미세한 변형이 가능 (복사 후 수정)
- 디자인 파일이 **원본(source of truth)** 역할 → 이식 시 컴포넌트 목록으로 활용

**규약**:
- 디자인 파일의 각 요소에 명확한 이름을 부여한다
- 이 이름이 프로덕션 컴포넌트 이름이 된다

```html
<!-- === ItemCard === -->
<div class="item-card">...</div>

<!-- === FilterChipGroup === -->
<div class="filter-chip-group">...</div>

<!-- === BottomSheet === -->
<div class="bottom-sheet">...</div>
```

## Phase 5. 페이지별 목업 제작

### 파일 구조

```html
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>페이지명</title>
  <link rel="stylesheet" href="shared.css">
  <style>
    /* 이 페이지 전용 스타일만 */
  </style>
</head>
<body>
  <!-- @route: /home -->
  <!-- @layer: 1 -->
  <!-- @auth: required -->

  <div class="device">
    <div class="device-inner">
      <!-- 페이지 콘텐츠 -->
    </div>
  </div>

  <script>
    // 이 페이지 전용 인터랙션만 (화면 전환, 탭 등)
  </script>
</body>
</html>
```

### 이식용 주석 규약

목업의 수정 비용은 0이면서, 이식 시 누락을 방지하는 주석 태그:

| 태그 | 용도 | 예시 |
|------|------|------|
| `@route` | 이 페이지의 경로 | `<!-- @route: /item/:id -->` |
| `@layer` | 접근 깊이 | `<!-- @layer: 2 -->` |
| `@entry` | 진입할 수 있는 곳 | `<!-- @entry: home, explore, profile -->` |
| `@nav` | 전환 방식 | `<!-- @nav: push -->` |
| `@auth` | 인증 필요 여부 | `<!-- @auth: required -->` |
| `@state` | 화면 상태 분기 | `<!-- @state: empty → "아직 항목이 없어요" + CTA -->` |
| `@data` | 필요한 데이터 | `<!-- @data: Item(user_id=me, limit=20) -->` |
| `@interaction` | 제스처/인터랙션 | `<!-- @interaction: swipe-left → bookmark -->` |
| `@sheet` | 시트 상세 | `<!-- @sheet: detents 280/50%/90% -->` |
| `@component` | 공통 컴포넌트 표시 | `<!-- @component: ItemCard -->` |

**규칙**:
- 주석은 **관련 요소 바로 위**에 작성
- 목업 수정 시 주석은 무시해도 된다 (이식 직전에 검토)
- 이식 담당자(사람 또는 LLM)는 이 주석을 스펙으로 취급

### 상태 표현

모든 목록/데이터 화면에 대해, **정상 상태 외에도** 다음을 목업에 포함하거나 주석으로 명시:

```html
<!-- @state: loading → 스켈레톤 카드 3개 -->
<!-- @state: empty → "아직 항목이 없어요" 일러스트 + "추가하기" CTA -->
<!-- @state: error → "불러오지 못했어요" + "다시 시도" 버튼 -->

<!-- 가능하면 목업에서 토글로 확인 가능하게 만든다 -->
<!-- 최소한 정상 + 빈 상태는 목업으로 구현, 나머지는 주석 -->
```

## Phase 6. 백엔드 스키마 추론

목업이 완성된 후, **프로덕션 코딩 시작 전**에 백엔드 구조를 확정한다.

### 6.1 페이지별 데이터 요구사항 추출

각 목업 페이지에서 **화면에 표시되는 모든 데이터**를 수집한다:

```
01_home.html
  - 추천 항목 목록: name, category, rating, image, distance
  - 최근 활동: item_name, created_at, rating, memo, photo
  - 인기 항목: name, tag, score

02_detail_item.html
  - 항목 정보: name, category, price_range, address, phone, hours
  - 하위 목록: name, price, photo, type
  - 평가: score_1, score_2, score_3 (도메인별 정의)
  - 리뷰 목록: user, rating, memo, photos, created_at
```

### 6.2 엔티티 도출

페이지에서 수집한 데이터를 **엔티티(테이블)** 로 정규화:

```
수집된 필드들    →  어떤 엔티티에 속하는가?
중복된 필드들    →  같은 엔티티를 참조하는 것 (FK 관계)
계산된 값들      →  computed 표시 (DB View 또는 앱 계산)
사용자별 다른 값  →  RLS 정책 필요
```

### 6.3 연결성 검증

**핵심: 모든 페이지를 교차 확인한다.**

```
Q. 홈에서 보여주는 "평점"과 상세 페이지의 "평점"은 같은 필드인가?
Q. 프로필의 "활동 횟수"와 상세의 "활동 횟수"는 같은 계산인가?
Q. 커뮤니티에서 공유된 항목과 홈의 항목은 같은 엔티티인가?
```

한 엔티티의 필드가 여러 페이지에서 다르게 표현되면 → 정규화 오류 또는 DTO 분리 필요.

### 6.4 최종 스키마 정의 파일

별도 문서(MD 또는 SQL)로 다음을 확정:

```
1. 엔티티 목록 + 필드 + 타입 + 제약조건
2. 관계 (1:N, M:N, 자기참조)
3. 인덱스 (목록 페이지의 정렬/필터 기준)
4. RLS 정책 (누가 어떤 데이터를 보는가)
5. Computed 필드 (어디서 계산되는가 — DB vs 앱)
6. API 엔드포인트 목록 (각 페이지가 호출하는 것)
```

### 6.5 LLM 활용 프로세스

```
1. 모든 목업 HTML을 LLM에 제공
2. "각 페이지에서 표시/입력되는 데이터를 모두 추출해줘"
3. "이 데이터들을 정규화된 엔티티로 정리해줘"
4. "페이지 간 같은 데이터를 참조하는 곳을 찾아 관계를 정의해줘"
5. "누락된 필드, 모순되는 타입, 불명확한 관계를 지적해줘"
6. 사람이 검토 → 확정 → 마이그레이션 작성
```

## 전체 워크플로 요약

```
Phase 1                    Phase 2                Phase 3
shared.css 작성     →     페이지 레이어 구분  →   네비게이션 설계
(토큰 + 프레임)           (L1: 탭 / L2: 진입점)  (전환 방식 확정)
      │                        │                       │
      ▼                        ▼                       ▼
Phase 4                    Phase 5                Phase 6
디자인 파일 제작    →     페이지별 목업 제작  →   백엔드 스키마 추론
(공통 컴포넌트 카탈로그)  (조합 + 주석)           (LLM 교차 검증)
      │                        │                       │
      ▼                        ▼                       ▼
                    ┌──────────────────────┐
                    │  프로덕션 이식 시작   │
                    └──────────────────────┘
```

### Phase별 소요 비중

```
Phase 1 (shared.css):        5%   ← 한번 만들면 거의 안 건드림
Phase 2 (레이어 구분):       5%   ← 기획 단계에서 자연스럽게 결정
Phase 3 (네비게이션):       10%   ← L1 확정 후 L2 연결
Phase 4 (디자인 파일):      15%   ← 컴포넌트 정의 + 상태별 변형
Phase 5 (페이지 목업):      55%   ← 실제 작업의 대부분
Phase 6 (스키마 추론):      10%   ← 목업 완성 후 LLM과 교차 검증
```

## 금지 사항

| 금지 | 이유 |
|------|------|
| shared.css에 페이지별 스타일 넣기 | 공통 파일이 비대해지면 수정 공포 발생 |
| JS import로 컴포넌트 재사용 | 로컬 서버 필요 + 목업 복잡도 증가 |
| 목업에서 실제 API 호출 | 목업은 독립적으로 동작해야 함 |
| 프로덕션 프레임워크 사용 (React 등) | 목업 수정 진입 장벽 증가 |
| 주석 없이 인터랙션 구현 | JS로 동작은 하지만 의도가 전달 안 됨 |
| 정상 상태만 목업에 포함 | 빈 상태/에러를 이식 시 100% 누락 |
| 스키마를 목업 제작과 동시에 확정 | 화면이 바뀌면 스키마도 바뀜 → 이중 작업 |
